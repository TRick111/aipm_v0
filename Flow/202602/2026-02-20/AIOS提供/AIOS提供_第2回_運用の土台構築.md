## 3. Flow / Stock / Meetings：情報の3つの置き場所

### 3.1 なぜ「置き場所」が大切なのか

第1回でCursorを使って成果物を作成しましたが、そのファイルはどこに保存されていたでしょうか？ 成果物が増えてくると、「あのファイルどこだっけ？」「前に作ったものをAIに参照させたいけど、どうすれば？」という問題が出てきます。

AIOSでは、情報の「状態」に応じてファイルを置く場所を3つに分けることで、この問題を解決しています。AIは「今作業していること」と「参照すべき確定情報」を明確に区別できるようになります。

### 3.2 3つの領域

```
AIOS Root/
├── Flow/          ← 日々の作業場所（作業中・一時的）
├── Stock/         ← 確定版の保管場所（AIが参照する"正"）
└── Meetings/      ← 会議記録の集約場所（横串管理）
```

| 領域 | 役割 | たとえるなら | 具体例 |
|:---|:---|:---|:---|
| **Flow** | 日々の作業場所（WIP） | 机の上の作業スペース | ドラフト、メモ、途中の分析結果 |
| **Stock** | 確定版の保管場所 | 整理された書棚 | 完成した提案書、運用ルール、背景情報 |
| **Meetings** | 会議記録の集約 | 会議室の議事録ファイル | 議事録、文字起こし、決定事項 |

### 3.3 情報の流れ

AIOSの基本サイクルは、以下の流れで回ります。

```
① AIに作業を依頼する
    ↓
② AIがStockの過去情報を自動参照し、Flowに成果物を作成する
    ↓
③ ユーザーが確認・修正する
    ↓
④ 完成したら「確定反映」でStockに保存する
    ↓
⑤ 次回の作業時、AIがStockの情報を参照して作業する（②に戻る）
```

このサイクルを回すことで、AIは作業を重ねるごとに「あなたのビジネスの文脈」を深く理解していきます。

### 3.4 Flowの構造

Flowは **月 → 日付** の階層で自動整理されます。

```
Flow/
├── 202602/
│   ├── 2026-02-18/
│   │   ├── 0218_daily_tasks.md      ← 今日のタスク一覧
│   │   ├── プロジェクトA/            ← プロジェクトごとの作業フォルダ
│   │   │   └── draft_proposal.md
│   │   └── プロジェクトB/
│   │       └── analysis_memo.md
│   └── 2026-02-19/
│       └── ...
```

**ポイント**: 作業を始めるとき、AIは自動的に「今日のFlow」配下にプロジェクト名の作業フォルダを作ります。ユーザーがフォルダを手動で作る必要はありません。

### 3.5 Stockの構造

Stockは **プログラム → プロジェクト** の階層で整理されます（次のセクションで詳しく説明します）。

```
Stock/
├── MasterIndex.yaml          ← 全体の地図（AI専用）
├── プログラムA/
│   ├── README.md             ← プログラムの背景・目的
│   ├── プロジェクト1/
│   │   ├── README.md         ← プロジェクトの背景・目的・現状
│   │   ├── ProjectIndex.yaml ← プロジェクト内の地図（AI専用）
│   │   ├── log.md            ← 更新履歴
│   │   └── ...（成果物ファイル）
│   └── プロジェクト2/
│       └── ...
└── プログラムB/
    └── ...
```

### 3.6 Meetingsの構造

Meetingsは日付とタイトルでフラットに管理されます。

```
Meetings/
├── 2026-02-10_Ponさん打ち合わせ.md
├── 2026-02-15_週次定例.md
└── ...
```

会議記録は複数のプロジェクトにまたがることが多いため、Stockの外に「横串」で保管します。関連するプロジェクトの `README.md` には、会議へのリンクと要点だけを追記します。

## 4. プログラム / プロジェクト：情報の骨格を設計する

### 4.1 プログラムとプロジェクトの関係

Stockフォルダの中は、**プログラム**と**プロジェクト**という2つの階層で情報を整理します。

| 概念 | 説明 | 例 |
|:---|:---|:---|
| **プログラム** | 事業やテーマの大きな枠組み | 「J-Beauty事業」「Rio's Innovation運営」 |
| **プロジェクト** | プログラムに紐づく具体的な取り組み | 「ASEAN市場調査」「Instagram運用」「AIOS導入」 |

> **イメージ**: プログラムは「部署」、プロジェクトは「その部署が担当している案件」と考えると分かりやすいです。

### 4.2 AIが参照する4つの重要ファイル

各プロジェクトには、AIが情報を理解するための4つの重要ファイルがあります。

| ファイル | 役割 | 誰のため |
|:---|:---|:---|
| **MasterIndex.yaml** | 全プロジェクトの地図。AIが「どのプロジェクトを見るべきか」を判断する | AI向け |
| **README.md** | プロジェクトの背景・目的・現状・次のアクション | 人間 + AI向け |
| **ProjectIndex.yaml** | プロジェクト内のファイル一覧。AIが「どのファイルを見るべきか」を判断する | AI向け |
| **log.md** | 更新履歴（いつ・何を・なぜ更新したか） | AI向け |

**AIの参照の流れ**:

```
① MasterIndex.yaml を読む → 対象プロジェクトを特定
    ↓
② README.md を読む → プロジェクトの背景・現状を把握
    ↓
③ ProjectIndex.yaml を読む → 必要なファイルを特定
    ↓
④ 必要なファイルを読み込んで作業開始
```

この仕組みにより、AIは「Ponさんの事業の背景を踏まえた上で」作業を進めることができます。

**考え方のヒント**:

### 4.4 【演習2】「プロジェクト」を追加する（10分）

各プログラムの中で、現在進行中あるいは今後予定している「プロジェクト」を書き出してみましょう。

```markdown
## プログラム名：〇〇〇

### プロジェクト一覧

1. プロジェクト名：
   - 概要：
   - 現在の状況：
   - キーワード：

2. プロジェクト名：
   - 概要：
   - 現在の状況：
   - キーワード：
```

> この構造をAIに伝えることで、AIは「〇〇プロジェクトについてだね」と、人間のように文脈を理解して作業を進めることができるようになります。

### 4.5 AIにプログラム/プロジェクトを作成してもらう

設計した構造をAIに伝えて、実際にフォルダとファイルを作成してもらいましょう。

**依頼の例**:
```
@workspace

以下のプログラムとプロジェクトを初期化してください。

プログラム: J-Beauty事業
  概要: タイ・ASEAN市場向けのJ-Beauty製品の展開事業
  キーワード: J-Beauty, ASEAN, タイ, バンコク, 美容

  プロジェクト1: ASEAN市場調査
    概要: タイを中心としたASEAN市場のJ-Beauty需要調査
    キーワード: 市場調査, 競合分析, 需要予測

  プロジェクト2: Instagram運用
    概要: J-Beauty製品のInstagramアカウント運用

```

AIは、フォルダ構造の作成、`MasterIndex.yaml` への登録、各プロジェクトの `README.md` / `ProjectIndex.yaml` / `log.md` の生成を自動で行います。

## 5. 日々のタスクを管理する

### 5.1 日次タスクファイルとは

AIOSでは、毎日の作業を `Flow/YYYYMM/YYYY-MM-DD/mmdd_daily_tasks.md` というファイルで管理します。毎朝、AIに「今日やること」を伝えるだけで、このファイルが自動で生成・更新されます。

### 5.2 タスク作成の依頼方法

**依頼の例**:
```
@workspace

今日やること：
- ASEAN市場調査の競合リストを作成（J-Beauty事業/ASEAN市場調査）
- Instagram投稿案を3つ考える（J-Beauty事業/Instagram運用）
- 来週の打ち合わせアジェンダを準備

以上の内容で日次タスクを作成してください。
```

AIは以下を自動で行います。

1. 今日の日付フォルダ `Flow/202602/2026-02-20/` を作成
2. `0220_daily_tasks.md` を生成
3. 各タスクと関連プロジェクトを紐付け
4. 必要に応じて、関連プロジェクトの `README.md` を参照して背景情報を把握

### 5.3 タスク管理のコツ

| コツ | 説明 |
|:---|:---|
| **プロジェクト名を添える** | 「〇〇プロジェクトの△△」と伝えると、AIが過去情報を参照できます |
| **毎朝の習慣にする** | Cursorを開いて最初にやることを「今日のタスク作成」にしましょう |
| **完了したらチェック** | タスク完了時にAIに報告すると、ログに記録されます |

## 6. 実践：確定反映

### 6.1 確定反映とは

「確定反映」は、AIOSの中核をなす最も重要なプロセスです。**Flowで作成した成果物を、正式な資産としてStockに保存する**作業を指します。

このプロセスをAIに依頼すると、AIは単にファイルを移動するだけでなく、以下の作業を自動で行います。

| ステップ | AIが行うこと |
|:---|:---|
| 1. ファイルの保存 | 成果物を適切なプロジェクトフォルダ（Stock）に保存 |
| 2. ProjectIndex.yaml の更新 | 新しいファイルの情報（概要・キーワード）を登録 |
| 3. README.md の更新 | プロジェクトの現状・次のアクションを更新 |
| 4. log.md の更新 | 「いつ・何を・なぜ」更新したかを記録 |
| 5. MasterIndex.yaml の更新 | 必要に応じてプロジェクトの概要・キーワードを更新 |

これにより、成果物はただのファイルではなく、**未来のAIが参照できる文脈付きの情報資産**となります。

### 6.2 【演習3】確定反映を体験する（15分）

**ステップ1**: 簡単な作業をAIに依頼し、Flow内に成果物を作成します。

```
@workspace

「AIOSを導入するメリット」について、以下の3つの観点で整理してください。
- 情報の蓄積と再利用
- AIの文脈理解の向上
- 作業の効率化

memo.md として保存してください。
```

**ステップ2**: 作成されたファイルの内容を確認し、必要があれば修正を依頼します。

```
@workspace

2つ目の項目に「過去の会議記録も参照できる」という点を追加してください。
```

**ステップ3**: 完成したファイルを「確定反映」するようAIに依頼します。

```
@workspace

先ほど作成した memo.md を、「〇〇/△△」プロジェクトに確定反映してください。
```

**確認ポイント**:
- [ ] Stock内の該当プロジェクトフォルダにファイルが保存されたか
- [ ] `ProjectIndex.yaml` に新しいファイルが登録されたか
- [ ] `README.md` が更新されたか
- [ ] `log.md` に更新履歴が追記されたか

## 付録：よくある躓きポイントと対処法

第2回で特に注意すべき躓きポイントをまとめました。

| 躓きポイント | 症状 | 対処法 |
|:---|:---|:---|
| **Cursorを開く習慣がつかない** | つい従来のChatGPTを使ってしまう | 毎朝「日次タスク作成」をCursorで行うことを習慣にしましょう |
| **Flow→Stock確定反映を忘れる** | Flowに作業結果が溜まり、Stockが更新されない | 作業が一区切りついたら「確定反映して」と依頼する習慣をつけましょう |
| **プロジェクトを指定せずに作業してしまう** | AIが過去情報を参照せず、毎回ゼロから作成 | 「〇〇プロジェクトの△△を作って」と、プロジェクト名を添えましょう |
| **フォルダ構造が理解できない** | どこに何を置けばいいか分からない | 最初はAIに「〇〇を保存して」と任せてOK。慣れてきたら構造を意識しましょう |
