# ペルソナ別ダッシュボード要件定義

**作成日**: 2026-03-03
**目的**: 飲食店オーナー・コンサルタント・店長の3ペルソナについて、それぞれの立場・課題・必要情報を整理し、ダッシュボード設計の基礎とする

---

## 1. ペルソナ定義

### 1.1 飲食店オーナー

| 項目 | 内容 |
|------|------|
| **代表的な人物像** | 1〜5店舗を経営する個人オーナー。現場には毎日は入らない。月に数回の店舗巡回と数字チェックで経営判断を行う |
| **ITリテラシー** | 低〜中。スマホは使えるがPCは苦手。LINEが主要コミュニケーション手段 |
| **利用デバイス** | スマートフォン（メイン）、タブレット（時々） |
| **利用頻度** | 毎日1〜2回のチラ見（LINE通知経由）。週1回じっくり見る |
| **現場との距離** | 遠い。店長経由で情報を得ることが多い |
| **意思決定の範囲** | 資金配分、メニュー価格改定、販促施策のGo/No-Go、人員増減、店舗の出退店 |

#### 最大の関心事

1. **「儲かっているのか？」** — 売上だけでなく、利益が出ているか。FL比率は適正か
2. **「お客さんは来ているのか？ 満足しているのか？」** — 客数の増減トレンドと口コミの方向性
3. **「現場は回っているのか？」** — 店長の日報から現場の温度感を把握したい
4. **「何をすべきか？」** — 自分で分析するのではなく、結論と提案がほしい

#### 抱えている課題

- 数字を見ても「だから何をすればいいのか」がわからない
- POS分析のグラフを見る時間も気力もない。結論だけ教えてほしい
- 複数店舗の状況を一目で比較したい
- 日報はLINEで読めるのが便利だが、蓄積された情報の活用ができていない

#### ほしい情報（優先度順）

| 優先度 | 情報 | 形式 | 頻度 |
|--------|------|------|------|
| **P0** | 今月の売上 vs 目標（達成率%） | 大きい数字1つ + プログレスバー | 毎日 |
| **P0** | 今月の利益率（FL比率） | 数字 + 信号色（緑/黄/赤） | 毎日 |
| **P0** | 日報サマリー（AIが要約した今日の現場状況） | テキスト2-3行 | 毎日 |
| **P1** | 口コミの最新評価と傾向 | 星の数 + 直近の口コミ1件 | 毎日 |
| **P1** | 前週比・前月比の売上/客数/客単価 | 矢印（↑↓→）付き数字 | 週次 |
| **P1** | AIからの改善提案 | 箇条書き2-3件 | 週次 |
| **P2** | 店舗間比較（複数店舗の場合） | ランキング形式 | 週次 |
| **P2** | 月次経営レポート（PDF） | AI生成の報告書 | 月次 |

#### やりたくないこと

- CSVのアップロード
- 経費の手入力
- 複雑なフィルター操作
- PCを開くこと

### 1.2 飲食コンサルタント

| 項目 | 内容 |
|------|------|
| **代表的な人物像** | 竹矢氏（Chef's Room）のようなコンサル。複数クライアント（5〜20店舗）を担当。飲食業の現場知識と経営ノウハウを持つ |
| **ITリテラシー** | 中。Excel/スプレッドシートは使える。分析ツールの操作は得意ではないが、データの意味は理解できる |
| **利用デバイス** | PC（メイン）、タブレット（クライアント訪問時）、スマホ（移動中） |
| **利用頻度** | 週2-3回。クライアント訪問前の準備、報告書作成時に集中的に使う |
| **クライアントとの関係** | 月1-2回の訪問 + 週次のLINE/メールでの報告 |
| **意思決定の範囲** | クライアントへの改善提案、分析方針の決定、報告内容の取捨選択 |

#### 最大の関心事

1. **「クライアントに何を報告するか？」** — 次回訪問で話す内容のネタがほしい
2. **「数字に裏付けられた提案を出したい」** — 感覚ではなくデータドリブンなコンサルをしたい
3. **「報告資料の作成時間を減らしたい」** — レポート作成に毎週何時間もかけている
4. **「複数クライアントを効率的に管理したい」** — クライアント切り替えがスムーズにできること

#### 抱えている課題

- クライアントごとにデータのフォーマットが違う（POS種類、経費の管理方法など）
- 分析に時間がかかりすぎる。売上データから「示唆」を出すのに毎回苦労する
- レポート作成が手作業（PowerPoint / Google Slides）
- 口コミ情報は手動で食べログやGoogle Mapsを見に行っている
- クライアントに渡す資料のクオリティを上げたいが時間がない

#### ほしい情報（優先度順）

| 優先度 | 情報 | 形式 | 頻度 |
|--------|------|------|------|
| **P0** | クライアント横断ダッシュボード | 店舗カード一覧（売上/利益/口コミのサマリー） | 常時 |
| **P0** | AIが生成した週次分析レポート | 要因分解付き。コピペで報告書に使えるテキスト | 週次 |
| **P0** | 口コミの推移と内容分析 | トレンドグラフ + AI要約（強み/弱み/改善点） | 週次 |
| **P0** | クライアント向けレポート自動生成 | PDF/HTML。クライアントにそのまま渡せる品質 | 週次/月次 |
| **P1** | 売上の要因分解（客数×客単価、曜日別） | 現在の週報分析のフォーマット | 週次 |
| **P1** | PL推移と目標差異分析 | FL比率トレンド、経費カテゴリ別の増減 | 月次 |
| **P1** | 日報の横断サマリー（全スタッフの声の傾向） | AI要約。「今週多かったキーワード」的な分析 | 週次 |
| **P2** | 業態別ベンチマーク比較 | 自社クライアント vs 業界平均 | 月次 |
| **P2** | アクション管理（前回提案した施策の実施状況） | チェックリスト形式 | 訪問時 |

#### やりたくないこと

- 生データの加工（CSV→集計の手作業）
- 手動でのグラフ作成
- 複数サイトを巡回しての口コミ収集
- ゼロからのレポート作成

### 1.3 店長（現場管理者）

| 項目 | 内容 |
|------|------|
| **代表的な人物像** | 店舗の営業責任者。毎日現場に立つ。調理・接客・シフト管理・発注まで兼任することも多い |
| **ITリテラシー** | 低。スマホのLINEとInstagramは使える。PC操作は基本的にしない |
| **利用デバイス** | スマートフォン（ほぼ唯一） |
| **利用頻度** | 毎日。営業終了後に日報を書くのがメインの接点。日中は見る余裕がない |
| **オーナーとの関係** | 週1回程度の報告。日報経由で日次のコミュニケーション |
| **意思決定の範囲** | 日々のシフト調整、仕入れ発注、スタッフへの指示、当日の営業判断 |

#### 最大の関心事

1. **「今日の売上は目標に届いたか？」** — 閉店後にサッと確認したい
2. **「明日の準備は大丈夫か？」** — 予約状況、仕入れ、シフトの過不足
3. **「スタッフの状態は？」** — 誰が何を感じているか（日報で把握）
4. **「オーナーに何を報告すればいいか？」** — 日報をAIがまとめてくれるのが助かっている

#### 抱えている課題

- 営業終了後は疲れているので、日報は最小限の入力で済ませたい
- データを見る時間がない。見ても何をすればいいかわからない
- 仕入れは経験と勘で行っている。廃棄が出ることもある
- シフトの穴埋めに毎週苦労する
- 口コミは見るが、対応する時間がない

#### ほしい情報（優先度順）

| 優先度 | 情報 | 形式 | 頻度 |
|--------|------|------|------|
| **P0** | 今日の売上（目標との差） | 大きい数字1つ。閉店直後に見る | 毎日 |
| **P0** | AI日報（質問→深掘り→まとめ） | 対話形式（現在のUI） | 毎日 |
| **P1** | 明日の予約件数 | 数字1つ | 毎日 |
| **P1** | 今週の売上推移（日別の棒グラフ） | シンプルなグラフ | 毎日 |
| **P1** | 直近の口コミ（新着があれば通知） | LINEプッシュ通知 + 内容プレビュー | 随時 |
| **P2** | スタッフの日報一覧（自店舗） | カード形式で一覧 | 毎日 |
| **P2** | 翌日の推奨仕入れ量（将来機能） | 食材リスト形式 | 毎日 |
| **P2** | 今月のシフト充足率 | %表示 | 週次 |

#### やりたくないこと

- PCを使うこと
- 長い文章を読むこと
- 複雑な操作（タップ2回以内でほしい情報に辿り着きたい）
- データの入力作業（日報以外）

## 2. ペルソナ間の比較

### 2.1 情報ニーズの重なりと違い

```
                深い分析・レポート
                     ↑
                     |
                コンサル
                     |
                     |
    戦略的判断 ←──────+──────→ 日常オペレーション
                     |
              オーナー|     店長
                     |
                     |
                     ↓
              サマリー・数字1つ
```

| 観点 | オーナー | コンサル | 店長 |
|------|---------|---------|------|
| **見たい粒度** | サマリー（結論だけ） | 詳細分析（根拠付き） | 要点のみ（今日の数字） |
| **見る頻度** | 毎日チラ見 + 週1じっくり | 週2-3回（準備時に集中） | 毎日（閉店後） |
| **デバイス** | スマホ | PC > タブレット | スマホ |
| **求めるアウトプット** | 「何をすべきか」 | 「何が起きているか + なぜか」 | 「今日の結果 + 明日の準備」 |
| **データ入力** | しない | しない | 日報のみ |
| **レポート** | 受け取る側 | 作る側（自動化したい） | 書く側（日報） |

### 2.2 画面の使い分け

| 画面 | オーナー | コンサル | 店長 |
|------|---------|---------|------|
| 統合ホーム | **メイン画面** | クライアント選択後に概要確認 | △ あまり見ない |
| 売上トレンド | 週1でチェック | 詳細分析に使用 | 今日の数字だけ見る |
| PL管理 | 月次で確認 | 月次レポートの素材 | × 見ない |
| 日報 | AI要約をLINEで読む | 横断サマリーとして活用 | **毎日書く** |
| 口コミ | たまに見る | **週次分析に必須** | 新着通知で確認 |
| 週報/月報 | 受け取ってLINEで読む | **作成・編集の主担当** | △ あまり見ない |
| アクション提案 | **最重要** | 提案の素材 | △ |

## 3. ダッシュボード設計方針

### 3.1 共通原則

1. **Mobile First**: 全ペルソナがスマホを使う。PCはコンサルだけ
2. **情報の階層化**: 概要 → 詳細 → 生データの順にドリルダウン
3. **AIファースト**: 数字の羅列ではなく、AIが「だから何」を解釈して提示
4. **LINE連携**: 重要な情報はLINEプッシュで届ける。アプリを開かなくても概要がわかる
5. **入力は最小限**: データの入力者は店長（日報）のみ。他は自動化

### 3.2 ペルソナ別の画面構成
