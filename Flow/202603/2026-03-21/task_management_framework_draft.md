# タスク管理・振り返りフレームワーク（設計ドラフト）

## 1. 全体像

```
バックログ（Backlog.md）
  ↓ 朝の計画で「今日やる」を選別
日次タスク（daily_tasks.md）
  ↓ 日中に進捗を記録
  ↓ 終業時に振り返り
日次振り返り（daily_tasks.md 内）
  ↓ 週末にまとめて俯瞰
週次振り返り（weekly_review.md）
  ↓ 月末にまとめて俯瞰
月次振り返り（monthly_review.md）
```

すべての起点は **AIとの対話**。ユーザーが「朝の計画をしたい」「振り返りをしたい」と言えば、AIが対話を通じて必要な情報を引き出し、ファイルを生成・更新する。

---

## 2. 日次サイクル

### 2.1 朝の計画（トリガー：「今日の予定を立てたい」等）

**目的**: 今日1日で「やること」と「やらないこと」を明確にし、集中すべきことを決める。

**AIとの対話の流れ**:

1. AIがバックログ（Backlog.md）から以下を抽出して提示する
   - 期日が今日以前のタスク（overdue含む）
   - 優先度が高いタスク（P0/P1）
   - 前日の「明日への申し送り」があればそれも提示
2. ユーザーが「今日やりたいこと」を自由に話す（箇条書き、口頭メモ、何でもOK）
3. AIが以下を整理して提案する
   - **やること**リスト（優先順位付き、各タスクのゴール条件付き）
   - **やらないこと**リスト（意図的に今日は外すもの。理由も添える）
   - 各タスクの見積もり時間
4. ユーザーが確認・調整する
5. AIが `daily_tasks.md` を生成する

**対話で意識すること**:
- 「それは今日やらないとまずいですか？」とAIが問いかけて、やることを絞る手助けをする
- ゴール条件が曖昧なタスクは必ず具体化する（「〜の状態になっていること」）
- 見積もり時間の合計が現実的か（1日の稼働時間を超えていないか）をAIが確認する

**生成物**: `Flow/YYYYMM/YYYY-MM-DD/mmdd_daily_tasks.md`

---

### 2.2 日中の進捗記録（トリガー：「〇〇が終わった」「進捗を報告」等）

**目的**: タスクの途中経過を記録し、予定と実績のズレを早期に把握する。

**AIとの対話の流れ**:

1. ユーザーが進捗を報告する（「〇〇終わった」「△△に着手した」「□□はブロックされている」等）
2. AIが `daily_tasks.md` を更新する
   - タスクの状態を更新（`[ ]` → `[>]` → `[x]` 等）
   - 実績時間を記録（分かれば）
   - 途中経過ログに時刻付きでメモを追記
3. 必要に応じてAIが状況を整理する
   - 「残りの時間であと〇〇は厳しそうですが、優先度を見直しますか？」
   - 「ブロックされている件、誰かに確認が必要ですか？」

**対話で意識すること**:
- ユーザーの報告は最小限でOK（「BL-0001終わった」だけで十分）
- AIが状態遷移を正しく反映する
- 予定と実績のズレが大きくなりそうなら、AIが早めにアラートする

**生成物**: `daily_tasks.md` の更新（状態・実績・ログの追記）

---

### 2.3 終業時の振り返り（トリガー：「今日の振り返りをしたい」等）

**目的**: 1日の予実を比較し、学びを抽出して翌日以降に活かす。

**AIとの対話の流れ**:

1. AIが `daily_tasks.md` の現状を読み取り、予実サマリを提示する
   - 計画タスク数 / 完了数 / 見送り数
   - 見積合計 vs 実績合計
   - 未完了タスクの一覧
2. AIが以下を問いかける
   - 「今日うまくいったことは何ですか？」
   - 「困ったこと・想定外だったことはありますか？」
   - 「明日以降に持ち越すものはありますか？」
3. ユーザーの回答をもとに、AIが振り返りセクションを構成する
4. 未完了タスクについてAIが確認する
   - バックログに戻すか（期日変更）
   - 明日に持ち越すか
   - 取りやめるか
5. AIが `daily_tasks.md` の振り返りセクションを追記し、必要に応じてバックログを更新する

**対話で意識すること**:
- 自己批判ではなく「事実の確認」と「次への改善」に焦点を当てる
- 見積もりのズレが繰り返されるパターンがあれば、AIが指摘する
- 「やらないこと」に入れたものが正しい判断だったかも振り返る

**生成物**:
- `daily_tasks.md` への振り返りセクション追記
- バックログ（Backlog.md）の更新（ステータス変更・期日変更等）

## 3. 週次サイクル

### 3.1 週次振り返り（トリガー：「今週の振り返りをしたい」等）

**タイミング**: 週末（金曜夕方 or 日曜夜など、ユーザーの習慣に合わせる）

**目的**: 1週間を俯瞰し、日次では見えないパターンや傾向を把握する。翌週の方向性を決める。

**AIとの対話の流れ**:

1. AIがその週の `daily_tasks.md` を全て読み込み、以下を集計・提示する
   - **週間の予実サマリ**
     - 計画タスク総数 / 完了数 / 見送り数 / 持ち越し数
     - プロジェクト別の稼働割合
     - 見積もり精度（見積合計 vs 実績合計）
   - **完了した主な成果**（何をやり遂げたか）
   - **持ち越し・未完了タスクの一覧**
   - **日次KPTからの傾向抽出**（繰り返し出ているProblem等）
2. AIが以下を問いかける
   - 「今週、一番インパクトがあった仕事は何ですか？」
   - 「来週に向けて、進め方を変えたいことはありますか？」
   - 「バックログの優先度を見直したい項目はありますか？」
3. ユーザーの回答をもとに、AIが週次レビューを構成する
4. 翌週に向けた調整
   - バックログの優先度見直し
   - 翌週の重点テーマの設定（任意）

**対話で意識すること**:
- 「忙しかった」ではなく「何に時間を使ったか」を可視化する
- プロジェクト間の時間配分が意図通りだったかを確認する
- 繰り返し持ち越されるタスクは、そもそもやる必要があるのかを問い直す

**生成物**:
- `Flow/YYYYMM/YYYY-MM-DD/mmdd_weekly_review.md`（その週の最終日のフォルダに配置）
- バックログ（Backlog.md）の更新（優先度変更・ステータス整理）

### 週次レビューの構成案

```
# YYYY-MM-DD 週次レビュー（MM/DD〜MM/DD）

## 週間サマリ
- 計画: X件 / 完了: Y件 / 見送り: Z件 / 持ち越し: W件
- 見積合計: Xh / 実績合計: Yh / 精度: XX%

## プロジェクト別稼働
| プロジェクト | タスク数 | 実績時間 | 備考 |
|---|---|---|---|

## 今週の成果
- ...

## 持ち越し・未完了
- ...

## 傾向と気づき
- （日次KPTから抽出したパターン）

## 来週に向けて
- 重点テーマ:
- 進め方の改善:
- バックログ優先度の変更:
```

## 4. 月次サイクル

### 4.1 月次振り返り（トリガー：「今月の振り返りをしたい」等）

**タイミング**: 月末〜翌月初

**目的**: 月単位でプロジェクトの進捗と自分の働き方を評価し、中長期の方向性を確認・修正する。

**AIとの対話の流れ**:

1. AIがその月の週次レビュー・日次タスクを読み込み、以下を集計・提示する
   - **月間の予実サマリ**
     - プロジェクト別の稼働時間・完了タスク数
     - 月初の計画 vs 実績の比較（月初に立てたゴールがあれば）
   - **主な成果・マイルストーン到達**
   - **未達・持ち越しの傾向**
   - **見積もり精度の推移**（週ごとの変化）
2. AIが以下を問いかける
   - 「今月、一番大きな進展があったプロジェクトはどれですか？」
   - 「想定外に時間を取られたことはありますか？」
   - 「来月の重点テーマ・ゴールは何ですか？」
   - 「やめたい・減らしたい仕事はありますか？」
3. ユーザーの回答をもとに、AIが月次レビューを構成する
4. 来月に向けた調整
   - バックログの棚卸し（不要タスクの削除・優先度の大幅見直し）
   - 来月のゴール設定（任意）

**対話で意識すること**:
- 日次・週次よりも「戦略的な視点」を持つ
- 「このプロジェクトにこれだけ時間をかける価値はあるか？」を問う
- 定量データ（タスク数・時間）だけでなく、定性的な「手応え」も聞く

**生成物**:
- `Flow/YYYYMM/mmYY_monthly_review.md`（月フォルダ直下に配置）
- バックログ（Backlog.md）の棚卸し結果

### 月次レビューの構成案

```
# YYYY年MM月 月次レビュー

## 月間サマリ
- 稼働日数: X日
- 計画タスク総数: X / 完了: Y / 完了率: XX%
- 見積合計: Xh / 実績合計: Yh

## プロジェクト別レビュー
| プロジェクト | 完了タスク | 実績時間 | 進捗評価 | 備考 |
|---|---|---|---|---|

## 今月の成果
- ...

## 未達・課題
- ...

## 見積もり精度の推移
| 週 | 見積 | 実績 | 精度 |
|---|---|---|---|

## 振り返り
### うまくいったこと
- ...
### 改善したいこと
- ...

## 来月に向けて
- 重点ゴール:
- やめる・減らすこと:
- バックログ棚卸し結果:
```

## 5. タスクの管理フロー

### 5.1 設計方針：バックログ一元管理

**`Backlog.md` がすべてのタスクの唯一の正（Single Source of Truth）**。

- プロジェクト別のタスク一覧ファイルは持たない
- プロジェクトごとに見たいときは `Program` / `Project` 列でフィルタする（backlog_tools.py で抽出可能）
- 各プロジェクトの `README.md` にはネクストアクション（概要レベル）を書くが、個別タスクの管理は行わない
- タスクの規模が大きくなった場合（目安：100件超）は二層構造（バックログ＋プロジェクト別）への移行を検討する

### 5.2 タスクのライフサイクル

```
発生（思いつき・依頼・会議での宿題など）
  ↓
バックログに登録（Backlog.md に1行追加）
  ↓
朝の計画で「今日やる」に選別 → daily_tasks.md へ転記
  ↓
日中に実行・進捗記録（daily_tasks.md を更新）
  ↓
完了 → バックログも done に更新
  or
未完了 → 振り返りで判断（明日持ち越し / バックログに戻す / 取りやめ）
```

### 5.3 タスクの状態遷移（daily_tasks.md 内）

| 記号 | 状態 | 意味 |
|:---:|---|---|
| `[ ]` | 未着手 | 朝の計画で登録されたまま |
| `[>]` | 進行中 | 作業に着手した |
| `[x]` | 完了 | ゴール条件を満たした |
| `[!]` | ブロック | 外部要因で進められない |
| `[-]` | 見送り | 今日はやらないと判断した |

### 5.4 バックログとの連携（各サイクルでの役割）

| サイクル | バックログに対する操作 |
|---|---|
| **朝の計画** | バックログから今日やるタスクを選び、daily_tasks.md に転記。バックログ側のステータスを `doing` に変更 |
| **日次振り返り** | 完了タスクは `done` に更新。未完了は期日を変更 or ステータスを `todo` に戻す |
| **週次振り返り** | 全体の優先度を見直し。長期間 `todo` のまま放置されているタスクを棚卸し |
| **月次振り返り** | 大掃除。不要タスクの削除、優先度の大幅見直し、`done` の整理（アーカイブ or 削除） |

### 5.5 タスク登録のタイミング

バックログへの登録は以下のタイミングで行う：
- 会議で宿題が出たとき（AIが会議反映時に自動追加）
- 作業中に「あとでやる」が発生したとき（ユーザーがAIに伝える）
- 朝の計画時に新しいタスクが出てきたとき
- 週次・月次レビューで新たなアクションが決まったとき

### 5.4 日次タスクファイルの全体構成

```markdown
---
date: YYYY-MM-DD
type: daily_tasks
---

# YYYY-MM-DD - 日次タスク

## 今日のフォーカス
### やること
- （最も重要な1〜3項目）

### やらないこと（意図的に外したもの）
- （項目と理由）

## タスクリスト

| 状態 | タスク | ゴール | 対象 | 見積 | 実績 | メモ |
|:---:|--------|--------|------|:---:|:---:|------|
| [ ] | ... | ... | ... | 1h | - | |

## 途中経過ログ
- HH:MM｜タスク名｜進捗メモ

## 振り返り（終業時）
### 予実サマリ
- 計画: X件 / 完了: Y件 / 見送り: Z件
- 見積合計: Xh / 実績合計: Yh

### 今日の学び
- うまくいったこと:
- 困ったこと・想定外:
- 次に活かすこと:

### 明日への申し送り
- ...
```

## 7. AIの振る舞い指針

### 朝の計画時
- ユーザーが「あれもこれも」と詰め込みすぎていたら、「今日の稼働時間に対して見積もりが超過しています」と伝え、優先度で絞る提案をする
- 「やらないこと」を明示させることで、判断の負荷を減らす

### 日中の進捗報告時
- ユーザーの報告は最小限でOK。AIが文脈から状態を推定して更新する
- ズレが大きくなりそうなら早めにアラートする

### 振り返り時
- 責めるトーンではなく、「事実確認→気づき→次のアクション」の順で進める
- 数字（予実）は淡々と示し、解釈はユーザーに委ねる
- ただし、繰り返されるパターン（常に見積もりオーバーなど）はAIから指摘する

### 週次・月次レビュー時
- 日次の積み上げデータを自動集計し、ユーザーの手間を最小化する
- 「木を見て森を見ず」にならないよう、俯瞰的な問いかけをする
