# タスク管理・振り返りフレームワーク（設計ドラフト v2）

## 0. このフレームワークが解決する課題

1. **複数タスクの並行管理が苦手** → AIが提案し、ユーザーはYes/Noの意思決定だけ。決まったら上から順にシングルタスクで処理する
2. **個人目標に向けた日々のトラッキング** → 仕事タスクと並列で「目標ログ」を毎日記録し、週次・月次で傾向を振り返る

### 設計原則

- **ユーザーの判断コストを最小化する**：AIが選択肢を絞って提案し、ユーザーは承認・却下・調整だけ行う
- **シングルタスク・フォーカス**：「次にやること」は常に1つ。リストの上から順に消化する
- **仕事と個人目標を一体管理する**：別々のツールに分散させない

---

## 1. 全体像

```
バックログ（Backlog.md）─── すべてのタスクの唯一の正
  ↓ AIが朝に候補を提案 → ユーザーがYes/Noで決定
日次タスク（daily_tasks.md）─── 今日やることだけ。上から順にシングルタスク
  ↓ 日中に進捗を記録
  ↓ 終業時に振り返り + 目標ログ記入
日次振り返り（daily_tasks.md 内）
  ↓ 週末にまとめて俯瞰
週次振り返り（weekly_review.md）
  ↓ 月末にまとめて俯瞰
月次振り返り（monthly_review.md）
```

## 2. 個人目標の管理

### 2.1 目標の定義場所

`Stock/定型作業/バックログ/Goals.md` に中長期の個人目標を管理する。

```markdown
# 個人目標

## 現在の目標

### 目標1: 体重・体脂肪率を減らす
- 指標: 体重(kg)、体脂肪率(%)
- 現在値: （記録開始時に記入）
- 目標値: （決まれば記入。未定でもOK）
- 期限: （決まれば記入）
- 日次で記録すること: 体重、食事の概要、運動の有無・内容

### 目標2: ...
```

### 2.2 日次での目標トラッキング

daily_tasks.md 内に「目標ログ」セクションを設ける。
AIが振り返り時に「今日の目標に関する活動を教えてください」と聞き、記録する。

### 2.3 週次・月次での目標レビュー

- **週次**: その週の目標ログを集計し、傾向を可視化（例: 体重の推移、運動した日数）
- **月次**: 目標の進捗を評価し、目標値・アプローチの見直しを行う

## 3. 日次サイクル

### 3.1 朝の計画（トリガー：「今日の予定を立てたい」「朝の計画」等）

**目的**: 今日やることを「AIの提案 → ユーザーの意思決定」で決め、上から順に取り組める状態にする。

**核心：ユーザーは「何をやるか」を考えなくてよい。AIが提案し、ユーザーは選ぶだけ。**

**AIの動き**:

1. **情報収集**（AIが自動で行う。ユーザーの入力は不要）
   - バックログ（Backlog.md）を読み込む
   - 前日の daily_tasks.md があれば「明日への申し送り」を読む
   - 今日が期日のタスク、overdue のタスクを抽出する
   - 個人目標（Goals.md）を読み込む

2. **提案**（AIがまとめて提示する）
   - 「今日やるべきタスク候補」を優先度順にリスト化して提示
     - 各タスクに「なぜ今日やるべきか」の理由を1行添える
     - 見積もり時間も付ける
   - 見積もり合計が稼働時間を超える場合は、AIが「ここまでが現実的です」のラインを引く
   - 目標に関する今日のアクション提案（例:「昨日は運動されていましたが、今日はどうしますか？」）

3. **ユーザーの意思決定**（最小限の判断）
   - 各タスクに対して: **やる / やらない / 順番を変える** だけ
   - 新しいタスクがあれば追加（AIがバックログにも登録）
   - 目標アクションの確認

4. **daily_tasks.md 生成**
   - 決定した順番でタスクリストを生成
   - **1番目のタスクを「NOW（今やること）」として強調表示**

**対話の例**:
```
AI: おはようございます。今日の候補を整理しました。

  🔴 期日超過
  1. BL-0004: Ponさんの日程調整（15min）— 期日を過ぎています

  📋 優先度順
  2. BL-0007: ダッシュボード改のUI作成（2h）— P2、長期滞留中
  3. BL-0006: 竹矢さんのインスタリール作成（1.5h）— P2

  合計: 3h45min（稼働時間内に収まります）

  🎯 目標: 前回の記録から3日空いています。今日の体重記録と、運動予定はありますか？

  この内容で進めますか？変更があれば教えてください。

ユーザー: 1と3でお願い。2は来週でいい。あと運動は夜にジム行く予定。
AI: 了解です。（daily_tasks.md を生成）
```

**生成物**: `Flow/YYYYMM/YYYY-MM-DD/mmdd_daily_tasks.md`

---

### 3.2 日中の作業（シングルタスク・フォーカス）

**目的**: 「次にやること」を常に1つだけにし、迷わず作業に集中できる状態を維持する。

**運用ルール**:
- タスクリストの上から順番に1つずつ取り組む
- 今やっているタスクが終わったら、次のタスクに自動的に移る
- 途中で別のタスクをやりたくなっても、原則として順番を守る（本当に必要なら順番変更をAIに依頼）

**AIの動き（進捗報告を受けたとき）**:

1. ユーザーの報告を受けて `daily_tasks.md` を更新する
   - 状態を更新（`[ ]` → `[>]` → `[x]` 等）
   - 実績時間を記録
   - 途中経過ログに追記
2. 完了時は **次のタスクを提示する**
   - 「お疲れさまです。次は〇〇（見積もりXXmin）です。」
3. ブロックされたときは代替案を提示する
   - 「〇〇がブロックされているので、先に△△を進めますか？」

**対話の例**:
```
ユーザー: Ponさんの日程調整終わった。メールで候補日送った。
AI: 完了にしました（実績15min）。
    次は「竹矢さんのインスタリール作成」（見積1.5h）です。
```

**生成物**: `daily_tasks.md` の更新

---

### 3.3 終業時の振り返り（トリガー：「今日の振り返り」「終わり」等）

**目的**: 1日の予実を確認し、目標の記録を行い、翌日に繋げる。

**AIの動き**:

1. **予実サマリを自動算出して提示**（ユーザーは眺めるだけ）
   - 計画タスク数 / 完了数 / 見送り数
   - 見積合計 vs 実績合計
   - 未完了タスクの状況

2. **未完了タスクの処理**（1件ずつ聞く）
   - 「〇〇は明日に持ち越しますか？ それともバックログに戻しますか？」

3. **目標ログの記録**（AIが質問して引き出す）
   - 「今日の体重を記録しますか？」
   - 「運動はしましたか？ 内容を教えてください」
   - 「食事で気をつけたことはありますか？」

4. **短い振り返り**（1〜2問だけ）
   - 「今日、一番良かったことは？」
   - 「明日に持ち越したいことや気をつけたいことはありますか？」

5. **daily_tasks.md に振り返りセクションを追記**、バックログを更新

**対話で意識すること**:
- 責めない。事実を確認して、次に繋げるだけ
- 質問は最小限（疲れている時間帯なので長々聞かない）
- 見積もりのズレが繰り返されるパターンがあれば、AIから軽く指摘する

**生成物**:
- `daily_tasks.md` への振り返りセクション追記
- バックログ（Backlog.md）の更新

## 4. 週次サイクル

### 4.1 週次振り返り（トリガー：「今週の振り返り」等）

**タイミング**: 週末（金曜夕方 or 日曜夜）

**目的**: 1週間を俯瞰し、「何に時間を使ったか」「目標は進んでいるか」を把握する。

**AIの動き**:

1. **データ集計**（AIが自動で行う）
   - その週の daily_tasks.md をすべて読み込み
   - Goals.md と各日の目標ログを読み込み
   - 以下を算出して提示:

   **仕事面**:
   - 計画タスク総数 / 完了数 / 見送り数 / 持ち越し数
   - プロジェクト別の稼働割合
   - 見積もり精度（見積合計 vs 実績合計）
   - 完了した主な成果
   - 繰り返し持ち越されているタスク

   **目標面**:
   - 目標に関する週間データ（例: 体重の推移グラフ的な表示、運動した日数/内容）
   - 前週との比較

2. **AIからの問いかけ**（3問以内）
   - 「今週、一番インパクトがあった仕事は何ですか？」
   - 「来週、進め方を変えたいことはありますか？」
   - 「目標に対して、来週意識したいことはありますか？」

3. **バックログの棚卸し**
   - 長期間 `todo` のまま放置されているタスクをリストアップ
   - 「これはまだ必要ですか？」と1件ずつ確認
   - 優先度の見直し提案

**生成物**:
- `Flow/YYYYMM/YYYY-MM-DD/mmdd_weekly_review.md`
- バックログ（Backlog.md）の更新

### 週次レビューの構成

```markdown
# YYYY-MM-DD 週次レビュー（MM/DD〜MM/DD）

## 仕事サマリ
- 計画: X件 / 完了: Y件 / 見送り: Z件 / 持ち越し: W件
- 見積合計: Xh / 実績合計: Yh / 精度: XX%

## プロジェクト別稼働
| プロジェクト | タスク数 | 実績時間 | 備考 |
|---|---|---|---|

## 今週の成果
- ...

## 持ち越し・未完了
- ...

## 目標トラッキング
### 目標1: （目標名）
- 今週のデータ:
  | 日付 | 体重 | 運動 | メモ |
  |---|---|---|---|
- 前週比:
- 気づき:

## 傾向と気づき
- ...

## 来週に向けて
- 仕事の重点:
- 目標のアクション:
- バックログの変更:
```

## 5. 月次サイクル

### 5.1 月次振り返り（トリガー：「今月の振り返り」等）

**タイミング**: 月末〜翌月初

**目的**: 月単位で「仕事の成果」と「目標の進捗」を評価し、方向性を確認・修正する。

**AIの動き**:

1. **データ集計**（週次レビューを積み上げて自動算出）

   **仕事面**:
   - プロジェクト別の稼働時間・完了タスク数
   - 見積もり精度の推移（週ごと）
   - 月初のゴール vs 実績（ゴールがあれば）
   - 主な成果・マイルストーン

   **目標面**:
   - 月間の目標データ推移（例: 体重の月間グラフ、運動頻度）
   - 目標に対する進捗評価
   - 前月との比較

2. **AIからの問いかけ**（戦略的な視点）
   - 「今月、一番大きな進展があったプロジェクトはどれですか？」
   - 「想定外に時間を取られたことはありますか？」
   - 「来月の重点ゴールは何ですか？」
   - 「やめたい・減らしたい仕事はありますか？」
   - 「目標のアプローチを変えたいですか？（例: 運動の頻度、食事の方針）」

3. **バックログの大掃除**
   - `done` の整理（アーカイブ or 削除）
   - 不要タスクの削除提案
   - 優先度の大幅見直し

4. **目標の見直し**
   - 目標値やアプローチの調整
   - 新しい目標の追加 / 達成した目標のクローズ

**生成物**:
- `Flow/YYYYMM/mmYY_monthly_review.md`
- バックログ（Backlog.md）の棚卸し結果
- Goals.md の更新（必要に応じて）

## 6. タスクの管理フロー

### 6.1 設計方針：バックログ一元管理

**`Backlog.md` がすべてのタスクの唯一の正（Single Source of Truth）**。

- プロジェクト別のタスク一覧ファイルは持たない
- プロジェクトごとに見たいときは `Program` / `Project` 列でフィルタする（backlog_tools.py で抽出可能）
- 各プロジェクトの `README.md` にはネクストアクション（概要レベル）を書くが、個別タスクの管理は行わない
- タスクの規模が大きくなった場合（目安：100件超）は二層構造への移行を検討する

### 6.2 タスクのライフサイクル

```
発生（思いつき・依頼・会議での宿題など）
  ↓
バックログに登録（Backlog.md に1行追加）
  ↓
朝の計画でAIが候補提案 → ユーザーがYes/Noで選別 → daily_tasks.md へ転記
  ↓
日中にシングルタスクで上から順に実行・進捗記録
  ↓
完了 → バックログも done に更新
  or
未完了 → 振り返りで判断（明日持ち越し / バックログに戻す / 取りやめ）
```

### 6.3 タスクの状態遷移（daily_tasks.md 内）

| 記号 | 状態 | 意味 |
|:---:|---|---|
| `[ ]` | 未着手 | 朝の計画で登録されたまま |
| `[>]` | 進行中 | 作業に着手した（NOW） |
| `[x]` | 完了 | ゴール条件を満たした |
| `[!]` | ブロック | 外部要因で進められない |
| `[-]` | 見送り | 今日はやらないと判断した |

### 6.4 バックログとの連携（各サイクルでの役割）

| サイクル | バックログに対する操作 |
|---|---|
| **朝の計画** | AIが候補を提案 → 選ばれたタスクを daily_tasks.md に転記、バックログを `doing` に変更 |
| **日次振り返り** | 完了は `done` に更新。未完了は期日変更 or `todo` に戻す |
| **週次振り返り** | 全体の優先度見直し。長期滞留タスクの棚卸し |
| **月次振り返り** | 大掃除。不要タスク削除、優先度大幅見直し、`done` 整理 |

### 6.5 タスク登録のタイミング

- 会議で宿題が出たとき（AIが会議反映時に自動追加）
- 作業中に「あとでやる」が発生したとき（ユーザーがAIに伝える）
- 朝の計画時に新しいタスクが出てきたとき
- 週次・月次レビューで新たなアクションが決まったとき

## 9. AIの振る舞い指針

### 最重要原則：ユーザーの判断コストを最小化する

- ユーザーに「何をやるか考えて」と丸投げしない。AIが提案し、ユーザーは承認・却下・調整するだけ
- 選択肢を出すときは推薦を添える（「この中では〇〇を先にやるのがよさそうです。理由は…」）
- 質問は1回にまとめる。何度も聞き返さない

### 朝の計画時
- AIが主導でタスク候補を優先度順に並べて提示する
- 見積もり合計が稼働時間を超えていたら、現実的なラインを引いて「ここまでが今日の分です」と示す
- 「やらないこと」もAIが提案する（「これは今週中で大丈夫なので、今日は外しましょう」）

### 日中の進捗報告時
- 完了報告を受けたら、次のタスクを1つ提示する（「次は〇〇です」）
- ユーザーが迷っている素振りを見せたら、「まず〇〇から手をつけましょう」と1つに絞る
- ブロック時は代替タスクを提示する

### 振り返り時
- 責めない。事実を確認して、次に繋げるだけ
- 終業時は疲れているので、質問は最小限にする
- 繰り返されるパターン（常に見積もりオーバー等）は軽く指摘する
- 目標ログは忘れていたらAIから声をかける

### 週次・月次レビュー時
- データ集計はAIが全自動で行い、ユーザーは結果を見て考えるだけ
- 俯瞰的な問いかけをする（「このプロジェクトに時間をかけすぎていませんか？」）
- 目標の推移を可視化し、傾向を伝える（「今週は運動が0日でした」等、事実のみ）
