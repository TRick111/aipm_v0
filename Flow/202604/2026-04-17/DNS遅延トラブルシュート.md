## 原因の仮説（確度順）

### 仮説1: ルーターDNSプロキシのキャッシュ消失 + 二重NAT遅延（確度: 高）

**メカニズム:**
1. Mac → ルーター(192.168.11.1)へDNSクエリ送信
2. ルーターのDNSプロキシがキャッシュを持っていない場合、上流の1.1.1.1に転送
3. この転送が**集合住宅ゲートウェイ(192.168.6.241)**を経由する二重NATを通る
4. ルーターのDNSキャッシュは容量が小さく（機種依存、通常数百件）、TTL切れで頻繁に消失
5. 「インターネット＠スタート」モードは接続方式の自動判別のため、DNSクエリに対してフォールバック処理が入る場合がある

**なぜ「初回だけ遅い」のか:**
- ルーターのDNSキャッシュにヒットしない初回はフル解決が必要
- 二重NATの往復 + ルーターDNSプロキシの処理オーバーヘッドが加わる
- 2回目はルーターキャッシュにヒットするため高速

**なぜMacで1.1.1.1直指定すると改善するのか:**
- ルーターのDNSプロキシを完全にバイパス
- 1.1.1.1はエニーキャストで地理的に近いサーバーから応答するため、二重NATを通っても高速

### 仮説2: IPv6 DNS解決のフォールバック遅延（確度: 中〜高）

**メカニズム:**
1. macOSはデフォルトでIPv6を優先してDNS解決を試みる
2. ルーターのIPv6は「インターネット＠スタート」で設定されているが、ステータス画面では**接続状態: 無効**
3. IPv6 DNSクエリ（AAAAレコード）がルーターで処理できず、タイムアウトまで待つ
4. タイムアウト後にIPv4にフォールバックして解決 → この待ち時間が「初回の遅さ」

**補強材料:**
- ステータス画面: IPv6接続状態=無効、グローバルアドレス=設定されていません
- IPv6アドレス配布状態=停止中
- macOSは積極的にIPv6を使おうとするため、応答不能な状態が遅延を生む

### 仮説3: 「インターネット＠スタート」の自動判別オーバーヘッド（確度: 中）

**メカニズム:**
- 「インターネット＠スタート」は接続方式を自動判別するBuffalo独自機能
- 集合住宅の上流ネットワーク構成が変わっている（二重NAT）ため、判別処理に時間がかかる場合がある
- 特にDHCPリース更新やDNSプロキシ再起動時に影響が出る可能性

## 現在の暫定対策（適用済み）

MacのDNS設定を手動で 1.1.1.1 / 1.0.0.1 に変更 → 改善を確認

---

## 推奨する恒久対策

### 対策A: ルーターのDHCP配布DNSを変更（推奨度: ★★★）

ルーター管理画面 → LAN → DHCPサーバー設定 [拡張設定] → DNSサーバーの通知:
- 現在: 「エアステーションのLAN側IPアドレス (192.168.11.1)」
- 変更先: 「指定したIPアドレス」→ プライマリー: 1.1.1.1 / セカンダリー: 1.0.0.1

**効果:** 全デバイスがルーターDNSプロキシをバイパスし、直接Cloudflare DNSに問い合わせる。
Mac以外のスマホやタブレットも自動的に改善される。

### 対策B: IPv6を無効化する（推奨度: ★★）

ルーター管理画面 → Internet → IPv6:
- 現在: 「インターネット＠スタートを行う」
- 変更先: 「IPv6を使用しない」

**効果:** IPv6のフォールバック遅延が解消される。
ただし、将来IPv6が集合住宅で有効になった場合に再設定が必要。

### 対策C: WAN接続方式を明示指定（推奨度: ★）

ルーター管理画面 → Internet:
- 現在: 「インターネット＠スタートを行う」
- 変更先: 「DHCPサーバーからIPアドレスを自動取得」

**効果:** 自動判別のオーバーヘッドを排除。
集合住宅の上流がDHCPであることはWAN IP (192.168.6.114) から確認済み。
