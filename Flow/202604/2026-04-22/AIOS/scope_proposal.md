# BL-0053 AIOSアップデート スコープ提案（3案）

- 作成日: 2026-04-22
- 前提: `discovery_notes.md` §1〜5 を読んだ上での提案
- 目的: 「アップデート」の範囲を **ユーザーが A/B/C から1つ選ぶだけ** で決められる状態にする
- 共通前提（どの案でも発生する作業）:
  - (P) **G-Brain実体の特定**（大地さんヒアリング or 共有依頼）※ G-Brainのタクソノミー (α)〜(ε) を確定
  - (Q) 新旧ルールの併存期間ポリシー（過去の Flow/Stock は変更しない＝新規分だけ新ルール適用、が既定）

---

## 案A — 最小スコープ（G-Brain統合の "型" だけ定義）

### 中身
1. G-Brain実体の特定（共通前提 P）
2. AIOSに **「外部共有ライブラリとの連携ルール」を1本だけ追記**（新規 ops 1本 or 00_core に1章追加）
   - (α)ルール集 / (β)コード / (γ)ナレッジ / (δ)ツール の4分類ごとに、AIOSからの **参照方法** を規定
   - 統合方式は原則「参照」（コピーせず外部パスを記載）、例外のみコピー
3. BL-0061で発生した **「PONさん案件方式（aipm_v0メタ + 実体は別リポ）」を非公式のまま放置**（ルール化しない）
4. 既存ルール（00〜12）の本文は **触らない**

### 出力
- 新ルール1本（例: `ops/13_external_library_integration.mdc`、2〜3KB想定）
- `00_aios_core.mdc` の運用フロー表に1行追加
- Backlog に BL-0054 / BL-0063 への影響メモを追記

### メリット
- **田中さん単独で着手可能な部分が多い**（大地さんヒアリングは最小限）
- 1〜2日で完了可能。BL-0054 のブロック解除が早い
- 既存ルールを壊さないので、過去ファイルに migration 不要

### デメリット
- 03/12 の責務重複・テンプレート配置などの **既存の歪みは残る**
- マルチリポ運用の型が "暫定" のまま → BL-0054 で結局詰め直しが必要
- G-Brain が (β) コードライブラリだと AIOS 側で出来ることが少ない（BL-0054 で吸収される）

### 所要時間（見積）
- G-Brain特定ヒアリング: 30min（大地さん対話）
- 新ルール起草＋PR: 2〜3h
- **合計: 半日〜1日**

### 向くケース
- 「まず BL-0054 を動かしたい」「G-Brain の詳細を待たずに前進したい」
- 今週末〜来週にかけて他案件（BL-0061 実装 / BL-0037 / BL-0062）が重い場合

## 案B — 中間スコープ（G-Brain統合 + AIOSルール整合性リファクタ）

### 中身
1. 案Aの全部
2. **ルール整合性のリファクタ**:
   - 03_daily_task_planning と 12_parallel_task_orchestration の **責務境界を明確化**（どちらに何を書くか）
   - `questions_to_user.md` 仕様の重複を一箇所に統合（12側に寄せる）
   - 05_finalize_to_stock と 06_meeting_close を **統合**（または役割をクリアに分離）
   - 09/10/11 レビュー系のテンプレート化（ルール文を短く）
3. **templates/ の整理**（サブディレクトリ化: `init/`, `daily/`, `project/`）
4. **マルチリポ運用の最小ルール**（PONさん案件方式を ops/14 として正式化）
5. 既存ルールに触るため **Changelog と migration note** をルートに配置

### 出力
- 新規 ops: 13（外部ライブラリ連携）/ 14（マルチリポ運用）の2本
- 既存 ops の書き直し: 03 / 05 / 06 / 09 / 10 / 11（約6本）
- templates のサブディレクトリ化
- `CHANGELOG.mdc`（ルール変更の履歴管理）

### メリット
- **G-Brain統合と同時に既存の歪みも解消**できる
- BL-0054 / BL-0063 が乗るマルチリポ運用の土台が整う
- 今後のルール追加時の "どこに書くべきか" 問題が減る

### デメリット
- 既存ルール改修のため、**エージェント挙動の変化が広範囲に波及**
  - 既存進行中タスク（BL-0061 実装 / BL-0037 要件詰め / BL-0062 要件詰め）のエージェントが新旧ルールの境目で混乱する可能性
  - リファクタ直後のデバッグ工数が読めない
- 大地さん合流が案Aより必須（マルチリポ運用の (β) 方向の決定に必要）

### 所要時間（見積）
- discovery拡張（責務重複の棚卸し詳細化）: 2h
- G-Brain特定＋大地さんすり合わせ: 1h
- 新ルール起草＋既存改訂＋templates整理: 6〜8h
- エージェント試運転（dry-run）: 2h
- **合計: 2〜3日**

### 向くケース
- **最近の並行タスク運用で "エージェントが迷う瞬間" を感じている** なら、この案が丁度良い
- BL-0054 を本格設計する前に AIOS 側を整えたい
- AIOS触るのは今回限り／次回は半年後、の心づもりで一気にやりたい

## 案C — 最大スコープ（AIOS再設計）

### 中身
1. 案A+Bの全部
2. **AIOSの "再定義"**:
   - 「単一リポ前提」から「**メタリポ（aipm_v0）+ 実体リポ群**」という多層モデルに設計変更
   - Program/Project/Task の用語を再検討（AI-Core / RestaurantAILab / 個人資産 の3層が混在している現状）
   - Index（MasterIndex / ProjectIndex）の **複数リポ対応スキーマ** への拡張（外部リポURL、同期方針）
3. **AI協働モデルの明文化**:
   - マスター（チャット）／要件詰めエージェント／実装エージェント／レビュワーの役割定義
   - フェーズ遷移（要件詰め → waiting_confirmation → 実装 → Stock反映）の標準化
4. **外部システム統合の抽象化**: G-Brain / OMI / Tachyon / Notion / Cursor を同じ抽象で扱える "統合ポート" の定義
5. 既存 Flow / Stock の **一括マイグレーション**（過去分も新ルールに合わせて整備）
6. AIOSの **バージョン番号**（例: `AIOS v2.0`）を導入し、以後の進化を管理

### 出力
- `00_aios_core.mdc` の全面書き直し（alwaysApplyなので慎重に）
- `01_aios_architecture.mdc`（メタ層/実体層の概念図）新設
- ops 全面改訂（12本 → 再編して10本前後）
- templates 全面再構築
- MasterIndex スキーマ v2（破壊的変更）＋ migration script（Python）
- `AIOS.VERSION` ファイル / CHANGELOG

### メリット
- **長期的な拡張性が最大**（OMI / Tachyon / G-Brain 等の外部統合が次々乗る想定に合う）
- マルチリポ + エージェント協働 + 外部統合を **一つの抽象で扱える**
- 後続の BL（BL-0054 / BL-0063 / BL-0049 等）の設計制約が大きく減る

### デメリット
- **所要時間が読みづらい**。最低でも数日、延びると1週間超
- 既存エージェントとの互換性が破壊される期間が発生 → その間 **他タスクが一時停止** するリスク
- 大地さんだけでなく、町田第一さん・他メンバーも巻き込む可能性（関係者合意が必要）
- MasterIndex 破壊的変更のため migration script のバグが致命的

### 所要時間（見積）
- 再設計ドラフト: 1〜2日（机上）
- 関係者合意: 半日
- 実装（ルール書き換え+マイグレーション）: 2〜3日
- 既存エージェントの再テスト: 1日
- **合計: 1週間〜1.5週間**

### 向くケース
- 「AIOS はこの先 1年以上使い倒す前提」で、**今週まとまった時間を投資してでも一気に土台を固めたい**
- OMI / Tachyon / G-Brain を同時期に全部入れる計画がある
- 田中さんに他重タスクが無く、集中的に投資できる週がある

## 推奨（AI提案）

- **第1推奨: 案B（中間スコープ）**
  - 理由1: 並行タスク運用（12_parallel_task_orchestration）が直近追加されたばかりで、**03との責務重複が既に実害（エージェントの質問一次受け場所の混乱）を生みかねない** 状態。早めに解消したほうが将来の手戻りが小さい
  - 理由2: BL-0054（共用リポ設計）が doing 状態で止まっているため、AIOS 側の「マルチリポ運用の型」を先に確定させる必要がある。案Aでは BL-0054 の設計制約が不十分
  - 理由3: 案Cは "今後AIOSを1年以上使う前提" のコミットメントが必要。現時点でそのコミットをするより、**案Bで整えた結果を1〜2ヶ月運用してから案Cを判断** する方が安全
- 第2推奨: 案A（G-Brainの実体判明が遅れそうなら、先に案Aで着地）
- 案C は **半日以上のまとまった時間を投資できる日** を確保できる場合のみ
