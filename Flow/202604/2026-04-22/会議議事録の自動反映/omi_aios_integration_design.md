## Problem Statement

OMI AI wearable を BL-0023 で受領済み。今日時点で録音・文字起こしの AIOS 連携は手動で未整備（BL-0026 も未着手、BL-0063 が今回のテーマ）。欲しい体験は受動的な議事録生成ではなく、日常的に OMI を装着している中で「思いついた瞬間の指示」や「これやっといて」という口頭依頼を、AIOS（aipm_v0 の Flow / Stock / Meetings）の文脈を理解したエージェントが実行し、成果物として残す状態。現状、この連鎖が何もないので、音声由来のタスク起動はゼロ、OMI の価値が「単なるライフログ機」に閉じている。

## What Makes This Cool

- 口頭で独り言のように投げた思いつきが、本当に成果物になって Flow に現れる
- AIOS の文脈（MasterIndex / ProjectIndex / Backlog）を持った Claude が実行するので、単なる transcript 起動ではなく「今進行中のプロジェクトを踏まえた」タスク実行になる
- OMI marketplace の既存 250 アプリは "transcript → Notion/Slack へ push" 型が主流。田中さんの用途は "voice front-end for an agent orchestrator" であり発想自体が異なる。
- Cockpit と OMI を繋ぐ層を持つことで、将来 OpenClaw ルート（外出先・Macスリープ中でも動く）への拡張口が自然に開く

## Constraints

- 個人用ツール。田中さん1人が使う想定。他ユーザー展開は今回スコープ外
- aipm_v0 は vault backup 運用。git remote 連携を前提にしない
- OMI からの webhook は OMI Cloud 発・HTTPS 必須・インターネット越し
- 田中さんの Mac はNATの内側。公開エンドポイントとしてTailscale Funnel / cloudflared 等が必要
- Cockpit / Claude Code / aipm_v0 は田中さんの Mac に既に存在。追加インフラは極小に
- 開発・検証・運用すべて1人でやるため、複雑な監視・多重冗長化は不要

## Premises

1. **[CORE VALUE]** OMI はパッシブな議事録生成器ではなく、voice-first の agent dispatcher の入力デバイス。BL-0026（議事録自動反映）は副産物として含まれる。
2. **[TRIGGER]** MVP は明示トリガー句（例:「エージェント」「OMIへ」等）で開始。誤検知回避のため。精度見えたら Phase 2 で LLM 自動判定に段階拡張（今回は Approach A の範囲外）。
3. **[EXEC TARGET]** 実行先は Cockpit サブタスク spawn で統一（MVP）。Mac 上の omi-bridge が `./cockpit task create ...` を subprocess exec する形。MVPでは dispatcher は単一関数（`dispatch_to_cockpit(prompt)`）。将来の OpenClaw / claude CLI 追加時にインターフェースを切り出す方針（先にクラス抽象化しない）。
4. **[CONTEXT INJECTION]** タスク起動時、プロンプトに AIOS 文脈（Approach A では `~/aipm_v0/Stock/MasterIndex.yaml` 全文 ≒ 13.8KB をベタ注入）を埋め込む。これが無いと、ただの Whisper+Cockpit ラッパーに堕ちる。動的 resolver は Approach B 以降。
5. **[WEBHOOK SOURCE]** OMI 公式 webhook の **memory-triggered** を採用（会話終了時に発火）。real-time transcript webhook は Phase 2。ファームウェア改造・独自STTは作らない。
6. **[OUTPUT PLACEMENT]** MVPでは成果物は常に `Flow/YYYYMM/YYYY-MM-DD/omi_inbox/` に固定着地。Project 推定ロジックは Phase 2 送り（夜の整理で田中さんが振り分ける）。
7. **[SCOPE]** 個人用ツール。プロダクト化・他ユーザー提供は考えない。「自宅Macで動くローカルwebhookサーバー + Cockpit」のみ。
8. **[IDEMPOTENCY]** 同一 `memory_id` の webhook は初回のみ処理。以降は 200 OK だけ返す（OMI Cloud のリトライ対策）。処理済み memory_id はローカルファイル（例: `~/.omi-bridge/processed.jsonl`）に追記。
9. **[AUTH]** MVP 認証は `X-OMI-Secret: <env 値>` ヘッダの shared secret 検証に固定。OMI 側が HMAC 署名対応していることが判明したら Phase 2 で切替。
10. **[LANGUAGE]** omi-bridge は Python 3.11 + FastAPI で実装（AIOSエコシステム他と分離した独立プロセス、保守容易性と subprocess/tempfile 周りの標準ライブラリ充実）。

## Approaches Considered

### Approach A: Minimum Shot（採用）
memory-triggered webhook のみ。固定トリガー句で検知し、AIOS文脈は MasterIndex 全文をベタ注入して `./cockpit task create` へ subprocess exec。

### Approach B: Phased Hybrid
A の上に real-time webhook + Haiku による intent 判定 + 動的 AIOS 文脈 resolver + Dispatcher 抽象化を段階投入。

### Approach C: Custom Tool First
OMI Custom Tool 機能に `aios_task` を登録し明示的に呼び出し。双方向UX。OMI Custom Tool API 成熟度が未確認のため MVP 選択肢からは外した。

## Recommended Approach

**Approach A: Minimum Shot を採用。**

- 2〜3時間で「喋ったら成果物が出た」を体験できる最短経路
- 実装が単純なので、故障時の切り分けが容易
- 田中さんが earlier で選んだ「トリガー句で始めて段階拡張」と整合
- Week2+ で B 要素（real-time, LLM判定, 動的resolver）、Week4+ で C 要素（Custom Tool 双方向 UX）を差し込む拡張路線を保つ

### 重要な実装詳細（レビュー反映）

- **subprocess 呼び出し具体形** — `--directory` 含む完全形:
  ```python
  # src/dispatcher.py
  def dispatch_to_cockpit(full_instruction: str, name: str) -> dict:
      """full_instruction には prompt_template.py で組み立てた最終文字列を渡す。"""
      result = subprocess.run(
          [
              "./cockpit", "task", "create",
              "--instruction", full_instruction,
              "--directory",   "/Users/rikutanaka/aipm_v0",
              "--agent-type",  "claude",
              "--name",        name,  # "OMI: <先頭30文字>"
          ],
          cwd="/Users/rikutanaka/.agi-tools/data/cockpit/master",
          capture_output=True, text=True, shell=False, timeout=30,
      )
      if result.returncode != 0:
          raise DispatchError(f"cockpit exit={result.returncode} stderr={result.stderr}")
      return json.loads(result.stdout)  # {"ok": true, "data": {...taskId...}}
  ```
- **プロンプト組立（`src/prompt.py`）** — 成果物が必ず `omi_inbox/` に着地することを**プロンプト側で明示する**:
  ```python
  PROMPT_TEMPLATE = """\
  あなたは AIPM (aipm_v0) 配下で動くエージェントです。
  OMI から届いた口頭指示を実行してください。

  ## 必ず守るルール
  1. 作業開始前に .cursor/rules/aios/00_aios_core.mdc を読むこと
  2. 成果物は必ず `Flow/{year_month}/{date}/omi_inbox/` 配下に保存すること
     （プロジェクトは推定せず、必ず omi_inbox/ 固定）
  3. ファイル名は内容が分かる snake_case、拡張子は .md を優先
  4. 生成後、作成したファイルのフルパスを最終出力に含めること

  ## AIOS 全体インデックス (MasterIndex.yaml、読み取り専用)
  ```yaml
  {master_index_content}
  ```

  ## ユーザーからの口頭指示
  {stripped_instruction}
  """
  ```
  Python 側で `{year_month}/{date}/{master_index_content}/{stripped_instruction}` を埋めて `dispatch_to_cockpit` に渡す。
- **idempotency の書込順序** — **dispatch 成功後に記録** (at-least-once 配達方針):
  ```
  1. memory_id が processed.jsonl に存在するか読取
  2. 存在すれば 200 OK を即返して終了（何もしない）
  3. 存在しなければ dispatch_to_cockpit() 実行
  4. 成功したら processed.jsonl に append し 200 OK
  5. 失敗したら processed.jsonl には書かず 500 を返す
     (OMI Cloud が retry すればもう一度試行される)
  ```
  書込時は `fcntl.flock()` で file lock（FastAPI async 並行対策）。Python `asyncio.Lock` でも可だがプロセス再起動時の安全側に倒して flock。
- **subprocess 引数長**: MasterIndex は 13.8KB、ARG_MAX (macOS: 1MB) の 1.3%。`subprocess.run(list, shell=False)` でリスト渡しすれば引用符エスケープも不要で安全。Week 0 step 1 で実機検証（万一 Cockpit が拒否した場合は stdin 渡し or tempfile+`--instruction-file` 追加を設計変更として提案）。
- **`--name` 生成**: `"OMI: " + トリガー句除去後の指示本文の先頭30文字`。例: 発話「エージェント、健康管理プロジェクトの今週の要約を作って」→ name「OMI: 健康管理プロジェクトの今週の要約を作」
- **トリガー句未検出時の挙動**: 200 OK を返し `~/.omi-bridge/ignored.log` に transcript 先頭50文字を debug レベルで記録。Cockpit タスクは作成しない。
- **トリガー句 regex 決定**: Week0 step 2 で OMI ダッシュボードから過去メモリの transcript を1件コピーしてサンプル化 → 句読点・表記揺れを見た上で `trigger.py` の regex を確定。サンプル化前に実装するとほぼ確実に誤判定。
- **ログのトークン保護**: `X-OMI-Secret` の値を含むヘッダは **stdout/stderr に出力しない**。ログに出す際は `Authorization: [REDACTED]` でマスク。
- **launchd 常駐は Week2 から**（Week1 は `make start` でターミナル前景起動、使用頻度検証後に launchd 化）。下記 plist は参考構成。
- **launchd plist** （`~/Library/LaunchAgents/com.rikutanaka.omi-bridge.plist`）— **`.env` はラッパスクリプトで読む**、spin-loop 回避:
  ```xml
  <?xml version="1.0" encoding="UTF-8"?>
  <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" ...>
  <plist version="1.0">
  <dict>
    <key>Label</key><string>com.rikutanaka.omi-bridge</string>
    <!-- ラッパシェル経由で .env をロード（launchd は .env を読まないため） -->
    <key>ProgramArguments</key>
    <array>
      <string>/bin/bash</string>
      <string>/Users/rikutanaka/code/omi-bridge/scripts/launch.sh</string>
    </array>
    <key>WorkingDirectory</key><string>/Users/rikutanaka/code/omi-bridge</string>
    <key>RunAtLoad</key><true/>
    <!-- spin-loop 回避: 正常終了(exit 0)以外のみ再起動、連続クラッシュ時はスロットル -->
    <key>KeepAlive</key>
    <dict><key>SuccessfulExit</key><false/></dict>
    <key>ThrottleInterval</key><integer>30</integer>
    <key>StandardOutPath</key><string>/Users/rikutanaka/.omi-bridge/stdout.log</string>
    <key>StandardErrorPath</key><string>/Users/rikutanaka/.omi-bridge/stderr.log</string>
  </dict>
  </plist>
  ```

  対応する `scripts/launch.sh`:
  ```bash
  #!/bin/bash
  set -euo pipefail
  cd "$(dirname "$0")/.."
  # .env を export して読み込み
  set -a
  [ -f .env ] && source .env
  set +a
  exec .venv/bin/python -m uvicorn src.main:app --host 127.0.0.1 --port 8787
  ```
- **Tailscale Funnel ポート制約**: Funnel が対応するのは 443/8443/10000。omi-bridge が :8787 で listen し、`tailscale funnel 443 http://localhost:8787` でマップ。OMI には `https://rikus-mac-mini-3.tailad7d87.ts.net/omi/webhook` を登録（Week 0 実測で tailnet DNS 名確定）。
- **Tailscale CLI パス（Week 0 発見）**: Tailscale は App Store 版で `tailscale` コマンドは PATH に無い。Makefile や launch.sh 内では絶対パス `/Applications/Tailscale.app/Contents/MacOS/Tailscale` を使うか、以下で symlink 作成:
  ```bash
  sudo ln -sf /Applications/Tailscale.app/Contents/MacOS/Tailscale /usr/local/bin/tailscale
  ```
- **tailnet クリーンアップ**: `rikus-mac-mini`（182日offline）が tailnet に残っている。Funnel URL 曖昧性排除のため Tailscale 管理画面で削除推奨（Week 0 副次発見）。

## Open Questions

1. **Tailscale Funnel か cloudflared か**: 田中さんが Tailscale を既に使っているなら Funnel。未使用なら cloudflared（Cloudflare アカウント1個で完結）。Week1 の最初に決定。
2. **「ついつい口が滑って起動」対策**: MVPはトリガー句一択なので誤検知は理論ゼロ。ただし「エージェントが〜」のような非意図発話でも反応してしまう可能性はある。使ってから判断（confirm phase の追加は Phase 2）。
3. **失敗時の OMI 側フィードバック**: omi-bridge が Cockpit 起動失敗した時、OMI 側に音声で返せないか（Custom Tool 経由なら可能）。Approach A では ログ記録のみ、Cockpit UI で気付く前提。
4. **BL-0026（議事録自動反映）との関係**: トリガー句なし発話を Meetings/ に溜める経路は Approach A では実装しない。BL-0026 として別タスクで設計。
5. **トークンコスト観測**: MasterIndex 13.8KB × 1日N回の呼び出しコスト見積もり。Claude Haiku/Sonnet/Opus どれが走るかで桁が変わる。Week 1 に実測して Approach B 拡張の判断材料にする。
6. **OMI memory 境界の精度**: 会話途中で切れた場合、トリガー句を含むが指示本体が途切れた発話の扱い。最初は「検知したがタスクとして不完全」に見える短文は 100文字未満なら reject する簡易ガードを入れる。
