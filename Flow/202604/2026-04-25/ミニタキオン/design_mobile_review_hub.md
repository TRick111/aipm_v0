## 位置付け

本プロジェクトは [Tachyon Workspace](~/tachyon-workspace) の「ミニ版」。
Tachyon が「会議駆動でのAI開発支援」という大きな仕組みを狙うのに対し、ミニタキオンは **個人の日次 AIPM 運用におけるAI成果物レビュー→次指示ループ** に特化したモバイル1画面ツール。作業効率化プログラムに属する。

## Problem Statement

田中さんの日々のタスク管理は AIPM (`/Users/rikutanaka/aipm_v0/Flow/YYYYMM/YYYY-MM-DD/`) のマークダウン運用 (INBOX.md / STATUS.md / daily_tasks.md) と AGI Cockpit のAIエージェント指示という2系統に分かれている。PC作業時は問題ないが、**モバイルから操作する場面（「結構多い」）で動線が破綻する**:

1. AIが出した markdown 成果物を iPhone で見たいが、Mac上の markserv は iPhone では開けない
2. 成果物を見ながら音声入力で「ここが違う」と指摘したいが、見るツール（markserv）と指示を出すツール（Cockpit）が別なので、音声入力しながら成果物を見る UX が成立しない
3. 複数プロジェクトを横断した「今日レビューすべきもの」の一覧が見えない
4. タスクの状態が Cockpit の3状態（running / waiting_confirmation / completed）だけでは「AIが手詰まり」「私のサブタスク」「私の作業後AI再開」といった細かい状態が表現できず、「今誰のターンか」が判らない

## What Makes This Cool

- **Tailscale × Cockpit CLI の組み合わせは、iPhone が Mac の内線端末になるのと同義**。認証もVPN設定もいらない。iPhone Safari から `http://rikus-mac-mini-3.tailad7d87.ts.net:3000` を開くだけ
- Cockpit が既に Electron アプリ + グローバル CLI として存在するので、webapp 側は CLI を叩くだけの薄いシェルでよい。実装は shadcn + Bun + Hono で 3-500 行程度
- 音声入力は iOS キーボードのマイクボタンを使うだけで完結。Whisper 等を自前で動かす必要なし
- **「インライン指摘（このパラグラフがおかしい）→音声で数十秒話す→送信→Cockpit タスクに次指示として流れる」までが1画面・数秒**で済むようになる

## Constraints

- AGI Cockpit 本体はインストール済み Electron アプリ、ソースには触れない。周辺ツールから CLI を叩く構成に限る
- AIPM のファイル構造は既存フィードバック (INBOX.md はBL ID順のプロジェクト別、状態更新は3ファイル同期) を壊さない
- 最低限のセキュリティは tailscale の VPN 内に閉じることで確保。外部公開はしない
- Mac mini (`rikus-mac-mini-3`) が常時起動している前提。これが落ちた時は諦める
- 自分だけが使う前提。マルチユーザー対応は考えない

## Premises

1. コアの未解決課題はモバイル動線（レビュー→音声コメント→次指示）。PCの動線は壊れていない — **合意済**
2. 成果物は AIPM `Flow/YYYYMMDD/<Project>/` のマークダウン。タスク状態は Cockpit。この二分は変えない — **合意済**
3. AGI Cockpit 本体は closed binary。CLI (`cockpit`) 経由での統合のみ — **合意済（PATH内に確認済み: `/Users/rikutanaka/.agi-tools/data/cockpit/master/bin/cockpit`）**
4. 必要な状態は5つ: 🟢作業中 / 👀レビュー待ち / ❓手詰まり / 🔧私の作業 / 🔄AIに戻す。Cockpit 3状態に手動で layer を乗せる — **合意済**

## Approaches Considered

### Approach A: Mobile Web Wrapper (Mac-hosted, Tailscale経由) — 採用
Mac 上で小さな webapp を立て、iPhone Safari から tailscale ホスト名で開く。Cockpit CLI と AIPM ファイルシステムを薄くラップする。音声入力は iOS キーボード純正のマイクボタン。

### Approach B: Notion をフロントエンド + Cockpit 同期デーモン
Notion database をタスク台帳にし、コメント投稿を Mac の同期デーモンがポーリングして `cockpit task send` に流す。却下理由: **async sync のエッジケース (削除・衝突・順序) が lake ではなく ocean 化する**。コメント→指示の遅延も秒単位出る。結局 bridge を書く必要があり、A より楽ではない。

### Approach C: Telegram Bot (メッセージング = UI)
Bot が朝ダイジェスト送信、音声返信を Whisper で文字起こしして `cockpit task send`。却下理由: 通知と音声は神だが、**マークダウン成果物をインラインで指摘する UX が成立しない**（長文スクロール地獄、段落参照不可）。本丸の痛点を解かない。

## Recommended Approach

**Approach A: Mobile Web Wrapper** を基本とし、将来的に Telegram Bot (C) を push 通知チャネルとして後乗せする。

### 主要画面

**`/today`（トップ）**
- 当日の INBOX.md をパース + `cockpit task list --json` をマージ
- プロジェクト別セクション（既存のBL ID順を維持）、各タスクに5状態 badge
- 状態フィルタタブ: `👀レビュー待ち` `❓手詰まり` `🔧私の作業` で絞り込み
- タスクタップで `/task/:id` へ

**`/task/:id`**
- 上半分 (60vh): 成果物 markdown レンダリング (`react-markdown` + GFM + シンタックスハイライト)
- 下半分 (40vh): 指示入力 textarea（ iOS キーボードのマイクで音声入力）
- 下部固定バー: 5状態トグル / 送信ボタン / `cockpit task complete` ボタン
- 送信時: `cockpit task send <id> --text <content>` を spawn。成功したら状態を `🟢作業中` に自動切替
- 過去のやり取りは下に縦スクロールで閲覧（`cockpit task get <id> --turns 10` で取得）

**`/projects/:slug`**
- プロジェクト単体ビュー (AIPM プロジェクトID単位)
- 成果物ファイル一覧 (Flow 配下のmd全部)、最新順

### 状態レイヤ設計

Cockpit の `running / waiting_confirmation / completed` を基底にし、sqlite で上書き状態を管理:

| ユーザー状態 | 自動判定ルール | 手動切替 |
|---|---|---|
| 🟢 作業中 | `running` | — |
| 👀 レビュー待ち | `completed` または `waiting_confirmation` （かつユーザー未確認）| デフォルト |
| ❓ 手詰まり | — | ユーザーが「AIが何か聞いてきた」と判断したら手動セット |
| 🔧 私の作業 | — | 完全に手動 |
| 🔄 AIに戻す | — | 完全に手動、送信で自動で🟢に遷移 |

自動判定より「手動で切り替え一発」を優先。精度より速度。

### スマホの音声入力フロー

1. `/task/:id` を開く（成果物が上に見える）
2. 下部 textarea をタップ → iOS キーボード出現
3. キーボードのマイクボタン（🎤）をタップ → 音声認識開始
4. 「このセクションの2番目の箇条書きが事実誤認。正しくは○○」と話す
5. textarea に文字起こしされる、微修正
6. 送信ボタン → `cockpit task send` が走り、Cockpit 側のタスクが `running` に遷移
7. webapp は自動で🟢に更新、次の通知を待つ

### 実装構成 (具体案)

- Runtime: Bun
- Framework: Hono (API) + 静的 HTML/JS（SPA不要、MPAで十分、初動速いほうが良い）
- UI: shadcn/ui の primitives を手動コピー（サーバーは Mac 内なので `cdn.jsdelivr` 依存でも可）
- Markdown: `marked` + `highlight.js`（react 不要）
- State: sqlite (`bun:sqlite`) でローカル保持
- Cockpit 連携: `Bun.spawn(["cockpit", ...])` で JSON レスポンスをパース
- 起動: launchd で常駐、Mac 起動時に自動で 3000 で立ち上がる
- Tailscale: 既設定 (`rikus-mac-mini-3.tailad7d87.ts.net`) をそのまま使用。iPhone は同じ tailnet にログイン済み想定

## Open Questions

1. **音声入力の文字起こし精度**: iOS キーボード純正で十分か、Whisper で後処理するか → 最初は純正のみ、不満が出たら Whisper 追加
2. **タスクIDと AIPM BL ID の紐付け**: Cockpit タスクには自動採番ID、AIPM には BL-XXXX。手動マッピングか、タスク作成時にBL IDを名前に含めるルールにするか → 運用ルールで「タスク名の先頭に BL-XXXX をつける」を推奨、コードは名前からパース
3. **外出中 Mac がスリープした時**: `caffeinate` で常駐するか、諦めるか → 当面諦める。スリープ中は Cockpit 自体が動かない
4. **通知**: push 通知がないとレビュー漏れが発生する → 初期リリースでは Cockpit のデスクトップ通知に依存。後から Telegram bot (Approach C) を push 通知だけに限定して追加

## Success Criteria

- 既存の PC 動線（markserv + Cockpit デスクトップ）は何も壊さない
- 2週末（計 4 日）以内に minimum viable が動く

## What I noticed about how you think

- **痛点の自己言語化が速い**。最初の質問で「AがBの組み合わせ」と即答できた。これは稀で、自分の workflow を内省的に観察しているからこそ出る回答。
- **「マークサーフ」「音声で普段入力」など、既にある程度の運用試行錯誤を経た跡**が見える。つまりこの設計は、机上ではなく実地の不満から来ている。これが builder として一番強い起点。
- **「Notion でもしかしたらいけるのかも」と自分で build vs buy を提起している**。完全自作バイアスも Notion 原理主義もなく、手段を目的化しない判断軸がある。この設計が決まるのが速かったのはここが原因。
- 3ファイル同期のフィードバック、BL ID プロジェクト別のフィードバックを既にメモリに刻んでいる。つまり**運用ルールを言語化して sync で回す**スタンスがある。今回の mobile hub もこの延長線に自然に収まる。
