## 位置付け

[Tachyon Workspace](~/tachyon-workspace) の「ミニ版」。
Tachyon が会議駆動でのAI開発支援という大きな仕組みを狙うのに対し、ミニタキオンは **個人の日次 AIPM 運用におけるAI成果物レビュー→次指示ループ** に特化したモバイル1画面ツール。

**重要: 当初スコープから拡張済**。INBOX.md / daily_tasks.md を廃止し、その責務を全てミニタキオンが引き受ける構成に決定 (2026-04-25 セッション内合意)。

## Problem Statement

田中さんの日々のタスク管理は AIPM (`~/aipm_v0/Flow/YYYYMM/YYYY-MM-DD/`) のマークダウン運用 (INBOX.md / STATUS.md / daily_tasks.md) と AGI Cockpit のAIエージェント指示という2系統に分かれている。PC作業時は問題ないが、**モバイルから操作する場面で動線が破綻する**:

1. AIが出した markdown 成果物を iPhone で見たいが、Mac上の markserv は iPhone では開けない
2. 成果物を見ながら音声入力で「ここが違う」と指摘したいが、見るツール（markserv）と指示を出すツール（Cockpit）が別なので、音声入力しながら成果物を見る UX が成立しない
3. 複数プロジェクトを横断した「今日レビューすべきもの」「私の番のもの」の一覧が見えない
4. タスクの状態が Cockpit の3状態（running / waiting_confirmation / completed）だけでは「AIが手詰まり」「私のサブタスク」「私の作業後AI再開」といった細かい状態が表現できず、「今誰のターンか」が判らない
5. **(発展課題)** AIPM の3ファイル同期ルール (STATUS.md / INBOX.md / daily_tasks.md) は人間とエージェント双方の認知負荷が高く、マークダウンを散文ベースで管理しているため構造化データとして扱いづらい

## What Makes This Cool

- **Tailscale × Cockpit CLI の組み合わせは、iPhone が Mac の内線端末になる**。認証もVPN設定もいらない、Safari で内線ホスト名を開くだけ
- Cockpit が既にグローバル CLI として `/Users/rikutanaka/.agi-tools/data/cockpit/master/bin/cockpit` で PATH 上にあり、ローカル webapp が subprocess で呼べる
- 音声入力は iOS キーボードのマイクボタンで完結。Whisper 等を自前で動かす必要なし
- **AIPM の3ファイル散文管理を、構造化データ (YAML) + UIベースのインタラクションに置き換える** — 人間にもエージェントにも扱いやすい統一モデルへ
- **「インライン指摘 → 音声で数十秒話す → 送信 → Cockpit タスクに次指示として流れる」までが1画面・数秒**で済む

## Constraints

- AGI Cockpit 本体はインストール済み Electron アプリ、ソースには触れない。CLI 経由での統合のみ
- Mac mini (`rikus-mac-mini-3.tailad7d87.ts.net`) が常時起動している前提
- 自分だけが使う前提。マルチユーザー対応は考えない
- 既存の AIPM 運用 (Flow/Stock の二層、BL-XXXX 命名) を継承
- 既存の Flow/INBOX.md データは過去ログ扱いで移行不要
- マークダウン以外の成果物 (画像、コード差分、Figma) は対象外。HTML は外部検証サーバーへの link out のみ

## Premises (確認済)

1. コアの未解決課題はモバイル動線（レビュー→音声コメント→次指示）。PCは既存ツールで足りる
2. AGI Cockpit 本体は closed binary、CLI 経由統合のみ
3. ミニタキオン は INBOX.md / daily_tasks.md の責務を完全に引き受け、新規スコープとなる
4. AIPM の Flow/Stock の二層構造を継承、BL-XXXX を引き続き正としつつ、その実体ファイルを ミニタキオン が管理

## Approaches Considered

### Approach A: Mobile Web Wrapper (Mac-hosted, Tailscale経由) — 採用
Mac 上で webapp を立て、iPhone Safari から tailscale ホスト名で開く。Cockpit CLI と AIPM ファイルシステムを薄くラップする。

### Approach B: Notion をフロントエンド + 同期デーモン
**却下**: async sync のエッジケース (削除・衝突・順序) が ocean 化、コメント→指示の遅延が秒単位、bridge を書く必要があり A より楽でない。

### Approach C: Telegram Bot
**却下**: 通知と音声は強いが、マークダウン成果物のインライン指摘 UX が成立しない。本丸の痛点を解かない。ただし将来 Approach A に push 通知チャネルとして後乗せ予定。

## Recommended Approach

**Approach A**。将来 Approach C を通知レイヤとして追加。

### 同期戦略

#### Flow ↔ Stock 同期 (STATUS.md のみ)

```
朝 (lazy):
  ・最初の操作 (webapp起動 or agent着手) で:
    - Flow/<today>/<project>/STATUS.md が無ければ Stock からコピー
    - Flow/<today>/<project>/deliverables.yaml が無ければ空で初期化
  ・前日 Flow が「未マージ」の場合 UI に警告バッジ

日中:
  ・エージェントが Flow の STATUS.md を更新 (タスク開始時/終了時)

夜:
  ・田中さんが手動で「日次マージ」ボタン → Flow の STATUS.md を Stock にマージ
  ・03:00 cron が走り、未マージなら自動マージ (セーフティネット)
  ・Flow STATUS.md はその日のスナップショットとして保持

マージロジック:
  - 履歴セクション: append (重複なし)
  - 完了済セクション: union
  - 次アクション/ブロッカー/概要: Flow の最新内容で上書き
  - frontmatter updated_at: max(stock, flow)
```

#### BL ファイル (同期なし、Stock 直接編集)

BL の Q&A は本質的に時系列累積、Flow 中間レイヤを挟まない。ミニタキオン が単一書き込み口になり、書き込みをシリアライズして競合回避。

```
ミニタキオン API: PUT /bl/:id
  1. ファイルロック (~/aipm_v0/Stock/.../<bl-id>.yaml.lock)
  2. ファイル読込
  3. 該当フィールドを更新
  4. atomic write (tmp → rename)
  5. ロック解放
```

#### deliverables.yaml (Flow のみ、Stock に上がらない)

成果物メタデータは「その日の出来事」として Flow に固定。Cross-day query は ミニタキオンが起動時に全 Flow をスキャンして in-memory index を構築。

## エージェント更新の信頼性メカニズム

「エージェントが必ず STATUS.md / BL / deliverables.yaml を更新する」を担保する仕組み:

### 1. Cockpit タスクテンプレート

```bash
./cockpit templates create \
  --name project_task \
  --instruction "$(cat <<'EOF'
プロジェクト: {project}
BL: {bl_id}

## このタスク開始時にやること
1. ~/aipm_v0/Flow/{today}/{project}/STATUS.md を読む
2. STATUS.md frontmatter の current_bl を {bl_id} に更新
3. STATUS.md「履歴」セクションに「{now} タスク開始 (cockpit-task-{cockpit_task_id})」を append

## このタスクの作業内容
{task_content}

## このタスク完了時にやること
1. 作成/編集した md ファイルを ~/aipm_v0/Flow/{today}/{project}/ 配下に保存
2. ~/aipm_v0/Flow/{today}/{project}/deliverables.yaml に該当エントリを追加 (新規) or last_modified 更新 (既存)
3. Stock の BL ファイル (~/aipm_v0/Stock/{category}/{project}/backlog/{bl_id}.yaml) の deliverable_refs に id を追加
4. STATUS.md の next_action / blocker / 進行中 セクションを最新化
5. STATUS.md「履歴」セクションに「{now} タスク完了」を append
EOF
)" \
  --variable "project:プロジェクト名:string:true:" \
  --variable "bl_id:BL ID:string:true:" \
  --variable "task_content:タスク内容:string:true:" \
  --variable "category:Stock カテゴリ:string:true:作業効率化"
```

ミニタキオン が `cockpit task create` する時、必ずこのテンプレを使う。

### 2. AIOS ルール追加

`.cursor/rules/aios/ops/13_project_status_protocol.mdc`:
- ミニタキオン経由でないエージェントも、プロジェクト配下で動く時は STATUS.md / deliverables.yaml の更新を強制

## 実装計画 (フェーズ分割)

設計スコープが当初の見積より広がったので、段階的に。**早期に動かして使い始める**を最優先。

### Phase 1: View-only MVP (週末1回 = 2日)

**ゴール:** iPhone から Stock/Flow を見られる。書き込みなし。
- `bunx create-next-app` で Next.js (App Router, TypeScript, Tailwind) スケルトン作成
- shadcn/ui 初期化 (`bunx shadcn@latest init`)、必要コンポーネント追加 (`button`, `card`, `badge`, `collapsible`)
- App Router の routes:
  - `app/page.tsx` ホーム (Server Component で Stock スキャン)
  - `app/projects/[slug]/page.tsx` プロジェクト詳細
  - `app/bl/[id]/page.tsx` BL 詳細
  - `app/deliverables/[id]/page.tsx` 成果物詳細 (react-markdown でレンダリング)
- `lib/stock.ts` `lib/flow.ts` `lib/cockpit.ts` の Server-side ヘルパー実装
- launchd で `bun next start` を Mac 起動時に自動起動 (port 3000、`HOSTNAME=0.0.0.0`)
- Tailscale 越しに iPhone Safari からアクセス確認

この時点で「読むだけだが、外出先から状況を把握できる」が手に入る。書き込みがないので何も壊れない。

### Phase 2: 書き込み機能 (週末2回目 = 2日)

**ゴール:** 成果物レビューと BL Q&A の応答が iPhone から完結する。
- Server Actions 実装:
  - `updateReviewState(deliverableId, newState, comment?)` — deliverables.yaml の review_state 更新 + review_log 追記
  - `replyToBL(blId, content)` — BL.yaml の thread 追記 + Cockpit task send
- 画面3, 画面4 の Client Components: textarea + iOS マイクボタン (`<input type="text">` で OS キーボード起動) + 送信ボタン (Server Action 呼び出し)
- 楽観的更新 (`useOptimistic`) でモバイルの体感を上げる
- `revalidatePath` で他クライアントにも反映
- `lib/lock.ts` でファイルロック実装 (`proper-lockfile` 等)

**この時点でモバイルの core loop が完成**。INBOX.md は当面まだ手で更新が必要。

### Phase 3: STATUS.md 自動更新 + Cockpit テンプレート (週末3回目 = 1日)

**ゴール:** エージェントが STATUS.md / BL / deliverables.yaml を確実に更新する。
- `cockpit templates create project_task` でテンプレ登録
- ミニタキオンの「タスク作成」UI からこのテンプレを使う
- AIOS ルール `13_project_status_protocol.mdc` 作成
- ミニタキオン UI に「STATUS更新漏れ警告」バッジ追加

### Phase 4: INBOX/daily_tasks 廃止 + 既存プロジェクト Stock STATUS bootstrap (週末4回目 = 1日)

**ゴール:** AIPM の3ファイル同期ルールを正式に終わらせる。
- 全 Stock プロジェクト (10カテゴリ) に `STATUS.md` の雛形を bootstrap (Cockpit エージェントに書かせる)
- AIOS ルール `12_parallel_task_orchestration.mdc` を更新 (INBOX/daily_tasks 廃止)
- メモリ `feedback_state_sync_3files.md` / `feedback_inbox_structure.md` を失効化、新メモリで置換
- 既存 INBOX.md / daily_tasks.md は Flow に残す (アーカイブ扱い、新規生成しない)

### Phase 5: 日次マージ + 自動セーフティネット (週末5回目 = 半日)

**ゴール:** Flow → Stock の STATUS.md 同期を自動化。
- ミニタキオン UI に「日次マージ」ボタン
- `./cockpit autorun create --type cron --expression "0 3 * * *"` で 03:00 cron タスクを登録
- マージロジック実装 + 競合解決

### Phase 6 (後回し): Telegram 通知 (週末6回目 = 1日)

ミニタキオン側で「BL state が awaiting_user に変わった」「未レビュー成果物が30分以上放置」を検知して Telegram bot で push。ただし通知だけ、UI は引き続き webapp。

### 合計見積もり

**実働 7-8 日 (約4-5週末)** で全機能完了。Phase 1 終了時点 (2日後) で既に「読むだけだが iPhone で見られる」が手に入るので、痛みは早期に大きく減る。

## Open Questions

1. **launchd vs nohup vs PM2**: シンプルさで launchd。OSに任せる。`bun next start -H 0.0.0.0` を ProgramArguments に。
2. **Next.js dev vs prod モード**: 本番運用は `next build && next start`。HMR が欲しい開発時のみ `next dev`。launchd は prod を起動。
3. **iOS音声認識精度**: 純正キーボードマイクで開始。不満が出たら Whisper 後処理を Phase 7 で。
4. **Tailscale ACL**: 現状 my-tailnet 内であれば誰でも 3000番が見える。実用上問題ないが、認証層は将来。
5. **過去の Flow が膨らんだ時**: 1年以上前の Flow を tarball にする運用は手動 (年1)。
6. **revalidatePath の粒度**: 書き込み時にどのパスを invalidate するか。BL 更新→そのBLページ + ホーム + プロジェクトページ、の3箇所が基本。

## What I noticed about how you think (改訂版)

- **痛点を「分断」と一語で切り取る抽象化能力**。最初は「Notion vs 自作」のフレームで考えていたのに、Q1 で「実は問題はモバイル動線」と切り直した瞬間、設計が一直線に決まった
- **Flow ↔ Stock パターンを既に内面化している**。STATUS.md の置き場所を聞いた時、迷わず「Flow に書いて夜にStockに反映」と即答できた。これは AIPM の運用を自分の頭で運用しているからこそ
- **「INBOX 廃止」を1行で決断できる胆力**。INBOX は数週間運用していたはずだが、ミニタキオンに統合する方が筋が通ると判断した瞬間に廃止を即決した。サンクコストに引きずられない
- **「データ構造を Flow 内の YAML で持つ」を自分で発想した**。これが今回の設計で一番の核心 — AIPM の「ファイルシステムが Source of Truth」哲学とミニタキオンの動的状態管理を自然に接続する答えで、私側からは出にくかった発想
- **「おまかせ」を効果的に使う**。重要でない判断 (BL-TBD 昇格 history 不要、YAML を選ぶ理由など) を「任せる」と切り捨てて、本質 (INBOX 廃止、ダッシュボード設計) に集中させた。意思決定リソースの使い方が巧み
