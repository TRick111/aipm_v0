## 位置付け

[Tachyon Workspace](~/tachyon-workspace) の「ミニ版」。
Tachyon が会議駆動でのAI開発支援という大きな仕組みを狙うのに対し、ミニタキオンは **個人の日次 AIPM 運用におけるAI成果物レビュー→次指示ループ** に特化したモバイル1画面ツール。

**重要: 当初スコープから拡張済**。INBOX.md / daily_tasks.md を廃止し、その責務を全てミニタキオンが引き受ける構成に決定 (2026-04-25 セッション内合意)。

## Problem Statement

田中さんの日々のタスク管理は AIPM (`~/aipm_v0/Flow/YYYYMM/YYYY-MM-DD/`) のマークダウン運用 (INBOX.md / STATUS.md / daily_tasks.md) と AGI Cockpit のAIエージェント指示という2系統に分かれている。PC作業時は問題ないが、**モバイルから操作する場面で動線が破綻する**:

1. AIが出した markdown 成果物を iPhone で見たいが、Mac上の markserv は iPhone では開けない
2. 成果物を見ながら音声入力で「ここが違う」と指摘したいが、見るツール（markserv）と指示を出すツール（Cockpit）が別なので、音声入力しながら成果物を見る UX が成立しない
3. 複数プロジェクトを横断した「今日レビューすべきもの」「私の番のもの」の一覧が見えない
4. タスクの状態が Cockpit の3状態（running / waiting_confirmation / completed）だけでは「AIが手詰まり」「私のサブタスク」「私の作業後AI再開」といった細かい状態が表現できず、「今誰のターンか」が判らない
5. **(発展課題)** AIPM の3ファイル同期ルール (STATUS.md / INBOX.md / daily_tasks.md) は人間とエージェント双方の認知負荷が高く、マークダウンを散文ベースで管理しているため構造化データとして扱いづらい

## What Makes This Cool

- **Tailscale × Cockpit CLI の組み合わせは、iPhone が Mac の内線端末になる**。認証もVPN設定もいらない、Safari で内線ホスト名を開くだけ
- Cockpit が既にグローバル CLI として `/Users/rikutanaka/.agi-tools/data/cockpit/master/bin/cockpit` で PATH 上にあり、ローカル webapp が subprocess で呼べる
- 音声入力は iOS キーボードのマイクボタンで完結。Whisper 等を自前で動かす必要なし
- **AIPM の3ファイル散文管理を、構造化データ (YAML) + UIベースのインタラクションに置き換える** — 人間にもエージェントにも扱いやすい統一モデルへ
- **「インライン指摘 → 音声で数十秒話す → 送信 → Cockpit タスクに次指示として流れる」までが1画面・数秒**で済む

## Constraints

- AGI Cockpit 本体はインストール済み Electron アプリ、ソースには触れない。CLI 経由での統合のみ
- Mac mini (`rikus-mac-mini-3.tailad7d87.ts.net`) が常時起動している前提
- 自分だけが使う前提。マルチユーザー対応は考えない
- 既存の AIPM 運用 (Flow/Stock の二層、BL-XXXX 命名) を継承
- 既存の Flow/INBOX.md データは過去ログ扱いで移行不要
- マークダウン以外の成果物 (画像、コード差分、Figma) は対象外。HTML は外部検証サーバーへの link out のみ

## Premises (確認済)

1. コアの未解決課題はモバイル動線（レビュー→音声コメント→次指示）。PCは既存ツールで足りる
2. AGI Cockpit 本体は closed binary、CLI 経由統合のみ
3. ミニタキオン は INBOX.md / daily_tasks.md の責務を完全に引き受け、新規スコープとなる
4. AIPM の Flow/Stock の二層構造を継承、BL-XXXX を引き続き正としつつ、その実体ファイルを ミニタキオン が管理

## Approaches Considered

### Approach A: Mobile Web Wrapper (Mac-hosted, Tailscale経由) — 採用
Mac 上で webapp を立て、iPhone Safari から tailscale ホスト名で開く。Cockpit CLI と AIPM ファイルシステムを薄くラップする。

### Approach B: Notion をフロントエンド + 同期デーモン
**却下**: async sync のエッジケース (削除・衝突・順序) が ocean 化、コメント→指示の遅延が秒単位、bridge を書く必要があり A より楽でない。

### Approach C: Telegram Bot
**却下**: 通知と音声は強いが、マークダウン成果物のインライン指摘 UX が成立しない。本丸の痛点を解かない。ただし将来 Approach A に push 通知チャネルとして後乗せ予定。

## Recommended Approach

**Approach A**。将来 Approach C を通知レイヤとして追加。

## Eng Review Decisions (2026-04-25 /plan-eng-review 適用済)

このセクションは設計書の全体に**優先される**。下位セクションと矛盾する場合はここを正とする。

### 1A: キャッシュ無効化 — chokidar fs watcher 採用

エージェントが webapp の外側で Stock/Flow YAML を直接書き換える運用なので、Next.js の `revalidatePath` だけでは UI がキャッシュされた古いデータを表示し続ける。対策:

- `chokidar` で `~/aipm_v0/Stock/` と `~/aipm_v0/Flow/` を再帰監視
- ファイル change/add/unlink イベントで該当の in-memory index を rebuild + `revalidatePath` を発火
- macOS では fsevents が backend、低オーバーヘッド
- **追加で 5分タイマーの fallback rebuild** を入れる (chokidar が event 取りこぼした時の救命、Failure Mode #7 対策)

実装場所: `lib/watcher.ts` (server-only モジュール)、`server.ts` か Next.js instrumentation hook で起動時に initialize。

### 1B: Cockpit subprocess エラーハンドリング — 構造化エラー

`lib/cockpit.ts` で以下を厳格に:

1. **起動時 PATH 確認**: `which cockpit` を実行、見つからなければ webapp 起動拒否 (process.exit)
2. **全 spawn に 30秒 timeout**: AbortController + `Bun.spawn`、超過したら kill + throw `CockpitTimeoutError`
3. **JSON parse 失敗時の構造化エラー**: stdout を `JSON.parse` し、失敗したら raw stdout/stderr を `error.detail` に格納して throw `CockpitParseError`
4. **三重 fallback (send → resume → create) 全失敗時**:
   - UI に「送信失敗」トースト + 再試行ボタン
   - 入力テキストは `localStorage` に下書き保存 (再送できる)
   - `console.error` ではなく `lib/log.ts` で構造化ログ

エラー型階層: `CockpitError` (base) → `CockpitNotFoundError` / `CockpitTimeoutError` / `CockpitParseError` / `CockpitExitError` / `CockpitFallbackExhaustedError`

### 1C: ロック + 競合 — Optimistic Locking

`lib/lock.ts` の `withLock(path, fn)` は以下を実装:

- `proper-lockfile` 設定: `stale: 30000, retries: 5, retryWait: 200`
- `withLock` 内で読込 → fn 実行 → 書出 をシリアライズ
- **追加: optimistic check**:
  - 読込時に `updated_at` を memo
  - fn が更新を返した後、**書き出す前に**ファイルを再読込
  - `updated_at` が memo と異なれば `OptimisticLockConflictError` を throw
  - UI 側は conflict をキャッチして「他から更新あり、最新を見て再送信してください」を表示
- `withLock` の戻り値型は `Result<T, OptimisticLockConflictError | LockError>`

### 4A: サーバーサイドキャッシュ — Module-level Singleton + BL駆動 load

**データアクセスパターンを修正**: 設計書当初の「全 Flow スキャン」→ **BL駆動 load** に変更:

```typescript
// lib/index.ts
type MiniTachyonIndex = {
  projects: Map<string, Project>      // Stock/<cat>/<proj>/STATUS.md
  bls: Map<string, BL>                // Stock/<cat>/<proj>/backlog/BL-*.yaml
  deliverables: Map<string, Deliverable>  // BL.deliverable_refs から逆引きで load した分のみ
}

// global singleton (Node.js single-threaded なので OK)
declare global { var __miniTachyonIndex: MiniTachyonIndex | undefined }
export const index = globalThis.__miniTachyonIndex ??= newIndex()

// 起動時:
// 1. Stock backlog/*.yaml 全部読み (~100 ファイル, ~50KB)
// 2. 各 BL.deliverable_refs から d-id 集合を作る
// 3. d-id がどの日の Flow にあるか解決して、その日の deliverables.yaml だけ load
//    (active BL 関連だけ、~50ファイル、~250KB、合計 ~200ms)
// 4. chokidar で変更検知 → 該当エントリだけ rebuild
```

**Active な BL に紐付かない過去の deliverables.yaml は load しない**。BL アーカイブ済 → そこに紐付く成果物も不可視で問題なし。

データアクセスのパフォーマンス: 1年後でも ~200ms (calendar-time に依存しない、active BL 数に依存)。Server Component からは `index.bls.get(id)` の O(1) 参照。

### Critical Gap 修正: chokidar silent staleness 対策

`lib/watcher.ts` に 5分ごとの全 rebuild タイマーを追加:

```typescript
// Failure mode #7 対策: chokidar が OS イベント取りこぼした場合の救命
setInterval(() => rebuildIndex(), 5 * 60 * 1000)
```

chokidar が動いていれば redundant、動いていなければ最大5分の遅延で復旧。

#### BL-XXXX.yaml スキーマ v2 (Stock 直接編集)

**v1 との差分**: `thread:` を廃止し、`cockpit_task_ids` + `pending_questions` + `decisions` の3層に分離。全会話履歴は Cockpit が単一の source of truth として保持、ミニタキオンは「意思決定に必要なサマリー」だけ持つ。

```yaml
# UI で「全会話を見る」を押した時に cockpit task get で fetch する対象

# 私への保留中 AI 質問 (回答すると consume されて decisions に移る)

    cockpit_task_id: "64200"
    content: |
      Bun のバージョン何を使いますか？ 1.1.x or 1.2.x?

# 決定事項ログ (append-only、curated、durable)
# pending_questions への回答、用方針変更、commitment などを残す
decisions:
  - at: 2026-04-25T15:45:00+09:00
    type: answer       # answer | commitment | scope_change | deferred
    by: user           # user | ai
    content: "1.3.x で行く (最新)"
  - at: 2026-04-25T16:00:00+09:00
    type: scope_change
    by: user
    content: "Hono は捨てて Next.js Route Handler に切り替え"

# このBL に紐づく成果物の参照 (deliverable id, deliverables.yaml の id を指す)
deliverable_refs:
  - d-20260425-001
  - d-20260425-002
```

**設計上の判断:**

- **全会話を BL.yaml に持たない理由**: Cockpit task が既に持っている。二重管理は整合性が壊れた時に死ぬ。BL寿命中に Cockpit task が複数になる場合 (Phase切替等)、BL.yaml に集約する仕様だと誰がどの会話を持つかメタが必要、結局 Cockpit を見るのと変わらない
- **pending_questions が「キュー」になる理由**: AI が複数質問を一度に出すケースがある (例: BL-0028 で「機種は？」「色は？」「保証は？」)。各質問を id で識別、回答済が consume されて decisions に移る
- **decisions が curated な理由**: 流れた会話を全部 dump するのではなく、durable な決定 (機種選定、納期決定、scope 変更等) だけを残す。append-only で時系列の decision log になる
- **「全会話を見る」UI**: BL 詳細画面のボタン → モーダルで `cockpit task get <ids>` を call、一時的に表示 (BL.yaml には保存しない)

#### スキーマ migration (v1 → v2)

`lib/migrate.ts` で v1 を読み込んだ時に自動変換:

```typescript
function migrateBLv1tov2(v1: BLv1): BLv2 {
  const cockpit_task_ids = unique(v1.thread?.map(t => t.cockpit_task_id ?? t.sent_to_cockpit_task).filter(Boolean) ?? [])
  // v1 thread の最後の AI message が role=ai で終わってたら pending_questions に
  const lastAi = findLastAiMessage(v1.thread)
  const pending_questions = (lastAi && !hasUserResponseAfter(v1.thread, lastAi))
    ? [{ id: generateId(), at: lastAi.at, cockpit_task_id: lastAi.cockpit_task_id, content: lastAi.content }]
    : []
  // user の発話で「決定」と読めるものを decisions に curate (heuristic、雑な v1 移行ロジック)
  const decisions = (v1.thread ?? []).filter(t => t.role === 'user').map(t => ({ at: t.at, type: 'answer' as const, by: 'user' as const, content: t.content }))
  return { ...v1, schema_version: 2, cockpit_task_ids, pending_questions, decisions, thread: undefined }
}
```

例データ (8 BL) は既に v2 で保存済み (master が手動 migration 完了 2026-04-25 21:00)。

### 同期戦略

```
  ・エージェントが Flow の STATUS.md を更新 (タスク開始時/終了時)

夜:
  ・田中さんが手動で「日次マージ」ボタン → Flow の STATUS.md を Stock にマージ
  ・03:00 cron が走り、未マージなら自動マージ (セーフティネット)
  ・Flow STATUS.md はその日のスナップショットとして保持

マージロジック:
  - 履歴セクション: append (重複なし)
  - 完了済セクション: union
  - 次アクション/ブロッカー/概要: Flow の最新内容で上書き
  - frontmatter updated_at: max(stock, flow)
```

#### BL ファイル (同期なし、Stock 直接編集)

BL の Q&A は本質的に時系列累積、Flow 中間レイヤを挟まない。ミニタキオン が単一書き込み口になり、書き込みをシリアライズして競合回避。

```
ミニタキオン API: PUT /bl/:id
  1. ファイルロック (~/aipm_v0/Stock/.../<bl-id>.yaml.lock)
  2. ファイル読込
  3. 該当フィールドを更新
  4. atomic write (tmp → rename)
  5. ロック解放
```

#### deliverables.yaml (Flow のみ、Stock に上がらない)

成果物メタデータは「その日の出来事」として Flow に固定。Cross-day query は **BL駆動 load**: Stock backlog/*.yaml を起動時に全部読み、その deliverable_refs から関連 deliverables.yaml だけを当日 Flow に解決して load する。Active BL と紐付かない過去の deliverables.yaml は load しない (孤児になるが、それは BL アーカイブ済み = 不可視で問題なし)。time-complexity O(BL数)、calendar-time 非依存。詳細は「Eng Review Decisions / 4A」参照。
