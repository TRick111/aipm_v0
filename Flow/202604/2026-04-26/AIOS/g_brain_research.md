|---|---|

## 5. 統合形態の検討メモ（案A / 案B / 案C）

> ※ 4/22 の `scope_proposal.md` で既に スコープ A/B/C（最小 / 中間 / 最大）が提示済み。本セクションは **「G-Brain との統合方式」だけ**を切り出して整理（4/22 の案A/B/C とは粒度が違うので注意）。

### 案A: そのまま import（コード/ライブラリ依存）

- G-Brain が (β) **コード共有ライブラリ** だった場合のメインルート。
- AIPM 配下のプロジェクト（ai-core-pl / Tachyon 等）の `package.json` / `requirements.txt` に `g-brain` を追加し、`import` して使う。
- AIOS 側は「G-Brain に依存するプロジェクトの初期化フロー」を ops に1条追記（どのリポを clone / install しておくかのチェックリスト）。
- **メリット**: コードレベルの再利用が最大。バージョン管理が ライブラリの版で一本化される。
- **デメリット**: G-Brain が (α) や (γ) なら **そもそも import できない** ので適用不可。
- **必要な事前確認**: G-Brain がパッケージ公開されているか、private npm registry / private GitHub Packages を使うか。

### 案B: 一部だけ参照（外部パス / submodule / ポインタ README）

- G-Brain が (γ) **ナレッジ集** や (α) **ルール集** で、**コピーは避けたい** ケースに有効。
- AIOS は `Stock/<新Program>/G-Brain/README.md` を **ポインタ**（外部リポURL + 利用ガイド）として置くだけ。実体は別リポ（大地さんメンテ）に置いたまま。
- もしくは git submodule で `.cursor/rules/g-brain/` などにマウントする方式。
- **メリット**: 実体管理を大地さんに任せたまま、AIPM 側からは「ある」状態にできる。MasterIndex 更新も最小。
- **デメリット**: aipm_v0 の git 履歴に G-Brain の更新が反映されない。submodule 派生のオペコストが乗る。
- **必要な事前確認**: G-Brain の更新頻度。頻繁に大地さんが書き換えるなら submodule、安定しているならポインタ README で十分。

### 案C: ルールに翻訳して取り込み（コピー + AIOS 形式に正規化）

- G-Brain が (α) ルール集 / (γ) ナレッジ集 で、**aipm_v0 の中で完結させたい** ケース。
- 大地さんから G-Brain のスナップショットをもらい、AIOS の `.cursor/rules/aios/` 形式に **正規化して取り込む**（frontmatter を AIOS 仕様に揃える、ファイル名規則を 00〜99 に揃える、等）。
- **メリット**: aipm_v0 だけで完結。新規メンバー（飯武さんなど）への展開時、リポ1つ渡せば動く状態にできる。AIOS のルール体系の中で一元管理。
- **デメリット**: コピー時点でフォークが発生 → 大地さん本家との同期メンテ責任が田中さん側に発生。**大地さんの版が更新されたら手動で取り込み直す必要**。
- **必要な事前確認**: G-Brain のライセンス・所有権。大地さんが「コピーしてOK」と言うか、原本をマスターとして残したいか。

### 案D（追加提案）: ハイブリッド（案B + 案C 部分採用）

- 「**ルール骨格は案C で正規化取り込み（安定する核）**」+「**頻繁に更新される事例集 / プロンプトは案B で外部参照**」という分割。
- 飯武さん導入（BL-0066）まで残り日数が少ない（4/29 期日）状況には現実的。
- 4/22 scope_proposal の **案B（中間スコープ）** とも整合する（マルチリポ運用の正式化）。

### 推奨（4/22 と同じく "B" 推奨だが条件付き）

- **G-Brain の実体が α / γ なら 案D ハイブリッド推奨**（飯武さん導入に間に合う形）
- **β（コード）なら 案A 一択**（AIOS 側の改修は薄い）
