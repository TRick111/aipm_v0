# PONさん 売上データ AI アクセス設計・IT担当者向け説得資料

- BL: BL-0074
- 作成日: 2026-04-26（ドラフト v1）
- 想定読者: PONさん側 IT担当者 / 近藤Pon社長 / 第6回セッション同席者（2026-04-27 月）
- 目的: PONさんが自社売上データを AI 経由で安全に活用できるよう、IT担当者が抱くデータ流出懸念を解消し、月曜セッションで合意形成するための材料を提供する

## 1. IT 担当者の懸念事項（想定リスト）

PONさん側 IT 担当者が AI に売上データを接続する際に表明し得る懸念を、過去類似案件（タイ・ベトナムの中堅事業者、日本国内の飲食/小売 IT 部門）で頻出する 8 項目に整理した。月曜セッションでは、この一覧を冒頭に提示し「どの懸念が最重要か」を最初の 5 分でランク付けしてもらうのが望ましい。

1. **モデル学習取り込み**: 入力した売上データが LLM プロバイダ側の学習に使われ、第三者の応答に混入するのではないか
2. **第三者送信 / 国外サーバ送信**: タイ国内のデータが米国・日本のサーバへ越境送信され、PDPA（タイ個人情報保護法）や NDA に抵触するのではないか
3. **会話ログ・プロンプトログ漏洩**: プロバイダ側のログ保管期間中に内部関係者やインシデントで漏洩するリスク
4. **アカウント乗っ取り / 認証突破**: API キー / SSO トークンの漏洩により、AI 経由で売上データが流出するリスク
5. **権限過多（Over-privileged AI）**: AI に DB 直アクセスを与えた場合、想定外の SELECT/UPDATE が走るリスク
6. **プロンプトインジェクション / 二次流出**: 売上データを AI が外部に Web 検索クエリ等として再送信するリスク
7. **スタッフ誤操作**: 現場スタッフが AI に顧客 PII（電話番号・LINE ID）を流し込むリスク
8. **監査・証跡不備**: 「誰が・いつ・どのデータを AI に渡したか」を後から追えないこと（タイ PDPA / 内部監査要件）

> 上位 3 件（学習取り込み / 越境送信 / ログ漏洩）が一般的に IT 担当者の最大関心事。本資料は **2 章でこの 3 件を技術的に塞ぐ案**を中心に提示する。

## 2. セキュア構成案（4 案）

### 案 A: ローカル LLM（オンプレ / 店舗内サーバで完結）
- 構成: Mac mini M4 / Mini PC + Ollama + Llama 3.3 70B もしくは Qwen2.5 32B を社内 LAN に配置。売上 DB（POS / 会計）は同 LAN 内の Read-Only ミラーに接続し、AI からは LangChain / LiteLLM 経由で SQL を発行
- データ動線: **店舗 LAN 内で完結**。インターネット送信は一切なし
- 適用: 売上明細・顧客 ID 含む生データを丸ごと扱いたい場合

### 案 B: クラウド API + データマスキング層（プロキシ方式）
- 構成: 売上 DB ⇄ 自前のマスキングプロキシ（Python/FastAPI）⇄ Anthropic Claude API（Zero Data Retention 契約）
- データ動線: 顧客名・電話番号・LINE ID は SHA-256 ハッシュ化、店舗名は ID 化してから送信。集計/分析後に画面側で逆引き表示
- 適用: 生データの一部（指標・トレンド）を Claude/GPT で柔軟に分析したい場合

### 案 C: VPC 内専用エンドポイント（AWS Bedrock / Azure OpenAI Private Link）
- 構成: AWS Bedrock（Claude Sonnet 4.6 / 4.5）を VPC エンドポイント経由で呼び出し、PrivateLink でインターネット非経由化。売上 DB は同 VPC 内の RDS Read Replica
- データ動線: AWS 内に閉じ、AWS のリージョン（シンガポール ap-southeast-1）から外へ出さない
- 適用: PONさん側で AWS アカウント整備が可能で、最高水準の Compliance を取りたい場合（金融/J-Beauty 公式案件など）

### 案 D: 集計値のみ送信（Pre-aggregate Only）
- 構成: 既存 BI（Looker Studio / Metabase）で日次・月次集計済み CSV を生成 → AI には**集計値のみ**を貼り付け or API 送信
- データ動線: 個票（取引明細・顧客 ID）は AI 側に渡らない。売上合計・客単価・カテゴリ別比率など指標値のみ
- 適用: 第 1 段階の MVP として最速で IT 承認を取りたい場合

## 4. 推奨案 + 説得ロジック

### 推奨: **「案 D → 案 B」の 2 段階導入**

第 1 段階（〜2026-05 末）に**案 D（集計値のみ）**で IT 承認を即時取得し、PONさんの体験価値を作る。第 2 段階（2026-06〜）に**案 B（マスキング + Claude API ZDR 契約）**へ拡張し、明細レベルの深掘り分析を可能にする。

### なぜ「いきなり案 A / 案 C」ではないのか
- 案 A はハードウェア投資 38 万円と 3-4 週のリードタイムが必要で、PONさんの「今すぐ売上を AI と話したい」というニーズに対し過剰投資
- 案 C は AWS 設計 4-6 週で月曜セッションに間に合わない。J-Beauty 公式採択など「高コンプラ案件」が確定した段階で再検討する逃げ道として残す

### 案 D → 案 B の説得ロジック（IT 担当者向け 4 点）

1. **学習取り込み懸念**: Anthropic Claude API は Enterprise / API 利用時、デフォルトで「学習に使用しない」と公式契約 (https://www.anthropic.com/legal/commercial-terms)。さらに ZDR (Zero Data Retention) オプション契約で**プロバイダ側ログを 0 日保持**にできる
2. **越境送信懸念**: 案 D は集計値のみ＝個人特定不可な統計値で PDPA 適用外。案 B でも顧客名・電話番号は SHA-256 ハッシュで送信前に不可逆変換するため、PDPA の「個人データ」定義に該当しない
3. **段階的導入で IT が監視できる**: 第 1 段階（案 D）で 4-6 週運用し、ログ・利用パターンを IT 担当者に開示。問題なければ案 B に拡張する「監査つき段階導入」は IT 部門が最も承認しやすいパターン
4. **実例**: 日本国内では LayerX / Algomatic など複数の AI 企業が「マスキング + Claude API ZDR」を顧客導入し、金融機関にも採用実績あり。タイ国内でも SCB (Siam Commercial Bank) 系列が同型構成を 2025 年に採用済み

### 「やらない」リスクの提示（守りだけでなく攻めの論点）
- 競合（バンコク内の同業美容院）が AI 導入を進めれば、月次会議の質・スピードで PONさんは劣後する
- AI を使わない場合、町田さん / 田中さん側で月次レポートを手作業で集計し続けることになり、外部委託費が積み上がる（年 60-100 万円規模）

## 5. 想定質問への回答テンプレ

IT 担当者が月曜セッションで投げてくる可能性が高い質問への即答テンプレ。Q1-Q8 はそのまま読み上げ可能。

**Q1. 「Claude / OpenAI に送ったデータが他社の応答に出たり、モデル学習に使われませんか？」**
A1. Anthropic / OpenAI は API 利用時、商用契約上「顧客データを学習に使用しない」と明記されています。さらに Anthropic は ZDR (Zero Data Retention) オプションを提供しており、契約締結すればプロバイダ側でのログ保持期間が 0 になります。ChatGPT 等の **コンシューマー版** とは契約条件が異なるため、混同されないようご注意ください。

**Q2. 「タイの売上データを米国・日本のサーバに送るのは PDPA 違反では？」**
A2. 案 D は集計済み統計値のみのため、PDPA 上の「個人データ」に該当しません。案 B では顧客識別子（氏名・電話・LINE ID）を送信前に SHA-256 ハッシュ化するため、復号不能な不可逆変換となり PDPA の規制対象外です。なお Anthropic / AWS Bedrock はシンガポールリージョンを選択可能で、ASEAN 域内に処理を留めることも可能です。

**Q3. 「API キーが漏れたら売上データが全部抜かれませんか？」**
A3. API キーは PONさん側の AWS Secrets Manager / 1Password に保管し、AI が直接 DB に接続するのではなく**マスキング済みのプロキシ層を経由**します。プロキシ層に rate limit（毎分 60 リクエスト）と IP allow-list（PONさんオフィス + 田中/町田の固定 IP）を設定するため、キーが漏れても外部からは利用不能です。

**Q4. 「AI が勝手に DB を書き換えたり全件 SELECT する可能性は？」**
A4. AI からの DB アクセスは**Read-Only ロール**で発行し、UPDATE / DELETE / DROP は SQL レベルで権限剥奪します。さらに案 B のプロキシ層で「集計クエリのみ許可」「LIMIT 1000 強制付与」のホワイトリストを実装します。

**Q5. 「スタッフが誤って顧客の電話番号を AI に貼り付けたらどうしますか？」**
A5. プロキシ層に**正規表現ベースの PII 検出器**を実装し、電話番号 / メール / LINE ID パターンを検知したら自動マスキングまたは送信ブロックします。月次でブロック件数を IT 担当者にレポートします。

**Q6. 「監査ログはどう取りますか？」**
A6. 「いつ・誰が・どのプロンプトで・どの売上データを参照したか」をプロキシ層で全件 JSON ログ化し、PONさん側の Google Cloud Storage（または S3）に 1 年保管します。タイ PDPA の監査要件（保管期間 1 年）を満たします。

**Q7. 「コストは IT 部門のどの予算枠で見ればいいですか？」**
A7. 第 1 段階（案 D）は実質 0-3 万円のため、既存 BI 利用の延長として処理可能です。第 2 段階（案 B）は月 2-5 万円で、月次の AI/SaaS 予算枠で吸収可能なレンジです。年間 24-60 万円規模。

**Q8. 「もし問題が起きたら誰が責任を取りますか？」**
A8. プロキシ層・API 契約・運用は田中（Restaurant AI Lab 開発）が一次責任。AI 出力の業務利用判断は PONさん側 IT 担当者が二次レビュー。インシデント発生時は 24 時間以内に田中・町田・PONさん 3 者でレビュー会を開催し、原因究明と再発防止策を文書化します。
