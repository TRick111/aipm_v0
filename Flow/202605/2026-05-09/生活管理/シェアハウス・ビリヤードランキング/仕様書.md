**Project**: 生活管理 / シェアハウス・ビリヤードランキング
**BL**: BL-0001
**Version**: 0.3 (MVP)
**Date**: 2026-05-09 (rev: 称号システム追加)

---

## 1. 概要

シェアハウス共有スペースのビリヤード台で行われる住民同士の真剣勝負を記録し、Glicko-2 ベースのレーティングでランキング化する Web アプリケーション。試合結果は LINE グループに自動投稿され、プレイヤーだけでなく非プレイヤー住民もコミュニティの話題として楽しめる。

### 1.1 目的

1. ビリヤードブームを一過性で終わらせず長続きさせる
2. 上達実感と目標を提供し、競技志向プレイヤーの熱量を保つ
3. 「ランキング戦」という社会的口実装置で 1on1 真剣勝負を仕掛けやすくする
4. 試合結果通知で非プレイヤーも巻き込みコミュニティを盛り上げる

### 1.2 想定スケール

- 想定参加人数: **20人以内**
- 想定対戦頻度: 不定 (人によって偏る前提)
- シェアハウス内の単一テーブルで運用

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| **プレイヤー** | システムに登録されたユーザー (シェアハウス住民) |
| **管理者** | 自分。プレイヤー登録・初期レート設定・シーズン管理権限を持つ |
| **アンカープレイヤー** | 外部 FargoRate を持っており、コミュニティの絶対実力換算の基準となるプレイヤー (= 管理者本人、Fargo ≈ 470) |
| **ランキング戦** | 開始前に「ランキング戦」として登録された真剣勝負試合のみ。ランキングに反映される |
| **カジュアル戦** | 記録対象外の遊び試合 |
| **レート** | Glicko-2 のレーティング値 (R) |
| **RD** | Rating Deviation。レートの不確実性 |
| **σ** | Volatility。レート変動の揺らぎ |
| **推定 FargoRate** | アンカーを基準に Glicko-2 レートから換算した推定 FargoRate |
| **シーズン** | ランキング集計の期間。終了時にアーカイブされる |
| **ハンデ** | 実力差を埋める試合条件 (フリーボール/ゲームボール拡張等) |

## 3. 機能要件

### 3.1 プレイヤー管理

- 管理者は Web UI からプレイヤーを登録・編集・無効化できる
- プレイヤー属性: `名前 / 表示名 / 初期レート / 初期RD / 初期σ / アクティブ状態 / 備考`
- 初期レートは管理者が見立てで決定 (デフォルト値: 1500、レンジ: 1000〜2200 程度)
- アンカープレイヤーは1名指定でき、`is_anchor: true` フラグを持つ

### 3.2 試合登録 (ランキング戦)

- Web UI のフォームから誰でも登録可能 (MVP では認証なし)
- 入力項目:
  - **対戦者A** (ドロップダウン)
  - **対戦者B** (ドロップダウン)
  - **勝者** (A/B のラジオボタン)
  - **ゲーム種別** (8ball / 9ball / 10ball / その他、自由テキスト) — レート計算には影響しない (Q4: 統合レート)
  - **ハンデ条件** (チェックボックス + 数値入力、後述)
  - **メモ** (任意自由テキスト)
- 登録時に Glicko-2 で両者のレートを更新し、結果を DB に保存
- 結果は LINE 公式アカウント経由でグループに自動 Push (3.5節)

### 3.3 ランキング表示

- 現シーズンのランキング: レート降順、上位から表示
- 各プレイヤー行に表示:
  - 順位 / 表示名 / 現レート / RD / 推定 FargoRate / 試合数 / 直近試合結果 (勝敗3件)
- ランキングは公開ページとして誰でも閲覧可能 (URL 共有可)

### 3.4 プレイヤー詳細

- 個別プレイヤーの履歴ページ
- レート推移グラフ (時系列)
- 試合履歴一覧 (対戦相手・勝敗・レート変動・ハンデ)
- 推定 FargoRate と Glicko-2 レートの両方を表示

### 3.5 LINE 通知

- LINE Messaging API (公式アカウント) でシェアハウスのグループに送信
- 試合登録時に下記フォーマットの flex message or text を Push
  ```
  🎱 ランキング戦結果
  勝者: {winner_name} ({winner_rate_before} → {winner_rate_after})
  敗者: {loser_name} ({loser_rate_before} → {loser_rate_after})
  ハンデ: {handicap_summary}
  種目: {game_type}
  順位変動: {rank_change_summary}
  ```
- メッセージ末尾にランキングページの URL を添付

### 3.6 シーズン管理

- 管理者がシーズンを作成・終了できる
- シーズン属性: `名前 / 開始日 / 終了日 / アーカイブ済みランキング`
- シーズン終了時:
  - 終了時点のランキングをスナップショットとして保存
  - 新シーズン開始時に各プレイヤーのレートを **70% リセット** (詳細 §5.5)
- 推奨シーズン長: **3ヶ月** (要検討、初回は様子見で 1〜2ヶ月でも可)

### 3.7 ハンデ自動提案 (Phase 1.5)

- 試合登録フォームで対戦者2人を選んだ際、レート差から推奨ハンデを提示
- §6.4 のテーブルを参照

## 5. レーティング仕様 (Glicko-2)

### 5.1 採用根拠

- 少人数・対戦数偏りに強い (Q1 比較で詳述)
- RD/σ により新規プレイヤーが急速収束、長期不在者は次回試合での変動が大きく許容される
- 業界実装が豊富 (Lichess, CS:GO 等)

### 5.2 パラメータ

| パラメータ | 初期値 | 説明 |
|---|---|---|
| R | 管理者指定 (default 1500) | 初期レート |
| RD | 350 | 初期不確実性 (Glicko-2 標準) |
| σ | 0.06 | 初期 volatility |
| τ | 0.5 | システム定数 (Glicko-2 標準) |
| Rating Period | **1試合ごと** | バッチではなく試合ごとに更新 |
| **DAMPING_FACTOR** | **0.5** | 全試合の変動量をブレンドで圧縮するグローバル係数 (§5.4 参照) |
| **HANDICAP_FACTOR** | **0.5** | ハンデ試合への追加圧縮 (DAMPING に上乗せ) |

### 5.3 計算

標準 Glicko-2 アルゴリズムをそのまま採用。npm `glicko2` または同等ライブラリ利用。試合ごとに `RatingPeriod = [対戦相手1名]` で更新。

### 5.4 変動量の圧縮 (DAMPING + ハンデ補正)

少人数 (20人以内) で対戦頻度バラつきが大きい運用環境を前提に、Glicko-2 標準の変動量だと「短期で暴れすぎる」問題があるため、計算結果を旧値とブレンドして圧縮する。

```
t = DAMPING_FACTOR                      // ハンデなし: 0.5
t = DAMPING_FACTOR * HANDICAP_FACTOR    // ハンデあり: 0.5 * 0.5 = 0.25

R_final  = R_old  + (R_new  - R_old)  * t
RD_final = RD_old + (RD_new - RD_old) * t
σ_final  = σ_old  + (σ_new  - σ_old)  * t
```

実効的な変動量の目安 (両者初期 RD=350, 同レートでの 1試合):

| 状況 | Glicko-2 標準 | DAMPING 適用 (本MVP) |
|---|---|---|
| ハンデなし試合 | ±60 | **±30** |
| ハンデあり試合 | ±30 (旧) | **±15** |
| 安定後 (RD≈80, ハンデなし) | ±10 | **±5** |
| アップセット (200diff, 弱者勝) | ±90 | **±45** |

**設計意図**:
- ブレが半分になるため、レートが「真の実力差」を反映するまでに試合数が増える
- 結果として "100 diff" がより意味のある実力差を表すようになる
- Glicko-2 の勝率計算式 (期待値 E) は不変なので、勝率の定義との整合性は保たれる
- ハンデ適用試合はさらに 50% 追加圧縮 (= 標準の 25% の動き) で、過剰ハンデ時のレート暴走を防止

### 5.5 シーズンリセット (Soft Reset)

シーズン切替時、各プレイヤーのパラメータを以下で更新:

```
R_new = 1500 + (R_old - 1500) * 0.7    -- 平均に向けて 30% 引き戻し
RD_new = max(RD_old, 200)              -- RD を200以上に膨らませる
σ_new = σ_old                          -- σ は据え置き
```

これにより前シーズンの成績が完全リセットされず、しかし新シーズンで実力差が再評価される余地を作る。

### 5.6 推定 FargoRate 換算

```
estimated_fargo(player) = anchor.external_fargo + (player.rating - anchor.rating) * K_fargo
```

- `K_fargo = 0.5` (初期値、要校正)
- アンカープレイヤーは常に実 FargoRate を表示
- 換算係数 K_fargo は年1回程度、アンカーの外部試合実績で校正

## 6. ハンデ仕様

### 6.1 ハンデの種類 (MVP)

MVP では実運用のシンプルさを優先し、**フリーボール権の回数のみ** をハンデ条件として扱う。
将来 (Phase 2) に向けて、以下のコード体系を予約だけ残しておく:

| コード | 名称 | MVP 採用 |
|---|---|---|
| H1 | ゲームボール拡張 | × (Phase 2) |
| **H2** | **フリーボール権** | ○ |
| H3 | 持ち点差 | × (Phase 2) |
| H5 | ブレイク優先権 | × (Phase 2) |
| H7 | 道具制限 | × (Phase 2) |

> Phase 1 の運用シンプル化のため、UI ではフリーボール権のみ入力。
> 将来追加された場合も `handicap` JSON カラム (§6.2) に追加するだけで済む。

### 6.2 データ表現

`Match.handicap` フィールド (JSON):

```json
{
  "strongerPlayerId": "<強者の player_id (= ハンデを背負う側)>",
  "freeballCount": 3
}
```

- `strongerPlayerId`: ハンデを背負う側 (= レートが高い側)
- `freeballCount`: 弱者 (= 強者の反対側) が試合中に行使できるフリーボール権の回数

ハンデなしの場合は `freeballCount: 0` または空オブジェクト。

**互換性**: 旧フォーマット (`h1GameBalls`, `h2FreeballCount`, `h3PointSpot`, `h5BreakPriority`, `h7Restrictions`) も `summarizeHandicap` で表示可能。新規入力は新フォーマットで保存される。

### 6.3 レートへの反映

- `freeballCount > 0` (または旧フォーマットのいずれか) が入っていれば §5.4 の HANDICAP_FACTOR が追加適用される
- ハンデの種類による補正は MVP では行わない (将来 Phase 2 で導入)

### 6.4 自動提案ロジック

校正方針: **「100 diff = 3 フリーボール」をアンカー** に、+100 diff ごとに +1 フリーボール。
これは「1 フリーボール ≈ 約 +5〜7% の弱者勝率ブースト」相当 (ビリヤードでの体感とも整合)。

| Glicko レート差 | 推定 Fargo差 | 推奨フリーボール |
|---|---|---|
| 0〜49 | 0〜25 | 0 (誤差レベル) |
| 50〜149 | 25〜75 | **3** (アンカー: 100 diff = 3) |
| 150〜249 | 75〜125 | 4 |
| 250〜349 | 125〜175 | 5 |
| 350〜449 | 175〜225 | 6 |
| 450〜 | 225〜 | 7 (上限) |

提案であって強制ではない。プレイヤーが合意したハンデで登録する。

### 6.5 校正の考え方 (補足)

- 勝率の定義 (§5: Glicko-2 期待勝率) と ハンデの効力 (§6.4) は **独立した別パラメータ**
- 100 diff の Glicko-2 期待勝率は強 64% / 弱 36% (差 28pp)
- 3 フリーボールで弱者を 50% に引き上げる仮定 = 14pp 補正 = 1 freeball あたり 約 4.7pp ブースト
- これは矛盾ではなく「ハンデ効力の校正値」として独立に決定したもの
- データが溜まったら (Phase 2) 実勝率から逆算して校正値を更新可能

## 7. 称号システム

レーティングは「変動が小さく安定して実力を表す」軸であるのに対し、称号は **1試合の勝敗で動く** 軸として、コミュニティの盛り上がりとライバル意識を作るために導入する。
全 8 種類: 保持型 4 + 算出型 4。

### 7.1 保持型称号 (4種)

タイトル防衛戦型: **現保持者が条件試合に出て負けた時のみ、勝者に移動**。
保持者が試合に出ない限り変動しない。空席なら最初の条件試合の勝者が取得。

| 称号 | 絵文字 | 移動条件 (qualifying match) | 保持資格 (eligibility) |
|---|---|---|---|
| **球聖** | 🥇 | ノーハンデ試合 | 全員 |
| **中ボス** | 👹 | ノーハンデ試合 | レート < 1700 のみ。 holder のレートが 1700 以上に達したら卒業 (vacate) |
| **ネイバーズカップ** | 🏆 | gameType = 8ball (ハンデ可) | 全員 |
| **ワールドカップ** | 🏅 | gameType = 9ball (ハンデ可) | 全員 |

ロジック (各 qualifying match 後):

1. 試合参加者に **現保持者がいない** → 称号は変化しない
2. 試合参加者に現保持者がいる + **保持者が勝った** → 防衛 (変化なし)
3. 試合参加者に現保持者がいる + **保持者が負けた** + **勝者が eligible** → **奪取**
4. 試合参加者に現保持者がいる + **保持者が負けた** + **勝者が eligible でない** → 失効 (空席へ)
5. **空席状態 + qualifying match の勝者が eligible** → 獲得
6. **(中ボス特殊)** 試合後、現保持者のレートが 1700 以上 → 卒業 (空席へ)。同試合で qualifying かつ勝者 eligible なら新規割当も行う

### 7.2 算出型称号 (4種)

各試合登録後 (および GET /api/titles 読み取り時) に DB を再スキャンして自動更新。

| 称号 | 絵文字 | 算出ルール | 補足 |
|---|---|---|---|
| **新人王** | 🌱 | 登録 2ヶ月以内 (createdAt > now - 60日) のプレイヤーで、登録以降の勝利数最多 | 0 勝なら空席 |
| **週刊王** | 📅 | 直近 7日間の勝利数最多 | 0 勝なら空席 |
| **戦闘強** | ⚔️ | 直近 30日間の勝利数最多 | 0 勝なら空席 |
| **連勝王** | 🔥 | 直近 30日間内で最長連勝 (≥ 2) | 連勝 1 のみは対象外 |

タイブレーク: 同点の場合は最初に達成した方 (= matches を時系列順に走査して先に勝った人)。

### 7.3 ワールドネイバーズ (2冠)

シェアハウス名「ワールドネイバーズ」に由来する特別称号扱い:

- **同一プレイヤーが「ワールドカップ」と「ネイバーズカップ」を同時保持した瞬間に達成**
- UI 上の派手演出:
  - 試合結果画面: フルスクリーン金色グラデーションカード + パルスアニメーション + 「{prefix} が2冠達成!!」
  - ホーム上部: 「✨ ワールドネイバーズ ✨」リボン (該当者がいる間ずっと表示)
  - LINE 通知: 「🏆🏆🏆 ワールドネイバーズ達成!! 🏆🏆🏆」を冒頭に追加

### 7.4 称号変更時の通知

LINE 通知の試合結果メッセージ末尾に「✨ 称号の動き ✨」セクションを追加。各エントリ:
- `🥇 球聖: 田中 が 山田 から奪取！` (奪取)
- `🏆 ネイバーズカップ: 鈴木 が獲得！` (新規・空席から)
- `👹 中ボス: 山田 が卒業 (空席)` (1700 到達)
- `🥇 球聖: 田中 から失効 (空席)` (eligible でない勝者)

### 7.5 データモデル

```
Title {
  code              // 主キー (kyusei | mid_boss | neighbors_cup | world_cup | rookie | weekly | sento_kyou | streak)
  name
  emoji
  category          // possession | computed
  description
  holderPlayerId    // 現保持者 (nullable)
  acquiredAt
  updatedAt
}

TitleHistory {
  id, titleCode, playerId, matchId, acquiredAt
  reason            // match | init | graduation | computed | vacated
}
```

### 7.6 試合登録時の処理フロー

1. レート計算 (Glicko-2) + Match レコード作成 (1トランザクション)
2. `updateTitlesAfterMatch(match, winner, loser)` を呼び出し
   - 保持型 4 種類を順番に評価し、変更があれば setHolder + TitleHistory にログ
   - 算出型 4 種類を再計算 (winner 集計 + 連勝計算)
3. ワールドネイバーズ判定 (winner が world_cup と neighbors_cup の両方を保持しているか)
4. 試合結果レスポンスに `titleChanges[]` と `isWorldNeighbors` を載せる
5. LINE 通知 (称号変動 + 2冠演出を含む)

| DB | SQLite (MVP) → 永続化が必要なら Postgres |

| Glicko-2 計算 | npm `glicko2` |
| LINE 通知 | `@line/bot-sdk` (Messaging API) |
| デプロイ | Vercel (or 自宅サーバー) |
| 認証 | MVP は管理画面のみ Basic Auth、将来 LINE Login |

→ 別途 BL を立てて選定確定。

## 10. 非機能要件

| 項目 | 要件 |
|---|---|
| 想定同時接続 | 20人 × 散発的アクセス。負荷は無視できる |
| 可用性 | ベストエフォート。ダウンしても住民に怒られない範囲 |
| データ保全 | DB は週次バックアップ (CSV ダンプ程度で十分) |
| セキュリティ | 管理画面は最低限のパスワード保護。試合登録は性善説 (将来 LINE Login で強化) |
| レスポンス | 主要画面 1秒以内 |
