# シェアハウス・ビリヤードランキングシステム MVP 仕様書

**Project**: 生活管理 / シェアハウス・ビリヤードランキング
**BL**: BL-0001
**Version**: 0.1 (MVP)
**Date**: 2026-05-09

---

## 1. 概要

シェアハウス共有スペースのビリヤード台で行われる住民同士の真剣勝負を記録し、Glicko-2 ベースのレーティングでランキング化する Web アプリケーション。試合結果は LINE グループに自動投稿され、プレイヤーだけでなく非プレイヤー住民もコミュニティの話題として楽しめる。

### 1.1 目的

1. ビリヤードブームを一過性で終わらせず長続きさせる
2. 上達実感と目標を提供し、競技志向プレイヤーの熱量を保つ
3. 「ランキング戦」という社会的口実装置で 1on1 真剣勝負を仕掛けやすくする
4. 試合結果通知で非プレイヤーも巻き込みコミュニティを盛り上げる

### 1.2 想定スケール

- 想定参加人数: **20人以内**
- 想定対戦頻度: 不定 (人によって偏る前提)
- シェアハウス内の単一テーブルで運用

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| **プレイヤー** | システムに登録されたユーザー (シェアハウス住民) |
| **管理者** | 自分。プレイヤー登録・初期レート設定・シーズン管理権限を持つ |
| **アンカープレイヤー** | 外部 FargoRate を持っており、コミュニティの絶対実力換算の基準となるプレイヤー (= 管理者本人、Fargo ≈ 470) |
| **ランキング戦** | 開始前に「ランキング戦」として登録された真剣勝負試合のみ。ランキングに反映される |
| **カジュアル戦** | 記録対象外の遊び試合 |
| **レート** | Glicko-2 のレーティング値 (R) |
| **RD** | Rating Deviation。レートの不確実性 |
| **σ** | Volatility。レート変動の揺らぎ |
| **推定 FargoRate** | アンカーを基準に Glicko-2 レートから換算した推定 FargoRate |
| **シーズン** | ランキング集計の期間。終了時にアーカイブされる |
| **ハンデ** | 実力差を埋める試合条件 (フリーボール/ゲームボール拡張等) |

## 3. 機能要件

### 3.1 プレイヤー管理

- 管理者は Web UI からプレイヤーを登録・編集・無効化できる
- プレイヤー属性: `名前 / 表示名 / 初期レート / 初期RD / 初期σ / アクティブ状態 / 備考`
- 初期レートは管理者が見立てで決定 (デフォルト値: 1500、レンジ: 1000〜2200 程度)
- アンカープレイヤーは1名指定でき、`is_anchor: true` フラグを持つ

### 3.2 試合登録 (ランキング戦)

- Web UI のフォームから誰でも登録可能 (MVP では認証なし)
- 入力項目:
  - **対戦者A** (ドロップダウン)
  - **対戦者B** (ドロップダウン)
  - **勝者** (A/B のラジオボタン)
  - **ゲーム種別** (8ball / 9ball / 10ball / その他、自由テキスト) — レート計算には影響しない (Q4: 統合レート)
  - **ハンデ条件** (チェックボックス + 数値入力、後述)
  - **メモ** (任意自由テキスト)
- 登録時に Glicko-2 で両者のレートを更新し、結果を DB に保存
- 結果は LINE 公式アカウント経由でグループに自動 Push (3.5節)

### 3.3 ランキング表示

- 現シーズンのランキング: レート降順、上位から表示
- 各プレイヤー行に表示:
  - 順位 / 表示名 / 現レート / RD / 推定 FargoRate / 試合数 / 直近試合結果 (勝敗3件)
- ランキングは公開ページとして誰でも閲覧可能 (URL 共有可)

### 3.4 プレイヤー詳細

- 個別プレイヤーの履歴ページ
- レート推移グラフ (時系列)
- 試合履歴一覧 (対戦相手・勝敗・レート変動・ハンデ)
- 推定 FargoRate と Glicko-2 レートの両方を表示

### 3.5 LINE 通知

- LINE Messaging API (公式アカウント) でシェアハウスのグループに送信
- 試合登録時に下記フォーマットの flex message or text を Push
  ```
  🎱 ランキング戦結果
  勝者: {winner_name} ({winner_rate_before} → {winner_rate_after})
  敗者: {loser_name} ({loser_rate_before} → {loser_rate_after})
  ハンデ: {handicap_summary}
  種目: {game_type}
  順位変動: {rank_change_summary}
  ```
- メッセージ末尾にランキングページの URL を添付

### 3.6 シーズン管理

- 管理者がシーズンを作成・終了できる
- シーズン属性: `名前 / 開始日 / 終了日 / アーカイブ済みランキング`
- シーズン終了時:
  - 終了時点のランキングをスナップショットとして保存
  - 新シーズン開始時に各プレイヤーのレートを **70% リセット** (詳細 §5.5)
- 推奨シーズン長: **3ヶ月** (要検討、初回は様子見で 1〜2ヶ月でも可)

### 3.7 ハンデ自動提案 (Phase 1.5)

- 試合登録フォームで対戦者2人を選んだ際、レート差から推奨ハンデを提示
- §6.4 のテーブルを参照

## 5. レーティング仕様 (Glicko-2)

### 5.1 採用根拠

- 少人数・対戦数偏りに強い (Q1 比較で詳述)
- RD/σ により新規プレイヤーが急速収束、長期不在者は次回試合での変動が大きく許容される
- 業界実装が豊富 (Lichess, CS:GO 等)

### 5.2 パラメータ

| パラメータ | 初期値 | 説明 |
|---|---|---|
| R | 管理者指定 (default 1500) | 初期レート |
| RD | 350 | 初期不確実性 (Glicko-2 標準) |
| σ | 0.06 | 初期 volatility |
| τ | 0.5 | システム定数 (Glicko-2 標準) |
| Rating Period | **1試合ごと** | バッチではなく試合ごとに更新 |

### 5.3 計算

標準 Glicko-2 アルゴリズムをそのまま採用。npm `glicko2` または同等ライブラリ利用。試合ごとに `RatingPeriod = [対戦相手1名]` で更新。

### 5.4 ハンデあり試合の変動量緩和

- ハンデが付いた試合は、計算後の変動量を **50% に圧縮** する
  ```
  Δrating = R_new - R_old
  if (handicap_applied) {
      R_final = R_old + Δrating * 0.5
      RD_final = sqrt(RD_old^2 + (RD_new^2 - RD_old^2) * 0.5)
      σ_final = σ_old + (σ_new - σ_old) * 0.5
  }
  ```
- これは Q2 案A (素直適用) + 緩和策

### 5.5 シーズンリセット (Soft Reset)

シーズン切替時、各プレイヤーのパラメータを以下で更新:

```
R_new = 1500 + (R_old - 1500) * 0.7    -- 平均に向けて 30% 引き戻し
RD_new = max(RD_old, 200)              -- RD を200以上に膨らませる
σ_new = σ_old                          -- σ は据え置き
```

これにより前シーズンの成績が完全リセットされず、しかし新シーズンで実力差が再評価される余地を作る。

### 5.6 推定 FargoRate 換算

```
estimated_fargo(player) = anchor.external_fargo + (player.rating - anchor.rating) * K_fargo
```

- `K_fargo = 0.5` (初期値、要校正)
- アンカープレイヤーは常に実 FargoRate を表示
- 換算係数 K_fargo は年1回程度、アンカーの外部試合実績で校正

## 6. ハンデ仕様

### 6.1 ハンデの種類 (MVP)

| コード | 名称 | 入力 |
|---|---|---|
| H1 | ゲームボール拡張 | 対象球リスト (例: 8,9) |
| H2 | フリーボール権 | 1試合あたり回数 (整数) |
| H3 | 持ち点差 | スタート点数 (弱者側) |
| H5 | ブレイク優先権 | boolean (常に弱者ブレイク) |
| H7 | 道具制限 | テキスト (マッセ禁止 等) |

> H4 (狙える球制限) と H6 (持ち時間制) は MVP では除外、必要が出たら追加。

### 6.2 データ表現

`Match.handicap` フィールド (JSON):

```json
{
  "advantaged_player_id": "<弱者のplayer_id>",
  "h1_game_balls": [8, 9],
  "h2_freeball_count": 1,
  "h3_point_spot": 2,
  "h5_break_priority": true,
  "h7_restrictions": "マッセ禁止"
}
```

ハンデなしの場合は `null` または空オブジェクト。

### 6.3 レートへの反映

- 6.2 のいずれかが入っていれば §5.4 の 50% 緩和を適用
- ハンデの種類による補正は MVP では行わない (将来 Phase 2 で導入)

### 6.4 自動提案ロジック (Phase 1.5)

試合登録フォームで対戦者を選んだ際、Glicko レート差から推奨ハンデを提示:

| Glicko レート差 | 推定 Fargo差 | 推奨ハンデ |
|---|---|---|
| 0〜100 | 0〜50 | なし |
| 100〜200 | 50〜100 | H5 (弱者ブレイク優先) |
| 200〜400 | 100〜200 | H1 (8/9 OK) または H2 (1回) |
| 400〜600 | 200〜300 | H1 + H3 (1点) |
| 600〜 | 300〜 | H1 + H3 (2点) + H2 (1回) |

提案であって強制ではない。プレイヤーが合意したハンデで登録する。

## 7. UI / 画面構成

| 画面 | URL | 内容 |
|---|---|---|
| ホーム / ランキング | `/` | 現シーズンランキング表 |
| プレイヤー詳細 | `/player/:id` | レート推移・試合履歴 |
| 試合登録 | `/match/new` | フォーム + ハンデ提案 |
| 試合一覧 | `/match` | 全試合履歴 |
| 管理 (admin) | `/admin` | プレイヤー登録・シーズン管理 (パスワード保護程度) |
| シーズンアーカイブ | `/season/:id` | 過去シーズンの最終ランキング |

### デザイン方針

- スマホ片手で 5秒以内に試合登録できる UX
- ランキング画面は LINE のリンクから流入する非プレイヤーが理解できる構成
- 「ランキング戦中」を明示する装飾 (現在登録中の試合を示すバナー等) は Phase 2

## 9. 非機能要件

| 項目 | 要件 |
|---|---|
| 想定同時接続 | 20人 × 散発的アクセス。負荷は無視できる |
