## P0：数字の整合性（最優先）

### P0-1【BUG】売上データが1ヶ月ずれている疑い

**調査したこと**:
- 4月度月報HTML（910行）の数値を、データスナップショット `pl_2026_PROD.json` / `sales_april_PROD.json` と突合
- Dashboard 本番DBスナップショット `pl_2026_PROD.json` 内 `daily_sales` 配列を date 昇順で全行精査
- Dashboard リポジトリの月レンジ計算ロジック（`lib/db/repositories/plDashboardRepository.ts` の `getMonthRange()`）を確認
- POS の `entryAt` カラムの型と取り扱いを Prisma schema（`prisma/schema.prisma`）で確認

**分かったこと** ✅:
1. 4月度月報PDFの「PL売上 ¥3,024,490」は、`daily_sales` の **2026-04-01 から 2026-04-30 のみ** の合計と**完全一致**。月報生成側で5月データの混入は**ない**。
2. ただし、本番DBスナップショット `pl_2026_PROD.json` には **2026-05-01 (¥48,870)** と **2026-05-02 (¥41,170)** の総売上エントリが含まれている（計 ¥90,040）。これは月跨ぎ営業（深夜会計）由来の可能性が高い。
3. Dashboard の月レンジ計算は次の構造:
   ```ts
   function getMonthRange(yearMonth: string): { start: Date; end: Date } {
     const [year, month] = yearMonth.split('-').map(Number);
     const start = new Date(year, month - 1, 1);   // ★ ローカルTZ
     const end = new Date(year, month, 0);          // ★ ローカルTZ
     return { start, end };
   }
   ```
   - **これは「サーバTZのローカルタイム」で月初・月末を生成**。Vercel本番(UTC)とローカル(JST)で挙動が異なる。
   - JSTサーバで動かすと `start` は `Mar 31 15:00 UTC` 相当、`end` は `Apr 29 15:00 UTC` 相当に丸まり、Prisma → PostgreSQL `@db.Date` カラムへの送信時に **境界の挙動が処理系依存** になる。
4. `SalesData.entryAt` は `Timestamptz` で UTC 保存される。早朝（0–9時 JST）の会計は UTC で **前日扱い**になるため、「営業日」概念で集計する場合に追加処理が必要。

**分からなかったこと** ❌:
- 「5/3 が4月として表示されている」具体的な再現条件。**現状のスナップショットには 5/3 のエントリ自体が存在しない**（5/1, 5/2 のみ）。MTG時点のダッシュボードを再現するには本番DBの当時のスナップショット or 直接照会が必要。
- 営業日基準 vs 会計時刻基準のどちらで集計されているか、Dashboard コード側で明示的に切り分けているかは未確認。

**仮説と推奨アクション** 🟡:
1. **境界バグの可能性大**: `getMonthRange()` を「`yearMonth` 文字列ベースで `date >= '2026-04-01' AND date <= '2026-04-30'` を発行」する形に書き換える（JS Date 経由を回避）。
2. **営業日定義の合意**: `entryAt` を JST に変換した上で **「JST 5:00 を日付分割点」** にする標準ロジックを `lib/utils/date.ts` に集約。
3. 確認用クエリ:
   ```sql
   SELECT date, sum(amount) FROM pl_daily_sales
   WHERE store_id = '<bfa-001>'
     AND date BETWEEN '2026-04-25' AND '2026-05-05'
   GROUP BY date ORDER BY date;
   ```

### P0-4-① P.9 平均客単価 ¥6,177 の正体

**調査したこと**:

**推奨注記**:

## 横断的に分かった「指示書には書かれていない」リスク

1. **取込パイプラインの分散**: 月報生成パイプライン（`Flow/.../月報/scripts/`）と Dashboard 本体（`~/RestaurantAILab/Dashboard/`）が**別リポジトリ**。同じ「4月度数字」が**2系統で別々に計算されている**。整合性担保のための SOT（Single Source of Truth）定義が未確立。
2. **TZ取り扱いの統一不足**: `entry_at @db.Timestamptz` と `date @db.Date` が混在。JS Date 経由のフィルタは boundary バグの温床（P0-1 参照）。`lib/utils/date.ts` での JST 正規化を**必須レイヤー**として強制すべき。
3. **assumptions.md と PDF の値の食い違い**: assumptions に「PL売上 ¥2,935,690」と書かれているが、PDFは ¥3,024,490。**差分 ¥88,800**。本番DBスナップショットの 5/1+5/2 ぶん（¥90,040）に近いが完全一致しない。**スナップショット取得タイミングの差**が原因の可能性。これも SOT 問題。
4. **メニューマスタの不備**: ボトル/グラスの販売形態、コース内訳商品、タグ商品（インバウンド等）の整備が `MenuMaster.menuType` レベルで未着手。**P1全件のボトルネック**になる。
