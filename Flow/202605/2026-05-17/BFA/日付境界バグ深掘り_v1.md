# 日付境界バグ 深掘り調査（P0-1）

- **作成日**: 2026-05-17
- **対象指摘**: 「ダッシュボードの『PL売上の見立てデータ』が 5/3 分を4月として表示している」（5/4 BFA定例 / 竹矢氏 / 文字起こし 31:50 付近）
- **前提**: 一次調査 v1（`一次調査_月報ダッシュボード改善_v1.md`）の P0-1 を構造レベルで追跡
- **手法**: コード静的解析 + Node.js での **TZ別 boundary 挙動の実機検証**

---

## 結論（先に）

| 項目 | 結論 |
|---|---|
| **Read 側の構造的バグ** | ✅ **存在する**。Dashboard の月集計クエリ `getMonthRange()` は **ローカルTZ依存の JS Date コンストラクタ**を使用 → **JST 環境で動かすと "April" 査読範囲が `2026-03-31 〜 2026-04-29` にずれる**ことを実機検証で確認。Vercel本番(UTC)では正しく動くため "production では再現しない" タイプの環境依存バグ。 |
| **Write 側の構造的バグ** | 🟡 **限定的に安全**。API ルートで `YYYY-MM-DD` 形式バリデーションがあり、`new Date('YYYY-MM-DD')` は **仕様上 UTC 解釈**のため、APIフォーム経由なら安全。**ただし** `lib/utils/date.ts:parseVisitDate()` は `YYYY/MM/DD` を **ローカルTZ解釈**しており、CSV経由のパスは危険。 |
| **コードベース全体の TZ統一** | ❌ **未統一**。`Date.UTC(...)` 系（安全）と `new Date(y, m-1, d)` 系（ローカルTZ依存）が**混在**。`lib/csv/mapper.ts` や `dailyStoreSummaryRepository.ts` は UTC 明示、`plDashboardRepository.ts` / `salesRepository.ts` / `pl/calculator.ts` / `line/sales-context.ts` は局所TZ依存。 |
| **「5/3 が4月に表示」の最有力仮説** | 🟡 **Write側で発生した日付シフト**（後述「シナリオC」）。Vercel本番(UTC)であれば Read 側の月境界は正しいが、その手前で **既にデータ書込時に `2026-05-01` が `2026-04-30` に丸められて保存**された可能性が高い。竹矢氏の言葉「5月3日分」は厳密日付ではなく "5月の頭の方のデータ" の包括表現と理解するのが自然。 |

## 1. コード上の問題箇所マップ

### 1.1 危険（ローカルTZ依存）

| ファイル | 行 | コード | 影響 |
|---|---|---|---|

| `lib/db/repositories/salesRepository.ts` | 187-188 | `new Date(year, month-1, 1)` / `new Date(year, month, 0, 23,59,59,999)` | POS月次集計（客単価、商品ランキング等の基礎） |

| `lib/db/repositories/plDashboardRepository.ts` | 28-33, 116, 188 | `getTodayDateStr()` は `now.getFullYear()/getMonth()/getDate()` | **ローカルTZ依存の "今日"** → 深夜帯にズレる |
| `lib/csv/airregiParser.ts` | (CSV取込) | （未確認） | エアレジCSVの日付パース。要追加調査 |

### 1.2 安全（UTC明示）

| ファイル | 行 | コード |
|---|---|---|
| `lib/csv/mapper.ts` | 80 | `new Date(Date.UTC(year, month, day))` |
| `lib/csv/transformer.ts` | 313 | `new Date(Date.UTC(year, month, day))` |
| `lib/db/repositories/dailyStoreSummaryRepository.ts` | 103-104 | `new Date(Date.UTC(year, month-1, 1))` |
| `lib/utils/date.ts:parseJSTtoUTC()` | 135-155 | 明示的 `+09:00` 付加で JST→UTC変換 |
| `scripts/import-pl-data.ts` | 251, 276, 303 | `new Date(dates[i] + 'T00:00:00Z')` ← **明示UTC** |

→ **同じ「月の最初の日」を取る処理が、ある場所では UTC、別の場所ではローカルTZで作られている**。これが本件の構造的バグの正体。

### 1.3 API レイヤでの保護

```ts
// app/api/pl/daily-sales/route.ts:89-93
if (!body.date || !/^\d{4}-\d{2}-\d{2}$/.test(body.date)) {
  return NextResponse.json({ ..., '日付形式が不正です (YYYY-MM-DD)' }, { status: 400 });
}
```

→ Write API は `YYYY-MM-DD` 文字列を強制。これが**ローカルTZ依存バグから救っている**唯一のレイヤ。`new Date('2026-05-03')` は ECMA-262 仕様で **必ずUTC midnight** に解釈されるため、これだけで日付がぶれない。

**ただし**: フロントの datepicker が `Date` オブジェクトを `JSON.stringify()` した場合（→ `toISOString()` → 'YYYY-MM-DDTHH:MM:SS.sssZ' 形式）API バリデーションは弾く。フロント実装が `formatISODate(date)`（`date-fns` の `format('yyyy-MM-dd')`）を経由しているかは要確認。

## 2. 実機検証（Node.js）

### 2.1 セットアップ

`getMonthRange()` の挙動を **TZ=UTC / TZ=Asia/Tokyo / TZ=America/Los_Angeles** の3パターンで実行して比較。
Prisma が `@db.Date` に対して送る値の挙動も再現（`Date.toISOString().split('T')[0]`）。

### 2.2 結果

#### TZ=UTC (Vercel本番のデフォルト)

```
getMonthRange("2026-04"):
  start: 2026-04-01T00:00:00.000Z  → Prisma @db.Date: 2026-04-01 ✓
  end  : 2026-04-30T00:00:00.000Z  → Prisma @db.Date: 2026-04-30 ✓

new Date('2026-05-03')            → @db.Date: 2026-05-03 ✓
new Date(2026, 4, 3)              → @db.Date: 2026-05-03 ✓
```

→ **問題なし**。

#### TZ=Asia/Tokyo (開発機 / 田中さんのMac等)

```
getMonthRange("2026-04"):
  start: 2026-03-31T15:00:00.000Z  → Prisma @db.Date: 2026-03-31 ✗  ←★
  end  : 2026-04-29T15:00:00.000Z  → Prisma @db.Date: 2026-04-29 ✗  ←★

new Date('2026-05-03')            → @db.Date: 2026-05-03 ✓ (YYYY-MM-DD is spec-UTC)
new Date(2026, 4, 3)              → @db.Date: 2026-05-02 ✗  ←★ (local TZ shifts back 9h)
new Date(2026, 4, 1)              → @db.Date: 2026-04-30 ✗  ←★ (May 1 stored as April 30!)
```

→ **致命的な3つのズレ**:
1. **April 月レンジが March 31 〜 April 29 にずれる** → 4/30 のデータが落ち、3/31 のデータが混入
2. **`new Date(2026, 4, 3)` (May 3 local) が May 2 として保存される**
3. **`new Date(2026, 4, 1)` (May 1 local) が April 30 として保存される** ← **これが「5月データが4月に紛れ込む」現象を直接生む**

#### TZ=America/Los_Angeles (参考)

```
getMonthRange("2026-04"):
  start: 2026-04-01T07:00:00.000Z  → Prisma @db.Date: 2026-04-01 ✓
  end  : 2026-04-30T07:00:00.000Z  → Prisma @db.Date: 2026-04-30 ✓
```

→ 西側TZでも読み側は問題なし（UTC との時差が東向きではないため）。

## 3. バグ発生シナリオの場合分け

### シナリオA: Vercel 本番（UTC）のみでデータ Read

- **読み側**: `getMonthRange()` は正しい（`2026-04-01 〜 2026-04-30`）
- **書き側**: API経由（`YYYY-MM-DD` バリデーション）なら正しい
- **結論**: バグは起きない。4月ビューに5月データは混入しない。

### シナリオB: ローカル開発機（JST）で Read

- **読み側**: April フィルタが `2026-03-31 〜 2026-04-29` に**狭まる**
- **結果**: April の合計が **4/30分だけ過少**、**3/31 分だけ過多** に出る
- **5月データの混入は無い**（範囲が逆方向にずれているため）
- → **これは「5/3 が4月に出る」現象とは方向が逆**

### シナリオC: 書き込み時に局所TZでDateを構築（JST環境）★ 最有力

- **書き側**: フロントエンドが `new Date(2026, 4, 1)` 形式で「5月1日」を表現 → `JSON.stringify` → `'2026-04-30T15:00:00.000Z'` を API へ送信
- **API バリデーション**: `/^\d{4}-\d{2}-\d{2}$/` 正規表現には**マッチしない** → 400エラーで弾かれるはず
- **ただし**: フロントが ISO文字列の頭9文字 `'2026-04-30'` だけ抽出して送信していたら、API は通る → DB は `2026-04-30` として保存
- **結果**: 「5月1日」と意図された売上が **4月30日のレコードとして格納** → April ダッシュボードビューに紛れ込む

→ 竹矢氏の指摘「5/3 が4月」は、厳密には **「5月初頭のデータが4月にずれて入っている」** を直感的に「5/3 分」と言った可能性が高い。実データには `pl_daily_sales` に `2026-05-01`(¥48,870) と `2026-05-02`(¥41,170) が**正しい日付で残っており**、本来なら4月ビューに混入しないはず。仮に上記シナリオCで「5月1日」が `2026-04-30` として保存され、その後修正されて `2026-05-01` の正しいレコードが追加された結果、**両方のレコード**が存在する状態になっていたら、4月ビューに ¥48,870 等が混入する形になる。**要 DB の重複検査**。

### シナリオD: スプレッドシートから手動コピペ / 月またぎCSV

- スプレッドシート (`新PL管理（移管前）`) のセル日付が **シリアル値で取り込まれ**、Node.jsの Date 変換時に JST解釈されたら、シナリオCと同じ事象が発生する。
- `import-pl-data.ts` は `'T00:00:00Z'` 明示なので**安全**。ただし**他に類似スクリプトが存在するか**は要確認（指示書には「BPO側で売上分析が止まっていた」とあり、独自スクリプト/手動運用の余地が大きい）。

### シナリオE: PL シート → ダッシュボード移管時の Date 変換

- 1〜3月は `pl_sheet_jan_mar.json`（スプレッドシート由来）、4月以降は本番DB（`plDailySales`）と**ソースが切り替わる**前提。
- 4月の途中で移管が起きた場合、**移管スクリプトのDate構築方法**が JST 局所TZ依存なら、移管直後の数日分がシフトする可能性。

## 4. 確認用クエリ（本番DBに直接当てるべきもの）

```sql
-- 4/25 ～ 5/5 のレンジで、総売上カテゴリの日次値を見る
SELECT
  date,
  SUM(amount) AS total_amount
FROM pl_daily_sales
WHERE store_id = (SELECT id FROM stores WHERE store_code = 'bfa-001')
  AND category_id = (SELECT id FROM pl_sales_categories
                     WHERE store_id = (SELECT id FROM stores WHERE store_code = 'bfa-001')
                       AND name = '総売上')
  AND date BETWEEN '2026-04-25' AND '2026-05-05'
GROUP BY date
ORDER BY date;
```

期待値（snapshot から）:
```
2026-04-25  239,200
2026-04-26   78,610
2026-04-27   88,310
2026-04-28  238,850
2026-04-29        — (no data)
2026-04-30  396,430
2026-05-01   48,870
2026-05-02   41,170
2026-05-03        ? (snapshot時点では未投入)
```

**チェックすべき異常パターン**:
1. `2026-04-30` の値が ¥396,430 と乖離している → **シナリオCが発生済**の証拠
2. `2026-04-30` に複数行ある（`SELECT date, COUNT(*)` での重複） → **同上**
3. `2026-05-01` 〜 `2026-05-03` 各日に、片方は数百円〜数千円、もう片方は数万円 等、二段書きされている → **シナリオCで4/30に書かれ、後で5/1に再投入された**痕跡

### 補助クエリ（重複検出）

```sql
SELECT date, category_id, COUNT(*) AS row_count, ARRAY_AGG(amount) AS amounts
FROM pl_daily_sales
WHERE store_id = (SELECT id FROM stores WHERE store_code = 'bfa-001')
```

## 5. 修正提案（実装ガイド）

### 5.1 即時パッチ（最小侵襲）

`lib/db/repositories/plDashboardRepository.ts` の `getMonthRange()` を **文字列直接生成**に変える:

```ts
// Before — ローカルTZ依存
function getMonthRange(yearMonth: string): { start: Date; end: Date } {
  const [year, month] = yearMonth.split('-').map(Number);
  const start = new Date(year, month - 1, 1);
  const end = new Date(year, month, 0);
  return { start, end };
}

// After — UTC固定（@db.Date と整合）
function getMonthRange(yearMonth: string): { start: Date; end: Date } {
  const [year, month] = yearMonth.split('-').map(Number);
  const lastDay = new Date(Date.UTC(year, month, 0)).getUTCDate();
  const start = new Date(`${yearMonth}-01T00:00:00.000Z`);
  const end   = new Date(`${yearMonth}-${String(lastDay).padStart(2,'0')}T00:00:00.000Z`);
  return { start, end };
}
```

同じパッチを次にも適用:
- `lib/db/repositories/salesRepository.ts:187-188`
- `lib/pl/calculator.ts:13-26`
- `lib/line/sales-context.ts:43, 67-74`
- `lib/utils/date.ts:getMonthStart/End`

### 5.2 構造的修正（推奨）

`lib/utils/date.ts` に **単一のソース・オブ・トゥルース**を作る:

```ts
/** 営業日基準で 'YYYY-MM-DD' を返す（JST 5:00 を日付分割点とする） */
export function toBusinessDate(timestamp: Date | string, splitHourJst = 5): string { ... }

/** 月レンジを @db.Date 互換の Date オブジェクトで返す（UTC固定） */
export function monthRangeUTC(yearMonth: string): { start: Date; end: Date } { ... }

/** 'YYYY-MM-DD' → UTC midnight の Date（datetime カラム用） */
export function parseDateOnlyUTC(s: string): Date { ... }
```

すべての repository / calculator / context 系を上記ユーティリティに集約。**`new Date(year, month-1, day)` 形式を ESLint で禁止**するルールも入れる（`no-restricted-syntax` で `NewExpression[callee.name="Date"][arguments.length>=3]` を禁止）。

### 5.3 営業日（business day）境界の明文化

- BFAの実営業時間は夜〜深夜。`entryAt` を **JST5時を分割点として日付化**するヘルパで統一。
- POS集計（客数、売上、出数）はすべて「営業日」基準で行う。
- 月境界は「**営業日の最初の日から最後の日**」で定義。

```ts
export function jstBusinessDate(ts: Date, splitHourJst = 5): string {
  const jst = new Date(ts.getTime() + 9 * 3600 * 1000);
  // 日付の hour < splitHour なら前日扱い
  if (jst.getUTCHours() < splitHourJst) jst.setUTCDate(jst.getUTCDate() - 1);
  return jst.toISOString().split('T')[0];
}
```
