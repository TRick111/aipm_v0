## 2. ★ 月合計クエリの実機 A/B 比較 — バグ確定

同じ「2026年4月の総売上合計」を、Dashboard 既存コードと同じ書き方 vs UTC 固定で投げた結果:

### A. Dashboard 既存実装 (`new Date(2026, 3, 1)` / `new Date(2026, 4, 0)`)

```
start = 2026-03-31T15:00:00.000Z  (= JST 2026-04-01 00:00:00)
end   = 2026-04-29T15:00:00.000Z  (= JST 2026-04-30 00:00:00)
rows = 26
sum  = ¥2,628,060   ← ★ 4/30 の ¥396,430 が落ちている
```

### B. 推奨実装 (`new Date('2026-04-01T00:00:00.000Z')` / `'2026-04-30T00:00:00.000Z'`)

```
start = 2026-04-01T00:00:00.000Z
end   = 2026-04-30T00:00:00.000Z
rows = 27
sum  = ¥3,024,490   ← 月報HTML値と一致
```

### 差分の正体

```
差分 = ¥3,024,490 − ¥2,628,060 = ¥396,430
4/30 単日値 (b715f786…) = ¥396,430
→ 完全一致
```

**つまり、JST 環境で Dashboard を動かすと、4月ビューから 4/30 の ¥396,430 が**完全に欠落する****。

### なぜ Vercel本番では正常に見えるか

- Vercel サーバランタイムは **UTC** が標準。
- UTC で `new Date(2026, 3, 1)` → `2026-04-01T00:00:00.000Z`（正常）
- UTC で `new Date(2026, 4, 0)` → `2026-04-30T00:00:00.000Z`（正常）
- → 本番では症状が出ない。**開発機（ローカル `pnpm dev`）でのみ症状が出る**典型的な**環境依存バグ**。

### 5/4 MTG の状況に照らすと

竹矢氏が見ていたダッシュボードが、

- (1) **田中さんの開発機 (JST) で起動された画面**だった場合 → 4月合計から 4/30 の ¥396,430 が落ちて表示されていた可能性が高い
- (2) **Vercel 本番にデプロイされた版**だった場合 → このバグは起きない（別の原因を探す必要あり）

どちらだったか、田中さん本人に要確認。

## 4. POS `sales_data` の UTC/JST 跨ぎ実態

4/28〜5/5 の POS 60レコード中、**UTC日付 ≠ JST日付** のものを抽出:

| entryAt (UTC) | UTC date | JST date | 跨ぎ | 金額 |
|---|---|---|---|---|
| 2026-04-28T15:18:58Z | 04-28 | **04-29** | ★ | ¥28,500 |
| 2026-04-28T15:34:23Z | 04-28 | **04-29** | ★ | ¥13,600 |
| 2026-04-28T15:50:52Z | 04-28 | **04-29** | ★ | ¥13,100 |
| 2026-04-28T16:30:06Z | 04-28 | **04-29** | ★ | ¥10,500 |
| 2026-04-28T17:02:59Z | 04-28 | **04-29** | ★ | ¥13,800 |
| 2026-04-28T17:03:52Z | 04-28 | **04-29** | ★ | ¥2,000 |
| 2026-04-28T17:05:05Z | 04-28 | **04-29** | ★ | ¥-12,800 (返品) |
| **2026-04-30T15:02:54Z** | **04-30** | **05-01** | ★ | ¥4,300 |
| **2026-04-30T15:21:09Z** | **04-30** | **05-01** | ★ | **¥50,000 (30人)** |
| 2026-05-01T15:12:21Z | 05-01 | **05-02** | ★ | ¥10,200 |
| 2026-05-04T15:01:22Z | 05-04 | **05-05** | ★ | ¥10,200 |

**重要な観察**:

1. **`RB0120260501002018000`**（accountId 上 5/1 0:20:18 を意味）の会計が、`entryAt=2026-04-30T15:21:09Z` で保存されている。これは **5/1 深夜営業の30人団体**（金額¥50,000）。
   - エアレジ的には「5/1 営業」
   - UTC 日付では「4/30」
   - JST 日付では「5/1」
   - accountId 由来日付では「5/1」
   - → **どの定義を採用するかで月帰属が変わる**

2. これらの月またぎ会計は、Dashboard 側で `entryAt` をそのまま UTC 日付で集計すると **「JST 5/1 の会計が UTC 4/30 として4月に算入される」** という現象を引き起こす。
   - **これこそが竹矢氏の言う「5月の会計が4月に表示されている」の正体**である可能性が高い

3. もし `dayOfWeek` カラムが JST ベースで保存されている（実データ確認: 上記 4/30 跨ぎは `dow=金`）一方、`entryAt` が UTC で保存されていると、**月選択フィルタの結果と曜日表示で齟齬が出る**。要追加検証。

## 6. 月またぎ営業の影響を試算

「JST 5/1 0:00〜4:59 の会計を **5/1営業日扱い**」とした場合の補正:

- UTC 4/30 の POS 集計 = 上記表より、UTC 4/30 で JST 5/1 跨ぎ分 = **¥4,300 + ¥50,000 = ¥54,300**
- 同じく UTC 5/4 で JST 5/5 跨ぎ分 = **¥10,200**
- → 4月の POS 集計を「UTC日付」基準でやってしまうと、5/1 営業（深夜）の **¥54,300 が4月に混入** している

これは PL daily_sales の話（categoryId='総売上' = ¥396,430）ではなく、**POS sales_data ベースの集計**に効く問題。`plDashboardRepository` は PL DB を見ているので**この問題は出ない**が、**POS 集計を使う章** (客単価、商品ランキング、時間帯分析) は影響を受ける。

月報HTMLの「客単価 ¥5,179」「POS売上 ¥3,024,490」がどちらの定義で計算されたかを再確認する必要がある（`assumptions.md` では「営業日: POS会計が1件以上発生した日（4月: 23日）」とあるが、UTC日付 vs JST日付の明示はない）。
