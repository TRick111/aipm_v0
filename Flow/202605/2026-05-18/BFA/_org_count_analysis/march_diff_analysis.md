## 2. 3月 旧 sales_202603.csv との取引No差分

旧データ ( `Flow/202605/2026-05-18/BFA/月報_202603/sales_202603.csv` 、`_convert_rawdata_to_airregi.py` 経由) は **193件**。
prod-clean 188件との diff:

### 2.1 旧データにあって prod-clean に無い: **7件**（全て **2026-02-28入店 / 会計日2026/03/01**）

| 取引No | 会計日 | 合計 | 人数 | 解釈 |
|---|---|---:|---:|---|
| RB0120260228183124000 | 2026/03/01 | ¥24,310 | 2 | 2/28 18:31入店、3/1未明会計 |
| RB0120260228183150000 | 2026/03/01 | ¥26,000 | 4 | 2/28 18:31入店、3/1未明会計 |
| RB0120260228190127000 | 2026/03/01 | ¥9,240 | 2 | 〃 |
| RB0120260228195615000 | 2026/03/01 | ¥16,390 | 2 | 〃 |
| RB0120260228200214000 | 2026/03/01 | ¥8,910 | 2 | 〃 |
| RB0120260228210316000 | 2026/03/01 | ¥17,490 | 4 | 〃 |
| RB0120260228221918000 | 2026/03/01 | ¥7,370 | 3 | 〃 |
| **小計** |  | **¥109,710** | **19** |  |

**原因**: これらは入店時刻が **2026-02-28 18:31〜22:19** で、業務日基準 (朝6時境界 / 5h offset) では 2026-02-28 のはず。
`_convert_rawdata_to_airregi.py` は `entry_jst - 5h` で `business_month` を出している。
2/28 18:31 - 5h = 13:31 → 2026-02 のはず。なのに sales_202603.csv に入っているのは、おそらく:
1. このコンバート時に target_month フィルタが正しく効いていなかった、または
2. rawdata.csv の `entry_at` カラム自体が exit_at (3/1 未明) を持っていた、または
3. 別系統で生成された CSV (例: 「会計日 = 3月のもの全部抽出」)

いずれにせよ **旧 sales_202603.csv は業務日基準と会計日基準が混在した汚染データ** で、これが「+22」の主要因 (うち +7)。

### 2.2 prod-clean にあって 旧データに無い: **2件**（両方 2026-03-31）

| 取引No | 会計日 | 合計 | 人数 | 解釈 |
|---|---|---:|---:|---|
| RB0120260331191710000 | 2026/03/31 | ¥81,770 | 13 | **13名大宴会** が欠落 |
| RB0120260331222919000 | 2026/03/31 | ¥4,680 | 0 | 0人 (テイクアウト?) |

**原因**: 旧データの基になった `rawdata.csv` は **2026-03-31 当日中にエクスポート** されたスナップショットで、当日夜の 2件はまだ DB に入っていなかった可能性が高い。

### 2.3 共通する 186件は **合計の不一致 0件** (完全一致)

→ rawdata.csv は時点違いと境界ミスを除けば prod DB と整合している。

## 7. 結論と次のアクション (要マスター確認)

### 7.1 確定したこと

1. **3月の「+22」のうち +5 は rawdata.csv 由来データの汚染が原因**
   - 2/28入店の7件 (会計日3/1) が誤って3月扱い
   - 3/31末の2件が rawdata snapshot 取得タイミングで欠落
2. **2月だけ pipeline ロジック「合計≠0」で airmate と完全一致**
3. **他3ヶ月は単一フィルタでは airmate と一致しない** — 別系統POS (0001-prefix) の扱いが鍵
4. **prod DB を使う限り、業務日基準 (朝6時境界) で取得すれば「件数の地ならし」は終わる** ので、以降の月報生成は prod-clean データを使うのが安全

### 7.2 マスターへの質問

> エアメイトの「117 + 53 = 170」 (3月の場合) は、内訳として何を分けていますか？
>
> - (案A) **店内 + テイクアウト** (店内=117 / テイクアウト=53)
> - (案B) **アラカルト + コース**
> - (案C) **男性 + 女性** (客数の話なら)
> - (案D) その他
>
> また、エアメイトの実画面 (1月〜4月) のスクリーンショットを共有可能でしょうか？
> 別系統POS (取引No `0001…`) のうち、どれを airmate が組数にカウントしているか確認したいです。

### 7.3 仮にここで方針判断するなら

最も筋が良いのは **「RB01 prefix のみカウントしつつ、0001 のうち "通常会計とみなせるもの" を補正で足す」** だが、補正ルール (試飲/イベント/CLP 一括の除外条件) は airmate のロジックを見ないと決められない。

暫定対応:
- pipeline 修正は「合計>0 & 人数>0 (パターン E)」で 1月だけは完全一致を取れる
- 2月・4月は ±1〜2 の差を受容
- 3月は **prod-clean データ + パターンE → 182** で旧 192 から 10件 改善 (まだ airmate +12 残)
- 0001 への対応は別タスク化
