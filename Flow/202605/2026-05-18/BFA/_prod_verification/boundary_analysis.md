## 検証手順 (`scripts/_bfa_boundary_analysis.mjs`)

各月について3つの境界ロジックで集計し、月初/翌月初の0:00〜6:00 dawn entries を全列挙：

```sql
-- (1) 暦日基準
WHERE entry_at >= 'M月1日 00:00 JST' AND entry_at < '翌M月1日 00:00 JST'

-- (2) 業務日5AM境界
WHERE entry_at >= 'M月1日 05:00 JST' AND entry_at < '翌M月1日 05:00 JST'

-- (3) 業務日6AM境界
WHERE entry_at >= 'M月1日 06:00 JST' AND entry_at < '翌M月1日 06:00 JST'
```

## 月別 dawn entries 実測

### 2026-01

- 当月1日 00:00〜06:00 JST: **2件 ¥7,750 / 2人**
  - 2026-01-01T02:10:02 JST sales=¥2,750 customers=1
  - 2026-01-01T02:10:58 JST sales=¥5,000 customers=1
  - → これは **2025-12-31 営業日**の取引。業務日基準では 12月にカウントされ 1月から除外される。
- 翌月1日 00:00〜06:00 JST: 0件

**差分 = 暦日 − 業務日 = ¥7,750 / 2人**（実測一致）

### 2026-02

- 当月1日 00:00〜06:00 JST: 0件
- 翌月1日 00:00〜06:00 JST: 0件
- どの境界でも同値 → 偶然 HTML と完全一致

### 2026-03

- 当月1日 00:00〜06:00 JST: 0件
- 翌月1日 00:00〜06:00 JST: 0件

### 2026-04

- 当月1日 00:00〜06:00 JST: 0件
- 翌月1日 00:00〜06:00 JST: **2件 ¥54,300 / 33人**

  - 2026-05-01T00:21:09 JST sales=¥50,000 customers=30

**差分 = 業務日 − 暦日 = ¥54,300 / 33人**（実測一致 — マスター仮説どおり ¥54,300）

## 5AM境界 vs 6AM境界の判定

実測した4ヶ月では **5AM境界と6AM境界の集計結果が完全一致**しています（5:00〜5:59 JST に entry がない）。
ただし共通ルール.md 準拠の正規ロジックは：

```python
# monthly_report_pipeline.py L73-75
def to_business_date(dt):
    return (dt - pd.Timedelta(days=1)).date() if dt.hour < 6 else dt.date()
```

**朝6時境界が正規**（00:00〜05:59 JST の取引は前日扱い）。これは `共通ルール.md` §1 / §18 に整合します。

## HTML 側の集計ロジック確認（F）

`monthly_report_pipeline.py` の `load_provided_csv()` → `calc_monthly_kpi()`：

1. 提供CSV (Shift-JIS) または rawdata.csv を読み込み
2. `business_date = to_business_date(dt)` 計算（dt は AirRegi: 会計日+時刻 / rawdata: exit_jst）
3. `df["business_month"] == target_month` でフィルタ
4. `hdr["合計"].sum()` で月次総売上、`hdr["人数"].sum()` で客数

**= 業務日基準（朝6時境界）でCSV全行を集計**。
コメントには「売上日時ベース（エアレジ売上分析画面と一致）」とあるが、実装は business_month フィルタ。エアレジ売上分析画面自体が業務日基準なので、コメントは「エアレジ画面と一致」が主旨。

## Phase 1 の集計が間違っていた点

Phase 1 (`query_prod_truth.mjs`) のクエリ：

```sql
WHERE (entry_at AT TIME ZONE 'Asia/Tokyo')::date BETWEEN '2026-04-01' AND '2026-04-30'
```

これは **暦日基準**。共通ルール上の業務日基準ではないため、HTML値と一致しなかった。

正しいクエリ（業務日朝6時境界）：

```sql
WHERE entry_at >= '2026-04-01 06:00:00+09'::timestamptz
  AND entry_at <  '2026-05-01 06:00:00+09'::timestamptz
```

または equivalent:

```sql
WHERE ((entry_at - INTERVAL '6 hours') AT TIME ZONE 'Asia/Tokyo')::date
      BETWEEN '2026-04-01' AND '2026-04-30'
```

## 次のフェーズ判断（提案）

仮説（α）が正しいと確認：

> **(α) HTML値（¥3,024,490等）を正として4ヶ月分prod DBアップロード（既定）**

理由：
1. HTML値は **共通ルール準拠の業務日基準**で集計されており、エアレジ売上分析画面と一致するため業務的に正
2. prod DB の sales_data 行自体は完全に保持されており、欠落なし。**ただ Phase 1 の集計クエリの WHERE 句が暦日基準で誤っていた**だけ
3. Jan/Feb/Mar も既存 HTML値で問題なし（Mar の -0.09% / +2人差は CSV と DB の微小同期差で、許容範囲内）
4. April の HTML ¥3,024,490 は 5/1 dawn entries 込みで業務日基準のあるべき値

**=> Phase 2 の再生成は不要。既存 HTML 4本をそのまま `_prod_verification/final/` にコピーし、Phase 3 で prod DB 直接アップロードへ進む。**

副次提案として（β）も：

> **(β) prod DB 側に業務日基準クエリを記録／標準化する**

`monthlyReportRepository.ts` 等で月次集計API がある場合は、業務日基準でフィルタするように修正する。本タスクのスコープ外なので別チケット化推奨。
