## 5. ⏳ Phase 6: PostgreSQL session TZ (任意・推奨)

Neon の Connection String に `options=-c timezone=Asia/Tokyo` を追加すると、
SQL 直書きで `NOW()` 等を使うときも JST 解釈になる。

```bash
# .env.production を Vercel ダッシュボードから編集 or Vercel CLI で
# DATABASE_URL を以下のように変更:

# Before:
# postgresql://...@ep-rough-bird-a1ynj8pb.ap-southeast-1.aws.neon.tech/neondb?sslmode=require

# After:
# postgresql://...@ep-rough-bird-a1ynj8pb.ap-southeast-1.aws.neon.tech/neondb?sslmode=require&options=-c%20timezone%3DAsia%2FTokyo
```

**注**: `@db.Date` 列の解釈は session TZ に依存しないため、PL データの整合性
には影響しない。アプリ層が JST で動くこととの一貫性のため推奨というだけ。

## 6. ⏳ Phase 7: `dayOfWeek` 再計算 (Phase 5 完了後)

`sales_data.dayOfWeek` カラムが UTC 暦日基準で保存されている可能性がある
ため、営業日付ベースで再計算する。

```sql
-- まずは件数確認 (どれぐらいズレているか)
WITH recalc AS (
  SELECT
    id,
    day_of_week AS old_dow,
    (
      CASE EXTRACT(DOW FROM
        (entry_at AT TIME ZONE 'Asia/Tokyo') - interval '5 hours'
      )
        WHEN 0 THEN '日' WHEN 1 THEN '月' WHEN 2 THEN '火' WHEN 3 THEN '水'
        WHEN 4 THEN '木' WHEN 5 THEN '金' WHEN 6 THEN '土'
      END
    ) AS new_dow
  FROM sales_data
  WHERE store_id = (SELECT id FROM stores WHERE store_code = 'bfa-001')
)
SELECT old_dow, new_dow, COUNT(*) FROM recalc
WHERE old_dow IS DISTINCT FROM new_dow
GROUP BY old_dow, new_dow
ORDER BY COUNT(*) DESC;

-- 問題なければ本反映 (cutoff=5 を直書き; 将来は store_settings から動的に)
UPDATE sales_data sd
SET day_of_week = (
  CASE EXTRACT(DOW FROM
    (sd.entry_at AT TIME ZONE 'Asia/Tokyo') - interval '5 hours'
  )
    WHEN 0 THEN '日' WHEN 1 THEN '月' WHEN 2 THEN '火' WHEN 3 THEN '水'
    WHEN 4 THEN '木' WHEN 5 THEN '金' WHEN 6 THEN '土'
  END
)
WHERE store_id = (SELECT id FROM stores WHERE store_code = 'bfa-001');
```

**注意**:
- 必ず**バックアップを取ってから**実行 (`pg_dump` or Neon の point-in-time
  recovery を確認)
- BFA 以外の店舗の cutoff が違う場合は、各店舗ごとに別実行する
- 監査トリガが入っているなら、UPDATE が全行に audit log を作るので注意

## 7. レコード再ダウンロードの判断

ユーザーから「レコードについてはダウンロードしなおしも検討」とあったが、
本番DB照会結果 (`本番DB照会結果_日付境界_v1.md`) と本実装で:

| データ | 状態 | 判断 |
|---|---|---|
| `sales_data.entryAt` | UTC タイムスタンプで正しく保存 | **再ダウンロード不要**。営業日カットオフを集計時適用 |
| `sales_data.dayOfWeek` | 一部 UTC 基準のため軽微にズレている可能性 | **SQL 一発で再計算 (Phase 7)** |
| `pl_daily_sales.date` | 営業日付として `@db.Date` で正しく保存 (重複0件) | **再ダウンロード不要** |
| `pl_daily_expenses.date` | 同上 | **再ダウンロード不要** |

→ **再ダウンロードは不要**。Phase 7 の `UPDATE` で十分。
