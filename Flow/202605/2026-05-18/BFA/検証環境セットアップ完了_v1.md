## 3. ★ 検証結果 (2026-04 POS集計, dev DB で実測)

`scripts/verify-bfa-verify.ts` で 3 通りに集計:

| 集計方法 | 売上 | 客数 | 客単価 | 会計件数 |
|---|---:|---:|---:|---:|
| **A. 営業日基準 (cutoff=5)** ※新ロジック | **¥3,024,490** | **584** | **¥5,179** | 166 |
| **B. 暦日基準 (UTC)** ※`@db.Date` 列基準 | **¥3,024,490** | **584** | **¥5,179** | 166 |
| **C. 旧バグ再現** (`new Date(2026,3,1)` JST) | ¥2,628,060 | 536 | ¥4,903 | 156 |

### AirMate 突合

| ソース | 客単価 | 判定 |
|---|---:|---|
| AirMate 4月実績 | **¥5,178** | — |
| 新ロジック (A or B) | **¥5,179** | ✅ **¥1差 (丸め誤差) で一致** |
| 旧バグ (C) | ¥4,903 | ✗ ¥276 のズレ |

→ **新ロジックは AirMate と一致**することが本番データのコピーで実証完了。

### A と B が同値なのはなぜか

BFA 4月の場合、たまたま以下が成立しているため:
- POS会計の `entryAt` で「JST 5/1 0:00〜5:00」に閉じた会計 (UTC 4/30 15:00〜20:00) は
  - UTC暦日 → `2026-04-30` → April に含まれる
  - 営業日 (cutoff=5) → `2026-04-30` → April に含まれる
- 「JST 4/1 0:00〜5:00」に閉じた会計 (UTC 3/31 15:00〜20:00) は存在しない (前月最終営業日と分離)

cutoff の効果が見える典型ケースは「**月またぎ深夜会計が複数日にまたがる**」場面。年末年始等では A と B の差分が見えるはず。

## 5. AirMate 突合の手順 (推奨)

1. AirMate 側で 2026-04 の以下を取得:
   - 売上合計
   - 客数
   - 客単価
   - (可能なら) 日別売上の Excel エクスポート
2. dev 環境の `bfa-verify` で同じ月を開き、上記4項目を並べる
3. 差分が発生した日があれば:
   - その日の `entryAt` を depDB 直接照会して JST跨ぎが原因か確認
   - 必要なら `cutoffHour` を 0 / 5 で切り替えて再集計

### cutoffHour の切替方法 (検証用)

```bash
cd ~/RestaurantAILab/Dashboard

# cutoff=0 (暦日基準) に切替
pnpm exec tsx -e "
import { PrismaClient } from '@prisma/client';
import { config } from 'dotenv';
config({ path: '.env.development' });
const p = new PrismaClient();
p.storeSettings.update({
  where: { storeId: '17d727c4-daee-4bed-a682-776fed723b6c' },
  data: { businessDayCutoffHour: 0 }
}).then(console.log).finally(() => p.\$disconnect());
"

# cutoff=5 に戻す場合は上記の 0 を 5 に
```
