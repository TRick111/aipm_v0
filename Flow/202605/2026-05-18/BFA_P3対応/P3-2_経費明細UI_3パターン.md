# P3-2 経費明細UI 3パターン比較

- 作成日: 2026-05-18
- 関連チケット: P3-2「経費ごとの詳細を見られるようにする」
- 元論点: `Flow/202605/2026-05-18/BFA/対応状況_5Arrowsフィードバック_v1.md`
- 対象: BFA（バー）月報ダッシュボード、PL明細ドリルダウン

---

## 1. 概要

### 背景
BFA月報では費目別の合計値（例: 「ドリンク 55万円」）は表示されているが、その内訳（どの取引先に・いつ・何の理由で支払ったか）に到達できない。経営判断のフィードバックを得るうえで「クリックして掘れる」状態が必要、というのが 5Arrows レビューでの指摘事項。

### DoD（このタスクの完了条件）
「ドリンク 55万円」をクリックすると、何の仕入れにいくら出たかが見える。具体的には、

- 費目（ドリンク仕入）→ 取引先別小計（○○酒販 32万円、△△ビバレッジ 18万円 …）
- 取引先別小計 → 日次明細（4/3 ○○酒販 4.2万円 メモ「ボトル補充」…）

までドリルできること。

### データ前提（既存 Prisma スキーマ）
- `PlExpenseCategory`（費目）: 名称・種別・FL原価フラグ
- `PlVendor` / `PlVendorCategory`（取引先・取引先カテゴリ）
- `PlDailyExpense`（日次経費）: 日付・費目・取引先・金額・種別（定常／例外）・メモ
- `PlExpenseCategoryGroup`（費目グループ）

つまり「費目 × 取引先 × 日付」の3軸データが既に揃っており、UIをどう切るかは自由度が高い。

---

## 2. 評価軸

3パターンを比較するための観点を先に定義する。

| # | 軸 | 説明 |
|---|---|---|
| 1 | 俯瞰性 | 月全体の経費構造（どの費目に偏っているか）が1画面でどれだけ見えるか |
| 2 | 詳細到達速度 | 「ドリンク」明細 → 個別取引まで最短何クリックか・認知負荷はどうか |
| 3 | モバイル適性 | スマホ縦画面（375px幅）でも破綻なく使えるか |
| 4 | 実装コスト | 既存テーブル／APIをどれだけ流用できるか |
| 5 | 予算比較適性 | 予算 vs 実績、前月比などの比較情報を載せやすいか |

## 3. パターンA：階層アコーディオン型（ドリルダウン）

### 3.1 画面構成

```
[月切替] 2026年4月 ▼

┌────────────────────────────────────────────────┐
│ ▶ ドリンク仕入             550,000   予算比 +5% │  ← クリックで開く
├────────────────────────────────────────────────┤
│ ▼ フード仕入               320,000   予算比 -3% │
│    ├ ▶ △△フーズ               180,000          │  ← さらに開く
│    │    ├ 4/02  鮮魚      24,500  メモ: 季節入荷│
│    │    ├ 4/09  鮮魚      18,200                │
│    │    └ 4/16  鮮魚      22,000                │
│    └ ▶ □□青果               140,000          │
├────────────────────────────────────────────────┤
│ ▶ 人件費                   980,000              │
│ ▶ 家賃                     250,000              │
│ ▶ 光熱費                   142,000              │
└────────────────────────────────────────────────┘
```

費目を行として並べ、`▶/▼` のディスクロージャーで「費目 → 取引先 → 日次」の3階層を同一カラム上で順次展開する。

### 3.2 操作フロー

「ドリンク 55万円 → 個別明細」までのクリック数:

1. 一覧表示（ドリンク行が見える）
2. ドリンク行をクリック → 取引先別小計が展開（1クリック）
3. ○○酒販 をクリック → 日次明細が展開（2クリック）

**最短2クリック**で日次レベルまで到達。閉じれば元の俯瞰に戻る。

### 3.3 表示する情報項目
- L1（費目行）: 費目名 / 月合計 / 予算比 / FL原価フラグ
- L2（取引先行）: 取引先名 / 月小計 / 構成比%（その費目内での）
- L3（日次行）: 日付 / 金額 / 種別（定常／例外） / メモ

### 3.4 向いている / 向いていない
- 向いている: 「気になった費目だけサクッと掘る」回遊探索、スマホでの片手操作、PLレビュー会で順に開きながら話す用途
- 向いていない: 複数費目を**同時に**比較したいケース（開きすぎると縦に伸びる）、日付軸での横断分析

### 3.5 実装難易度
**低**。既存の費目別集計APIに、取引先別と日次別のサブクエリを足すだけ。フロントは `<details>` 要素か `useState` の開閉ステートで十分。最短実装。

## 4. パターンB：マスター・ディテール（2ペイン）型

### 4.1 画面構成

```
┌─────────────┬────────────────────────────────────┐
│ 費目         │  ドリンク仕入   550,000  予算比 +5% │
│             ├────────────────────────────────────┤
│ ドリンク仕入 │ 取引先別                            │
│ ●550,000 31%│  ○○酒販      320,000   58%        │
│             │  △△ビバレッジ 180,000   33%        │
│ フード仕入   │  ボトル直送便  50,000    9%        │
│  320,000 18%│                                    │
│             │ 日次推移（4月）                     │
│ 人件費       │ ▮ ▮▮ ▮  ▮▮▮▮  ▮▮ ▮  ▮▮▮     │
│  980,000 55%│ 4/1 ...........................4/30│
│             │                                    │
│ 家賃         │ 日次明細                            │
│  250,000 14%│  4/03  ○○酒販      42,000        │
│             │  4/08  △△ビバレッジ 38,500        │
│ 光熱費       │  4/15  ○○酒販      55,200        │
│  142,000  8%│  4/22  ○○酒販      48,300        │
│             │  4/28  ボトル直送便  50,000        │
└─────────────┴────────────────────────────────────┘
   左ペイン         右ペイン（選択中の費目の中身）
```

左に費目リスト（縦長スクロール）、右に**選択中費目の詳細**を全部出す（取引先別＋日次グラフ＋日次明細を1画面同居）。

### 4.2 操作フロー

「ドリンク 55万円 → 個別明細」までのクリック数:

1. 一覧表示時に左ペインから「ドリンク仕入」を選択（1クリック）
2. 右ペインに取引先・日次・グラフが**すべて同時表示**

**1クリック**で必要情報がすべて出る。費目切替も1クリック。

### 4.3 表示する情報項目
- 左ペイン: 費目名 / 月合計 / 全費目に対する構成比% / 選択状態
- 右ペイン上部: 費目名・月合計・予算比・前月比
- 右ペイン中段: 取引先別テーブル（取引先名 / 小計 / 構成比%）
- 右ペイン中段: 日次バーチャート（仕入れ集中日が一目でわかる）
- 右ペイン下段: 日次明細テーブル（日付 / 取引先 / 金額 / メモ）

### 4.4 向いている / 向いていない
- 向いている: PCで1費目に集中して深掘りしたいケース、選んだ費目について**取引先・日次・グラフを同時に見たい**ケース、月次レビュー時の主席分析者向け
- 向いていない: スマホ（左ペイン＋右ペインのレイアウトが破綻）、複数費目を横並び比較

### 4.5 実装難易度
**中**。レイアウトはflex 2カラムで素直。ただし右ペインに「取引先別」「日次グラフ」「日次明細」を載せるので、データ取得APIを1リクエストにまとめるか3つに分けるかの設計が必要。グラフライブラリ（Recharts 既存）の流用で済む。スマホ用の「左ペイン→右ペイン遷移」モバイルレイアウト分岐が追加コスト。

## 5. パターンC：ピボット（マトリクス）テーブル型

### 5.1 画面構成

```
              4/1  4/2  4/3  ... 4/29 4/30  合計
─────────────────────────────────────────────────
[原価]
ドリンク仕入    -    -   42K   -   48K   -   550K
フード仕入     24K   -    -    -    -   22K  320K
─────────────────────────────────────────────────
[人件費]
人件費          -    -    -    -    -    -   980K
─────────────────────────────────────────────────
[固定費]
家賃            -    -    -   250K  -    -   250K
光熱費          -    -    -    -    -   142K  142K
─────────────────────────────────────────────────
合計          24K   0   42K  250K 48K  164K 2.24M
```

行＝費目、列＝日付（または取引先に切替可）、セル＝金額。費目はカテゴリグループ（原価／人件費／固定費）で区切り、行末・列末に合計。

セルをクリックすると、その費目×その日の取引先内訳とメモがツールチップ／右ドロワーで出る。

### 5.2 操作フロー

「ドリンク 55万円 → 個別明細」までのクリック数:

- 一覧時点で「いつ・どの費目に支払いが集中しているか」は**0クリックで**目視可能（仕入集中日が色濃度／太字で浮かぶ）
- ドリンク行の特定セル（例: 4/15）をクリック → ドロワーで「○○酒販 55,200円 メモ: 週末ストック」（1クリック）

**1クリック**で個別の日×費目に到達。ただし「ドリンクの月合計 55万円が**どの取引先で構成されているか**」だけを知りたい場合は、列を「取引先」に切り替える必要あり（モード切替UI必要）。

### 5.3 表示する情報項目
- 行頭: カテゴリグループ見出し / 費目名 / 月合計
- 列ヘッダ: 日付（曜日付き、土日は薄色）
- セル: 金額（0は `-` 表示、上位N%は太字／背景濃度）
- 行末: 月合計 / 予算比
- 列末: 日次合計
- セルクリック時のドロワー: 取引先別内訳 / メモ / 種別

### 5.4 向いている / 向いていない
- 向いている: 月全体の経費の**カレンダー的俯瞰**、仕入集中日・支払いの山谷の把握、複数費目の同時比較、Excel慣れしたユーザー
- 向いていない: 取引先軸の分析（列を切り替えないと見えない）、スマホ（横スクロール必須）、費目数が多い店舗で行が長くなる

### 5.5 実装難易度
**高**。行×列のピボット集計（既存データを `groupBy(費目, 日付)` で再構築）、ヘッダ固定スクロール、セル背景濃度の計算、ドロワー実装、列軸切替（日付↔取引先）が必要。最も実装が重い。

## 6. 比較サマリ

| 観点 | パターンA<br>アコーディオン | パターンB<br>2ペイン | パターンC<br>ピボット |
|---|:---:|:---:|:---:|
| 俯瞰性（月の経費構造が見える） | △ 順次展開 | △ 1費目のみ | ◎ 一画面で全部 |
| 詳細到達速度 | ○ 2クリック | ◎ 1クリック | ◎ 1クリック（日×費目） |
| モバイル適性 | ◎ 単一カラム | × PC前提 | × 横スクロール必須 |
| 実装コスト | ◎ 低 | ○ 中 | △ 高 |
| 予算比較適性（予算vs実績） | ○ 行に並記 | ◎ 右ペイン上部に大きく | △ 行末のみ |

**総合**: 

- A は「とりあえずDoDを最短で満たす」最有力
- B は「PC レビュー会用のメイン画面」候補
- C は「経営層向けカレンダーサマリ」としての別ビュー

## 7. 推奨案

### 7.1 最初に作るもの
**パターンA（アコーディオン型）を P3-2 の第一弾としてリリース**。理由:

1. DoD（ドリンクをクリックして内訳が見える）を**最小コストで**満たす
2. スマホでも自然に動くため、外出先で経費確認したいオーナー要望にも応える
3. 既存の月報レイアウトを大きく崩さず、既存APIへの差分が小さい

### 7.2 第二弾として追加
**パターンB（2ペイン）をPCの月次レビューモードとして追加**。理由:

1. アコーディオンだけでは「1費目をじっくり分析」したいユースケースに対し縦スクロールが冗長
2. 日次バーチャートを右ペインに置けるので、仕入れ集中日の可視化が可能になり、Cの代替にもなる
3. モバイルでは A、PCでは B、というレスポンシブ切替で両立可能

### 7.3 第三弾／オプション
**パターンC（ピボット）は「月次レポート印刷」や「経営会議スナップショット」用の別画面**として後追い。日常UIにはせず、PDFエクスポートとセットで設計する。

### 7.4 組み合わせの可能性
- A の費目行に「詳細を別ペインで開く」アイコンを置き、B のディテールビューに飛ばす（A→B のシームレス連携）
- B の右ペインに「マトリクス表示に切り替え」トグルを置き、選択中費目だけのカレンダーマトリクス（C のミニ版）を出す
