# グループ全体ダッシュボード 要件・ワイヤフレーム

作成日: 2026-05-18
対象: RestaurantAILab / Dashboard
ステータス: モック向け要件ドラフト（v0.1）

---

## 1. 背景と目的

### 1.1 現状

現行ダッシュボードは「店舗ごとの管理」を前提に作られている。

- `/store/[storeId]/sales-dashboard` 売上ダッシュボード
- `/store/[storeId]/pl` PL管理
- `/store/[storeId]/upload-attendance` 勤怠／日報入力
- `/store/[storeId]/weekly-reports/[weekId]` 週報
- `/store/[storeId]/monthly-reports/...` 月報
- PDF / スライド形式で店長が現場運営に使う

各店長が自店のコスト意識と現場状況を持つ「個店最適化」のための構成。

### 1.2 課題

10〜20店舗を展開するグループ／FCオーナー視点の機能が欠けている。

- 全店の合算売上・PLが一目で見えない
- どの店が「異常」なのかが見抜けない（人がCSVを並べて比較するしかない）
- 店長日報の中の「トラブル・クレーム・設備故障」など、オーナーが拾うべき情報が個店UIに埋もれる
- 週次・月次で店舗を横並びで比較できない
- グループ全体レポート（投資家・本部用）を作る作業が手作業

### 1.3 ゴール

「グループオーナー／エリアマネージャーが、毎朝5分で全店の健康状態を把握し、注意すべき店舗・トピックにドリルダウンできるダッシュボード」を提供する。

今回のスコープはモック制作のための **UI要件とワイヤフレーム** にフォーカスする。データ設計・集計ロジックは後続で詰める。

## 2. ペルソナとユースケース

### 2.1 主要ペルソナ

| ペルソナ | 想定 | 利用頻度 | 主目的 |
|---|---|---|---|
| **グループオーナー** | 10〜30店舗を保有する経営者 | 週1〜2 | 全店パフォーマンスと異常検知。投資・撤退判断 |
| **エリアマネージャー** | 5〜10店舗を担当する管理職 | 毎日 | 担当店の前日確認・週次レビュー・店長対応 |
| **本部スタッフ（管理）** | 経理／オペレーション | 毎日〜週次 | 数値抽出・レポート出力・本部資料作成 |

### 2.2 主要ユースケース（ジョブストーリー）

1. **朝のチェック**: 出社直後にスマホ/PCで「昨日の全店サマリー」を開き、赤信号の店だけ詳細を見たい
2. **異常店舗の特定**: 売上が前年同曜日比 -20% を超えた店、原価率が異常な店を一覧で見たい
3. **日報から拾うべきトピックの発見**: 全店の日報の中で「クレーム」「設備故障」「人員不足」などのキーワードを含む報告を横断検索したい
4. **店舗間比較**: 同業態同規模の店舗を横並びで売上・客単価・原価率・人件費率で比較したい
5. **週次レビュー**: 月曜朝に先週1週間の全店ランキング＋トピック要約をPDFで取得し、定例会で共有したい
6. **月次経営会議**: 月初に前月の全店PL・トレンド・問題店舗をスライド出力し、経営会議で配布したい

## 3. 情報設計（IA）

### 3.1 ルート構成案

```
/group/[groupId]/
├── overview            # グループサマリー（トップ）
├── stores              # 店舗一覧（ランキング・比較）
├── compare             # 店舗間比較ビュー（任意の店舗を選んで横並び）
├── anomalies           # 異常検知センター
├── daily-reports       # 日報横断ビュー（フィード）
├── pl                  # グループPL
├── weekly-reports      # 週報（全店統合）
└── monthly-reports     # 月報（全店統合）
```

既存の `/group/select` は単なる店舗選択画面 → 「グループ単位の入口」に格上げする想定。

### 3.2 ナビゲーション

- 画面上部に **スコープ切替バー**: `グループ全体 / エリア / 店舗` をワンクリック切替
- スコープを切り替えても **同種の指標** が見られる（売上・PL・日報・週月報）
- 店舗単位に潜ったときは既存の `/store/[storeId]/...` に着地

## 6. 異常検知・アラートの考え方（モック向け）

実装は後回しでも、UIに「もっともらしい数字」を出すために、以下のルールをモックの仮ロジックとして提示する。

| 指標 | しきい値（例） | 重大度 |
|---|---|---|
| 売上達成率 | <80% | 🔴 |
| 売上達成率 | 80–90% | 🟡 |
| 原価率 | 業態基準 +5pt以上 | 🔴 |
| 人件費率 | 業態基準 +5pt以上 | 🔴 |
| 客単価 | 前年同月比 -10%以上 | 🟡 |
| 日報タグ | `トラブル` / `クレーム` / `設備` 含む | 🔴 |
| 日報タグ | `クレーム` が3日連続 | 🔴 |
| 連続未達 | 3日連続で売上達成率<90% | 🔴 |

「ルール」と「統計外れ値（前年同曜日比z-score>2 など）」の両方をハイブリッドで使える設計余地を残す。

## 7. 出力（PDF / スライド）要件

- 既存単店レポートのテンプレートをグループ層に転用する
- グループレポートは **A4縦 1〜2枚／16:9スライド 6〜8枚** を上限の目安に
- 表紙 → サマリー → トピック → 異常店舗 → 来期フォーカス の流れを統一
- PDFは「定例会で配布」、スライドは「投影」、メールは「関係者に配信」を想定
- ファイル名規約: `group-{groupId}-{weekly|monthly}-{YYYYMM(W)}.{pdf,pptx}`

## 8. 既存機能との関係（既存を壊さない原則）

- 個店UIは現状維持。グループUIは「親」として上に被せる構造
- グループUI内の店舗名・店舗カードは **すべて既存の `/store/[storeId]/...` にディープリンク**
- 既存の日報フォームに対し、AIタグ付け（バックエンド処理）を後付けで足す前提
- グループPLは既存単店PLの集計ビューであり、入力UIは増やさない
