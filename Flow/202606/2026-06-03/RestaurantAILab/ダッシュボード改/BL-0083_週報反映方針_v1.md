## 2. 実装方針 (案)

### 2.1 集計ロジック (月報からの移植)

月報の `build_html_2026XX.py` 内のロジックをそのまま週単位 (月→週) に変換:

```python
# 既存月報ロジック (例: build_html_202605.py)
df_acc["is_inbound"] = df_acc["メモ"].fillna("").str.contains("海外", na=False)
p1_1_inbound = aggregate(df_acc[df_acc["is_inbound"]])
# 集計: 会計件数 / 入客数 / 売上 / 売上構成比

# 週報移植 (新規)
# 期間を月 → 週 (週次集計範囲: 月〜日 もしくは BFA 朝6時境界)
# それ以外のロジックは同一
```

実装場所: 既存週報スクリプト `Stock/RestaurantAILab/月報/Scripts/deep_dive_weekly_analysis.py`
あるいは新設 `Stock/RestaurantAILab/週報/Scripts/customer_segment_analysis.py` のどちらにするかは Section 5 Q-B。

### 2.2 サンプル数小の取り扱い (週次特有の論点)

月次 (1 ヶ月分) では「インバウンド 34 件 / 84 人」あった BFA 2026-05 実績も、
週次 (1 週間) では **0〜数件しかヒットしない週** が高頻度で発生する見込み。

- 0 件週: スライドごと非表示 / 「該当なし」表示 / 4 週平均で代替 の 3 案
  → Section 5 Q-C

### 2.3 月報ビルドへの組込み

> 「今日の一手 = 反映ロジック実装 → 月報ビルドに組込み → 動作確認」

月報側は既に組込み済 (5/18 リリース、6/2 の 2026-05 月報で稼働確認済)。
→ **月報側は今日追加作業なし**。週報側の実装に集中。

## 3. 週報テンプレへの挿入提案

### 3.1 推奨案: 第 3 部後に「第 3.5 部: 客層分析 (2 枚)」を新設

| No. (新) | タイトル | 内容 |
|:-:|---|---|
| 23.5 | 客層分析①: インバウンド比率 | ドーナツ (インバウンド vs 非インバウンド) + 左右パネル (会計件数 / 入客数 / 売上)。集計対象: 今週、抽出キー: メモ「海外」部分一致 |
| 23.6 | 客層分析②: CLP社員比率 | ドーナツ (CLP関連 vs その他) + 左右パネル (同上)。抽出キー: メニュー名「CLP」部分一致 |

理由:
- 第 3 部 (商品・カテゴリ分析) と第 4 部 (総括) の間 = 客層という別軸の分析として独立性が高い
- 月報の Slide-7/8 と同じレイアウトを採用すれば、見た目の一貫性 + 実装流用が効く

### 3.2 代替案: 第 1 部 (全体サマリー) に組込み

KPI カードの 1 枚に「うちインバウンド比率 X%」を追加するだけの軽い反映。
- メリット: スライド総数が増えない (28 枚維持)
- デメリット: 詳細 (件数 / 客数 / 売上構成比) が見えない、月報との比較が難しい

### 3.3 何もしない案

サンプル数小で意味のある分析にならない週が多いため、まずは月報のみ運用 (=K-5 維持)、
来店データ集約 (例: 月跨ぎ 4 週移動平均) を用意してから週報に組込む。

### Q-B. 客層フラグの対象範囲

BL-0083 description には「常連 / インバウンド / 社員」とあるが、
現状実装済 (月報) は **インバウンド (メモ「海外」)** と **CLP社員 (メニュー名「CLP」)** のみ。

- (1) 週報も同じ 2 系統 (インバウンド + CLP社員) で揃える
- (2) 「常連」も追加する (※その場合、Airレジ メモ欄での運用ルールを別途決める必要あり)
- (3) インバウンドのみ (CLP は外す)

### Q-C. サンプル数小 (週次特有) の取り扱い

週次集計だとインバウンド 0 件 / CLP 0 件の週が頻発する想定。

- (a) 0 件週はスライド非表示
- (b) 0 件でも「該当なし」と明示してスライドは出す
- (c) 単週ではなく **4 週移動平均** で集計 (週次サマリ内に月次相当の参考値として表示)

### Q-D. 週報テンプレへの挿入位置

- (軽量) Section 3.2: 第 1 部 KPI に 1 行追加 (枚数維持)
