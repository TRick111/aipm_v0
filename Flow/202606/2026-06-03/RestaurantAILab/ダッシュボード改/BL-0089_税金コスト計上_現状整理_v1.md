## 1. 実装済の内容 (dev ブランチ uncommitted)

### 1.1 スキーマ (`prisma/schema.prisma`)

新規モデル `PlSalesDeductionRule` を追加:

```
salesCategoryId   : 控除対象の売上カテゴリ
ratio (Decimal 5,2): 控除割合 (% 単位、0 < ratio <= 100)
expenseCategoryId : 振替先費用カテゴリ (null = 振替なし、単純控除)
startDate / endDate: 有効期間 (endDate inclusive、null = 無期限)
isActive          : 有効/無効
```

制約 (アプリ層で検証):
- 同一 (storeId, salesCategoryId, expenseCategoryId) で期間重複禁止
- 同売上カテゴリの有効ルール ratio 合計 > 100% 禁止
- expenseCategoryId に指定されたカテゴリへの `PlDailyExpense` 手動入力禁止

### 1.2 DB マイグレーション (`scripts/sql/add_pl_sales_deduction_rules.sql`)

- 生 SQL でテーブル作成 + 3 つの index
- `prisma db execute` で適用する想定 (Prisma migration ファイルにはなっていない)
- **ローカル dev / 本番 (Neon) への適用ステータスは未確認**

### 1.3 集計ロジック (`lib/pl/salesDeduction.ts` 127行)

`calculateSalesDeduction(sales, rules, todayStr)` を export:
- 日次売上 × ルール一覧 → 売上カテゴリ別控除額 / 費用カテゴリ別振替加算額 (累計 + 当日) を Map で返す
- 有効期間内 (start/end inclusive) のルールのみ適用
- 純関数 (Prisma 依存無し)、ただし **単体テスト未整備**

### 1.4 リポジトリ (`lib/db/repositories/salesDeductionRulesRepository.ts` 453行)

CRUD + 期間重複チェック + ratio 合計上限チェック + 有効期間判定 `isManualExpenseAllowed` 等。
規模が大きい (453行) ので、ロジック分離のリファクタ候補。

### 1.5 API

- `app/api/pl/sales-deduction/route.ts` — list / create
