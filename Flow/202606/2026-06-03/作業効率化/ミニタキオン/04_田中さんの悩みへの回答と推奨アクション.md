## 悩み 1. 「1 日の初めのタスクを作る前にすでにいくつかタスクが表示されている」

### 答え (実装ベース)

「今日の選択」リストの判定ロジック (`lib/today-selected.ts:90-100`) は **3 つの OR 条件**:

| 条件 | 内容 |
|---|---|
| (a) 朝の採択 | 今日の `_orchestration/今日のタスク案.md` で `[x]` が付いた BL |
| (b) 今日の decisions | BL.decisions[] に **今日 (logicalNow 基準、朝 5:00 切替)** で append されたエントリがある BL |
| (c) own active | BL-TBD-XXX で in_progress / awaiting_user の BL は **常時可視** |

→ 朝のタスク案が未生成でも (b) と (c) は **常時マッチし得る**。だから「朝の前にすでにある」現象が起きる。

これは **設計どおりの挙動**。意図は:
- 田中さんが起き抜けに UI を開いた瞬間「昨日続きの BL は何だっけ」を一覧したい
- ミニタキオン自身の Phase X の dogfood BL は朝の整理を経ずに着手したい

### 改善案

**A. UI で「朝確定済」「自動可視」を区別**  
朝のタスク案を通った [x] BL は ★、(b)/(c) で勝手に出ているものは ☆ にラベル分け。

**B. 朝のタスク案を見るまで「今日の選択」を表示しない mode (toggle)**  
ホームの設定で「朝の整理前は空表示」を選べる。

**C. (c) own active の廃止 / 緩和**  
BL-TBD-XXX special-case を外し、「ミニタキオン project の active を常時出す」も廃止。ミニタキオンも他の project と同じ扱いに。

→ **推奨は A + C**。 (B は morning が遅れた時に「今日のタスクが見えない」になるので fragile)

## 悩み 2. 「バックログの番号の振りに一貫性がない」

### 答え (実装ベース)

**田中さんの直感は正しい**。実装上 2 つの問題:

1. `nextNumericBLId(category, project)` は **指定 project の backlog/ dir だけ** scan して max + 1 する。global 採番ではない (`lib/api/bl-paths.ts:50-65`)
2. ID キーの index Map は global ユニーク前提で構築されているので、複数 project に `BL-0001.yaml` があると **silently 上書き**される (`lib/stock.ts: bls.set(bl.id, bl)`)

実際 `BL-0001.yaml` は **11 個** 存在 (詳細 02 文書参照)。

加えて `BL-TBD-XXX` と `BL-XXXX` の **二系統 ID** が併存し、`BL-TBD-` は特別扱い (always-visible) されているのに、新フォーマット (`BL-0095`) は同じ ミニタキオン project でも特別扱いされない。

### 改善案

**A. `nextNumericBLId` を global スキャンに変更** (最小修正、即効)
```ts
// 現状: project dir だけスキャン
// 変更: ~/aipm_v0/Stock/**/backlog/BL-*.yaml と Backlog.md の最大値 + 1
```
※ index-store の rebuild と整合させる必要あり

**B. 中央 Backlog.md を canonical ID 採番表 として復活**  
新規 create 時は Backlog.md に行追加 + per-project YAML を atomic に作る (既に `mergeCentralBLs` は in-flight でやっているが、create 時に書き込んでいないので二重管理になっている)

**C. BL-TBD-XXX を BL-XXXX に rename**  
12 個の BL-TBD-XXX を BL-0097 〜 BL-0108 (例) に renumber。`isMiniTachyonOwnBLActive` special-case を撤廃。`deliverable_refs` などの逆参照は鎖で書き換え。**スクリプト書けるが ref 切れリスクあり、要 dry-run + 田中さん承認**

**D. 既存重複 BL-0001 〜 を renumber**  
11 個の BL-0001 を global ユニーク (例: 中央 Backlog.md にある ダッシュボード改/BL-0001 は残し、他は次の空き番号へ) に振り直し。**これは大手術なので相談しながらやる**

→ **推奨順: A → C → D → B**。A だけで「これから増える BL は global ユニーク」になり、田中さんの困惑の主因 (= 新規追加の番号が場所によって違う) は解消する。

## 悩み 3. 「新しいバックログの追加をミニタキオン上からできない、cockpit などからやっていて不整合を生んでいる」

### 答え (実装ベース)

**正しい**。`POST /api/bl` は実装済だが UI に form / button **なし** (03 文書 § 4 参照)。

現状の代替手段:
- mt CLI 直接 (`mt bl create ...`) — 田中さんが Mac terminal で
- cockpit エージェント経由 — AI が中央 Backlog.md を見て採番、`mt bl create --id BL-XXXX` で渡す
- 朝のタスク案で AI が新規 BL 候補を出して finalize 経由

cockpit 経由は AI が独自に採番を決めるので、悩み 2 と組み合わさって ID が散らかる。

### 改善案

**A. UI に「+ 新規 BL」モーダルを追加** (実装難度: 低、推奨)
- ホーム or projects 一覧の上部に「+ 新規 BL」ボタン
- モーダル: title (req) / project (select req) / category (select req) / priority (default normal) / description (optional)
- submit で POST /api/bl → chokidar が反映 → 一覧に追加

**B. 既存 BL-TBD-007 (Phase 3d UI 拡張) に統合**  
TBD-007 は「📅 今日の運用」UI 拡張だが、ついでに「+ 新規 BL」も乗せられる。スコープ追加で良い。

**C. iPhone / Tailscale でも使えるよう mobile UI**  
新規 BL は iPhone で立てたい場面も多い (移動中の閃き)。モーダルは mobile-first で。

→ **推奨: A + C**。B はおまけ。「悩み 2 の修正 (global 採番)」とセットで入れると、cockpit からの ad-hoc 採番に頼らなくて済む状態になる。

## 悩み 4. 「AIOS の中身をタスクがどれぐらい読み込んでいるかわからない」「ダッシュボードなど開発物は別リポなのにどう動いているか不明」

### 答え (実装ベース)

**読み込んでいない**ものが多い (詳細 03 文書 § 5 参照):

| 対象 | 読み込み |
|---|---|
| AIOS rule 13 | ❌ 本文は inject されず、絶対パス参照のみ → AI 判断で Read |
| AIOS rule 01〜12 | ❌ 参照すらなし |
| `~/aipm_v0/CLAUDE.md` | ✅ (Claude Code が自動 inject、cockpit + codex などでは未確認) |
| 中央 Backlog.md | ✅ 朝エージェントが必ず |
| BL.yaml | ✅ `mt bl get` で |
| STATUS.md (Flow / Stock) | △ `has_status` のみ伝わる、本文は AI 判断 |
| Flow 過去成果物 | △ 前日 wrap-up しか |
| USAGE.md / INTEGRATIONS.md | ❌ |
| 別リポ (Dashboard 等) のコード | ❌ |

ダッシュボードについては:
- 状態管理 (BL / STATUS / Flow 成果物) は `~/aipm_v0/Stock/RestaurantAILab/ダッシュボード改/` に置く
- 実コードは `~/RestaurantAILab/Dashboard*` (別リポ複数)
- cockpit task は **常に AIPM_ROOT (=`~/aipm_v0`) で起動**。エージェントが Dashboard 実コードに触りたければ手動で `cd ~/RestaurantAILab/Dashboard`
- 成果物 .md は AIPM_ROOT 配下にしか register できない

### 改善案

**A. instruction header に「読み込み推奨ドキュメント」リストを動的注入**  
BL に紐付く `code_repo` フィールド (新設) を BL.yaml に持たせ、`buildMtAgentInstruction` で `# 関連リポジトリ:` セクションに追加注入する。

**B. STATUS.md の本文を mt CLI で取れるエンドポイント追加**  
`mt project status <slug>` で STATUS.md frontmatter + body を JSON で返す。エージェントが「読みに行く」コストを下げる。

**C. 「AIOS rule の本文を inject」オプション**  
`mt bl get <id> --with-aios-rules` などで rule 13 と関連 rule (BL の category 別に rule 04 / 07 など) を JSON に詰めて返す。長い prompt にはなるが「読み忘れ」が無くなる。

**D. 別リポを STATUS.md frontmatter で宣言**
```yaml
---
project: ダッシュボード改
code_repo:
  - path: ~/RestaurantAILab/Dashboard
  - path: ~/RestaurantAILab/Dashboard-Kai
---
```
mini-tachyon UI で表示 + エージェント prompt に inject。

**E. エージェントの読み込みログ** (verification)  
cockpit task の output log を解析し「rule 13 を実際に Read したか」「STATUS.md を読んだか」を可視化。これは観測機構なので実装中重い。

→ **推奨順: D → A → C → B → E**。D が一番すぐ効く (BL の category 別ルール / 別リポを 1 箇所に集約)。
