## 0. このドキュメントは何か

田中さんの「AI 主導 + 人間レビュー」方針転換を、**タスクの生成 〜 完了のライフサイクル**として整理する。
最初に田中さんの意図を再構成し、その上で複数のワークフロー案を提示する。

## 1. 田中さんが言っていることの整理

### 1.1 現状認識
- 最近の仕事は コーディング / 調査 / 要件定義 が多い
- それらの **実行はすでに AI に寄せられている**
- 結果、**コスト中心が「作業時間」から「レビュー時間 + AI への要件伝達時間」に移った**

### 1.2 目指す姿
- 「AI に一部を手伝ってもらう」→「**AI が作業の base、人間はレビュー + 要件定義**」
- 人間が触る部分を意図的に絞り、田中さんの時間を最も価値の高い 2 点に集中させる

### 1.3 検討したい論点
- 個別タスクのライフサイクル: 生成 → 完了までのループ
- どこで人が介在するか
- 何をトリガに AI が動き出すか
- タスクの生成元の多様性: ミーティング / 思いつき / 完了タスクの派生 / WBS の次行
- プロジェクト視点での WBS 分解単位
- AI と人間のフィードバックループ設計

→ **個別タスク軸** と **WBS（プロジェクト）軸** の 2 軸で考える必要がある

## 2. 「人間の時間」をどこに集中させるべきか

### 2.1 人間が行う判断の種類

| 判断 | 質問 | 1 回の所要 | 頻度 |
|---|---|---|---|
| スコープ判断 | やる価値はあるか / どこまでやるか | 1〜3 分 | BL 起票時 |
| アプローチ判断 | どの方針で進めるか | 5〜15 分 | 着手前 / 詰まった時 |
| 受け入れ判断 | この出力で完成か | 3〜10 分 | 完成時 |
| 横展開判断 | この知見を次にどう活かすか | 1〜5 分 | 完了直後 |

### 2.2 削るべき時間（コスト中心）

- ❌ AI に要件を **ゼロから書き起こす** 時間 → AI に下書きを書かせて田中さんが添削する形に置換
- ❌ 個別 BL の **進捗確認** に毎回頭を使う時間 → UI が状態を見せ、田中さんは介入が必要な瞬間だけ動く
- ❌ レビュー時に **「全体を読み直して文脈を取り戻す」** 時間 → AI 出力に「何を見れば判断できるか」のサマリ強制
- ❌ 完了後に **「次に何をすべきか」を都度考える** 時間 → 完了 BL から次 BL が自動提案される

### 2.3 残すべき時間（削ってはいけない）

- ✅ スコープと受け入れ基準の判断（譲ると全体品質が落ちる）
- ✅ 「これは AI に任せて良いか」のメタ判断
- ✅ 戦略 / 優先順位（緊急タスクと長期投資の配分）

## 3. タスクライフサイクル（一般形）

7 段階。👁 が「人間が触る」段階。

```
┌───────────────────────────────────────────────────────────────┐
│ 1. CAPTURE   捕捉    ← 思いつき/ミーティング/前BL派生/WBS次行  │
│    出力: 1 行の goal                                            │
├───────────────────────────────────────────────────────────────┤
│ 2. DEFINE    定義    ← AI 下書き、人間が添削                  👁│
│    出力: spec (goal / acceptance_criteria / constraints / refs) │
├───────────────────────────────────────────────────────────────┤
│ 3. DECOMPOSE 分解    ← AI 提案、人間が approve / 修正         👁│
│    出力: sub-task 一覧 または「1 BL でいい」判断                │
├───────────────────────────────────────────────────────────────┤
│ 4. EXECUTE   実行    ← AI 自律実行                              │
│    出力: deliverable (.md, code, PR, etc.)                      │
├───────────────────────────────────────────────────────────────┤
│ 5. REVIEW    レビュー ← 人間が spec と照合                    👁│
│    出力: accept / revise / split / abandon の判断                │
├───────────────────────────────────────────────────────────────┤
│ 6. CLOSE     クローズ ← AI が状態更新、人間は 1 クリック        │
│    出力: BL.state = completed                                    │
├───────────────────────────────────────────────────────────────┤
│ 7. CASCADE   派生    ← AI が「次の BL 候補」を提案            👁│
│    出力: 新 BL(draft) 群 / 「次なし」判断                       │
└───────────────────────────────────────────────────────────────┘

👁 = 人間が触る段階（4 段階、それぞれ短時間）
```

### 3.1 各段階で AI を起動するトリガ

| 段階遷移 | トリガ |
|---|---|
| 1 → 2 | BL 作成イベント (state=`captured`) で自動 |
| 2 → 3 | 人間が spec を done で承認 |
| 3 → 4 | 人間が decomposition を done で承認 |
| 4 → 5 | AI が deliverable register で完了報告 |
| 5 → 6 | 人間が done / needs_revision で判断 |
| 6 → 7 | BL.state = completed への遷移 |
| 7 → 1 | 提案 BL の accept |

---

## 4. タスクの生成元（CAPTURE のサブパターン）

全部段階 1 に入る。違いは「事前に何が分かっているか」と「人間が何をするか」。

| 生成元 | 事前情報 | 必要な人間アクション |
|---|---|---|
| 🗣 ミーティング | 議事録 + 文脈 | AI 抽出した action item を triage |
| 💡 思いつき | 1〜2 行のメモ | 1 ボタンで capture（spec は 2 で AI が起こす） |
| ⛓ 派生 | 完了 BL の出力 | AI 提案を accept / decline |
| 🗂 WBS 次行 | プロジェクト計画 | 自動（前 BL 完了 = 次 BL が active 化）|
| 🔔 外部イベント | bug report, 顧客連絡 | inbound 検知 → AI が下書き起票 |

→ **すべてのチャネルが段階 2 (DEFINE) に合流する** = capture チャネルが増えてもライフサイクルは 1 本で良い

---

## 5. WBS（プロジェクト軸）の扱い

### 5.1 BL と WBS の関係
- **BL** は「1 セッションで AI が完結できる単位」
- **WBS** は「プロジェクトを BL に分解した木構造」
- BL と WBS の繋ぎ方が方針案の最大の分岐点（後述）

### 5.2 分解の粒度の原則
- 1 BL = AI が **1 セッション (≤ 数時間)** で deliverable を出せる
- 1 BL = **1 deliverable** が原則（途中で人間レビューが入らない）
- 1 BL の spec は **A4 1 枚以内**（書ききれないなら分解不足）
- 親子関係は必要な時だけ持つ（強制しない）

### 5.3 WBS 進化の 2 モード
- **Pre-planned WBS**: プロジェクト開始時に全 BL を起こす（waterfall 寄り）
- **Just-in-time WBS**: 完了した BL の cascade で次が湧き出る（agile 寄り）
- 現実解は **ハイブリッド**: マイルストーン（粗 BL）は事前、その下の細 BL は JIT

## 6. ワークフロー案（5 つ）

### 案 α: Spec-First Loop（仕様先行型）

- **要点**: 人間の介在を「spec 承認」と「最終 review」の 2 点に集中
- **流れ**:
  1. CAPTURE
  2. AI が spec 下書き（goal / acceptance_criteria / constraints / refs）
  3. 人間が spec を 5〜10 分でレビュー、修正、approve
  4. AI が EXECUTE を完走（途中で詰まったら BL.pending_questions に貯める）
  5. AI が deliverable register
  6. 人間が REVIEW（spec の受け入れ基準と照合）
  7. accept / revise
- **人間タッチポイント**: **2 回**（spec 承認 + 最終 review）
- **得意**: 要件が固まる コーディング / 設計ドキュメント
- **苦手**: 探索的調査（spec を書き切れない）

### 案 β: Streaming Review Loop（小刻みレビュー型）

- **要点**: spec を最初に固めず、AI が小さい刻みで進めて都度確認
- **流れ**:
  1. CAPTURE
  2. AI が次の 1 ステップ (≤ 15 分) を実行
  3. 人間が「続けて / 方向修正 / 完了」を **30 秒で 1 タップ**
  4. 2〜3 を繰り返す
  5. 人間が「完了」したら CLOSE
- **人間タッチポイント**: 多い（が 1 回 30 秒）
- **得意**: 調査 / 思考整理 / 不確実性が高い領域
- **苦手**: 並列実行（待ち時間が散らばる）

### 案 γ: Two-Stage WBS Loop（WBS 先行型）

- **要点**: プロジェクト開始時に WBS を 1 回固定、各 BL は案 α で回す
- **流れ**:
  1. プロジェクト goal 入力
  2. AI が WBS（マイルストーン + BL 一覧）を提案
  3. 人間が WBS を 10〜15 分で review、修正
  4. 各 BL について 案 α を適用
  5. WBS は AI が完了に応じて更新（変更があれば 1 タッチ承認）
- **人間タッチポイント**: WBS 承認 1 回 + 各 BL の α タッチ
- **得意**: 中〜大規模プロジェクト（数週間〜数ヶ月）
- **苦手**: 探索フェーズ（WBS が書けない）

### 案 δ: Discovery-Then-Lock Loop（発見 → 固定型）

- **要点**: 不確実な領域は探索 → 結果を見て scope 確定 → 案 α
- **流れ**:
  1. CAPTURE（ぼんやりした goal）
  2. AI が DISCOVERY: 選択肢 / リサーチ / 制約洗い出し（1 セッション）
  3. 人間が discovery 結果を見て scope を確定（10 分）
  4. AI が確定 spec を起こす
  5. 以降は 案 α
- **人間タッチポイント**: scope 確定 1 回 + 案 α の 2 回 = **3 回**
- **得意**: 新規領域 / 要件不明瞭 / 技術検証
- **苦手**: 「すぐ終わるはず」のタスクには重い

### 案 ε: Meeting-Driven Inbox Loop（ミーティング駆動型）

- **要点**: ミーティング後に AI が action item + 下書き spec を生成、人間はまとめて triage
- **流れ**:
  1. ミーティング録音 / 議事録
  2. AI が抽出: action item + 担当 + 期日 + 下書き spec
  3. 人間が mini-tachyon の「ミーティング inbox」を 5〜10 分で triage（採用 / 却下 / 統合）
  4. 採用 BL は 案 α または 案 γ に流す
- **人間タッチポイント**: triage 1 回 + 各 BL のタッチ
- **得意**: クライアント案件 / 定例で action が多い
- **苦手**: 個人プロジェクト（meeting がない）

## 7. 案の使い分け（推奨）

「全部 α」ではなく、**仕事タイプで案を振り分ける** のが現実的。

| 仕事タイプ | 推奨案 | 理由 |
|---|---|---|
| コーディング（実装、bugfix） | α | spec が書ける、レビューは PR/diff |
| 要件定義 / 設計ドキュメント | δ → α | 最初は discovery、固まったら α |
| 調査 / リサーチ | β | 不確実性高く spec が書きづらい |
| プロジェクト立ち上げ | γ | WBS から入る |
| クライアントワーク | ε → α | meeting trigger が多い |
| 雑事 / 思いつき | β の最小版 | spec を書く時間がもったいない |

→ mini-tachyon は **BL ごとに「どの案で回すか」を選べる** UI を持つべき（後述 § 9）

---

## 8. AI 自律実行を伸ばす仕組み（共通機能）

人間タッチポイントを減らすために、AI が独力で進められる範囲を広げる:

1. **Spec を構造化スキーマで持つ**
   `goal / acceptance_criteria[] / constraints[] / references[]`
   → AI が「これで合ってるか」を spec と照合で判断可能 → REVIEW が高速化
2. **Cascade ルールを宣言できる**
   「設計 BL 完了 → 実装 BL を draft 起票」のようなルールを BL.yaml に書ける
3. **質問のバッチ化**
   `BL.pending_questions[]` に貯め、田中さんは 1 日 1 回まとめて回答
4. **失敗 budget**
   AI が N 回試して詰まったら escalate（無限ループ防止）
5. **依存解決の自動注入**
   前 BL の deliverable を後続 BL の `references` に自動注入
6. **「介入レベル」を BL ごとに宣言**
   `intervention_mode: full | spec_only | streaming | discovery` で AI の挙動を切り替え

## 10. 全体まとめ

```
仕事タイプで案を選ぶ → ライフサイクル 7 段階に流す → 人間タッチ 2〜4 回で完結
                                    ↑
                          mini-tachyon が「介入が必要な瞬間」を露出する
```

**結論として最小限の意思決定:**

1. 個別 BL は **α + β を first-class で回す**（共通基盤だけで OK）
2. WBS（γ）は **マイルストーン BL = 親、実 BL = 子** の親子関係を BL に持たせる（軽量実装）
3. δ（discovery）は α の前段として **同じ BL の中で 2 段階回す**（discovery deliverable → spec deliverable → execute deliverable）
4. ε（meeting inbox）は **後回し**（録音文字起こしが整ってから）

## 11. 開く論点（決めるべきこと）

1. **WBS データモデル**
   - 親子 BL (`parent_id`) vs 別ファイル `wbs.md` vs プロジェクト STATUS.md 内の section
   - 推奨: `parent_id` フィールドを BL.yaml に追加（schema_version: 3）
2. **Spec の粒度**
   - 田中さんが書く / AI が書いて添削 / 両方混ぜる
   - 推奨: AI が下書き、田中さんが受け入れ基準だけ書き換える
3. **Cascade の自動度**
   - 提案のみ（accept 必要）vs draft 自動作成 vs 直接 active 化
   - 推奨: 初期は「draft 自動作成、accept で active 化」
4. **Discovery と Spec の境界**
   - どこまでが「調査」で、いつ「実装に進む」のか判断ルール
   - 推奨: discovery deliverable に「3 つの選択肢 + 推奨 1 つ」を強制 → 田中さんは選ぶだけ
5. **ミーティング inbox の入力**
   - 録音文字起こし / 議事録ペースト / Calendar 連携
   - 推奨: 当面は議事録ペーストのみ
6. **失敗 budget の閾値**
   - 何回詰まったら escalate するか
   - 推奨: 3 回 retry → pending_questions に escalate
7. **「案を BL ごとに選ぶ」UI**
   - 起票時に選ばせる vs AI が推奨 vs 後から変更可
   - 推奨: AI が推奨を出して、田中さんは変更したい時だけ触る
8. **介入モード切替の名前**
   - α/β/γ/δ/ε を内部名のまま運用するか、`spec_first`, `streaming`, `wbs`, `discovery`, `meeting` などにリネーム
9. **Spec 構造化スキーマ**
   - YAML フィールドとして固定（zod 検証）vs Markdown 内のフロントマター
10. **「人間タッチを必要としない」自律 BL を許すか**
   - 完全自動（review skip）の経路を作るか、必ず 1 タッチは挟むか
