# マーケティングダッシュボード 実装方針（Phase 1）

最終更新: 2026-05-19
発端: `Flow/202605/2026-05-19/RestaurantAILab/ダッシュボード追加要件一覧_竹矢さんヒアリング_v1.md`

## 概要

竹矢さん（Chef's Room）からの追加要望を受け、既存の売上分析／PL管理ダッシュボードに **「マーケ媒体の月次取り込み＋媒体横断ビュー＋外注コンサル向け権限」** を増築する。Phase 1 は「外注化スタートに最低限間に合わせる」スコープ。

### 要件IDの対応

要件一覧（v1） §2 より:

- 取り込み: M-01（食べログ）／M-03（食べログノート）／M-04（ホットペッパー）／M-08（媒体別アップロード画面）
- 分析: A-01（媒体横断KPIサマリー）
- 権限: S-01（コンサル用ロール）／S-02（コンサル店舗単位ACL）／S-03（媒体ログイン情報の社内集中管理：運用ルール）
- 横断: X-01（店舗ごとの契約媒体フィーチャーフラグ）／X-02（運用フロー定義書）

## 1. データモデル（Prisma スキーマへの追加）

既存 schema.prisma との衝突を避け、`Marketing*` プレフィックスで新規モデルを追加する。命名は既存の `Pl*` / `DailyReport*` パターンに合わせる。

### 1.1 媒体マスタ（`MarketingChannel`）

| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | PK |
| code | VARCHAR(40) | `tabelog` / `tabelog_note` / `hotpepper` / `gbp` / `instagram` |
| displayName | VARCHAR(80) | 表示名 |
| category | VARCHAR(20) | `reservation` / `discovery` / `sns` |
| isActive | Boolean | グローバル有効/無効 |

> 媒体追加（GBP/Instagram など）は Phase 2 以降だが、マスタは Phase 1 で先に用意する。

### 1.2 月次レポート（`MarketingMonthlyReport`）

媒体×店舗×月のサマリーを1行にまとめる集計テーブル。

| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | PK |
| storeId | UUID | FK → stores |
| channelId | UUID | FK → marketing_channels |
| month | VARCHAR(7) | `YYYY-MM` |
| metrics | Json | 媒体ごとの指標（PV/予約数/フラット来店/タップ数 等） |
| sourceBlobUrl | String | アップロードしたCSVの Vercel Blob URL |
| importedAt | TIMestamptz | 取り込み実施日時 |
| importedBy | UUID | 実施ユーザー |

UNIQUE: `(storeId, channelId, month)`

> `metrics` を Json にしておくことで、媒体ごとのカラム差異（食べログのページ別タップ数 / ホットペッパーの流入導線など）に対応する。固定カラムは持たない方針。

### 1.3 取り込みログ（`MarketingImportLog`）

既存の `CsvImportLog` と同等思想。媒体ごとに分けて運用しやすくする。

| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | PK |
| storeId | UUID | |
| channelId | UUID | |
| month | VARCHAR(7) | |
| status | VARCHAR(20) | `success` / `failed` / `partial` |
| errorMessage | Text? | |
| createdAt | Timestamptz | |
| createdBy | UUID | |

### 1.4 既存 `StoreSettings.featureFlags` への追加

X-01（店舗ごとの契約媒体）は **新規テーブルを作らず**、既存の店舗別機能設定の枠組みに乗せる（`docs/01_design/店舗別機能設定.md` 参照）。

```ts
export interface FeatureFlags {
  dailyReportPosition?: boolean;
  // 追加
  marketingChannels?: {
    tabelog?: boolean;
    tabelogNote?: boolean;
    hotpepper?: boolean;
    gbp?: boolean;
    instagram?: boolean;
  };
}
```

> 管理者のみがON/OFFを切り替える。設定値（媒体側URL等）が必要になったら `customSettings.marketing` 配下に格納。

### 1.5 既存 `StoreGroup` への相乗り（S-02）

S-01 のコンサル用ロールは **新規ロール `consultant` を作らず**、既存の `StoreGroup` に `groupType = 'consultant'` を持たせる方針で初期実装する（Phase 1 を最小コストで完了するため）。

- 既存 `StoreGroupMember` の多対多で「コンサル割当店舗」を表現
- コンサル用グループは PL／売上の詳細ではなく **マーケビューと月次レポートのみ閲覧可能** にミドルウェアで絞る

`StoreGroup` に追加するカラム:

| カラム | 型 | 既定値 | 説明 |
|---|---|---|---|
| groupType | VARCHAR(20) | `'general'` | `'general'` または `'consultant'` |
| viewScope | Json | `{}` | 閲覧可能ビューのホワイトリスト（例: `{ marketing: true, pl: false, sales: false }`） |

> マイグレーション時、既存レコードは `groupType='general'` ・`viewScope='{}'` で埋める。

## 2. APIルート設計

既存 `app/api/` 配下に `marketing/` セグメントを新設する。エラー形式は既存規約に準拠（`{ success, data | error }`）。

| メソッド | パス | 内容 |
|---|---|---|
| GET  | `/api/marketing/channels` | 媒体マスタ一覧 |
| GET  | `/api/marketing/reports?storeId&month` | 媒体横断サマリー（A-01） |
| GET  | `/api/marketing/reports/:storeId/:channel/:month` | 個別媒体の月次レポート |
| POST | `/api/marketing/upload` | 媒体CSVアップロード（M-08） |
| GET  | `/api/marketing/months?storeId` | 取り込み済み月一覧 |
| DELETE | `/api/marketing/reports/:id` | 取り込み取り消し（管理者のみ） |

### 2.1 アップロードAPI仕様

```ts
// Request (multipart)
POST /api/marketing/upload
fields: {
  storeId: string,
  channel: 'tabelog' | 'tabelog_note' | 'hotpepper',
  month: 'YYYY-MM',
}
file: <CSV>

// Response
{ success: true, data: { reportId: string, metrics: Record<string, number> } }
```

処理フロー:
1. 認証チェック（admin or 該当 storeId 所属の store/group）
2. CSV を Shift-JIS / UTF-8 自動判定（既存 `lib/csv/` の `detectEncoding` を再利用）
3. 媒体別パーサで指標を抽出 → `metrics` Json に格納
4. 原本CSVを Vercel Blob に `marketing/{storeId}/{channel}/{month}.csv` で保存
5. `MarketingMonthlyReport` に UPSERT（同 storeId/channel/month は上書き）
6. `MarketingImportLog` に成否を記録

### 2.2 媒体別パーサ（Phase 1 スコープ）

- `lib/marketing/parsers/tabelog.ts`：PV／予約数／アクセスレポート由来の月次合計
- `lib/marketing/parsers/tabelogNote.ts`：予約件数／フラット来店件数／キャンセル
- `lib/marketing/parsers/hotpepper.ts`：PV／予約／流入経路

> いずれも **「人が画面からCSVをダウンロードした原本」を想定**。媒体のCSV仕様が変わる前提で、`channelVersion` フィールドをパーサ内で持ち、将来のフォーマット差を吸収。

## 3. 画面構成

### 3.1 マーケダッシュボードタブ

`/store/[storeId]/marketing/[month]` に新ページを追加。既存のダッシュボード（`/dashboard/[month]`）の並びに **「マーケ」タブを追加**。

タブ構成（Phase 1）:

| タブ | 内容 |
|---|---|
| 横断サマリー | A-01：契約媒体の主要KPIを1枚カード／表で表示 |
| 食べログ | M-01：PV／予約／タップ数の月次推移＋当月詳細 |
| 食べログノート | M-03：予約数とフラット来店比、キャンセル率 |
| ホットペッパー | M-04：PV／予約／流入導線の月次 |

> フィーチャーフラグで「契約していない媒体タブは非表示」にする（X-01）。

### 3.2 アップロード画面（M-08）

`/store/[storeId]/marketing/upload` を新設。既存 `/upload`（売上）を雛形に媒体別に複製。

UI要素:
- 媒体タブ（食べログ／食べログノート／ホットペッパー）
- 対象月セレクト
- ファイルアップロードエリア（ドラッグ＆ドロップ）
- アップロード結果の媒体別プレビュー（取り込んだ指標を一覧表示）
- 取り込み履歴（最近5件）

### 3.3 媒体フィーチャーフラグ管理（X-01）

既存「管理者設定」画面（`/admin` 配下）に **「店舗の契約媒体」セクション**を追加し、`featureFlags.marketingChannels` を編集可能にする。

## 4. 権限制御（S-01／S-02）

### 4.1 ミドルウェア追加

`/store/[storeId]/marketing/...` および `/api/marketing/...` に対して以下を判定:

| ロール | 判定 |
|---|---|
| admin | 常に許可 |
| store | `session.storeId === paramStoreId` のみ許可 |
| group (`general`) | `viewScope.marketing` が true、かつ `StoreGroupMember` に storeId を含む |
| group (`consultant`) | 上に加えて **PL／売上APIは拒否**（`/api/pl/*`、`/api/data` 等） |

### 4.2 コンサルロールの拒否対象

Phase 1 では以下の既存ルートをコンサルから明示的にブロック:

- `/api/pl/**`（PL関連すべて）
- `/api/data`（売上明細）
- `/api/upload`（売上アップロード）
- `/api/store-reports/**`（個別レポート）

→ 「マーケビュー」「月次レポート」「自身がコメントを書く欄」のみアクセス可能。

### 4.3 媒体ログイン情報の集中管理（S-03）

これは **アプリの機能ではなく運用ルール** として `docs/02_operations/媒体ログイン情報管理.md` に切り出す（本書とセット）。RAL側で1Password等のVaultに集中保管し、外注へは渡さない。ダッシュボードには媒体側のURLとアカウント識別名のみ表示し、ID/PWは表示しない。

## 8. リスクとフォールバック

- **媒体CSV仕様の変動**: 食べログ・ホットペッパーは管理画面の改修が頻繁。`channelVersion` フィールドでバージョン管理し、パーサ差し替え可能にしておく。
- **CSV取得が手動である継続前提**: 自動スクレイピングは Phase 1 ではやらない。失敗時はマニュアル再アップで回復できる UI を必ず用意する。
- **コンサル割当の取り違え**: ACL の閲覧スコープ Json を破壊した場合に全店舗閲覧されないよう、`viewScope` 未設定時は **拒否デフォルト** にする。
- **既存テストの破壊**: 既存 `__tests__` の認証/アップロード関連テストに、`groupType='general'` が前提のケースを足してリグレッションを検知する。
