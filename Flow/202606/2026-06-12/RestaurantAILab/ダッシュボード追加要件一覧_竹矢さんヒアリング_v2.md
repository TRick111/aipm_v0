## 0. 背景と要望の構造

竹矢さんが既存ダッシュボード（売上分析・PL）に対して「マーケ／集客系のデータを束ねたい」という追加要望をしているのが今回のヒアリングの主旨。背景には以下がある。

1. **CMOプラン／CFOプランの外注化**を今月中に開始予定。外注コンサルにクライアントの食べログ／ホットペッパー等の **ログインID・パスワードを直接渡したくない**（支払情報など機密情報まで見えてしまう）。
2. **マーケティングパネル（自社の改善ロジック）** はあるが、現在は各媒体に手作業でログインして数字を集計しており、コンサル品質と運用負荷が律速になっている。
3. **アウトプットはダッシュボード単体ではなく、スライド形式**で進行したい（既に週報・月報の納品形態がスライド中心へ移行している流れと一致）。
4. **更新頻度は月1回のアップロードで十分**（マーケはデイリーで動かす世界ではないため）。
5. **店舗ごとに契約媒体が異なる**ため、全店舗で同じUIを強制するより「各店舗が契約している媒体だけ表示」する柔軟性が必要。

→ つまり今回のスコープは **「各媒体のレポートを月次CSV等で取り込み、横断ビューにし、スライド形式で出力する」マーケ向け拡張**。

## 1. 既存ダッシュボードの位置づけ（前提整理）

| 領域 | 既存状態 | 今回の要望との関係 |
|---|---|---|
| 売上分析（ホーム／曜日・時間帯／メニュー／商品詳細／月次推移／全体サマリー／客数／客単価） | 実装済（POS：Airレジ / Dinii / URegi など） | 実装済み。今回は触らない |
| PL管理（日次入力・月次予実・経費カテゴリ） | 実装中（律速：客単価計算修正・予実UI・経費カテゴリ・PC日次入力） | 実装済み。今回は触らない |
| 認証（管理者／店舗ユーザー／店舗グループ） | 実装済 | **外注コンサル用ロール**を新規追加する必要あり |
| アップロード（CSV、Vercel Blob） | 売り上げについてのみ実装済 | **マーケ系CSV取り込み**に枠組みを流用 |
| 日報・LINE連携 | 実装済 | 今回は触らない |
| アウトプット | ダッシュボードUI中心 | **スライド／PDF出力**を新規追加 |

→ 既存の認証・アップロード・月単位データ管理の枠組みを活かし、**「マーケ媒体取込→集計→スライド出力」レーンを増築**する形になる。

## 2. 要件一覧（カテゴリ別）

ID は M（マーケ取込）、A（分析）、O（アウトプット）、S（セキュリティ）、X（横断）で分類。
**本書スコープ＝★★以上のみ**。★（将来／保留）は §2-6 に参考掲載のみ。

## 3. フェーズ分け（★★以上のみ・2フェーズ）

判断基準: **(a) 外注コンサル運用を始められる最小単位を Phase A に寄せる**、**(b) Phase B はクライアント体験（スライド納品・打ち手提示）を完成させる**。

---

### Phase A：運用開始までの最短機能

**目的**: 外注コンサルが「ID/PW を渡されずに、月次でクライアントのマーケ数字を見て、ミーティング進行できる」状態を作る。

**含まれる要件**:
- **権限・運用ルール**: S-01／S-02／S-03／X-01／X-02
- **アップロード基盤**: M-08
- **媒体取込（主要予約媒体）**: M-01／M-03／M-04／**M-09／M-10**（食べログ・食べログノート・ホットペッパー・**一休・オズモール**）
- **横断ビュー**: A-01（媒体横断KPIサマリー1枚）

**Phase A 完了時にできること**:

| 役割 | 具体的にできるようになること |
|---|---|
| 外注コンサル | コンサル用ログインで自分の担当店舗だけにアクセス。媒体ID/PW は見えない。担当店舗の今月の食べログ／食べログノート／ホットペッパー／一休／オズモールの月次数字をダッシュボードで一覧でき、ミーティングを進行できる |
| RAL 内（吉田・町田・田中） | 店舗ごとに契約媒体の ON/OFF を管理画面から切り替えできる。新しい店舗を追加する手順がドキュメント化されている |
| 店舗（クライアント） | 自店舗ユーザーでログインしたとき、マーケタブが追加され、契約媒体の数字が見える |
| データ運用 | 月1回、担当者が「媒体別CSV→アップロード画面」で取り込み。失敗時は履歴から再アップロードで回復可能 |

**Phase A 完了時にまだできないこと**:
- スライド／PDF 出力（手作業でスクショして納品が必要）
- マーケパネル準拠の縦並びレイアウト（数字は出るが、打ち手提示の構造化はまだ）
- GBP／Instagram の取り込み
- 媒体ログイン情報集中管理の **実運用フロー**（ルールは定義済だが、誰が CSV をダウンロードするかは §4-A で要確定）

---

### Phase B：追加機能

**目的**: クライアントへの納品物（スライド形式の月次レポート）と、マーケパネルに基づく打ち手提示まで一気通貫させる。Phase A の運用が回り始めた直後から並行で着手。

**含まれる要件**:
- **マーケパネル化**: A-02／A-03
- **媒体取込の拡充**: M-02／M-05（GBP）／M-06（Instagram）
- **近隣比較**: A-04
- **Instagram分析**: A-07
- **アウトプット**: O-01（スライド）／O-02（PDF）
- **監査**: S-04（アクセスログ）

**Phase B 完了時にできること**:

| 役割 | 具体的にできるようになること |
|---|---|
| 外注コンサル | クライアントMTG用のスライド／PDFをダッシュボードからワンクリックで生成。マーケパネルに沿った順番で数字が並び、弱い指標と打ち手が自動でハイライト |
| 外注コンサル | GBP の検索キーワード・周辺シェア、Instagram の投稿別エンゲージメント、食べログ由来の近隣店舗比較まで含めて打ち手を提案できる |
| RAL 内 | コンサルがダッシュボードで何を見たか／何を出力したかの監査ログが残る（情報漏洩リスクの早期検知） |
| クライアント | 月次納品物が「ダッシュボードの数字＋打ち手スライド」のセットになり、外注コンサルが入ってもサービス品質が均質化 |

**Phase B 完了後に残るもの（スコープ外）**:
- 周辺店舗データ統合ビュー（A-05、エリア内クライアント数の条件付き）
- エリア人口・通行人数連動（A-06、外部データ選定後）
- SaaS型SNS分析ツール連携（X-03、社内のInstagram運用方針が固まった後）

## 4. 確認事項（竹矢さん／吉田さんに確認）

### 4-A. 媒体ログイン情報を外注に渡さない場合の「データ入力業務」をどう設計するか ★最重要

S-03 で「媒体 ID/PW は社内集中管理し、外注コンサルには渡さない」と決めると、各媒体の管理画面から **月次CSVをダウンロードしてアップロード画面に投げる作業** が必ず発生する。誰がやるかで運用コストと実装範囲が変わる。

| 案 | 担い手 | コスト | リスク | 本書での扱い |
|---|---|---|---|---|
| **案①: 社内（RAL／Chef's Room）に人を回す** | 田中／町田／Chef's Roomメンバー | 媒体数 × 店舗数 × 月1回。例: 5媒体 × 5店舗 = 25回／月。1回10分なら 4〜5時間／月 | 担い手が休んだ時の冗長性なし | Phase A の **運用フロー（X-02）で人手前提のSOP** を起こす |
| **案②: AI操作（Playwright 等）でCSV自動取得** | 開発（田中） | 初期開発: 媒体1つあたり2〜4日程度。媒体側UI変更時に保守コスト発生 | 媒体規約でスクレイピング禁止のものがある（食べログは要確認）／2要素認証時の取り扱い | **Phase A では取らない**（運用開始を遅らせない）。Phase B 以降に **段階導入**を検討し、本書では §4-A 論点として保留 |
| **案③: クライアント本人が CSV をダウンロードして渡す** | 各店舗側 | 店舗側の作業負荷／教育コスト | 月次CSV取得が抜け落ちる店舗が出る | 採用しない（クライアント体験が悪化、CMO/CFO外注化の前提と矛盾） |

**推奨**: **案①を Phase A で確定**して運用開始 → **回数の多い媒体から案②（Playwright自動取得）に置き換える Phase B 後半タスク**として乗せる。
- 案②に倒す前提なら、Playwright スクリプトの実行基盤は既存の `scripts/` ／ `skills/` 配下に置けるか（Airレジ自動抽出RPAの先例あり）を事前確認。
- 媒体規約・2要素認証の整理は別途タスクとして発生（食べログ／一休／オズモールの利用規約レビュー）。

**→ 確認したいこと**: 案①／②／併用、いずれで Phase A をスタートさせるか？

### 4-B. その他の確認事項

1. **外注化のスタート時期**: Phase A の最終納期はそこに合わせていいか？
2. **対象店舗の優先順位**: 麻布しき旗の台・本店／BFA からスタートでよいか？（店舗ごとの契約媒体構成によって M-09／M-10 の優先順位も決まる）
3. **アウトプットの第一形態**: O-01 は HTMLスライド（週報パイプラインと同じD3.js HTML）でよいか、PPT必須か？
4. **媒体CSVの原本サンプル**: 各媒体（食べログ／食べログノート／ホットペッパー／**一休／オズモール**）の月次レポートCSV原本を各1ファイル欲しい（パーサ実装の前提）。
5. **マーケティングパネル本体の共有**: A-02／A-03 を実装するためにパネルの全項目・閾値・打ち手をJSON化したい。次回MTGで原本を共有してもらえるか？
6. **GBPアカウントの権限委譲**: M-05 をAPIで取りに行く場合、各クライアントのGBPに対するOAuth委任が必要。これを集中管理する運用が成立するか？
