## 3. 媒体 × KPI マッピング（★最重要：ここを確認してください）

> 凡例:
> - ✅ **Phase 1 で取得**（運用開始時に必須）
> - 🟡 **Phase 2 で取得**（運用が回ってから追加）
> - ❓ **要確認** — 媒体の管理画面で CSV／レポートに含まれているか、サンプル確認が必要
> - ❌ **取得しない**（媒体の仕様上そもそも提供されない／別媒体の管轄）
> - **─** **対象外**（その媒体に該当機能がない・概念がない）

#### 3-0-1. このマトリックスから読み取れること（PO 向け解説）

- **食べログノートが Phase 1 で最も多くの KPI を提供する**（公式ヘルプに分析機能が詳細明示）→ 食べログノート契約店舗での運用が一番厚みのある提案になる
- **PV と予約数は食べログ／食べログノート／ホットペッパーで確実に取れる** → 「媒体横断のPV／予約数推移」は Phase 1 で確実に実現
- **一休・オズモールは Web 情報が極端に薄い** → サンプル CSV／管理画面スクショの提供が最優先。これがないとマトリックスの精度が上がらない
- **ホットペッパーは「マンスリークライアントレポート」がレポート源** → 取り込みは PDF/Excel 想定。営業担当から月次でもらう前提が成立しないと運用設計が崩れる
- **食べログの予約経路内訳は契約条件あり**（050番号 + PR/集客契約） → 全店舗で同じ深さの分析が成立するわけではない
- **ランキング・周辺シェアは食べログだけ** → 「順位の動き」を語る材料は食べログ依存
- **フラット来店・キャンセルは食べログノートだけ** → 「予約と実来店のギャップ」は食べログノート契約店舗でのみ語れる
- **プラン別予約・客単価は一休が主役**（公式に確認） + 食べログノートも CSV で全件取得可

#### 3-0-3. サンプル確認待ち（PO への依頼項目）

Web では確証できなかったセルを、優先度順にまとめます。**§6-B「サンプル CSV 提供」のときに、ここを見ながら確認**できる形にしてあります。

| 確認したい媒体 / KPI | 優先度 | 確認のヒント（店舗管理画面で何を見れば確定できるか） |
|---|:---:|---|
| **一休：PV／流入経路／ページ別タップ** | high | 店舗管理画面に「アクセス解析」「効果測定」相当のメニューがあるか。送客レポートCSVのカラムに「PV」「閲覧数」が含まれるか |
| **一休：客単価／プラン別予約／在庫消化率／ポイント還元率** | high | RESZAIKO/店舗管理画面の月次CSVに「コース別予約数」「予約単価」「ポイント還元率」カラムがあるか |
| **オズモール：PV／予約経路内訳／特集枠経由予約／口コミ集計** | high | 月次成果報酬レポートCSVに「特集」「編集記事」「予約経路」のフラグがあるか |

| **食べログ：CSV ダウンロード可否（全レポート共通）** | high | 「アクセス数レポート」「ページ別レポート」「来店指標」画面にCSV/Excelダウンロードボタンがあるか（PR/集客サービス契約店舗で） |
| **ホットペッパー：キャンセル件数／キャンセル率／ノーショー** | med | 予約リストCSV（レストランボード連携経由）のキャンセル理由カラムを集計可能か |

## 4. Phase 1：運用開始までの最短機能

### 🎯 Phase 1 のゴール

> **「外注コンサルが、ID/PW を渡されずに、毎月クライアントとのミーティングを進行できる」**

### Phase 1 で何ができるようになるか（PO 視点）

| 立場                   | できるようになること                                                                         |
| -------------------- | ---------------------------------------------------------------------------------- |
| **外注コンサル**           | 自分の担当店舗のログインで、契約媒体（食べログ／食べログノート／ホットペッパー／一休／オズモール）の **月次主要KPI**（§3 の ✅ 項目）を一画面で見られる |
| **外注コンサル**           | 媒体のログインID・パスワード・支払情報は **一切見えない**                                                   |
| **外注コンサル**           | 月1回のミーティングで「先月のPVと予約はこうでした、来月はこの打ち手を提案します」が即座に語れる                                  |
| **店舗（クライアント）**       | 自店舗ユーザーでログインしたとき、マーケタブが追加されており、契約媒体の数字を自分でも確認できる                                   |
| Chef's Room          | 店舗ごとに「契約している媒体」を ON/OFF できる。新店舗の追加手順が文書化されている                                      |
| **データ運用担当（人 or AI）** | 月1回、各媒体の月次CSVをアップロード画面から流し込むだけで取り込み完了。失敗したら再アップで回復                                 |

### Phase 1 でカバーする媒体・KPI

- **食べログ**: PV／予約数／ページ別タップ数／エリア内ランキング

- **一休**: PV／予約数／プラン別予約／平均客単価
- **オズモール**: PV／予約数／特集枠経由数／レビュー件数・点数

→ つまり、**「予約系媒体5つの基本KPIで媒体横断のミーティングが成立する」状態** を Phase 1 で作ります。

### Phase 1 のアウトプット（追加スコープ：2026-06-15）

  - 週報パイプラインの D3.js HTMLスライド生成モジュールを再利用
  - 「外注コンサルが ID/PW なしでクライアントに月次レポートをスライド納品できる」状態を Phase 1 で完成させるため

### Phase 1 でまだできないこと（あえて Phase 2 に回すもの）

## 6. 確認事項（PO 判断が必要）

### 6-A. 媒体ログイン情報を外注に渡さない場合の「データ入力作業」をどう設計するか ★最重要

外注コンサルに媒体ID/PWを渡さない方針を取ると、**月次CSVをダウンロードしてアップロード画面に投げる作業を誰がやるか** を決める必要があります。

| 案 | やる人 | 月次の負荷 | リスク | 推奨 |
|---|---|---|---|---|
| **①社内人手で回す** | RAL or Chef's Room メンバー | 5媒体 × 5店舗 = 25回／月。1回10分なら 4〜5時間／月 | 担当者が休んだ時の冗長性なし | **Phase 1 推奨** |
| **②AI自動取得（Playwright）** | 自動化 | 媒体側UI変更時に保守発生 | 媒体規約（特に食べログ）でスクレイピング禁止のものあり／2要素認証時の運用 | **Phase 2 後半で段階導入** |
| **③クライアント本人が CSV をダウンロード** | 各店舗 | 店舗側に新たな作業負荷 | 月次取り込みが抜け落ちる店舗が出るリスク | **採用しない** |

> **田中の推奨**: Phase 1 は **案①で始める** → Phase 2 で **案②（Playwright自動取得）に置き換え** が現実的。媒体規約レビューと2要素認証の運用整理を並行で進める。

**→ 竹矢さんへの確認**: この方針で OK か？ それとも別案ありか？
