# マーケダッシュボード WBS（エピック／フィーチャー／ユーザーストーリー）v1

- 作成日: 2026-06-13 ／ 更新: 2026-06-13（縦切り再構成）
- 根拠資料: `ダッシュボード追加要件一覧_竹矢さんヒアリング_v3.md`／`0519_マーケダッシュボード実装方針.md`
- 階層: **エピック（Phase）→ フィーチャー（機能単位）→ ユーザーストーリー**
- Phase 1 の定義: **最低限、予約系5媒体すべてからKPIが取得できる**こと
- 凡例: 🔶 = サンプルCSV確認待ち（v3 §3-0-3）に依存し、スコープが変動するストーリー

## 構成方針（縦切り）

- 旧「マーケデータ基盤」「アップロード共通」フィーチャーは廃止し、**1媒体目（食べログノート）に統合**。最初の媒体を end-to-end で貫通させることで基盤を成立させる（トレーサーバレット方式）
- パーサの共通化は先に設計しない。**2媒体目（食べログ）を通すときに初めて抽出**する（rule of three）。3媒体目（ホットペッパー＝PDF/Excel）が抽象の耐性テストになる
- ただし **`metrics` Json の契約（UNIQUEキー・channel code体系・月の表現）だけは5媒体のサンプル確認後に凍結**する。実装は1媒体ずつでも、スキーマ判断は全媒体分のサンプルを見てから
- `StoreGroup` のスキーマ変更は基盤ではなく **ACLフィーチャーの所掌**

# エピック E1: Phase 1 — 5媒体KPI取得と外注コンサル運用開始

> ゴール: 外注コンサルが、ID/PWを渡されずに、毎月クライアントMTGを進行できる

## F1-1. 食べログノートKPI取得＋マーケデータ基盤（トレーサーバレット）

> 完了定義: 食べログノートのCSVを「アップロード → 取込 → 媒体タブで閲覧」まで貫通。以降の媒体はこの上に乗るだけ

### 基盤（このフィーチャー内で構築）

| ID | ユーザーストーリー |
|---|---|
| US1-1-1 | 開発者として、**5媒体すべてのサンプルを確認した上で** `metrics` Json契約（UNIQUE `(storeId, channelId, month)`・channel code体系）を凍結できる ★実装に先行する唯一の全媒体横断判断 |
| US1-1-2 | 開発者として、媒体マスタ（`MarketingChannel`、7媒体: 5媒体＋GBP/Instagram）をseed投入できる |
| US1-1-3 | 開発者として、`MarketingMonthlyReport`／`MarketingImportLog` をdev環境に適用できる（dev→staging→prod の適用順を厳守。`StoreGroup` 変更は含まない→F1-6） |
| US1-1-4 | データ運用担当として、媒体・店舗・対象月を選んでファイルをD&Dアップロードできる |
| US1-1-5 | データ運用担当として、取込直後のプレビューで原本と数値を検算できる |
| US1-1-6 | データ運用担当として、同月の再アップロードで上書き復旧できる（UPSERT） |
| US1-1-7 | データ運用担当として、取込履歴（5件）と取込済み月一覧を確認できる |
| US1-1-8 | 開発者として、Shift-JIS/UTF-8を自動判定して取り込める（既存 `lib/csv/detectEncoding` 再利用） |
| US1-1-9 | 管理者として、誤った取込をレポート単位で取り消せる |
| US1-1-10 | 開発者として、原本を Vercel Blob（`marketing/{storeId}/{channel}/{month}`）に保全し、後から再パースできる |

## 補足: 実装順序の依存関係（Phase 1）

```
【先行ゲート】5媒体サンプル収集 → metrics Json 契約凍結（US1-1-1）
        │
F1-1 食べログノート＋基盤（end-to-end貫通）
        │
F1-2 食べログ（2媒体目・共通部を抽出リファクタ）
        │
F1-3 HP ／ F1-4 一休 ／ F1-5 オズモール（共通基盤の上に並列実装可）
        │
F1-8 横断サマリー（取れた媒体から順次表示）

F1-6 ACL（完全独立・いつでも並列可）
F1-7 フラグ（F1-1完了後、軽量）
F1-9 運用ルール（実装と独立、ただし納品物に含む）
```

- 🔶付きストーリーはサンプル確認の結果次第で**削除も含めて即断**する。スコープ協議に時間を使わない
- F1-3（ホットペッパー）のみ US1-3-1 という**アプリ外の運用前提ストーリー**を持つ。ここが最大の外部依存
- 納品最低ライン: F1-1＋F1-2＋F1-3（縮退可）＋F1-6＋F1-8。一休・オズモールはUAT週へのスリップ許容
