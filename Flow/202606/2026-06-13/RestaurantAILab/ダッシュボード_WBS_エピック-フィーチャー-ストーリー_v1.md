# エピック E1: Phase 1 — 5媒体KPI取得と外注コンサル運用開始

> ゴール: 外注コンサルが、ID/PWを渡されずに、毎月クライアントMTGを進行できる

## F1-1. マーケデータ基盤

| ID | ユーザーストーリー |
|---|---|
| US1-1-1 | 開発者として、媒体マスタ（`MarketingChannel`、7媒体: 食べログ／食べログノート／ホットペッパー／一休／オズモール／GBP／Instagram）をseed投入できる。Phase 2 媒体もマスタだけ先行登録する |
| US1-1-2 | 開発者として、媒体×店舗×月のKPIを `MarketingMonthlyReport.metrics`（Json）に格納できる。媒体ごとのカラム差異を固定カラムなしで吸収する |
| US1-1-3 | 開発者として、取込の成否を `MarketingImportLog` で追跡できる |
| US1-1-4 | 開発者として、スキーマ変更を dev→staging→prod の順で安全に適用できる（既存テストのリグレッションなし） |

## F1-2. 媒体CSVアップロード機能（M-08 共通基盤）

| ID | ユーザーストーリー |
|---|---|
| US1-2-1 | データ運用担当として、媒体・店舗・対象月を選んでファイルをD&Dアップロードできる |
| US1-2-2 | データ運用担当として、アップロード直後に「取り込まれた指標の一覧」をプレビューで確認し、原本と一致しているか検算できる |
| US1-2-3 | データ運用担当として、取込に失敗しても同月の再アップロードで上書き復旧できる（UPSERT） |
| US1-2-4 | データ運用担当として、直近の取込履歴（5件）と取込済み月の一覧を確認できる |
| US1-2-5 | 開発者として、Shift-JIS/UTF-8 を自動判定して取り込める（既存 `lib/csv/detectEncoding` 再利用） |
| US1-2-6 | 管理者として、誤った取込をレポート単位で取り消せる（DELETE、管理者のみ） |
| US1-2-7 | 開発者として、原本ファイルを Vercel Blob（`marketing/{storeId}/{channel}/{month}`）に保全し、後から再パースできる |

## F1-8. 外注コンサル向けACL（S-01/S-02）★情報漏洩リスク直結・省略禁止

| ID | ユーザーストーリー |
|---|---|
| US1-8-1 | 管理者として、コンサル用グループ（`StoreGroup.groupType='consultant'`）を作成し、担当店舗を割り当てられる |

| US1-8-4 | 開発者として、`viewScope` 未設定時は拒否デフォルトになることをテストで保証できる |
| US1-8-5 | 開発者として、コンサルロールでの全ルート総当たりアクセステストを自動化できる |
| US1-8-6 | 外注コンサルとして、媒体ログインID/PW・支払情報は画面上のどこにも表示されない |

## 補足: 実装順序の依存関係（Phase 1）

```
F1-1 基盤 ──→ F1-2 アップロード共通 ──→ F1-3〜F1-7 媒体別パーサ（並列可）──→ F1-10 横断ビュー
                                          ↑ 各媒体のサンプル入手が前提
F1-8 ACL（独立・並列可）
F1-9 フラグ（F1-1後、軽量）
F1-11 運用ルール（実装と独立、ただし納品物に含む）
```

- 🔶付きストーリーはサンプル確認の結果次第で**削除も含めて即断**する。スコープ協議に時間を使わない
- F1-5（ホットペッパー）のみ US1-5-1 という**アプリ外の運用前提ストーリー**を持つ。ここが最大の外部依存
