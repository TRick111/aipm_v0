# マーケダッシュボード WBS（エピック／フィーチャー／ユーザーストーリー）v1

- 作成日: 2026-06-13 ／ 更新: 2026-06-13（縦切り再構成＋モック確認フェーズ追加・表現見直し）
- 根拠資料: `ダッシュボード追加要件一覧_竹矢さんヒアリング_v3.md`／`0519_マーケダッシュボード実装方針.md`
- 階層: **エピック（Phase）→ フィーチャー（機能単位）→ ユーザーストーリー**
- Phase 1 の定義: **最低限、予約系5媒体すべてからKPIを取得できる状態**にすること
- 凡例:
  - 🔶 = サンプル確認の結果によってスコープが変わるストーリー（v3 §3-0-3 の「要確認」項目に依存）
  - 「凍結」 = **これ以降は変更しない、と決めて確定させること**（凍結するまでは自由に変えてよい）

## 進め方の方針

初動は次の3つを回す。

1. **取得できるデータの確定が最初**。5媒体すべてのサンプルレポートを入手し、「どの媒体からどのKPIが取れるのか」を確定させる（F1-1）
2. 確定したKPIをもとに**画面モックを作り、竹矢さん（PO）に確認**してもらう。画面の合意を実装より先に取ることで、UAT段階での大きな手戻りを防ぐ（F1-2）
3. モック確認と**並行して、アップロードなど基盤部分の実装に着手**する（F1-3）。基盤は画面デザインの影響を受けにくいため、PO確認を待たずに進められる

実装の組み立て方は「縦切り」とする。

- 「データ基盤」を独立したフィーチャーにはせず、**1媒体目（食べログノート）の実装に含めて、アップロードから画面表示まで一気通貫で動かす**。最初の1媒体を端から端まで通し、それを以降の土台にする進め方（トレーサーバレット）
- パーサの共通化は最初から設計しない。**2媒体目（食べログ)を実装するときに、共通部分が見えてから切り出す**。3媒体目のホットペッパーはPDF/Excel取込で毛色が違うため、切り出した共通基盤が通用するかの試金石になる
- ただし **`metrics` Json の設計（行の一意性・媒体コード体系・値の持ち方）だけは、5媒体すべてのサンプルを見てから凍結する**。1媒体のCSVだけを見て決めると、後続媒体のデータが入れ物に収まらず作り直しになるため
- `StoreGroup` のスキーマ変更は、データ基盤ではなく **ACLフィーチャー（F1-8）の担当範囲**とする

# エピック E1: Phase 1 — 5媒体KPI取得と外注コンサル運用開始

> ゴール: 外注コンサルが、媒体のID/PWを渡されることなく、毎月のクライアントMTGを進行できる

## F1-1. 取得可能KPIの確定（全媒体サンプル調査）★すべての起点

> 完了定義: 5媒体それぞれについて「取得するKPIの一覧」が確定し、データの入れ物（スキーマ）の設計判断が下せる状態

| ID | ユーザーストーリー |
|---|---|
| US1-1-1 | RALとして、5媒体の管理画面にログインし、直近2ヶ月分のレポート／CSVサンプルを入手できる（ID/PWは入手済み） |
| US1-1-2 | RALとして、ホットペッパーの営業担当にマンスリークライアントレポートのサンプル提供を依頼し、受け取れる ★入手待ちが最も長引きやすい外部依存。最初に動かす |
| US1-1-3 | 開発者として、要件一覧v3で「要確認（❓）」だったKPIをサンプルと突き合わせ、媒体ごとの「取得できるKPI一覧」を確定できる |
| US1-1-4 | RALとして、食べログの来店指標の前提条件（050番号＋PR/集客サービス契約）が対象店舗で成立しているかを確認できる |
| US1-1-5 | 開発者として、全媒体のサンプルを踏まえて `metrics` Json の設計——「店舗×媒体×月で1行」という一意性ルール、媒体コードの命名体系、値の持ち方——を凍結できる ★実装に先行して全媒体を横断的に見る必要がある、唯一の設計判断 |

## F1-2. 画面モック作成とPO確認

> 完了定義: 竹矢さんが「この画面で外注コンサルの月次運用が成立する」と判断し、表示KPIと画面構成に合意している状態

| ID | ユーザーストーリー |
|---|---|
| US1-2-1 | 開発者として、F1-1で確定したKPI一覧をもとに、媒体横断サマリーと媒体別タブの画面モックを作成できる |
| US1-2-2 | 開発者として、アップロード画面（媒体・対象月の選択 → 取込結果プレビュー）のモックを作成できる |
| US1-2-3 | 竹矢さん（PO）として、モックを見て「コンサルがこの画面だけでクライアントMTGを進行できるか」を実装前に判断できる |
| US1-2-4 | 開発者として、モックへのフィードバックを反映し、表示KPI・画面構成を「合意済み」として確定できる（以降の画面変更はUATでの軽微な修正に限る） |

> 進め方: モックの作成・確認は **F1-3（基盤実装）と並行**で行う。基盤（スキーマ・アップロードAPI・取込処理）は画面デザインの影響をほぼ受けないため、POの回答待ちで実装を止めない

## 補足: 実装順序と依存関係（Phase 1）

```
F1-1 取得可能KPIの確定（全媒体サンプル調査）★すべての起点
   │
   ├─→ F1-2 画面モック作成 → 竹矢さん確認 → 表示KPI・画面構成の合意 ──┐
   │        （POの回答待ちの間も下の実装は止めない）                │
   └─→ F1-3 食べログノート＋基盤（一気通貫で実装）←── 合意内容を画面実装に反映
              │
        F1-4 食べログ（共通部分を切り出し）
              │
        F1-5 ホットペッパー ／ F1-6 一休 ／ F1-7 オズモール（共通基盤の上で並列実装可）
```
