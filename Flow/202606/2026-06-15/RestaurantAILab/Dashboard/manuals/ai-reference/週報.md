# 週報・月報機能 詳細仕様

> マスター: `../AI参照用_仕様マニュアル_v1.md`

---

## 1. 機能の役割

「ダッシュボードはあるけどレポートにならない」という現場課題への対応。
**管理者（RAL）が編集した HTML レポートを Blob にアップロード →  店舗オーナーがダッシュボードから閲覧** する読み物形式のレポート。

- ダッシュボードがリアルタイム可視化なのに対し、週報・月報は **「今週/今月見るべき論点を絞った読み物」** として配信
- 1ファイル = 1HTML（CSS/JS inline、自己完結型）

---

## 2. 画面構成

| パス | 用途 |
|------|------|
| `/store/{storeId}/weekly-reports` | 週報・月報の **統合一覧画面**（時系列マージ） |
| `/store/{storeId}/weekly-reports/{weekId}` | 週報の詳細閲覧（iframe + 印刷ボタン） |
| `/store/{storeId}/monthly-reports/{monthId}` | 月報の詳細閲覧（iframe + 印刷ボタン） |

> 注: `/monthly-reports/`（IDなし）は 404。一覧アクセスは `/weekly-reports` から（週報・月報がマージ表示される）。

---

## 3. 一覧画面（/store/{storeId}/weekly-reports）

### 3.1 表示

- 週報と月報を **`endDate` 降順でマージ** して時系列表示
- 年でグルーピング（年単位のアコーディオン）
- 1カードに: タイトル、期間（startDate〜endDate）、レポート種別バッジ（週報/月報）

### 3.2 関連API

```
GET /api/store-reports?storeId=...
  → 週報+月報を endDate 降順でマージして返す（UI一括取得用）

GET /api/weekly-reports?storeId=...    週報のみ
GET /api/monthly-reports?storeId=...   月報のみ
```

---

## 4. 詳細画面（閲覧）

### 4.1 表示構成

- iframe で `/api/weekly-reports/[id]/html` を読み込み
- 「印刷」ボタン → `/api/weekly-reports/[id]/printable` に遷移 → ブラウザの印刷ダイアログ → 「PDFとして保存」
- admin の場合は「削除」ボタンも表示

### 4.2 関連API

```
GET /api/weekly-reports/[id]              週報メタデータ（blobUrl 含む）
GET /api/weekly-reports/[id]/html         iframe 表示用 HTML プロキシ（Content-Type: text/html）
GET /api/weekly-reports/[id]/printable    印刷用CSS注入版（window.print() で PDF 保存用）

GET /api/monthly-reports/[id]             月報メタデータ
GET /api/monthly-reports/[id]/html        月報 HTML
GET /api/monthly-reports/[id]/printable   月報印刷用
```

### 4.3 印刷用CSS

`/printable` は元の HTML に以下のような CSS を注入してから返す:

```css
@media print {
  body { background: white; margin: 0; }
  .no-print { display: none; }
  /* 改ページ制御等 */
}
```

ブラウザの `window.print()` を呼ぶことで、紙ではなく **PDF として保存** する想定。

## 5. アップロード（admin のみ）

### 5.1 週報アップロード

```
POST /api/weekly-reports/upload (multipart/form-data)
  file:      HTMLファイル
  storeCode: 配信先店舗
  year:      年（YYYY）
  week:      週番号（1-53）
  startDate: 週の開始日
  endDate:   週の終了日
  title:     表示タイトル
```

**処理フロー**:
1. HTMLファイルを Vercel Blob にアップロード（`store-reports/{storeCode}/weekly/{year}/{week}.html`）
2. `weekly_reports` テーブルにメタデータ INSERT
3. **年×週で一意管理**（同じ年・同じ週で再アップロードすると上書き）

### 5.2 月報アップロード

```
POST /api/monthly-reports/upload (multipart/form-data)
  file:      HTMLファイル
  storeCode: 配信先店舗
  year:      年
  month:     月（1-12）
  startDate: 集計期間開始
  endDate:   集計期間終了
  title:     表示タイトル
```

同様に Blob にアップロード → `monthly_reports` テーブルに INSERT。

### 5.3 削除

```
DELETE /api/weekly-reports/[id]       週報削除（Blob + DB）
DELETE /api/monthly-reports/[id]      月報削除（Blob + DB）
```

admin のみ実行可能。

| 操作 | admin | store | group |
|------|:-----:|:-----:|:-----:|
| 一覧閲覧 | 全店舗 | 自店舗のみ | 所属店舗のみ |

| 印刷／PDF保存 | ○ | ○ | ○ |
| アップロード | ○ | × | × |
| 削除 | ○ | × | × |

---

## 8. 運用フロー（管理者側）

### 8.1 通常運用

1. 管理者（RAL）が週次／月次で HTML レポートを作成（テンプレ準拠）
2. Vercel ダッシュボードや CLI 経由でアップロード API を叩く、または admin の管理UIから（実装によりUIまたはCLIアップロード）
3. 店舗オーナーがダッシュボードから閲覧

### 8.2 同じ週／月の再アップロード

UNIQUE 制約により、同じ `(storeId, year, week)` または `(storeId, year, month)` で再アップロードすると **Blobとメタデータの両方が上書き** される。

---

## 9. LINE連携との関係

週報・月報のアップロード時点では **LINE通知は送信されない**（日報の店長締めとは別系統）。

別途、PLダッシュボード経由で店舗に「新しい週報が届きました」のような通知を出す場合は、フィーチャーフラグやカスタム実装で対応する（現状の標準機能には含まれない）。

## 10. よくある質問

### Q. 店舗ユーザーが週報をアップロードできるか

A. **不可**。アップロードは admin のみ。

### Q. 週報のフォーマットは決まっているか

A. HTMLファイルである以外の **フォーマット規定は無い**。管理者が編集者として体裁を整える。CSS/JS は inline 推奨（外部依存があるとブラウザによっては読み込み失敗）。

### Q. PDFを直接アップロードできるか

A. 現状の API は HTML 前提。PDFをアップロードする場合は、それを iframe で読めるラッパーHTMLを作成して上げる必要がある。

### Q. 配信履歴／配信先を限定したい

A. 1HTML = 1店舗 のため、店舗ごとに別ファイルを上げる運用。同じ内容を複数店舗に配信する場合は、店舗数分アップロードする。

### Q. 印刷時にデザインが崩れる

A. `/printable` エンドポイントは印刷用CSSを注入する。`@media print` ブロックを HTML 側にも入れておくとさらに崩れにくい。

### Q. 週番号の定義は？

A. アップロード時のフォームで指定する `year` / `week` をそのまま使う。アプリ側で ISO 週番号を強制してはいない。**運用上は ISO 8601 週番号** を推奨。

### Q. レポートを削除したらBlobは残るか

A. 削除APIは **Blob とメタデータの両方** を削除する。

### Q. 過去のレポートはずっと残るか

A. 削除されない限り **無期限保存**。保持期間の自動制限はない。

### Q. 月報の startDate / endDate は1日始まり末日終わりか

A. アップロード時に指定する。カレンダー月でも会計月（例: 月末締めではなく10日締め）でも自由。

---

## 11. 制約・注意点

1. **HTMLは自己完結型推奨**。外部CSS/JS依存はブラウザの読み込みポリシーに依存
2. **画像は base64 inline か Blob URL を埋め込む**（外部参照は推奨しない）
3. **同一週／同一月のアップロードは上書き**（誤って上げてもやり直し可）
4. **削除は完全削除**（Blob＋DB）。復旧不可
5. **iframe表示のためサンドボックス制約あり**。フォーム送信や外部リダイレクトは動作しない場合あり
