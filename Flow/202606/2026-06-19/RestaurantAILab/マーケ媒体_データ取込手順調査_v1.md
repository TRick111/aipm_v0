## 0. エグゼクティブサマリー

### 0.1 全媒体に共通する結論

| 観点 | 結論 |
|---|---|
| 公式 API（店舗オーナーが自店 KPI を取れるもの） | **7 媒体すべてで提供なし**。リクルートのグルメ Web Service／ぐるなび API は店舗マスタ検索用で KPI 取得不可。Retty Food Data Platform は B2B マクロ用途 |
| CSV/Excel 出力 | **媒体ごとに大きく差**がある。レストランボード／食べログノート／RESZAIKO は CSV 出力が比較的整備。食べログ本体／ホットペッパー本体／オズモール／Retty／ぐるなびは出力範囲が限定的 or 不明 |
| 規約上の自動取得 | **7 媒体すべてで「スクレイピング・Bot による無断自動取得は規約違反リスクが高い」**。食べログ第 9 条、ホットペッパー第 12 条、ぐるなび第 7 条、Retty 第 14 条 等が該当 |
| 2 要素認証 | 食べログ本体／食べログノート／ぐるなび／オズモール／Retty／ホットペッパー店舗管理画面は **2FA なしの ID/PW 単一要素**。AirID（レストランボード）は 2FA 任意設定可。一休は宿泊側で 2FA 必須化済み・レストランも強化方向 |
| 推奨実装方式 | **「店舗オーナーが月次で CSV を手動 DL → ダッシュボードへアップロード」のセミ手動運用が共通の安全解**。Playwright 自動取得は媒体ごとの規約レビュー後にホワイトリスト方式で段階導入 |

## 1. 食べログ

### 1.1 公式 API
- **店舗オーナー向けの公式マーケ KPI 取得 API は提供なし**。
- 過去に存在した「料理メニュー API（一般公開）」は 2014 年頃終了。
- ebica／RESZAIKO／レスラク 等とは「API 連携」しているが、これは食べログ側がパートナー事業者に閉じて提供する予約双方向同期用で、店舗オーナーが直接叩ける形では非公開。
- アクセス数（PV）、来店指標、エリア内ランキング、クチコミ点数を取得できる公式 API・Webhook・データダウンロードバッチは公式ヘルプで確認できない。

### 1.2 CSV / Excel エクスポート
| レポート | 出力可否 | 備考 |
|---|:---:|---|
| トップページ「直近1ヶ月のアクセス数（PV）」 | ✕ | 画面表示のみ |
| 来店指標（電話成立数・ネット予約組数・クーポン/地図PV・電話番号クリック数） | ✕ | 画面表示のみ。PR/ネット予約/集客契約店舗のみ表示 |
| エリア内アクセスランキング | ✕ | 画面表示のみ |
| クチコミ（件数・平均点・本文一覧） | ✕ | 一覧の閲覧と返信のみ。エクスポートボタンなし |
| 食べログノートの分析（予約数／卓稼働率／経路／新規・リピート 等） | ◯ | §2 参照 |
| 「月間ページレポート」という名称の独立レポート | ❓ | 公式ヘルプに該当ページなし。管理画面トップの月次推移ビューを指す可能性 |

### 1.3 メール配信
- 月次マーケ KPI レポートをメール添付で配信する公式機能は確認できない。

### 1.4 利用規約上の自動取得可否
**食べログ利用規約**（https://tabelog.com/help/rules/）に以下の禁止条項：

- **第 6 条 (1) 第 1 号**: 「営業活動その他の営利を目的とした行為またはそれに準ずる行為やそのための準備行為を目的として、利用又はアクセスしてはならない」
- **第 9 条 (1) 第 1 号**: 「当社の事前の同意なく、複写、…再生、複製、…保管」を禁止
- **第 9 条 (1) 第 12 号**: 「その他、当社が不適切と判断すること」（包括的禁止）

→ 公開ページ／管理画面いずれも自動取得は規約違反リスク高。本番運用には食べログ側への書面同意が必要。

### 1.5 2 要素認証
- ログイン URL: `https://owner.tabelog.com/owner_account/login/`
- **ID + パスワードのみ。2FA / SMS / 認証アプリ / CAPTCHA の公式記述なし**
- パスワードリセットはメール経由（24 時間有効 URL + 再設定コード）

### 1.6 追加契約要件
- **「来店指標」全項目の表示には PR/ネット予約/集客 のいずれかの契約必須**
- 050 番号（食べログ予約専用番号）は **PR プレミアム 10／プレミアム 5 契約時のみ設定可能**
- 料金（2026/06 時点・税込）: ベーシック 27,500 円／プレミアム 5 = 55,000 円／プレミアム 10 = 110,000 円。ネット予約従量はランチ 110 円/人、ディナー 220 円/人

### 1.7 推奨取込方式
1. **【正規ルート】食べログノートの分析 CSV を活用**（§2）
2. **【現実解】管理画面のスクリーンショット → 人手 or Vision-LLM で抽出**（PV、エリアランキング、クチコミ件数・平均点）
3. **【条件付き】連携先 SaaS（ebica/RESZAIKO 等）の CSV/API 経由**
4. **【非推奨】Playwright 自動ログイン**: 規約 6/9 条抵触リスク、Bot 対策強化で停止リスク

実装方針: **「食べログノート CSV（半自動） + 管理画面スクショ手動転記」のハイブリッド** が最適バランス。

### 1.8 主要参照 URL
- 食べログ利用規約: https://tabelog.com/help/rules/
- 店舗会員ヘルプ: https://owner-support.tabelog.com/
- 来店指標を確認する: https://owner-support.tabelog.com/来店指標（電話数・ネット予約数など）を確認する-64ce834e68386a001c51a485
- 食べログ予約専用番号（050）について: https://owner-support.tabelog.com/食べログ予約専用番号（050から始まる電話番号）について-64ce834f2dc506001b804f50
- 食べログ予約システム × 外部連携の裏側: https://tech-blog.tabelog.com/entry/tabelog-reservation-system-external-integration

## 2. 食べログノート

### 2.1 公式 API / 外部連携
- **公開 API なし**（内部では GraphQL+RoR を利用）。
- 外部連携は **公式パートナー連携のみ**：
  - 予約台帳: ebica / Resty / TableCheck
  - サイトコントローラー: レスラク / RESZAIKO
  - メール取り込み: OZmall, ぐるなび, ホットペッパーグルメ, ヒトサラ, Retty, 一休, OMAKASE
  - POS: 食べログオーダー → スマレジ
- 汎用連携窓口（Webhook、自社 SaaS への push）は **提供なし**。
- 連携時の認証情報: 「食べログレストラン ID + ログイン ID + パスワード + API 認証キー（管理画面で発行）」

### 2.2 CSV / Excel エクスポート
CSV は **「分析」画面（管理画面 → その他 → 予約受付設定 → 左サイドバー「分析」）からのみ提供**。Excel 直接出力は不可。

| レポート | 主な指標 |
|---|---|
| ダッシュボード（当月予約サマリ） | 総客数、予約数、ウォークイン、卓稼働率、キャンセル率、平均人数、平均予約リードタイム |
| 予約数・総客数 | 日別／月別の来店データ |
| 卓稼働率 | 月単位 |
| 予約コース | TOP20 が画面表示、**CSV は全件出力可** |
| 予約経路 | 食べログネット予約／席のみ予約／電話／他媒体の比率 |
| 新規・リピート | 新規 vs リピート比率 |
| 予約キャンセル | キャンセル件数・キャンセル率 |
| 人数割合 | 平均人数の人数別構成 |
| リードタイム | 平均予約リードタイム |

- 集計単位: **1 日（06:00〜翌 06:00）**
- 過去月: **毎月月初に前月実績が確定**、当月・未来はリアルタイム
- **重要な制約**: ebica / Resty / RESZAIKO / レスラク と連携している店舗は **分析機能・CSV エクスポート自体が利用不可**
- 「予約一覧」画面そのものから一覧 CSV をダウンロードする公式手順は未掲載

### 2.3 利用規約上の自動取得可否
食べログ全体規約が準用される（§1.4 と同じ）。**スクレイピング名指しの禁止条項は無いが、第 6 条・第 9 条で実質禁止**。本番運用にはカカクコムへの書面同意必須。

### 2.4 2 要素認証
- ログイン URL は食べログ店舗管理画面と共通: `https://owner.tabelog.com/owner_account/login/`
- **ID + パスワードのみ、2FA なし**

### 2.5 Webhook / 自動連携
- **店舗オーナー向け Webhook なし**
- 公式パートナーとのサーバ間同期のみで、第三者ダッシュボードは受信側に立てない
- 定期エクスポート（S3 dump、scheduled CSV mail）も提供なし

### 2.6 アカウント体系（食べログ本体との関係）
- 食べログ店舗管理画面と **同じ「食べログ店舗会員」アカウントで一元管理**
- 食べログノートは **食べログネット予約契約店舗の付帯サービス**（単独契約不可・料金無料）
- 顧客向け食べログ（tabelog.com）とは完全別系統。SSO なし

### 2.7 推奨取込方式
1. **【第一推奨】店舗オーナーに月次で「分析」CSV を手動 DL → ダッシュボードにアップロード**
   - 対象 CSV: ダッシュボード月次サマリ／予約コース／予約経路／新規・リピート／予約キャンセル／リードタイム／人数割合／卓稼働率（最大 8 ファイル）
2. **【条件付き第二】店舗が ebica/RESZAIKO/レスラク を既に使用している場合 → その SaaS 経由**
3. **【非推奨】ヘッドレスブラウザ自動 CSV DL**: 規約抵触リスク高。書面合意必須

### 2.8 主要参照 URL
- 食べログノート 分析機能について: https://owner-support.tabelog.com/【食べログノート】分析機能について-64ce834f2dc506001b804ecf
- 食べログノート かんたんマニュアル: https://owner-support.tabelog.com/【食べログノート】かんたんマニュアル-64ce834fe3a7d7001c90f551
- RESZAIKO サポート（食べログノート連携手順）: https://reszaiko.zendesk.com/hc/ja/articles/23513881898259

## 3. ホットペッパー / レストランボード

### 3.1 公式 API / 外部連携
- **店舗オーナー向け公式 API は存在しない**。
- リクルート公開の **ホットペッパーグルメ Web Service** は **店舗マスタ検索専用**で、PV／予約数／クチコミ／リピート率などの店舗 KPI は取得不可。
- レストランボードは **公開された外部連携 API を持たない**（開発者ポータルなし）。連携は **同一リクルート系内製連携（Airレジ／Airレジ オーダー／ハンディ／ホットペッパーグルメ）に限定**。
- 食べログ／ぐるなび／Retty 等の予約取込は「自動取込」機能で実現されるが、外部 BI/CRM が KPI を pull できる API ではない。

### 3.2 CSV / Excel エクスポート（レストランボード）
**公式 FAQ**: https://faq.restaurant-board.airregi.jp/hc/ja/articles/900004601743

- **対象画面**: 予約台帳 → 予約リスト / スケジュール（**PC 版のみ、ホットペッパーグルメ掲載店舗に限定**）
- **手順**: サイドメニュー → 予約台帳 → 予約リスト / スケジュール → 「CSV 出力」 → 期間（複数日 OK）と複数店舗選択 → 「CSV ファイルを出力する」 → DL
- **出力上限**: 1 回 10,000 件
- **主な出力カラム**: 予約受付日、ステータス、予約経路（電話／ネット／ホットペッパー／他）、店舗名、来店日、人数、テーブル、コース名、決済確定金額、メモ、顧客名、新規/リピート区分 等（20+ 項目）
- **「集計分析」カテゴリ**（ダッシュボード／月次予約数・総客数／コース別／経路別／新規・リピート／キーワード／キャンセル／テーブル稼働率）は **画面 UI 中心で、CSV 出力ボタンの公式記載は未確認**。実質は「予約リスト CSV を再集計」or「画面スクショ／PDF 印刷」
- **顧客リスト一括 DL の公式マニュアルは見当たらず**（アップロード用テンプレは公開）
- **PV、クチコミは レストランボード側では取得不可**（ホットペッパー店舗管理画面側の指標）

### 3.3 マンスリークライアントレポート の入手手順
- **ホットペッパー店舗管理画面**（`https://www.cms.hotpepper.jp/CLN/login/`）から **クライアントレポートを DL 可能**（公式仕様ページはログイン後のみ）
- **担当営業から月次の詳細レポート（PDF/Excel）が配布される運用が一般的**。エリア平均・競合分析等は営業経由
- 配信形態: 管理画面 DL（PDF/CSV）＋ 営業からのメール添付（PDF/Excel）のハイブリッド
- **プログラムから安定取得できる API/フィードは無い**

### 3.4 利用規約上の自動取得可否
**ホットペッパーグルメ利用規約 第 12 条**（https://cdn.p.recruit.co.jp/terms/hpg-t-1002/index.html）:

- 第 12 条 1 項 [18]: 「不正なプログラム・スクリプトなどを用いて、サーバーに負荷を与える行為」
- 第 12 条 1 項 [19]: 「本サービスの全部又は一部を、当社に無断で、転載、転送、蓄積、…複製、複写する行為」

→ スクレイピング名指しはないが、上記により **自動取得は実質禁止**。
→ 加盟店規約は非公開だが業界慣行から **店舗管理画面の自動ログイン/DL（RPA 含む）はリクルートの明示許諾なしには規約違反リスクが高い**。本番運用は営業窓口経由で個別契約が必要。

### 3.5 2 要素認証 / 認証方式（AirID）
- **レストランボード / Air 系**は **AirID** でログイン
- **AirID の 2 段階認証は任意設定**: 認証アプリ（Google Authenticator 等）／SMS（国内のみ）／メール認証は不可
- **ホットペッパー店舗管理画面は AirID と別系統**（`cms.hotpepper.jp`、加盟店契約ベースの独立 ID）。SSO 無し。レストランボードとの連携はレストランボード設定画面でホットペッパー ID/PW を登録して紐付け
- **2 系統の認証情報管理が必要**

### 3.6 連携プラットフォーム
- **Airマーケット**（https://market.airregi.jp/）に外部ベンダー枠はあるが、レストランボードの連携先は **リクルート社内サービスのみ**（Airレジ／Airレジ ハンディ／ホットペッパーグルメ）
- 外部 BI／会計／CRM 向けのオープン連携（OAuth、Webhook、データ取得 API）は **公開なし**
- freee／弥生連携は Airレジ側の機能で、レストランボードの予約／顧客／集計データには到達しない

### 3.7 推奨取込方式
1. **【第一推奨】レストランボード PC 版「予約リスト CSV」を月次手動 DL → ETL**
   - 取得可能: 月次予約数、予約経路内訳、リピート区分、コース別、キャンセル、客名、人数、金額
   - 10,000 件上限注意（多店舗・繁盛店は分割 DL）
   - **店舗オーナーが手動 DL → ダッシュボードへアップロードする半自動方式が規約上クリーン**
2. **【第二推奨】ホットペッパー店舗管理画面「クライアントレポート」月次手動 DL（PDF/Excel）+ パーサ**
   - 取得 KPI: PV、流入キーワード、クチコミ件数・平均点、エリア比較
   - PDF はレイアウト依存で壊れやすいので Excel 形式優先（営業経由で依頼）
3. **【第三推奨】マンスリークライアントレポート（営業配布版）を Google Drive 等に集約 → 構造化抽出**
4. **【非推奨】店舗管理画面のスクレイピング/RPA 自動ログイン**: 第 12 条 [18][19] 抵触、AirID 2FA で自動不能、アカウント停止リスク
5. **【補助】グルメ Web Service API**: 店舗マスタ（住所/ジャンル/座席数/URL/写真）同期にのみ利用

### 5.6 2 要素認証 / 認証方式
- 店舗管理画面（Login.aspx）は **ログイン ID + パスワード + 「ログイン情報を保存」チェックのみ**

- パスワード再発行は専用ページ。ID 忘却時は `r-toiawase@ozmall.co.jp` 宛にメール問い合わせ
- 動作対象: Windows10/11 Chrome、macOS Safari/Chrome
- セッション Cookie ベースの ASP.NET 認証と推定。**Playwright によるヘッドレス自動ログインは技術的に可能**

### 5.7 成果報酬型課金レポートとの関係
- OZmall レストランは **完全成果報酬型**（月額固定費 0 円、初期費 0 円）
- 掲載審査あり（ディナー客単価 5,000 円以上／ランチ 2,500 円以上、首都圏・関西・名古屋中心）
- 課金額 =「月次の予約成立件数 × プラン単価」。料率は店種別非公開
- **請求書／課金明細 = 月次予約数の正の集計**
- ただし課金レポートは確定・成立予約ベースで、キャンセル・no-show・PV・特集流入比率・お気に入り などのマーケ KPI は別途取得が必要

### 5.8 推奨取込方式
1. **【第一推奨】店舗管理画面のヘッドレスブラウザ自動化（Playwright/Selenium）+ HTML パース**
   - 認証は ID/PW のみ・2FA なしのため自動ログイン可能
   - 自社店舗の自分用データであれば規約リスクは比較的低いが、運用前に **書面確認・レート制限・夜間バッチ**
2. **【第二推奨】予約台帳（ebica/レストランボード/TableSolution 等）経由の二次取込**
   - 「予約元 = OZmall」ラベル付き予約数・キャンセル・客単価が取れる。ただし **PV／流入／レビュー／お気に入り／特集枠は取れない**
3. **【第三推奨】専任セールス経由の月次データレビュー（メール／PDF）の手動／OCR 取込**
   - 特集枠経由予約・全体ランキング順位など、画面に出ない高粒度データはここでしか得られない可能性
4. **【補助】公開ページのスクレイピング**（口コミ件数・平均点・お気に入り表示数）
   - ログイン不要範囲のみ。自店ページのみ。テキスト本文の再配布なし

**結論**: マーケ KPI 全項目（月次予約数・特集枠経由予約数・レビュー・PV・流入経路・キャンセル・客単価・お気に入り）を 1 系統で揃える公開 API は無く、**(1) 管理画面ヘッドレス自動化を主、(3) 営業担当の月次レポートを補完** とする二系統運用が現実的。

## 6. Retty

### 6.1 公式 API / 外部連携（過去・現在）

### 7.1 公式 API（パートナー向け / 店舗向け）
- **無料版「ぐるなび Web Service（レストラン検索 API）」は 2021/06/30 で新規受付を含めサービス終了**。旧バージョンは 2019/09/03 に廃止
- **法人向け有料 API（「ぐるなび API」）として継続**: ぐるなびマーケティングソリューションが「飲食店・施設情報を自社サービスでユーザーに配信する法人」向けに **月額 10 万円程度**（3 か月無料トライアル）で提供。提供データは公開情報で、**店舗オーナーが「自店の PV」「予約件数」「クチコミ点数」をプログラマブルに取得する API は公開されていない**
- ネット予約の「予約 API（Light/Standard）」が存在するが、対象は **外部予約サービス事業者がぐるなびへ予約を流し込む方向**
- **店舗オーナー向けの「自店データ取得 API」は提供なし**

### 7.2 CSV / Excel エクスポート（店舗管理画面）
- 加盟店管理画面（manage.gnavi.co.jp/Pro）の「**ぐるなび GON → アクセスレポート**」で、PC / SP / アプリ別に **月別 PV を CSV 出力可能**。集計期間を「月別」に切り替え、「CSV 形式で出力」ボタンから DL
- 予約管理・顧客管理は「**ぐるなび台帳**」で行うが、CSV エクスポート機能の明示は見当たらず。台帳画面のダッシュボードでは予約・売上分析の **PDF 出力**（目標管理含む）が可能
- **クチコミ／平均点に関する加盟店向け CSV エクスポートは公式ドキュメントには確認できず**、管理画面の口コミ管理画面で目視確認が基本

### 7.3 月次レポート機能
- 加盟店管理画面「アクセスレポート」が月別集計に対応（事実上の月次 PV レポート）
- 「ぐるなび台帳」のダッシュボードで時間別・曜日別・月別の分析と **PDF 出力**
- **店舗向けに月次 KPI を PDF で自動メール配信する公式サービスは確認できず**。月次レポートは管理画面に営業担当が随時持参・メール送付するケースが主

### 7.4 利用規約上の自動取得可否
**ぐるなびサイト利用規約 第 7 条（禁止事項）**:

- (13) 号: ぐるなびサイト・サービス・コンテンツの **無断転載・蓄積・販売・複製・複写**
- (15) 号: ぐるなび／提携企業の **サーバー・ネットワーク機能の破壊・妨害**（高頻度クローラー該当）
- (16) 号: 方法を問わず **他のお客様の個人情報を収集し蓄積する行為**

**ぐるなび PRO 会員規約 第 7 条**: 「営利目的でぐるなびから直接的・間接的に得た情報を他に転用・流用する行為」「不正行為等」を禁止

→ 「スクレイピング」の語は明記されていないが、**自店データを管理画面から自動収集して外部システムに転用する行為は規約に抵触するリスクが高い**
→ API についても専用の「ぐるなび Web サービス利用規約」あり、用途・配信先が制限される

### 7.5 2 要素認証 / 認証方式
- ログインは「**ぐるなび PRO 会員 ID（メールアドレス）＋ 半角 6〜16 文字英数字パスワード**」のみ（manage.gnavi.co.jp/Pro/member/login/）
- 公式ヘルプ・規約に **2 要素認証 / SMS 認証 / TOTP 必須の記述なし**
- ヘッドレスブラウザでのログイン自動化自体は技術的に困難ではないが、規約ブロッカーが存在

### 7.6 連携サービス（予約台帳、POS 等）
- **ぐるなび台帳**（pro.gnavi.co.jp/gnavi_daicho/）: ぐるなびネット予約と自動連携、顧客台帳・予約台帳・分析ダッシュボード。iOS アプリあり
- **ぐるなび POS+**（パートナー POS）: ぐるなび台帳と連携し、来店客の注文履歴を顧客台帳に紐付け
- **ぐるなび経営サポート**: 実証実験で発表、店舗の各種データを集約し経営支援に活用する構想
- **他社 POS 連携**: ぐるなびネット予約 → 外部予約管理システム（レストランボード等）への取り込みは個別パートナー実装が中心
- **データライブラリ**: pr.gnavi.co.jp/promo/gdl_sys/ で外食ビッグデータ提供（マクロ）

### 7.7 推奨取込方式
1. **【本命】「ぐるなび GON → アクセスレポート」CSV を運用者が月次 DL → ダッシュボードへインポート**（月次 PV）。手動運用だが規約面でクリーン、実装コスト最小
2. **【予約件数・プラン別予約数】ぐるなび台帳の管理画面表示を運用者が転記、または台帳の PDF レポートをアップロードして OCR/手入力**
   - CSV 出力可否は店舗の契約プランで要確認 → **初回ヒアリング項目に含めるべき**
3. **【クチコミ件数・平均点】公式の機械取得手段がないため、運用者が管理画面で月次目視転記する Excel テンプレートを用意**
4. **【非対象】API 自動取込**: 店舗向けの自店データ API が存在せず、スクレイピング自動化は規約抵触リスクが高い。どうしても自動化する場合は **ぐるなび営業（マーケティングソリューション部門）に「店舗向けデータ提供契約」を個別相談**
5. **【将来の選択肢】**: 「ぐるなび経営サポート」や台帳の公式エクスポート拡充の動向を継続ウォッチ。実装としては「CSV アップロード窓口」と「手入力フォーム」のハイブリッド UI を用意し、加盟店プランの差異に耐えられる設計

## 9. まとめ

### 9.1 結論

- **Phase 1（手動 CSV アップロード運用）は実現可能**。7 媒体すべてに「店舗オーナーが月次で何らかの形式（CSV／PDF／スクショ）を取得 → ダッシュボードへアップロード」のルートが存在する。
- **Phase 2（Playwright 自動取得）は媒体ごとに大きく難易度が異なる**:
  - **比較的容易**: OZmall（2FA なし、規約明示禁止なし、自店データに限定）
  - **書面合意で道がある**: 食べログノート（カカクコム合意）、RESZAIKO（一休合意）
  - **困難**: 食べログ本体・ホットペッパー・Retty・ぐるなび（規約上のリスクが高い）
- **2 要素認証は AirID と一休が現時点のハードル**。AirID は任意設定なので店舗オーナーに「2FA 無効化」を依頼するのは現実的でなく、人手フォールバックが必須。

### 9.2 田中の最終推奨

1. 要件定義書 §10 の **B 案（段階リリース）を改めて推奨**
2. Phase 1.0 リリース時点では **全 7 媒体を「手動 CSV アップロード + 目視転記」運用** で着手
3. Phase 1.1 で **OZmall を皮切りに自動取得を段階導入**。書面合意の取得状況に応じて食べログノート／RESZAIKO に拡大
4. 食べログ本体・ホットペッパー・ぐるなび・Retty は Phase 1 期間中は **自動取得対象外** とし、Phase 2 以降に検討
