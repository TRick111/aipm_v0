## 1. サマリ

オズモール（OZmall）はスターツ出版が運営する女性向けレストラン／ホテル／ビューティ予約サイトで、編集部キュレーションと特集枠連動が特徴の **成果報酬型** プラットフォーム。店舗向けには `restaurantweb_client`（旧 ASP.NET 製の店舗管理画面）が提供されており、以下が公開情報・第三者記事から確認できた。

- **店舗管理画面に「予約一覧」「レビュー一覧」「在庫管理」「連携先（アライアンスパートナー）管理」が存在することは確認できる**が、CSV/Excel 形式での **公式エクスポート機能が公開ドキュメントから確認できない**。OZmall ヘルプ／媒体資料・第三者支援記事のいずれも「CSV ダウンロード」に明確に触れた一次情報は見つからなかった。→ **要管理画面実機確認**。
- 一方、OZmall は **RESZAIKO（サイトコントローラー）／ebica（予約台帳）／トレタ／レストランボード（Air）／TableCheck／かんざし／サロンコネクト** 等と公式 API またはメール連携でつながっており、**月次の予約集計 CSV はこれら連携 SaaS 側から取得するのが現実解**。特に **RESZAIKO の「予約集計」画面では、媒体別予約件数・予約人数・時間帯／曜日／リードタイム／予約経路を CSV 出力可能（直近 1 年分）**。
- 編集部企画／特集枠経由予約の「特集ID／キャンペーンID」単位の分解レポートを店舗側 CSV で取得できる、という記述は **どの一次情報からも確認できず**。プラン単位（OZ限定プラン、OZの日プラン等）の予約件数を、RESZAIKO や ebica の媒体別／プラン別集計から間接的に追う形が現実解と推察される（推測）。
- 二要素認証は **OZmall 店舗管理画面では未確認**（公開情報なし）。ID＋パスワード＋（連携時のみ API 認証キー発行）方式は確認できる。
- マーケダッシュボード設計上は **「OZmall 直 CSV は不確定」「RESZAIKO 連携店舗なら CSV あり」** を前提に、まずは OZmall 担当窓口（`r-toiawase@ozmall.co.jp`）への直接ヒアリングと、現場店舗での画面キャプチャ取得を強く推奨する。

## 2. 各レポート詳細

### 2.1 OZmall ネイティブ店舗管理画面：予約一覧（推定パス: `restaurantweb_client/reserve/List.aspx`）

- **出力形式**: 画面表示のみ確認。CSV／Excel 出力ボタンの有無は **公開情報からは断定不可（要画面確認）**。OZmall 公式ヘルプ・媒体資料・第三者運営支援記事のいずれにも CSV ダウンロード機能の明示記述なし。
- **主要項目（推定）**: 予約番号／来店日時／プラン名（OZ限定プラン含む）／人数／予約者氏名・連絡先／ステータス（予約成立／キャンセル／来店）／備考。
  - ebica・トレタ API 連携で「OZmall の予約詳細情報が自動転記される」「変更・キャンセル情報も取り込み」と明記されており、最低限これら項目は管理画面側にも保持されている。
- **取得手順（推定）**:
  1. `https://www.ozmall.co.jp/restaurantweb_client/Login.aspx` に店舗 ID／パスワードでログイン
  2. グローバルメニューから「予約一覧」（推定 URL: `/restaurantweb_client/reserve/List.aspx`）に遷移
  3. 期間／ステータス等で絞り込み
  4. （CSV ボタンがあれば押下、なければ画面コピー／RESZAIKO 連携経路へ）
- **契約プラン要件**: 「OZのプレミアム予約」掲載店舗であれば標準提供と推察。プラン別の CSV 機能差は要確認。
- **取得頻度**: 画面参照は随時可能。CSV 出力可否は未確認。
- **備考**: OZmall ヘルプ（ChatPlus 提供）に「店舗側で来店日を変更しても OZmall システムは元の来店日のまま」との記述があり、店舗管理画面の予約レコード編集機能は限定的な可能性。

### 2.2 OZmall ネイティブ店舗管理画面：レビュー一覧（確認パス: `restaurantweb_client/report/review/ListNew.aspx`）

- **出力形式**: 画面表示のみ確認。CSV ダウンロードは **要画面確認**。URL パスに `report` が含まれており、レビューレポートの一覧表示機能は存在。
- **主要項目（推定）**: 投稿日時／投稿者属性（年代等）／総合評価（星）／カテゴリ別評価（料理・サービス・雰囲気等の項目構成は要確認）／コメント本文／プラン名／来店日。
  - OZmall 公式ランキング「口コミ年間ランキング」や「OZmallアワード」が「実際の予約数」と「ユーザーレビューポイント」を集計対象としており、ユーザー側評価項目は星評価＋総合点形式である旨が確認できる。
- **取得手順**:
  1. 店舗管理画面ログイン
  2. メニュー「レポート」→「レビュー（口コミ）」→「一覧」（推定）
  3. 期間で絞り込み
- **取得頻度**: 随時。月次の平均点／件数を抽出するならスクレイピング or 手動コピーが現実解（推測）。

### 2.3 OZmall ネイティブ店舗管理画面：在庫管理（プラン／空席）

- **出力形式**: 公開ヘルプ記載は「在庫情報＞在庫管理画面」表記のみ。CSV 出力は未確認。
- **主要項目**: プラン別在庫（席数）／日付別残数。月次 KPI として直接ダウンロードする対象ではない。
- **備考**: RESZAIKO や TableCheck と連携している場合、在庫はそちら側で一括管理される。

### 2.4 OZmall ネイティブ店舗管理画面：連携先（アライアンスパートナー）管理

- **目的**: 外部 SaaS（RESZAIKO 等）に API 認証キーを発行するための画面。CSV 機能ではない。
- **取得手順（RESZAIKO 公式ヘルプより確認）**:
  1. OZmall 店舗管理画面ログイン
  2. 「トップ」→「連携先管理」をクリック
  3. 連携したい SaaS（例: RESZAIKO）の「設定」を「する」に変更
  4. 画面に表示された **API 認証キー** をコピー
  5. 連携先 SaaS（RESZAIKO 等）の「API 認証キー」欄に貼り付けて連携有効化

### 2.5 RESZAIKO 連携時：予約集計レポート（CSV 出力可）

- **出力形式**: 画面表示＋ **CSV 出力可（各カテゴリ右上タブから）**。出力できる CSV データは **直近 1 年分**。
- **主要項目／分析軸**（RESZAIKO 公式ヘルプ「予約集計」より）:
  - 予約件数／予約人数／予約件数と予約人数（推移）
  - 時間帯ごとの予約数
  - 曜日ごと／日ごと／累計
  - **予約経路**（媒体別：OZmall／食べログ／ぐるなび／ホットペッパー／一休／Retty／ヒトサラ／OMAKASE 等）
  - **予約のリードタイム**（来店日の何日前に予約が入ったか）
  - 予約が入った時間帯（受付時間分布）
- **取得手順**:
  1. RESZAIKO 管理画面ログイン
  2. 「集計」メニュー →「予約集計」
  3. 期間を【今週】【今月】【今年】または【期間を指定】で選択
  4. 集計カテゴリのタブを選択し、右上の **CSV 出力ボタン** を押下
- **契約プラン要件**: RESZAIKO サイトコントローラー契約と、OZmall 側「アライアンスパートナー管理」での連携有効化が必要。
- **取得頻度**: 任意（月次／週次／日次いずれも自由に期間指定可）。
- **備考**: **OZmall 単独契約のみの店舗ではこの CSV は使えない**。RESZAIKO 等のサイトコントローラー導入が前提。

### 2.6 RESZAIKO／ebica／トレタ／レストランボード 経由の予約データ取り込み

- ebica：OZmall 公式 API 連携で「OZmall のレストラン予約から受け付けた予約が **リアルタイムに ebica 予約台帳へ自動転記**」「変更・キャンセル情報も取り込み」される。ebica 側で予約台帳・顧客台帳・媒体別集計レポートが利用可能（CSV 出力の有無は ebica 機能仕様要確認）。
- トレタ：2015 年に OZmall と公式 API 連携。OZmall 予約の自動取り込み。さらに「レスラク連携」オプションで OZmall を含む 4 媒体（食べログ／ホットペッパー／ぐるなび／OZmall）の予約・在庫一元管理が可能。
- レストランボード（Air レジ系）：OZmall は **API ではなく「グルメサイトメール取り込み」経由**で予約データを取り込む方式。設定は「サイドメニュー」→「設定」→「グルメサイト連携」→「グルメサイトメール取り込み」→「OZmall」→「メール取り込み設定」。OZmall は他媒体と異なりグルメサイト管理画面側での認証チェックは不要（Step2 非アクティブ）。
- TableCheck：OZmall がプレスリリースで RESZAIKO・TableCheck の組み合わせを公開。TableCheck 側のマンスリー分析レポート（来店回数／最終来店日／利用金額／予約経路／客単価）と OZmall 予約の媒体別紐づけが可能。
- かんざし／サロンコネクト：主にビューティ側だがレストラン在庫連動も同系統。

これらは **OZmall 自身が CSV を出さない代わりに、店舗が既に契約している予約台帳／サイトコントローラー側で CSV を取るのが標準的な運用**であることを示している。

## 3. 編集部企画／特集枠のレポーティング

| 項目 | 状況 |
|---|---|
| 編集部企画の存在 | 確認済。OZmall は編集部キュレーション（年間 30＋本の特集記事、年間約 50 件のイベント）が看板。月間特集記事への掲載＋特集連動の「OZ限定プラン」が運用されている。 |
| 「OZの日」プラン | 確認済。毎月 2〜4 日限定の OZ 限定プラン枠が走り、店舗側はこの枠でプラン作成・在庫設定する。 |
| 特集枠経由予約の店舗向けレポート | **公開情報からは確認不可**。「この予約がどの特集記事／編集タイアップ経由か」を CSV/UI で分解して見せる店舗向け機能の記述はない。**プラン単位（OZ限定プラン名）での予約集計はおそらく可能**だが、「特集 ID」「キャンペーン ID」までブレークダウンしたレポートは要管理画面確認（推測）。 |
| 編集タイアップの料金 | 1 週間枠 60 万円、2 週間枠 120 万円（グロス・税別、制作費込み）程度の単価で運用（媒体資料時点）。 |
| 成果報酬モデル | 月額掲載料・維持費ゼロ、特集／記事掲載も無料、来店確定ベースの送客手数料モデル（料率非公開）。 |

**マーケダッシュボード設計への含意（推測）**:
- 「特集枠経由予約数」を月次 KPI として正確に取りたい場合、**OZmall 側ではプラン名（OZ 限定プラン名）で分解 → 店舗側でプラン名を「特集名」にマッピング**する運用設計が現実解。
- 自動化（Selenium／Playwright での画面スクレイピング）に踏み込まないと、特集 ID 単位の自動取得は困難な見込み。

## 4. RESZAIKO 連携時の挙動（該当する場合）

### 4.1 連携セットアップ手順

1. RESZAIKO 管理画面で「媒体連携」→「OZmall を連携する」を選択
2. OZmall 店舗 ID、ログイン ID、パスワードを入力
3. OZmall 店舗管理画面にログインし、「トップ」→「連携先管理」を開く
4. RESZAIKO の「設定」を「する」に切り替えると **API 認証キー** が発行・表示される
5. 認証キーをコピーし、RESZAIKO の「API 認証キー」欄へ貼り付け
6. 「連携を有効にする」にチェック → 「保存」

### 4.2 連携後にできること

- OZmall の在庫を RESZAIKO で一元管理（食べログ・ぐるなび等と並行）
- 予約発生時の在庫自動消し込み（売り止め）
- **「予約集計」画面で OZmall 経由予約を媒体別に分解 → CSV 出力**
- 媒体別の予約件数／人数／時間帯／曜日／リードタイム／予約経路の年単位集計
- 期間指定（今週・今月・今年・任意期間）→ CSV ダウンロード

### 4.3 留意点

- CSV 出力は **直近 1 年分の制限あり**（連続 1 年超のヒストリカルが要件の場合は、月次で自動取得 → DB 蓄積する設計が必要）。
- **客単価（金額）／キャンセル件数** の集計が RESZAIKO 予約集計でどこまで含まれるかは公開ヘルプから断定不可（要画面確認）。基本は「予約件数・人数」が中心と推測される（推測）。
- **レビュー件数・平均点／月次 PV／流入経路（広告経路）** は RESZAIKO のスコープ外（予約管理 SaaS のため）。OZmall 側または別途取得が必要。

## 5. 既知の制約・自動化への留意点

- **OZmall ネイティブ画面の CSV 出力可否はオフィシャルに公開されていない**。マーケダッシュボード要件定義の精度を上げるには、対象店舗の管理画面ログインキャプチャ／OZmall 担当窓口（`r-toiawase@ozmall.co.jp`）への直接ヒアリングが必須。
- **特集枠経由の予約 ID 分解は店舗向けには公開機能なし（公開情報範囲）**。プラン名マッピングによる近似が現実解。
- **二要素認証（2FA）の有無は未確認**。RPA／スクレイピング設計時は SMS／メール OTP の有無も合わせて要確認。
- **OZmall 画面の URL は ASP.NET（`.aspx`）由来**。Google 検索結果に `reserve/List.aspx`, `report/review/ListNew.aspx` といったパスが露出している点から、画面構成は伝統的なフォームベース推察（SPA ではない）。スクレイピング側からは比較的扱いやすい一方、ViewState 等の ASP.NET 特有のトークン管理が必要になる可能性（推測）。
- **「特集 PV」「お気に入り登録数」「ページ流入経路」「広告経由予約」など、いわゆるサイト分析系 KPI は店舗管理画面では未確認**。これらが必要な場合、OZmall 側の媒体担当からエクセル形式の月次レポートを個別に受領するスキーム（電通・博報堂等の編集タイアップ運用に近い）の打診が現実的（推測）。
- **第三者情報の「管理画面でできること」記述は文章テンプレートのコピペが多く、一次情報の信頼度は低い**。本ドキュメントの「推定」「未確認」ラベルが付いた項目は実機確認で必ず上書きすること。
- **OZmall ヘルプセンター（`ozmall.custhelp.com`／ChatPlus）** は来店者（消費者）向け FAQ が中心で、店舗向け技術仕様はほぼ未公開。

## 7. ネクストアクション提案

1. **対象店舗での管理画面ログイン実機確認**（最優先）
   - `restaurantweb_client` のメニュー構成、予約一覧の CSV ボタン有無、レビュー一覧の CSV ボタン有無、二要素認証の有無、月次集計画面の有無を画面キャプチャ取得
2. **OZmall 担当窓口へのヒアリング**
   - 連絡先: `r-toiawase@ozmall.co.jp`（店舗専用問い合わせ）
   - 質問項目: ①予約一覧 CSV／Excel エクスポート可否 ②特集枠経由予約の店舗向けレポート機能の有無 ③編集タイアップ実施店舗向け月次レポート提供有無 ④API 公開有無（アライアンスパートナー以外）
3. **RESZAIKO／ebica 利用有無の店舗側棚卸し**
   - 既に RESZAIKO／ebica／トレタ／TableCheck を契約済の店舗は、OZmall 直接スクレイピングよりこれら SaaS の CSV／API 経由のほうが安定。マーケダッシュボードの初期データ取り込みパスはここを優先する設計案を推奨。
4. **特集枠ロジック設計**
   - 「特集 ID」を直接取れない前提で、プラン名（OZ 限定プラン名）→特集名のマッピングテーブルを店舗側で運用するスキームを設計
