## 1. サマリ

- **一休.com レストラン本体**は店舗管理画面（`restaurant.ikyu.com` 配下）に予約管理・店舗情報設定・特集応募などの機能を持つが、**「CSV／Excel ダウンロード可能なレポート」についての公開情報はほぼ存在しない**。公式 PDF マニュアル『管理画面マニュアル（基本編）』（2021/05/20 版）は HTTP 403 で外部からは取得不可（要ログイン or 加盟店配布）。
- ただし一休加盟店は**付帯サービスとして RESZAIKO Lite（サイトコントローラー）を無償利用可能**で、ここに「予約一覧 CSV エクスポート」「予約集計 CSV 出力」「顧客データ出力」がまとまっている。**マーケダッシュボードのデータソースは実質 RESZAIKO に寄せるのが現実的**。
- 一休本体の管理画面 UI には**予約一覧画面・期間検索・予約詳細画面**が存在し、画面上での件数集計・絞り込みは可能だが、**CSV エクスポートは確認できなかった**（公開情報ベース。一部の高度な権限／メール通知連携で対応している可能性は残る → 現地確認が必要）。
- ログインは **Yahoo! JAPAN ID 連携が任意**で、店舗管理画面自体は**店舗 ID + パスワード**ベース。SMS による 2 要素認証は一般会員側で導入済みだが、**店舗管理画面側の 2FA は公開情報なし**（要・店舗オーナーへヒアリング）。
- 「PV／特集枠別流入／プラン別表示回数」などのマーケ KPI を CSV で得る公式手段は **2026-06 時点で未確認**。一休営業担当からのレポートメールが主流の可能性が高い。
- **推測（要検証）**：マーケダッシュボード側は「(1) RESZAIKO の予約集計 CSV を一次データ」「(2) 一休本体の予約一覧をスクレイピング or 営業レポートからの手入力で補完」する二段構成が現実解。

## 2. 一休.com レストラン 各レポート詳細

> 注意：以下は公開情報・推測・他媒体（食べログ／ぐるなび等）との一般的構成からの類推を含む。**最終確定には店舗管理画面に実際にログインして確認する必要がある**。

### 2.1 予約一覧（管理画面：予約管理メニュー）

- **出力形式**: 画面表示のみ。**CSV／Excel ダウンロードボタンは公開情報では確認できず**（推測：標準機能としては未提供の可能性が高い）。
- **主要項目**（画面表示・推測）:
  - 予約番号
  - 来店日時 / 来店人数
  - 予約者氏名 / 連絡先（電話・メール）
  - 利用プラン名 / プラン金額
  - 予約ステータス（予約成立／キャンセル／来店済）
  - 予約経路（特集枠 or 通常検索 or カテゴリ）※媒体内分類
  - 支払方法（現地・事前カード）
  - ポイント利用（一休ポイント／PayPay）
  - 備考・要望
- **取得手順**:
  1. `https://restaurant.ikyu.com/` にアクセス（店舗用ログインページに遷移）
  2. 店舗 ID + パスワードでログイン
  3. メニュー「予約管理」→「予約一覧」を開く
  4. 期間・ステータス等で絞り込み
  5. （CSV ダウンロードボタンがあれば押下。**現状は未確認**）
- **契約プラン要件**: 一休はすべての加盟店が成果報酬型単一プラン。プランによる差はないと推測。
- **取得頻度**: 任意。リアルタイム反映。
- **備考**:
  - 予約完了メールに 8 桁の認証コードが付与される運用がある。
  - PayPayグルメと管理画面が共通化されている。
  - **要・店舗オーナーへのヒアリング**：実際の CSV ダウンロード可否、対象期間の上限。

### 2.2 月次成果レポート（営業担当からの送付？）

- **出力形式**: Excel または PDF（推測。確証なし）。
- **主要項目**（推測）:
  - 月間予約件数 / 月間来店件数 / キャンセル件数
  - 月間送客手数料額（成果報酬の請求書相当）
  - プラン別予約件数
  - 特集掲載別の予約件数
  - 月間 PV / プラン詳細ページ閲覧数 / コンバージョン率
- **取得手順**:
  1. 一休営業担当からメール送付される（推測）
  2. または管理画面の「レポート」「請求書」セクションからダウンロード（要確認）
- **契約プラン要件**: 加盟店全体に提供と推測。
- **取得頻度**: 月次。
- **備考**:
  - 一休の Developers Blog からは内部に DWH（SQL Server + Redash）があり、施設運営者へのフィードバックループを重視している記述あり。プラン閲覧数・滞在日数を毎月振り返る運用がメンションされている → **何らかのレポート提供がある可能性は高い**。
  - 公式 PDF『一休ポイント・クーポン・PayPay マニュアル』は存在するが、月次レポート仕様の公開はなし。

### 2.3 予約詳細／メール通知

- **出力形式**: 画面表示 + メール本文（plain text／HTML）。CSV ではない。
- **主要項目**: 予約一覧と同等。
- **取得手順**: 予約成立時に自動的に登録メールアドレスへ通知。レストランボード等の予約台帳サービスは、この通知メールをパースして自動取り込みする仕組みを採用している（『グルメサイトメールの取り込み機能』）。
- **備考**:
  - **2025/04/24 のリリース**で一休.com／PayPayグルメのメール取り込み機能に再設定依頼が出ている → **メールフォーマット変更があった**ことを示唆。スクレイピング設計時は注意。

## 3. RESZAIKO 各レポート詳細

> RESZAIKO は一休本体の付帯サービス（Lite は無償／Pro は有償）。サポートサイト（`reszaiko.zendesk.com`）に公式マニュアルあり。**こちらが実質的なデータソース候補**。

### 3.1 予約集計（CSV 出力）

- **出力形式**: **CSV（公式）**。
- **主要項目**:
  - 予約件数（累計／時間帯別／曜日別／日別）
  - 予約人数
  - 予約経路（媒体別：一休／食べログ／ぐるなび等）
  - リードタイム（来店日の何日前に予約されたか）
  - 時間帯ごとの予約分布
- **取得手順**:
  1. RESZAIKO 管理画面（`https://reszaiko.com/`）にログイン
  2. 左メニュー「集計」→「予約集計」を開く
  3. 期間指定（[今週] [今月] [今年] [期間を指定]）。**最大 1 年間**。
  4. 集計したいカテゴリ（予約件数／予約人数／時間帯／予約経路 等）のタブを選択
  5. 各カテゴリ右上の `[・・・]` メニュー →「CSV 出力」をクリック
- **契約プラン要件**:
  - 一休加盟店なら Lite が**無料**（当面）。
  - **2025 年 4 月から TableCheck 連携は有料化**。
  - RESZAIKO Pro（台帳）は別契約。
- **取得頻度**: 任意（オンデマンド）。
- **備考**:
  - **媒体別＝予約経路で「一休／食べログ／ぐるなび／PayPayグルメ」等が分離して取れる**。これはマーケダッシュボードにとって極めて有用。
  - 「客単価」「売上金額」については予約集計には含まれていない（推測）。RESZAIKO Pro の「オーダー集計」が必要な可能性が高い。

### 3.2 顧客集計／顧客データ出力

- **出力形式**: CSV。
- **主要項目**（推測）:
  - 顧客 ID、氏名、連絡先、来店回数、初回来店日、最終来店日、累計予約人数、累計金額
- **取得手順**:
  1. RESZAIKO 管理画面で「顧客」メニューを開く
  2. `[・・・]` →「顧客データ出力」→「出力する」
- **契約プラン要件**: 台帳機能を使うため、**RESZAIKO Pro（有償）** が主な利用層。Lite では制限あり可能性。
- **取得頻度**: 任意。
- **備考**: 既存顧客と新規顧客の比率を出すのに使える。

### 3.3 オーダー集計

- **出力形式**: CSV（RESZAIKO Pro 機能）。
- **主要項目**（推測）:
  - メニュー別の注文件数
  - 売上金額
  - 客単価
- **取得手順**:
  1. 「集計」→「オーダー集計」
  2. 期間指定 → カテゴリ選択 → `[・・・]` →「CSV 出力」
- **契約プラン要件**: **RESZAIKO Pro 必須**。
- **取得頻度**: 任意。
- **備考**: POS／オーダー入力をしている店舗のみ意味のあるデータが取れる。一休主体の店舗では未導入のことも多い。

### 3.4 予約一覧（台帳ベース・トレタ／TableCheck 移行用 CSV）

- **出力形式**: CSV（インポート／エクスポート両対応）。
- **主要項目**:
  - 月単位エクスポート：系列店を含む未来の予約全件 + 当月から 2 ヶ月前までの過去予約
  - 日単位エクスポート：店舗を指定した上での過去全期間の日別予約
  - 顧客情報 CSV：氏名・連絡先・属性
- **取得手順**:
  1. 「予約・顧客設定」メニュー
  2. 「予約・顧客取り込み」セクション
  3. 移行元（トレタ／TableCheck）から落とした CSV をアップロード or 自身のデータを出力
  4. **必ず「顧客情報 → 予約情報」の順でインポート**（順番を間違えると予約が顧客に紐付かない）
- **契約プラン要件**: 移行ツールのため、初期セットアップ時に利用。
- **取得頻度**: 移行時のみが基本。
- **備考**:
  - 月単位エクスポートで **「当月より 2 ヶ月前まで」しか過去は出ない** 制約に注意。長期過去データは日単位エクスポートで取得する必要あり。

### 3.5 連携一覧（媒体接続状況の確認）

- **出力形式**: 画面表示。
- **取得手順**: 「メディア連携」「連携一覧」メニュー。
- **備考**: マーケダッシュボードがどの媒体からの予約を取り込めているかの監査用に有用。CSV ではない。

## 4. 2 要素認証（Yahoo!ID 連携）への対処

### 4.1 一休本体の認証構成（既知）

- **個人会員側**:
  - メールアドレス／携帯番号 + パスワードでログイン。
  - SMS 認証コード（6 桁）による 2 要素認証あり。普段と異なる端末・ブラウザ・ネットワーク環境からのログイン時にトリガー。
  - Yahoo! JAPAN ID／Google／Facebook／LINE／Apple との ID 連携が可能。連携時は一休側のパスワードを入力して紐付け。
- **店舗管理画面側**:
  - 公開情報では **店舗 ID + パスワード** が基本。
  - **店舗側 2FA／SMS 認証の公開情報なし**。GMO トラスト・ログイン（SSO 製品）の連携投稿があるため、**SSO で組織管理する想定の設計**にはなっていそう。
  - Yahoo! JAPAN ID で店舗管理画面に直接ログインできるかは**未確認（要・現地確認）**。一般的には店舗管理は別系統。

### 4.2 Playwright 自動化の難所と回避策

| 難所 | 影響度 | 推奨対処 |
|---|---|---|
| 普段と違う環境からのアクセスで SMS 認証コードが飛ぶ | 高 | (a) 固定 IP（プロキシ）+ 固定 User-Agent で「いつもの環境」を維持。(b) 認証済みセッションの Cookie/Storage State を保存して再利用。(c) SMS 到達時は半手動で介入できる UI を用意（運用者へ Slack 通知 → コード入力フォームで受け取る） |
| Yahoo! JAPAN ID 連携経由のログイン | 高 | Yahoo 側は reCAPTCHA／不正検知が強い。**極力 Yahoo 経由を避け、店舗 ID + パスワードの直接ログインを使う**。Yahoo ID しか持っていない店舗の場合は、最初に一休側に「メールアドレス + パスワード」も登録してもらう。 |
| reCAPTCHA / hCaptcha | 中〜高 | 2Captcha・CapSolver 等の有料 CAPTCHA 解決サービス併用が現実的。または手動介入を許容する設計。 |
| ログイン画面の URL／フォーム構造の変更 | 中 | E2E テストを毎日 1 回回し、失敗時に Slack 通知。スナップショットを取って差分検出。 |
| 店舗管理画面のセッションタイムアウト | 中 | スクレイピング処理を 1 セッション 5〜10 分以内に収まるようジョブ分割。 |
| メール取り込み連携の仕様変更（2025/04/24 リリース） | 中 | メールパース方式を採用するなら、フォーマット変更検知ロジック（必須フィールドが取れないと即アラート）を入れる。 |

### 4.3 RESZAIKO のログイン

- ログイン URL: `https://reszaiko.com/`
- 認証方式: 公開情報では**メールアドレス + パスワード**（標準的な SaaS）。SMS 2FA の公開情報なし。
- 一休管理画面より自動化のハードルは低いと推測。**RESZAIKO 経由でデータ取得する方針が圧倒的に楽**。

## 5. 既知の制約・自動化への留意点

### 5.1 一休本体側

- **CSV ダウンロード機能の公開情報がほぼ皆無**。スクレイピングが必須になる可能性が高い。
- 画面上で取得できるが CSV エクスポート不可の場合、HTML をパースして集計する必要がある。
- **加盟店規約上、店舗のログイン情報を第三者システム（ダッシュボード）に共有することがグレー〜NG**となる可能性。利用規約を確認し、店舗オーナーから明示的な同意を取得すること。
- 月次成果レポートは営業担当からの送付（メール／PDF）の可能性が高く、**SaaS 化が困難**な部分。OCR/PDF パースの仕組みを別途用意するか、店舗オーナーに手入力 UI を提供して回避するのが現実解。
- **PV／プラン別表示回数**は一休内部の DWH にあるが、店舗側 UI に出ているかは未確認。出ていない場合は取得不能。

### 5.2 RESZAIKO 側

- **予約集計 CSV は最大 1 年間**しか出ない。長期トレンドはダッシュボード側で月次スナップショットを蓄積して構築する必要あり。
- 「客単価」「売上金額」は予約集計に含まれない可能性。**Pro（有償）契約がない店舗では客単価が取れない**。
- TableCheck 連携が 2025/04 から有償化。連携先構成によって取得経路が変わる可能性。
- 過去の予約（トレタ／TableCheck 移行系統）の月単位エクスポートは**当月から 2 ヶ月前まで**の制約。
- インポート時の順番ルール（顧客 → 予約）はマイグレーション時のみ関係。日次取得には影響しない。

### 5.3 自動化アーキテクチャの推奨

- **データソース優先順位（推奨）**:
  1. **RESZAIKO 予約集計 CSV**（月次媒体別予約数）← マーケダッシュボードの主軸
  2. **RESZAIKO 顧客集計／オーダー集計 CSV**（客単価・リピート率・売上）
  3. **一休本体の予約完了メール**を IMAP で取り込み → パース（リアルタイム性が必要な場合）
  4. **一休営業担当の月次レポート**（PV／特集枠別）→ 手入力 or PDF OCR
- **自動化方式**:
  - RESZAIKO は ID/PW + Playwright で日次バッチ → CSV 取得 → 取り込みが妥当。
  - 一休本体スクレイピングは「店舗オーナーが自分のローカルブラウザで実行」する形（クライアントサイド常駐 or 専用ヘルパー Chrome 拡張）にすると、利用規約・2FA・SSO 問題を一括回避できる。
- **CAPTCHA／2FA 介入 UI**:
  - 自動化失敗時に運用者へ Slack/メール通知 → 通知から起動するブラウザ UI で手動コード入力 → 自動化再開。

## 付録：次のアクション（提案）

1. **店舗オーナー（Daichi さん or 既存パートナー）に依頼**：実際の一休店舗管理画面を共有してもらい、**予約一覧画面に CSV ダウンロードボタンがあるかをスクリーンショット**で確認。
2. **RESZAIKO アカウントを 1 店舗で作成 → 予約集計の CSV カラムを実物で確認**。仕様の不確実性を一気に解消できる。
3. **一休営業担当に「月次レポートの送付サンプル」を依頼**。送付があるなら、その項目をダッシュボード KPI 設計に取り込む。
4. **マーケダッシュボード側のデータインジェスト**：
   - フェーズ 1.0：RESZAIKO CSV 手動アップロード（最速）
   - フェーズ 1.1：RESZAIKO Playwright 自動取得 → 日次バッチ
   - フェーズ 2.0：一休本体スクレイピング or メール取り込み
