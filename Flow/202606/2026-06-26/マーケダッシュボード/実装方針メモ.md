## 1. 既存実装の踏襲ポイント

### 1.1 スライド出力（O-01）の踏襲対象

既存ダッシュボードには **月次レポート（MonthlyReport）** の HTML 配信実装があり、これを踏襲する。

| 既存ファイル | 役割 | マーケでの踏襲方針 |
|---|---|---|
| `prisma/schema.prisma` の `MonthlyReport` モデル | 月次レポートのメタデータ＋ Blob URL を保持 | `MarketingMonthlyReport` 側で **そのままの構造を使わず**、媒体ごとの KPI は `metrics(Json)` で保持。**スライド配信用の MonthlyReport 自体は流用しない**（既存は手動で HTML を upload する想定のため）。代わりに、媒体集計値からスライド HTML を **動的生成** し、Blob にキャッシュする方式とする |
| `app/api/monthly-reports/upload/route.ts` | Admin が HTML ファイルをアップロード → Blob 保存 → DB に紐付け | M-08 のアップロード API のリファレンス。`getSession()` での admin 認可・`uploadXxxToBlob()`・`createXxx()` の構造をマーケ用に踏襲 |
| `app/api/monthly-reports/[id]/html/route.ts` | Blob から HTML を取得し `Content-Type: text/html` で iframe 表示用にプロキシ | マーケスライド表示の URL 戦略を踏襲。生成済み HTML を Blob から返すか、その場で動的生成 |
| `app/api/monthly-reports/[id]/printable/route.ts` | 印刷向け HTML 出力 | PDF（O-02）対応時に再利用 |
| `app/store/[storeId]/monthly-reports/[monthId]/page.tsx` | 月報表示ページ（iframe で HTML レンダリング） | マーケスライドビューア URL 構造を踏襲（例：`/store/[storeId]/marketing/monthly-reports/[month]`） |

> **T-15 の実装方針**: 上記既存のテンプレート構造・ZIP 化方式・iframe プロキシ方式を **そのまま踏襲**し、マーケ媒体の月次集計値（`MarketingMonthlyReport`）から HTML スライドを動的生成する。新規にレンダリングエンジンは作らない。

### 1.2 認証・認可の踏襲

| 既存ファイル | 役割 | マーケでの踏襲方針 |
|---|---|---|
| `lib/auth/session.ts`（`getSession()`） | セッション取得 + ロール判定（admin / store / group） | コンサルロール（S-01）は `StoreGroup.groupType='consultant'` で表現し、`getSession()` の戻り値に `groupType` を追加するか、別途 `getConsultantSession()` を新設する判断は T-16 で確定 |
| `StoreGroup` モデル（schema.prisma 536〜） | 店舗グループ + パスワード認証 + セッション | **そのまま拡張**。`groupType('general'\|'consultant')` と `viewScope(Json)` を追加するマイグレーションで対応 |
| `StoreGroupSession` | グループログイン用セッション | コンサルログインにそのまま流用 |

### 1.3 CSV 取り込みパターンの踏襲

| 既存ファイル | 役割 | マーケでの踏襲方針 |
|---|---|---|
| `CsvImportLog` モデル | CSV 投入履歴 | `MarketingImportLog` は別テーブルで作るが、フィールド命名と運用パターン（status / errorMessage / importedBy）を踏襲 |
| `lib/storage/blob.ts`（`uploadXxxToBlob`） | Vercel Blob への永続化 | マーケ CSV 原本の保管に流用 |
| `scripts/upload-monthly-report-prod-direct.mjs` | 本番 Blob 直アップロード | M-14e（自動取得スキルからの API 投入）設計の参考 |

### 1.4 PL ダッシュボードのモデル化パターン

既存の `PlDailySales` / `PlDailyExpense` / `PlMonthlyTarget` 等で「店舗×日付×カテゴリ×値」構造が確立している。マーケ側も同型で **店舗×媒体×月** の集計を `MarketingMonthlyReport` に保持する設計とする。

## 2. 新規追加するファイル群（予定）

### 2.1 Prisma スキーマ追加（T-04）

```prisma
// ─── マーケダッシュボード ───

enum MarketingChannelCategory {
  REVIEW      // 食べログ・Retty 等
  RESERVATION // 食べログノート・一休 等
  AGGREGATE   // ホットペッパー・オズモール 等
}

model MarketingChannel {
  id          String   @id @default(uuid()) @db.Uuid
  code        String   @unique @db.VarChar(40)            // 'tabelog' 等
  displayName String   @map("display_name") @db.VarChar(100)
  category    MarketingChannelCategory
  isActive    Boolean  @default(true) @map("is_active")
  sortOrder   Int      @default(0) @map("sort_order")
  createdAt   DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt   DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  reports     MarketingMonthlyReport[]
  importLogs  MarketingImportLog[]

  @@map("marketing_channels")
}

model MarketingMonthlyReport {
  id            String   @id @default(uuid()) @db.Uuid
  storeId       String   @map("store_id") @db.Uuid
  channelId     String   @map("channel_id") @db.Uuid
  month         DateTime @db.Date                          // 該当月の 1 日
  metrics       Json     @default("{}")                    // 媒体固有 KPI を JSON で保持
  sourceBlobUrl String?  @map("source_blob_url") @db.VarChar(500)
  importedAt    DateTime @default(now()) @map("imported_at") @db.Timestamptz(6)
  updatedAt     DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  store   Store            @relation(fields: [storeId], references: [id], onDelete: Cascade)
  channel MarketingChannel @relation(fields: [channelId], references: [id])

  @@unique([storeId, channelId, month])
  @@index([storeId, month])
  @@index([channelId, month])
  @@map("marketing_monthly_reports")
}

model MarketingImportLog {
  id            String   @id @default(uuid()) @db.Uuid
  storeId       String   @map("store_id") @db.Uuid
  channelId     String   @map("channel_id") @db.Uuid
  month         DateTime @db.Date
  status        String   @db.VarChar(20)                   // 'success'|'failed'|'partial'
  source        String   @db.VarChar(20)                   // 'manual'|'auto-local'
  errorMessage  String?  @map("error_message") @db.Text
  importedById  String?  @map("imported_by_id") @db.Uuid
  createdAt     DateTime @default(now()) @map("created_at") @db.Timestamptz(6)

  channel MarketingChannel @relation(fields: [channelId], references: [id])

  @@index([storeId, channelId, month])
  @@index([createdAt])
  @@map("marketing_import_logs")
}
```

## 3. 媒体ごと metrics JSON スキーマ（v1.1 §6 から導出）

各媒体の `metrics` JSON は **媒体ごとに固定キー** を持つ Zod スキーマで管理する（パーサもこのスキーマに合わせる）。

### 3.1 食べログ（M-01）

```ts
{
  monthly_pv: number,
  monthly_reservations: number,
  page_taps: { menu: number, photo: number, map: number, phone: number, reserve: number },
  visit_metrics: { phone_clicks: number, map_print_pv: number, calls_050: number, online_reservations: number },
  area_rank: number | null,           // トップ100位置
  review_count: number,
  review_avg_score: number,
}
```

### 3.2 食べログノート（M-03）

```ts
{
  monthly_reservations: number,
  walk_in_count: number,
  reservation_walkin_ratio: number,
  cancellation_count: number,
  cancellation_rate: number,
  seat_utilization_rate: number,
  table_utilization_rate: number,
  avg_party_size: number,
  avg_lead_time_days: number,
  net_reservation_course_ratio: number,
  new_repeat_ratio: { new: number, repeat: number },
  plan_top20: Array<{ plan_name: string, reservations: number }>,
}
```

### 3.3 ホットペッパー（M-04）

```ts
{
  monthly_pv: number,
  monthly_reservations: number,
  reservation_channel: { phone: number, web: number },
  inflow_keywords: Array<{ keyword: string, count: number }>,
  review_count: number,
  review_avg_score: number,
  rb_summary: { repeat_rate: number, monthly_trend: Array<{ month: string, count: number }>, courses: Array<{ name: string, count: number }> },
}
```

Retty／ぐるなび／一休／オズモールの詳細スキーマは T-01 サンプル取得後に確定する。

## 5. 既知の懸念点（実装着手前のメモ）

1. **既存 main の未コミット変更との衝突**: `app/login/page.tsx`、`prisma/schema.prisma` の PL 関連変更が main 上にある。**`prisma/schema.prisma` の編集は競合する可能性が高い**ため、T-04 着手時は最新 main を取り込んでからマイグレーション生成する
2. **既存 `MonthlyReport` モデルとの命名衝突**: 既存は HTML 月報用。マーケ側は `MarketingMonthlyReport` と明確に分け、混在を避ける
3. **スライド出力の動的生成 vs 事前生成**: 既存は事前 upload 方式。マーケ側はデータが日々更新されるため動的生成が望ましいが、性能を考慮して **「スライド表示時に生成し Blob にキャッシュ／データ更新時にキャッシュ無効化」** で対応予定（T-15 着手時に最終判断）
4. **店舗認証 vs グループ認証 vs コンサル認証の整理**: 既存に store / group の二系統あり、コンサル（group の派生）を入れることでセッション判定ロジックが複雑化する可能性。T-16 で `getSession()` を慎重に拡張する
