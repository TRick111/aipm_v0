# 商品検索機能 実装方式検討 v1

- 日付: 2026-07-02
- 対象リポジトリ: `/Users/rikutanaka/RestaurantAILab/voice-beauty-advisor`
- ステータス: 検討中（方式未決定）

---

## 1. 背景と要件

### やりたいこと
音声美容アドバイザー（Mia）で、現在3件のサンプル商品しかないカタログを**実運用規模の商品DB**に置き換え、会話中のユーザーの悩みに合った商品を提案できるようにする。

### 要件
| 項目 | 内容 |
|---|---|
| 規模 | 約10社 × 各50SKU = **約500SKU** |
| 商品ジャンル | スキンケア / ヘアケア / ネイルケア / ボディケア / メイクアップ など美容商品全般（コーセー等のメーカーが持つラインナップのイメージ） |
| 提案トリガー | 会話中に話者が語った**悩み**（乾燥、毛穴、髪のパサつき、爪の割れ…）に合致する商品を提案 |
| UX制約 | 音声会話なので**レイテンシが致命的**。検索は数百ms以内に返したい |

---

## 2. 現状アーキテクチャ（2026-07-02 時点）

- 商品カタログは `src/config/mia-prompt.ts` にハードコード（`ALL_PRODUCTS` 配列 + システムプロンプト内テキストの**2箇所重複**）
- 提案は OpenAI Realtime API（gpt-realtime）の function calling `recommend_product` のみ。**検索・DB・埋め込みは一切なし**
- function call の処理はクライアント側 `src/hooks/useRealtimeSession.ts:323-363` に集約
- サーバーは `server.ts`（Next.js カスタムサーバー + OpenAI への WS プロキシ）。API Route（`src/app/api/`）は Tavus 用が1本あるのみ

### スケール時の根本問題
- 500SKU をプロンプトに全載せすると **40k トークン前後**になり、gpt-realtime のコンテキスト上限を超える／会話履歴を圧迫する → **現行方式は不可**
- よって「モデルが商品を知っている」前提を捨て、「**モデルが検索ツールを呼ぶ**」構成（`search_products` ツールの追加）への転換がどの案でも共通の前提になる

### 共通アーキテクチャ（全案共通）

```
ユーザー発話「最近、髪がパサついて…」
   │
   ▼
gpt-realtime が function call: search_products({ query, category, concerns })
   │
   ▼
Next.js API Route /api/products/search  ← ★ここの検索方式が本ドキュメントの論点
   │
   ▼
上位3〜5件の商品JSONを function output として返却
   │
   ▼
モデルが結果から選んで recommend_product → UIカード表示（既存フロー流用）
```

## 3. データモデル設計（全案共通の基盤）

**どの検索方式を選ぶかより、SKUごとの構造化メタデータの質が精度を決める。** 500SKU 規模では「検索技術」より「タグ設計」が支配的。

```jsonc
// data/products.json（1レコードの例）
{
  "sku_id": "KSE-SK-001",
  "brand": "コーセー風サンプルA社",        // 10社
  "product_name": "モイストリペア セラム",
  "category": "skincare",                 // ★第1階層: skincare / haircare / nailcare / bodycare / makeup / uv / lipcare
  "sub_category": "serum",                // 美容液 / 化粧水 / シャンプー / トリートメント / ネイルオイル…
  "target_area": ["face"],                // face / scalp / hair / nail / hand / body / lip
  "concerns": ["乾燥", "小じわ", "ハリ不足"],  // ★悩みタグ（統制語彙。20〜30語で管理）
  "skin_type": ["乾燥肌", "敏感肌"],       // 該当カテゴリのみ
  "price": 4200,
  "description": "セラミド配合で角質層まで…",   // 全文検索・埋め込み対象
  "usage": "洗顔後、化粧水の前に2プッシュ",
  "image": "/images/products/kse-sk-001.png"
}
```

### 悩みタグの統制語彙（例）
- スキンケア系: 乾燥 / 毛穴 / ニキビ / くすみ / シミ / 小じわ / たるみ / 敏感 / テカリ
- ヘアケア系: パサつき / 枝毛 / うねり / ボリューム不足 / 頭皮のかゆみ / フケ / カラーダメージ
- ネイル系: 二枚爪 / 割れ / ささくれ / 甘皮 / ツヤ不足

> ユーザーの生の発話（「なんか最近肌がゴワゴワして」）→ 統制タグ（乾燥・くすみ）への正規化は **gpt-realtime 自身に function call の引数生成でやらせる**（enum で縛る）。これがルールベース検索でも高精度に効く理由。

## 4. 実装案

### 案1: プロンプト全載せ（現行方式の延長）— 比較基準

システムプロンプトに全商品を記載し、モデルの記憶だけで提案する。

- ✅ 実装ゼロ。レイテンシ最小（検索往復なし）
- ❌ **500SKU では容量的に不可能**（〜40kトークン）。圧縮しても会話コンテキストを圧迫
- ❌ ハルシネーション（存在しない商品名・価格の捏造）リスク大
- **判定: 不採用**。ただし「各カテゴリの代表商品10件だけプロンプトに置き、詳細は検索」という折衷は案5で回収

### 案2: ルールベース構造化フィルタ（検索エンジン不使用）

`search_products` の引数を enum で構造化し（category, concerns[], price_max…）、サーバー側は **JSON をメモリに載せて素朴に filter + スコアリング**するだけ。

```ts
// スコア例: concernsの一致数 × 2 + カテゴリ一致 + 価格帯適合
products
  .filter(p => p.category === args.category)
  .map(p => ({ p, score: intersect(p.concerns, args.concerns).length }))
  .sort(byScore).slice(0, 5)
```

- ✅ 実装が最も軽い（半日〜1日）。500件のメモリフィルタは**1ms未満**でレイテンシ最強
- ✅ 挙動が完全に説明可能・デバッグ容易。埋め込みAPI費用ゼロ
- ✅ 「悩み→タグ正規化」をモデルにやらせるので、実質セマンティック検索に近い精度が出る
- ❌ タグに載っていない語彙・ニュアンス（「彼女へのプレゼント」「さっぱりした使用感」）を拾えない
- ❌ タグ設計・付与の運用コストがデータ側に寄る

### 案3: 全文検索（BM25 / 転置インデックス）

description 等のテキストに対して全文検索。500件規模なら外部エンジン不要で、ライブラリで完結。

- 候補: **MiniSearch / FlexSearch**（JSメモリ内）、**SQLite FTS5**（永続化するなら）
- ⚠️ **日本語の分かち書きが必要**（bi-gram トークナイズ or TinySegmenter）。ここが英語圏ライブラリの落とし穴
- ✅ タグにない自由語彙（成分名「セラミド」、ブランド名指し）を拾える
- ❌ 「肌がゴワゴワ」→「乾燥」のような**言い換えに弱い**（字面一致ベース）
- ❌ 単体では案2より悩みマッチ精度が落ちやすい。補助検索向き

### 案4: ベクトル検索（シンプルRAG）— ご提案の案①

全SKUの `name + description + concerns` を埋め込み（`text-embedding-3-small`）、クエリ埋め込みとのコサイン類似度で上位k件。

- **500件なら vector DB 不要**。ビルド時に埋め込みを `embeddings.json` に事前計算し、メモリ内で総当たりコサイン計算（500件×1536次元 ≒ 数ms）
- ✅ 言い換え・ニュアンスに強い。「プレゼント向きの上品なやつ」も拾える
- ✅ タグ設計をサボっても一定の精度が出る（データ整備コストが低い）
- ❌ クエリ埋め込みのAPI往復で **+100〜300ms**（音声UXでは体感される可能性）
- ❌ カテゴリ誤爆リスク: 「乾燥」で頭皮用と顔用が混ざる等。**説明可能性が低く**チューニングが手探り
- ❌ 埋め込みモデルの更新・再計算の運用が増える

### 案5: ハイブリッド（ルールベース分割 + セマンティック検索）— ご提案の案②【推奨】

**第1段: ルールベースで検索空間を絞る**（category / target_area は function call の enum 引数で確定 → 500件 → 50〜80件）
**第2段: 絞った集合内でスコアリング**（concernsタグ一致 + ベクトル類似度 or BM25 の合成スコア）

```
search_products({ category: "haircare", concerns: ["パサつき"], query: "カラーの後の傷み" })
  → ①category でハードフィルタ（500→50件）
  → ②concernsタグ一致でベーススコア
  → ③query の埋め込み類似度を加点（or FTSスコア）
  → 上位5件返却
```

- ✅ カテゴリ誤爆を構造的に防げる（顔の悩みに頭皮ケアが出ない保証）
- ✅ タグで説明可能性を担保しつつ、自由語彙はベクトルで補完。**両方の弱点を相殺**
- ✅ 段階導入と相性が良い: **まず案2として動かし、後からベクトル加点を足すだけ**でこの案に育つ
- ❌ スコア合成の重み調整が必要。実装は案2+案4の合算（2〜3日）

### 案6: 2段LLMリランク（参考・発展形）

第1段のルールフィルタで50件程度に絞った後、**小型LLM（gpt-4o-mini / Haiku）に候補リストと会話文脈を渡して選定・理由生成までさせる**。

- ✅ 精度は最高クラス。「この人は敏感肌と言っていたから低刺激を優先」など文脈横断の判断ができる
- ❌ +500ms〜1s のレイテンシ。音声リアルタイムでは会話の間（ま）が崩れるリスク
- **位置づけ**: 音声中の即時検索には過剰。ただし「会話終了後のまとめ提案画面」には最適

## 6. 推奨と段階導入ロードマップ

**推奨: 案5（ハイブリッド）を最終形として、案2から段階的に育てる。**

理由:
1. 500SKU 規模では検索技術より**タグ設計が精度を支配**する。まず案2で「function call 引数の enum 化 + タグ設計」という土台を作るのが最短
2. 案2はそのまま案5の第1段になるので**捨てコードにならない**
3. 音声UXのレイテンシ制約上、常時LLMリランク（案6）は不向き。ベクトル加点（+100-300ms）は許容圏

### Phase 0: データ基盤（先行着手・全案共通）
- [ ] `data/products.json` のスキーマ確定（§3）
- [ ] 悩みタグの統制語彙を確定（20〜30語）
- [ ] サンプルデータ生成: 架空10社 × 50SKU をLLMで生成（実在メーカー名・実在商品名は使わない。「コーセー風」の品揃え構成だけ参考にする）
- [ ] `mia-prompt.ts` の商品ハードコード2箇所を廃止し、単一ソース化

### Phase 1: 案2（ルールフィルタ）で E2E を通す
- [ ] `search_products` ツール定義追加（category/concerns は enum）、`recommend_product` は sku_id 参照に変更
- [ ] `/api/products/search` 実装（メモリ内 filter + タグスコア）
- [ ] `useRealtimeSession.ts` に search_products のハンドラ追加

- [ ] UI: 3枚固定の `ProductPanel` を検索結果ベースの動的表示に改修

### Phase 2: 案5 へ拡張（ベクトル加点）
- [ ] ビルド時埋め込み事前計算スクリプト（`text-embedding-3-small` → `embeddings.json`）
- [ ] 検索APIに類似度加点を合成（重みは タグ:ベクトル = 7:3 あたりから調整）
