# ビリヤード俯瞰アプリ — 技術設計書

## 1. 対象：日本で一般的なポケットビリヤード台

日本のビリヤード場では **9フィート台**（プロ・トーナメント標準）が主流。バーや漫画喫茶等では **7フィート台** も使われる。本設計では9フィートを主対象とし、7フィートにも対応する。

| サイズ | プレイ面（鼻先〜鼻先） | 備考 |
|---|---|---|
| 9フィート | 1270 × 2540 mm | ビリヤード場の標準。JPBF公式戦 |
| 7フィート | 991 × 1981 mm | バー・アミューズメント施設 |

全サイズ共通で **長辺:短辺 = 2:1**。

代表的な台：Brunswick Gold Crown、Diamond Pro-AM。
ラシャ：Simonis 860（トーナメントブルーが最多、グリーンも多い）。

### ボール

| 項目 | 値 |
|---|---|
| 直径 | 57.15 mm（2-1/4"） |
| 半径 (r) | 28.575 mm |
| 台面からボール中心の高さ | 28.575 mm（= r） |
| セット | 1〜15番 + キューボール = 計16個 |

---

## 2. レールとラシャの3D構造（重要）

これが正確な3D再構成の鍵。**ラシャはプレイ面だけでなくクッション面も覆っている**ため、色検出の境界線は台面（Z=0）にはない。

### レール断面図（プレイ面からレール外側への断面）

```
   外側                                            内側（プレイ面）

    |<-------- レールキャップ（木材） -------->|<-- クッション面 -->|
    |                                          |                    |
    |     +-ダイヤモンド--+                    |                    |
    |     |  (はめ込み)   |                    |                    |
    |=====|===============|====================|=======*            |
    |     |               |   フェザーストリップ      / |            |
    |     |   93.7mm      |   （ラシャ固定溝）      /  |            |
    |     |  （鼻先から） |                       /   |            |
    |     +---------------+                     /    |            |
    |                                          /     |            |
    |      露出した木材面                ラシャ/      | Z ≈ 36mm   |
    |     （ラシャなし）                （クッ/ション | （鼻先）    |
    |                                    面を/覆う）  |            |
    |                                      /  K-66   |            |
    |                                    /   ラバー   |            |
    |                                  /     （約66°）|            |
    |                                /               |            |
    |________________________________/________________|____________|===== Z = 0
                                                     |            |
                                                レール|  プレイ面  |
                                                木材  | （スレート |
                                                      |  + ラシャ）|
```

### Z座標の整理

| 構造要素 | Z高さ (mm) | 備考 |
|---|---|---|
| **プレイ面（スレート + ラシャ）** | **0** | 基準面 |
| **ボール中心** | **28.575** | = 半径 r |
| **クッション鼻先** | **≈ 36** | ボール直径の約63.5% |
| **レールキャップ上面** | **≈ 44.5** | Brunswick/Diamond の場合 |
| **ダイヤモンドサイト** | **≈ 44.5** | レールキャップ面に埋め込み |

### ラシャが覆っている範囲

ラシャ（同じ色の布）は以下をすべて覆う：

1. **プレイ面全体**（Z = 0、水平）
2. **クッション面全体**（Z = 0 〜 ≈36mm、約66°の傾斜面）
3. 鼻先を越えてレール裏側まで巻き込み、レール底面にステープル留め

**レールキャップ（木材面）にはラシャがない。** フェザーストリップ溝を境界に、内側がラシャ、外側が露出した木材。

### 色検出で何が見えるか

カメラから見た場合、**ラシャ色（青/緑）の領域**は：

```
 +--[ダイヤ]--+--[ダイヤ]--+--[ダイヤ]--+   ← 木材レールキャップ（茶/黒）
 |============ フェザーストリップ境界 =====|   ← ラシャ色の上端（Z ≈ 44mm）
 |\\\\\\\\\\\ クッション面のラシャ \\\\\\\ |   ← 傾斜面、同じ色
 |  =========== クッション鼻先 ===========|   ← Z ≈ 36mm
 |                                         |
 |           プレイ面のラシャ               |   ← Z = 0（水平）
 |               (ボールはここ)             |
 |                                         |
```

**色検出で得られる境界は Z ≈ 44mm（フェザーストリップ付近）であり、Z = 0 ではない。**

これが重大な意味を持つ：
- ラシャの色境界＝プレイ面の端 ではない
- ラシャの色境界を「台の4隅」として Z=0 のホモグラフィに使うと大きな誤差が出る
- 色境界のZ高さを正しく考慮する必要がある

## 3. 利用可能な参照点とそのZ座標

PnP（カメラ姿勢推定）に使える参照点を、Z座標別に整理する。

### ポケット顎（Jaw）ポイント：12点、Z ≈ 36 mm

各ポケットの開口部の両端（クッション鼻先が途切れる点）。最も位置が明確に定義されている。

```
  コーナーポケット（4箇所 × 2点）:      サイドポケット（2箇所 × 2点）:

      長辺レール                           長辺レール
  =========* ← 顎点               ========*        *========
             \                              \      /
              \ 142°                  104°   \    /  104°
               \                              \  /
                +  ポケット                    +  ポケット
               /   (114-117mm)                    (127-133mm)
              /
  =========* ← 顎点
      短辺レール
```

**9フィート台の顎ポイント座標（プレイ面上の投影、mm）：**

ポケット開口（鼻先〜鼻先）を考慮した近似座標。原点 = 長辺左下隅の鼻先交点。

| ポケット | 顎1 (X, Y, Z) | 顎2 (X, Y, Z) |
|---|---|---|
| 左下コーナー | (0, ≈58, 36) | (≈58, 0, 36) |
| 左上コーナー | (≈58, 2540, 36) | (0, ≈2482, 36) |
| 右上コーナー | (1270, ≈2482, 36) | (≈1212, 2540, 36) |
| 右下コーナー | (≈1212, 0, 36) | (1270, ≈58, 36) |
| 左サイド | (0, ≈1204, 36) | (0, ≈1336, 36) |
| 右サイド | (1270, ≈1336, 36) | (1270, ≈1204, 36) |

※ 開口幅やアングルは台の型番で若干異なる。

### ダイヤモンドサイト：18点、Z ≈ 44.5 mm

レールキャップ上面に埋め込まれた目印。白色/真珠色で視認性が高い。

9フィート台のダイヤモンド間隔 = **317.5 mm**（12.5"）

ダイヤモンドの中心はクッション鼻先から **93.7 mm** 外側（レールキャップ上）。

```
  短辺: 3個（D1, D2, D3）

  顎--D1---D2---D3--顎
      |317.5|317.5|

  長辺: 各6個（D1〜D6）（サイドポケットで分断）

  顎--D1--D2--D3--顎[サイドポケット]顎--D4--D5--D6--顎
```

### クッション鼻先ライン：連続線、Z ≈ 36 mm

クッションが途切れるポケット開口部を除き、プレイ面を囲む連続的な線。ラシャの折れ目（傾斜面→水平面の変わり目）として画像中に存在する。

## 4. 問題の本質（改訂版）

単純なホモグラフィが不十分な理由は **2つ** ある：

### 問題1：ラシャ色境界は Z = 0 ではない

色検出でラシャ領域を抽出すると、その輪郭はフェザーストリップ付近（Z ≈ 44mm）。これを「台の角」としてZ=0の平面変換に使うと、カメラの視線方向に応じた **数十mmの投影誤差** が発生する。

### 問題2：ボールは台面より高い位置にある

ボール中心は Z = 28.575mm。Z=0 のホモグラフィで変換すると **15〜60mm** の位置ズレ（視差誤差）が生じる。

**→ 両方の問題を解決するには、各参照点・各対象物のZ座標を正しく指定した3D再構成が必要。**

## 5. アプローチ比較

### アプローチ A：ホモグラフィのみ

| 項目 | 評価 |
|---|---|
| 実装難度 | 低 |
| カメラキャリブレーション | 不要 |
| ボール位置精度 | **悪い（15〜60mm + ラシャ境界誤差）** |
| 問題1（ラシャ境界Z）| 未対応 |
| 問題2（ボール高さ） | 未対応 |

**結論：** 両方の問題を無視するため精度が出ない。

---

### アプローチ B：PnP + レイキャスト【推奨】

**方法：**
1. 複数の参照点を **正しいZ座標付き** で定義
2. カメラ内部パラメータをEXIFから推定
3. `solvePnP` でカメラ姿勢（R, t）を算出
4. 各ボールの画像座標からレイキャストで **Z = 28.575mm** の平面と交差

**アプローチAとの決定的な違い：**
- 参照点が **異なるZ高さに存在してよい**（ポケット顎 Z≈36mm、ダイヤモンド Z≈44.5mm）
- むしろ非共面の参照点がある方が PnP の解が安定する（共面4点だと2解の曖昧性がある）
- ボール、ダイヤモンド、ポケット顎、それぞれのZ座標で個別にレイキャストできる

**核心のアルゴリズム：**

```
入力: 画像座標 (u, v)、対象のZ座標 h、K、R、t

1. 正規化座標:  p_cam = K^(-1) * [u, v, 1]^T
2. カメラ原点:  C = -R^T * t
3. レイ方向:    d = R^T * p_cam
4. Z = h 平面との交差:
     λ = (h - C_z) / d_z
     P_world = C + λ * d
5. 俯瞰位置 = (P_world.x, P_world.y)
```

**参照点ごとのZ指定：**

```javascript
// ポケット顎ポイント（Z = 36mm）
const jawPoints3D = [
  [58, 0, 36], [0, 58, 36],       // 左下コーナー
  [0, 2482, 36], [58, 2540, 36],   // 左上コーナー
  // ... 計12点
];

// ダイヤモンドサイト（Z = 44.5mm）
const diamondPoints3D = [
  [-93.7, 317.5, 44.5],  // 短辺のD1（鼻先から93.7mm外側）
  // ... 計18点
];

// solvePnP に渡す（混合Z座標でOK）
const allPoints3D = [...jawPoints3D, ...diamondPoints3D];
const allPoints2D = [...jawPoints2DInImage, ...diamondPoints2DInImage];

cv.solvePnP(allPoints3D, allPoints2D, K, distCoeffs, rvec, tvec);

// ボール位置の復元（Z = 28.575mm）
for (const ball of detectedBalls) {
  const pos = backprojectToPlane(ball.u, ball.v, K, R, t, 28.575);
  // pos.x, pos.y がボールの俯瞰位置
}
```

| 項目 | 評価 |
|---|---|
| 実装難度 | 中 |
| カメラキャリブレーション | 必要（EXIFから推定可能） |
| ボール位置精度 | **5〜15 mm** |
| 問題1（ラシャ境界Z）| **対応（各参照点に正しいZを指定）** |
| 問題2（ボール高さ） | **対応（Z=28.575mmでレイキャスト）** |
| PnP安定性 | **非共面点で曖昧性が解消** |

---

### アプローチ C：ホモグラフィ + 高さ補正

| 項目 | 評価 |
|---|---|
| 実装難度 | 中 |
| ボール位置精度 | 厳密: 5〜15 mm / 簡易: 10〜25 mm |

**結論：** Bよりも回り道になる。参照点のZ混在を自然に扱えない。

## 6. 推奨アプローチ：B（PnP + レイキャスト）の詳細設計

### 処理パイプライン

```
[写真入力]
    │
    ▼
[1. 前処理]
    ├── EXIF読み取り → カメラ内部パラメータK推定
    └── 画像リサイズ・正規化
    │
    ▼
[2. 参照点検出]  ← レール構造を考慮した検出
    ├── (a) ラシャ色領域のセグメンテーション → 台のおおまかな位置
    ├── (b) ラシャ/木材の境界検出 → フェザーストリップライン
    ├── (c) ポケット開口部の検出 → 顎ポイント（Z ≈ 36mm）
    ├── (d) ダイヤモンドサイト検出 → 白/真珠色の点（Z ≈ 44.5mm）
    └── (e) 台サイズ選択（9ft / 7ft）→ ワールド座標確定
    │
    ▼
[3. カメラ姿勢推定]
    ├── 参照点の 3D座標（Z混在）と 2D画像座標を対応付け
    ├── solvePnP → R, t 算出
    │   ・非共面点により曖昧性なし
    │   ・参照点が多いほど精度向上（最低4点、推奨6点以上）
    └── 検証: 参照点を再投影してピクセル誤差を確認
    │
    ▼
[4. ボール検出]
    ├── プレイ面領域マスク生成（クッション面を除外）
    │   ・鼻先ライン（Z≈36mm）の内側のみを対象
    │   ・色検出後にクッション傾斜面を除去するため、
    │     台面の矩形をR,tで再投影してマスク生成
    ├── 非ラシャ領域（ボール候補）の抽出
    ├── 輪郭分析: 面積・真円度フィルタ
    └── minEnclosingCircle → 中心座標 + 半径
    │
    ▼
[5. ボール位置の3D復元]
    ├── 各ボール中心 (u,v) → レイキャスト → Z = 28.575mm 平面と交差
    └── 俯瞰座標 (X, Y) を取得
    │
    ▼
[6. 描画]
    ├── 台の矩形描画（正確なスケール）
    ├── ポケット・ダイヤモンドサイト描画
    ├── 各ボールを俯瞰位置に描画
    └── （オプション）ボール番号・色の識別
```

### 参照点検出の戦略

**最も検出しやすい参照点の優先順位：**

1. **ダイヤモンドサイト** — 白/真珠色の小さな点。木材レールキャップ上にあり、ラシャ色と木材色の両方と高コントラスト。サイズと間隔が既知
2. **ポケット開口部の顎ポイント** — ラシャ色が途切れてポケットの暗い開口部になる境界。エッジがシャープ
3. **ラシャ/木材境界（フェザーストリップライン）** — ラシャ色と木材色の境界線。直線的で検出しやすいが、Z座標の精密な割り当てが必要

**推奨する最小構成：**
- ポケット顎ポイント 4〜8点（コーナーポケットの顎）: Z = 36mm
- ダイヤモンドサイト 4点以上: Z = 44.5mm
- 合計 8〜12点以上の非共面参照点

### ボール検出領域の精密化

カメラ姿勢（R, t）が得られた後、**プレイ面の矩形をカメラ画像に再投影** して精密なマスクを作れる：

```
プレイ面の4隅（Z=0）:
  (0, 0, 0), (W, 0, 0), (W, L, 0), (0, L, 0)

これを projectPoints で画像座標に変換
→ この四角形の内部だけでボールを探索
→ クッション面の誤検出を排除
```

## 7. iPhoneカメラ内部パラメータの推定

### EXIFからの算出

```
fx = fy = (画像幅px × 実焦点距離mm) / センサー幅mm
cx = 画像幅px / 2
cy = 画像高さpx / 2
```

### 主要モデルの参考値

| モデル | 解像度 | センサー幅 | 実焦点距離 | fx (px) | 35mm換算 |
|---|---|---|---|---|---|
| iPhone 13/14 (広角) | 4032×3024 | ~6.86 mm | ~5.7 mm | ~3350 | 26mm |
| iPhone 13 Pro | 4032×3024 | ~7.6 mm | ~5.7 mm | ~3024 | 26mm |
| iPhone 14 Pro/15 Pro | 4032×3024 (12MP) | 9.8 mm | ~5.3 mm | ~2180 | 24mm |

### 実装方針

1. EXIFの `FocalLength`（実mm）と `FocalLengthIn35mmFilm`（35mm換算）を取得
2. 35mm換算がある場合: `センサー幅 = 36mm × 実焦点距離 / 35mm換算焦点距離`
3. `fx = 画像幅px × 実焦点距離 / センサー幅`
4. 歪み係数はゼロと仮定（iPhoneは広角カメラでソフトウェア補正済み）

## 8. ボール検出の詳細

### 推奨手法：色セグメンテーション + 輪郭解析

```
1. HSVに変換
2. ラシャの色範囲でマスク生成
   ・ブルー系: H=90-130, S>50, V>50
   ・グリーン系: H=35-85, S>50, V>50
   ※ 台の色は写真のホワイトバランスで変動するため、
     最大面積の色クラスタから動的に範囲推定するのが理想
3. プレイ面マスク（PnPで再投影した矩形）で AND → クッション面を除外
4. マスク反転 → 非ラシャ領域 = ボール候補
5. findContours → 各輪郭に対し:
   - 面積フィルタ（小さすぎ/大きすぎ除外）
   - 真円度フィルタ: circularity = 4π × 面積 / 周長² > 0.7
   - minEnclosingCircle → 中心 + 半径
```

### 将来的な改善

- ボール番号・色の識別（HSVヒストグラムでソリッド/ストライプ分類）
- YOLO等のDLモデルで検出精度向上（Roboflowにデータセットあり）

### OpenCV.jsの注意点

- `calib3d` モジュールは `@techstark/opencv-js` に含まれる（古いビルドでは未同梱の場合あり）
- メモリ管理は手動：すべての `cv.Mat` に `.delete()` が必要
- `findHomography` の入力は `Point2fVector` でなく `cv.Mat` を使用すること
- `solvePnP` は非共面点に対応（IPPE法は共面専用なので、非共面点には `SOLVEPNP_ITERATIVE` や `SOLVEPNP_SQPNP` を使用）

## 11. 期待精度

| 対象 | PnP + レイキャスト | ホモグラフィのみ |
|---|---|---|
| 台の参照点 | < 3 mm | < 1 mm（同一平面前提時） |
| ボール位置 | **5〜15 mm** | 15〜60 mm |
| ダイヤモンドの再投影 | < 5 mm | N/A（Z不一致で使えない） |

PnP法なら、ボール同士の相対位置やポケットとの距離関係を正確に把握できるレベル。

## 12. 実装ステップ（案）

| Phase | 内容 | 成果物 |
|---|---|---|
| Phase 1 | EXIF読み取り + カメラパラメータ推定 | K行列の自動算出 |
| Phase 2 | 参照点検出（ポケット顎 + ダイヤモンド） | 画像上の参照点座標リスト |

| Phase 6 | Web UIの整備・UX改善 | 完成版Webアプリ |

### Phase 2 の補足：半自動アプローチ

初期実装では参照点の検出を **半自動** にすることを推奨：
- 自動検出で候補を提示
- ユーザーがクリック/ドラッグで微修正
- 台サイズ（9ft / 7ft）はユーザーが選択

完全自動化は後のフェーズで追加。まずは3D再構成の精度を確実にすることが重要。
