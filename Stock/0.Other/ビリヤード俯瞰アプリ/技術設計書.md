## 2. 問題の本質：なぜ単純なホモグラフィでは不十分か

ホモグラフィ（射影変換）は **同一平面上の点** を正しく変換する。しかし、ビリヤードのボールは台面（Z=0）の上に乗った球体であり、ボールの見かけの中心は台面から **28.575 mm** 浮いた位置（Z = r）にある。

この高さの違いにより **視差（パララックス）誤差** が発生する：

- カメラ角度が急であるほど誤差が大きい
- カメラから遠いボールほど誤差が大きい
- 9フィート台で斜め30〜45度から撮影した場合、**15〜60 mmの位置ズレ**（ボール直径の約25〜100%）

**つまり、台面のホモグラフィだけではボール位置を正確に復元できない。**

## 3. アプローチ比較

### アプローチ A：ホモグラフィのみ（参考：以前の手法）

**方法：** 台の4隅を検出 → ホモグラフィ行列Hを算出 → 全点をHで変換

```
s * [u, v, 1]^T = H * [X, Y, 1]^T
```

| 項目 | 評価 |
|---|---|
| 実装難度 | 低 |
| カメラキャリブレーション | 不要 |
| ボール位置精度 | **悪い（15〜60 mm誤差）** |
| ライブラリ | OpenCV.js `getPerspectiveTransform` |

**結論：** 台面のマーキング（フットスポット等）の変換には十分だが、ボール位置には不適。

---

### アプローチ B：PnP（Perspective-n-Point）によるカメラ姿勢推定 + レイキャスト【推奨】

**方法：**
1. 台の4隅（またはポケット6点）のワールド座標と画像座標の対応を取得
2. カメラ内部パラメータ（焦点距離等）を推定
3. `solvePnP` でカメラの外部パラメータ（回転R + 並進t）を求める
4. 各ボールの画像座標からカメラ光線を逆算し、**Z = 28.575 mm の平面**と交差させる

**核心のアルゴリズム（レイ-平面交差法）：**

```
入力: ボールの画像座標 (u, v)、カメラ内部行列 K、回転 R、並進 t

1. 正規化座標:  p_cam = K^(-1) * [u, v, 1]^T
2. カメラ原点:  C = -R^T * t
3. レイ方向:    d = R^T * p_cam
4. Z = h 平面との交差:
     λ = (h - C_z) / d_z
     P_world = C + λ * d
5. ボールの俯瞰位置 = (P_world.x, P_world.y)
```

ここで `h = 28.575 mm`（ボール中心の高さ）。

| 項目 | 評価 |
|---|---|
| 実装難度 | 中 |
| カメラキャリブレーション | 必要（EXIF情報から推定可能） |
| ボール位置精度 | **良好（5〜15 mm誤差）** |
| ライブラリ | OpenCV.js `solvePnP`（IPPE法）、gl-matrix |

**補足：画像上のボール中心 ≠ 3D中心の投影**

斜め視点では球の見かけ中心が3D中心の投影と微妙にずれるが、1〜2m距離での57mmボールの場合は1〜2ピクセル程度であり、実用上無視可能。

---

### アプローチ C：ホモグラフィ + 高さ補正（ハイブリッド）

**方法：**
1. 台面のホモグラフィHを算出
2. Hを分解してカメラ姿勢（回転R）を推定
3. ボール画像座標に高さ補正を適用してから、Hで変換

**高さ補正の数学：**

台面上の点Pの投影: `p = K(r1*X + r2*Y + t) = H*[X,Y,1]^T`

高さhにある点の投影: `p = H*[X,Y,1]^T + K*r3*h`

よって補正: `p_surface = p_detected - K * r3 * h`

**簡易近似（カメラ位置が台のほぼ上方の場合）：**

```
Δx = h * (X - X_cam) / D
Δy = h * (Y - Y_cam) / D
```
（D = カメラの台面からの高さ、X_cam/Y_cam = カメラ直下の台面座標）

| 項目 | 評価 |
|---|---|
| 実装難度 | 中 |
| カメラキャリブレーション | 厳密補正には必要 / 簡易近似なら不要 |
| ボール位置精度 | 厳密: 5〜15 mm / 簡易: 10〜25 mm |
| ライブラリ | OpenCV.js `decomposeHomographyMat` |

**結論：** Bの簡略版。内部パラメータが信頼できない場合のフォールバック。

---

### アプローチ D：深層学習ベース

**関連研究：**

- **Van Zandycke & De Vleeschouwer (CVPR 2022W)**: CNNでボールのピクセル直径を回帰 → `depth = (実直径 × 焦点距離) / ピクセル直径` で3D位置を推定
- **Rodriguez-Lozano et al. (2023)**: 複数カメラによるビリヤード3D再構成（1mm精度、ただしハイスピードカメラ）
- **Roboflow**: プールボール検出データセット（353枚、YOLO学習済み）あり

| 項目 | 評価 |
|---|---|
| 実装難度 | 高 |
| カメラキャリブレーション | アプローチによる |
| ボール位置精度 | 検出は95%+、3D位置は20〜50 mm |
| ライブラリ | TensorFlow.js, ONNX.js |

**結論：** ボール検出には有用だが、3D位置特定はジオメトリベース（B/C）に劣る。補助的に使用するのが最適。

## 4. 推奨アプローチ：B（PnP + レイキャスト）

### 処理パイプライン

```
[写真入力]
    │
    ▼
[1. 前処理]
    ├── EXIF読み取り → カメラ内部パラメータK推定
    └── 画像リサイズ・正規化
    │
    ▼
[2. 台検出]
    ├── HSV色空間でラシャ検出
    ├── エッジ検出 → Hough直線 or 輪郭検出
    └── 4隅（または6ポケット）の座標取得
    │
    ▼
[3. カメラ姿勢推定]
    ├── ワールド座標定義（台サイズ選択: 7/8/9フィート）
    ├── solvePnP (IPPE法) → R, t 算出
    └── 検証: ワールド点を再投影してズレを確認
    │
    ▼
[4. ボール検出]
    ├── 台領域マスク生成
    ├── 非ラシャ領域（ボール候補）の抽出
    ├── 輪郭分析: 面積・真円度フィルタ
    └── minEnclosingCircle → 中心座標 + 半径
    │
    ▼
[5. ボール位置の3D復元]
    ├── 各ボール中心 (u,v) からカメラ光線を逆算
    ├── Z = 28.575mm 平面との交差計算
    └── 俯瞰座標 (X, Y) を取得
    │
    ▼
[6. 描画]
    ├── 台の矩形描画（正確なスケール）
    ├── ポケット・ダイヤモンドサイト描画
    ├── 各ボールを俯瞰位置に描画
    └── （オプション）ボール番号・色の識別
```

### なぜBが最適か

1. **物理的に正しい**: ボールの高さを明示的にモデル化するため、急角度撮影でも正確
2. **必要な情報は揃っている**: 台サイズ（既知）、ボールサイズ（既知）、カメラ内部パラメータ（EXIFから推定可能）
3. **OpenCV.jsで完結**: `solvePnP`, `findContours`, `inRange` 等すべて利用可能
4. **拡張性**: ダイヤモンドサイトや追加特徴点を使えばさらに精度向上可能

## 5. iPhoneカメラ内部パラメータの推定

### EXIFからの算出方法

```
fx = fy = (画像幅px × 実焦点距離mm) / センサー幅mm
cx = 画像幅px / 2
cy = 画像高さpx / 2
```

### 主要モデルの値

| モデル | 解像度 | センサー幅 | 実焦点距離 | fx (px) | 35mm換算 |
|---|---|---|---|---|---|
| iPhone 13/14 (広角) | 4032×3024 | ~6.86 mm | ~5.7 mm | ~3350 | 26mm |
| iPhone 13 Pro | 4032×3024 | ~7.6 mm | ~5.7 mm | ~3024 | 26mm |
| iPhone 14 Pro/15 Pro | 4032×3024 (12MP) | 9.8 mm | ~5.3 mm | ~2180 | 24mm |

### 実装方針

1. 写真のEXIFから `FocalLength`（実焦点距離mm）と `Model`（機種名）を取得
2. 機種名からセンサー幅を引く（内蔵テーブル or 35mm換算焦点距離から逆算）
3. `fx`, `fy`, `cx`, `cy` を算出
4. 歪み係数はゼロと仮定（iPhoneの広角カメラはソフトウェア補正済み）

## 6. ボール検出の詳細

### 推奨手法：色セグメンテーション + 輪郭解析

```
1. HSVに変換
2. ラシャの色範囲でマスク生成（緑系: H=35-85, S>50, V>50）
3. マスク反転 → 非ラシャ領域 = ボール候補
4. findContours → 各輪郭に対し:
   - 面積フィルタ（小さすぎ/大きすぎ除外）
   - 真円度フィルタ: circularity = 4π×面積 / 周長² > 0.7
   - minEnclosingCircle → 中心 + 半径
```

**なぜHough円検出でなくこちらか：**
- Hough円検出はパラメータ調整が敏感で、照明条件によりロバストでない
- 色セグメンテーションはラシャとの高コントラストを活用でき安定

### 将来的な改善

- ボール番号・色の識別（HSVヒストグラムでソリッド/ストライプ分類）
- YOLO等のDLモデルで検出精度向上（Roboflowにデータセットあり）

## 7. 技術スタック

| レイヤー | 技術 | 理由 |
|---|---|---|
| フレームワーク | Next.js or Vite + React | SPA、高速開発 |
| 画像処理 | @techstark/opencv-js (v4.9+) | solvePnP, findContours等を含む |
| 行列演算 | gl-matrix | 軽量（~10KB）、3x3/4x4行列演算 |
| EXIF読み取り | exifr or exif-js | ブラウザ対応 |
| 描画 | Canvas API or SVG | 俯瞰図のレンダリング |
| UI | React + Tailwind CSS | — |

### OpenCV.jsの注意点

- `calib3d` モジュールは `@techstark/opencv-js` パッケージに含まれている（古いビルドでは未同梱の場合あり）
- メモリ管理は手動：すべての `cv.Mat` に `.delete()` が必要
- `findHomography` の入力は `Point2fVector` でなく `cv.Mat` を使用すること

## 9. 期待精度

| 対象 | 推定誤差 |
|---|---|
| 台の隅・マーキング | < 1 mm（サブピクセル精度の検出前提） |
| ボール位置（PnP法） | 5〜15 mm（ボール半径28.6mm以下） |
| ボール位置（ホモグラフィのみ） | 15〜60 mm（ボール1個分のズレが起きうる） |

PnP法なら、ボール同士の相対位置やポケットとの距離関係を正確に把握できるレベル。
