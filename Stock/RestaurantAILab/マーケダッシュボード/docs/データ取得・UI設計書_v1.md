## 1. 前提（信頼境界）

### 1.1 データソースの確定

| 区分 | データソース | 取得方法 | 月次粒度の可否 |
|---|---|---|---|
| 予約情報（生データ） | **食べログ** + **ホットペッパー** | レストランボード経由で 1 ファイル統合（推奨）／媒体管理画面個別取得（fallback） | ◯ 期間指定 CSV |
| クチコミ情報 | **食べログ** + **ホットペッパー** | 食べログ＝Playwright DOM 抽出、HPG＝管理画面 DOM 抽出 + 月別 xlsx の累計値 | ◯ |
| 来店実績（総組数・総客数） | **POS レジ** | 既存ダッシュボードの売上分析側データを参照（直結 or 既存 ETL の中間テーブル経由） | ◯ |
| 媒体内アクセス（PV） | 食べログ・HPG 各管理画面 | 食べログ＝Playwright DOM／HPG＝Excel | ◯（食べログ ページ別のみ「13ヶ月集計しか取れない」制約あり） |
| クチコミ / SEO 補助 | HPG 流入ワード CSV、食べログエリアランキング | 媒体管理画面 | ◯ |

### 1.2 v1 スコープ外（明示）

| 除外 | 理由 |
|---|---|
| 一休 / ぐるなび / オズモール / Retty | 本書時点で実機サンプル未取得。**v1 では「予約はレストランボード側 `予約経路` カラムで識別」のみ対応**（媒体管理画面の独自KPI＝PV等は v1 では取らない） |
| 食べログノートのウォークイン・卓稼働率・予約経路・新規リピート等の分析CSV | 「店舗の手入力依存」で値が信用できないため v1 では非採用（§2 で詳述） |
| HP マンスリークライアントレポート | 2026-06-25 PO 決定で「取得不可」確定済 |
| 自社直予約（電話・LINE 等の店舗手打ち） | レストランボードに登録されていれば取れる、なければ取れない（運用依存） |

### 1.3 月次集計の原則

- マーケダッシュボードの 1 行 = 1 店舗 × 1 月。
- 集計タイムゾーン: **00:00〜24:00 の暦月** に統一（食べログノート独自の「06:00〜30:00」境界は v1 では使わない）。
- 月締めタイミング: **翌月 3 日**（食べログ・HPG の前月確定 + レストランボード遅延吸収）。
- 当月途中の値は「速報値」扱いで UI 上に明示。

## 2. ウォークイン比率と「予約比率」の取り扱い ── 本設計の核

### 2.1 食べログノートが提供する「ウォークイン数」「予約比率」が信用できない理由

食べログノート 2.2 予約数・総客数 CSV、2.1 ダッシュボード CSV は「ウォークイン人数」「ウォークイン組数」を提供する。だがこれらは:

- **店舗側がノート画面に「ウォークイン来店」を手入力していないとカウントされない**
- 「電話予約」「自社サイト予約」「LINE 予約」も手打ち登録しない限り「ウォークイン or 未計上」になる
- 結果として **「ウォークイン比率 = 100% × 入力率」と「真のウォークイン率」の判別ができない**

実例: 月の総客数 200 人 / 食べログノート上のウォークイン記録 5 人 → ノート上の「ウォークイン比率 = 約 4%」だが、店舗が POS に記録した総客数が 350 人だとすると、真のウォークイン + 未計上予約 = 145 人（41%）になる。**100 人以上の客が「どこにも記録されていない」状態**。

### 2.2 解決方針 ── POS を Single Source of Truth に置く

POS レジが計上する「総組数」「総客数」を **唯一の真実値** とし、媒体予約データはその内訳として突き合わせる。

```
月次総組数（POS）                              ← Single Source of Truth
  ├─ 食べログ予約組数（媒体取得・キャンセル除く）
  ├─ ホットペッパー予約組数（媒体取得・キャンセル除く）
  ├─ その他媒体予約組数（レストランボード `予約経路` カラム）
  ├─ 自社直予約組数（電話・LINE・自社サイト等、レストランボードに登録された分のみ）
  └─ ウォークイン組数（= POS総組数 − 上記予約合計）   ← 引き算で算出
```

### 2.3 重要な前提条件と運用要件

| 前提 | 説明 |
|---|---|
| POS の総組数 ≥ 全媒体予約合計 | **常に成立**するはず。逆転したらデータ異常（突合監視で検知） |
| ノーショーの扱い | 媒体予約データは「キャンセル除外後」で集計 → POS 側で「来店なし」のものは存在しない前提で計算可 |
| ダブルカウント | 同一予約が複数媒体に重複登録される運用はないものとする（v1 では検知しない、運用ルールで担保） |
| 当日予約 vs ウォークイン | 食べログ・HPG 経由の「当日予約」も予約組数に含む。POS 突合時はウォークインから除外される |

### 2.4 媒体別「予約構成比」の定義

| 指標 | 定義 |
|---|---|
| 食べログ予約構成比 | 食べログ予約組数（月） ÷ POS総組数（月） |
| HPG 予約構成比 | HPG 予約組数（月） ÷ POS総組数（月） |
| 自社直予約構成比 | 自社直予約組数（月） ÷ POS総組数（月） |
| ウォークイン比率 | (POS総組数 − 全媒体予約合計) ÷ POS総組数 |
| 媒体予約合計比率 | 1 − ウォークイン比率 |

→ **これが「ウォークイン入力依存問題」を回避した最終形**。媒体ごとの「予約取得 KPI 正解値」と、店舗運営側の「ウォークイン取り込み力」を独立に評価できる。

## 4. 媒体横断 KPI 体系（月次）

### 4.1 KPI 階層

```
[Tier 1] 経営判断 KPI (3-5指標、月次変化で意思決定する)
├─ 総組数（POS）                          → 来店力
├─ 媒体予約構成比                          → 媒体ミックスの健全性
├─ 平均クチコミ評価（媒体加重平均）         → 顧客満足代理指標
└─ エリア順位（食べログ）                  → 競合内ポジション

[Tier 2] 媒体運用 KPI (媒体ごとに月次トラッキング)
├─ 媒体PV                                 → 露出
├─ 媒体予約 CVR (= 予約組数 ÷ PV)         → 媒体ページの集客効率
├─ 媒体クチコミ件数                       → 媒体内ソーシャル力
├─ 媒体平均評価                           → 媒体内評価
└─ HPG 流入ワード TOP10                   → SEO 改善材料

[Tier 3] 突合・整合性 KPI (異常検知)
├─ POS総組数 − 媒体予約合計 = ウォークイン組数  → 0 以下は異常
├─ 媒体予約合計 / POS総組数 = 媒体予約比率     → 急変は媒体側障害の可能性
└─ 当月途中値 vs 前月同時点値                  → トレンド異常検知
```

### 4.2 各 KPI の定義式

| KPI | 計算式 | データソース |
|---|---|---|
| 月次総組数 | sum(POS 日別 組数) | POS |
| 月次総客数 | sum(POS 日別 客数) | POS |
| 平均人数 | 総客数 ÷ 総組数 | POS |
| 食べログ予約組数 | count(レストランボード予約 where `予約経路`=食べログ系 and `ステータス`!=キャンセル) | レストランボード |
| HPG 予約組数 | 同上 where HPG系 | レストランボード |
| ウォークイン組数 | POS総組数 − 全媒体予約合計 | 突合算出 |
| 食べログ予約構成比 | 食べログ予約組数 ÷ POS総組数 | 突合算出 |
| 食べログ 媒体内 CVR | 食べログ予約組数 ÷ 食べログ全体PV | レストランボード + 食べログCV |
| HPG 媒体内 CVR | HPG 月別レポート「ネット予約組数」÷ HPG 月別「全体PV」 | HPG xlsx |
| クチコミ件数 / 媒体 / 月 | count(クチコミ where 来店月 = 対象月) | 食べログクチコミ / HPG クチコミ |
| 平均評価 / 媒体 / 月 | mean(評価 where 来店月 = 対象月) | 同上 |
| エリア順位 | scrape(`access_ranking` where `is_own_shop`=true).順位 | 食べログ |
| HPG 流入ワード | 流入ワードCSV TOP10 + 流入割合% | HPG freeword.csv |

## 5. システム保存データ設計

### 5.1 三層レイヤー

```
[L1: raw]   媒体ごとの生データ。取得日タイムスタンプ付きで「全履歴」を残す（再集計・遡及補正用）
   ↓ ETL（媒体別正規化）
[L2: stg]   全媒体の予約・クチコミ・アクセスを共通スキーマに正規化（媒体横断集計のための中間層）
   ↓ ETL（月次集約）
[L3: mart]  店舗 × 月で 1 行のマート。マーケダッシュボード UI が直接参照
```

### 5.2 テーブル一覧

#### L1: raw（生データ）

| テーブル名 | 1 行の粒度 | 取得元 |
|---|---|---|
| `raw_tabelog_access_daily` | 店舗 × 取得日 × 日付 | 食べログ アクセスレポート（日別） |
| `raw_tabelog_page_pv` | 店舗 × 取得日 × ページ種別 | 食べログ ページ別 PV（13ヶ月集計） |
| `raw_tabelog_conversion_monthly` | 店舗 × 取得日 × 年月 | 食べログ 来店指標 CV |
| `raw_tabelog_reviews` | 店舗 × 取得日 × クチコミ ID | 食べログ クチコミ一覧 |
| `raw_tabelog_ranking` | 店舗 × 取得日 × エリア × 順位 | 食べログ エリア内ランキング |
| `raw_hpg_access_daily` | 店舗 × 取得日 × 日付 | HPG daily.xlsx |
| `raw_hpg_summary_monthly` | 店舗 × 取得日 × 発行月 | HPG monthly.xlsx |
| `raw_hpg_freeword_monthly` | 店舗 × 取得日 × 順位 × ワード | HPG freeword.csv |
| `raw_hpg_reviews` | 店舗 × 取得日 × クチコミ ID | HPG クチコミ管理画面 |
| `raw_restaurant_board_reservations` | 店舗 × 取得日 × 予約番号 | レストランボード予約一覧 |
| `raw_pos_daily` | 店舗 × 日付 | POS（既存ETL中間テーブル参照） |

→ raw は **「取得日 = 取り直し可能」**。同じ年月のデータを 2 回取った場合は両方残し、最新取得日を `is_latest` フラグで管理。

#### L2: stg（正規化中間層）

| テーブル名 | 1 行の粒度 | 概要 |
|---|---|---|
| `stg_reservations` | 店舗 × 予約ID | 全媒体予約を共通スキーマで統合（`source` カラム: tabelog / hpg / direct / other） |
| `stg_reviews` | 店舗 × クチコミID | 食べログ + HPG クチコミ統合（`source`, `visit_month`, `rating`, `posted_date`） |
| `stg_access_daily` | 店舗 × 日付 × 媒体 | 食べログ日別 + HPG日別を縦持ち統合 |
| `stg_pos_daily` | 店舗 × 日付 | POS 日別組数・客数（raw からのコピー or ビュー） |

#### L3: mart（月次マート）

| テーブル名 | 1 行の粒度 | 用途 |
|---|---|---|
| `mart_monthly_pos` | 店舗 × 年月 | POS 月次集計（総組数 / 総客数 / 平均人数） |
| `mart_monthly_reservation_by_media` | 店舗 × 年月 × 媒体 | 媒体別予約組数・人数・キャンセル数 |
| `mart_monthly_walkin` | 店舗 × 年月 | ウォークイン組数 = `mart_monthly_pos.total - sum(mart_monthly_reservation_by_media.組数)` |
| `mart_monthly_funnel_by_media` | 店舗 × 年月 × 媒体 | PV → 予約 → 来店 のファネル |
| `mart_monthly_reviews_by_media` | 店舗 × 年月 × 媒体 | 件数 / 平均評価 / 評価分布 |
| `mart_monthly_ranking` | 店舗 × 年月 | 食べログエリア順位（時系列） |
| `mart_monthly_seo` | 店舗 × 年月 × 媒体 × 順位 | HPG 流入ワード TOP20 |
| `mart_monthly_dashboard` | 店舗 × 年月 | ダッシュボード KPI タイル用 1 行集約 |

### 5.3 主要テーブルの列定義

#### `stg_reservations`（全媒体予約統合）

| 列 | 型 | 説明 |

### 5.4 主キー設計と冪等性

- raw 系: `(shop_id, fetched_at, *自然キー*)` で UNIQUE → 同じ取得結果を2回入れても1行のみ。
- stg 系: 自然キー（`reservation_id` 等）で UPSERT。最新の取得分で上書き。
- mart 系: `(shop_id, year_month)` で UPSERT。月確定後でも遡及補正が入れば再計算で上書き可。

→ **冪等性が確保されているので、月初の自動取得が一部失敗しても、その媒体だけ再実行すれば mart が更新される**。

## 6. UI 画面定義

### 6.1 ナビゲーション構造

```
マーケダッシュボード（既存売上分析の隣にタブとして追加）
│
├── [1] 月次サマリ（トップ画面・デフォルト表示）
├── [2] 集客・予約ファネル
├── [3] 媒体別深掘り（食べログ / HPG）
├── [4] クチコミビュー
├── [5] 競合・SEO
└── （月選択ピッカーは全画面共通でヘッダに常駐）
```

期間ピッカー: `YYYY-MM` プルダウン。デフォルト = 「前月」（最新確定月）。

### 6.2 [1] 月次サマリ画面

**目的**: 店舗1か月の「健康診断」。1 画面で経営判断 Tier 1 KPI が一目で分かる。

**レイアウト**: KPI タイル 8 個（4 列 × 2 行）+ 推移グラフ 1 つ

| タイル | 表示値 | 補助情報 |
|---|---|---|
| 総組数 | `pos_total_groups`（大きく） | 前月比 / 前年同月比 / 12ヶ月スパークライン |
| 総客数 | `pos_total_customers` | 同上 |
| 平均人数 | `pos_avg_party_size` | 同上 |
| 媒体予約比率 | `1 - walkin_groups/pos_total_groups`（%） | 食べログ / HPG / 直 の積み上げバー |
| ウォークイン組数 | `walkin_groups`（POS突合算出） | 「※POS総組数 − 予約合計で算出」注記 |
| 食べログ予約 CVR | `tabelog_cvr`（%） | 前月差・スパークライン |
| HPG 予約 CVR | `hpg_cvr`（%） | 同上 |
| エリア順位 | `tabelog_area_rank`（食べログ） | 前月差（▲5 等）・スパークライン |

下部: **媒体予約構成 12 ヶ月推移グラフ**（積み上げ面グラフ、X軸 = 月、Y軸 = 組数、色分け = 媒体）

警告バナー:
- データ取得未完了の媒体がある場合: 「⚠ HPG xlsx 未取得・データは速報値です」
- POS突合異常（媒体予約合計 > POS総組数）の場合: 「⚠ データ整合性エラー」

### 6.3 [2] 集客・予約ファネル画面

**目的**: 「PV はあるのに予約に繋がっていない」「予約はあるのに来店していない」を媒体別に可視化。

**レイアウト**: 媒体タブ（食べログ / HPG）+ ファネル図

```
食べログ ファネル（2026-06）
┌─────────────────────┐
│  食べログ全体 PV   │  12,345
└─────────┬───────────┘
          │  CVR_予約: 0.8%
┌─────────▼───────────┐
│  ネット予約組数     │  98
└─────────┬───────────┘
          │  来店率: 92% (POS突合)
┌─────────▼───────────┐
│  来店組数（推定）   │  90
└─────────────────────┘
```

「来店率」は **同月の予約 vs POS で実際に来店した推定** という注記付き（=キャンセル除外後の予約組数 ÷ POS の食べログ経由来店、ただし POS で媒体識別できない場合は媒体予約組数を上限とする）。

**Phase 1.0 では「来店率」段は省略可**（POS と媒体予約の紐付け運用が未整備のため）。Phase 1.1 で予約→来店の紐付けを実装後に追加。

### 6.4 [3] 媒体別深掘り画面

**目的**: 媒体運用担当が個別媒体の月次推移を追う。

**レイアウト**: 媒体タブ（食べログ / HPG）切替で同じテンプレを使い回し。

各媒体ビューに表示する 12 ヶ月推移グラフ:
- PV（全体・PC/スマホ/アプリ別、食べログのみ）
- 予約組数
- CVR
- クチコミ件数
- 平均評価

食べログタブ専用: ページ別 PV ランキング表（13ヶ月集計のため「期間: 2025-05〜2026-05 累計」と注記）

HPG タブ専用: クライアントレポートの主要指標（クーポン枚数 / 集客人数 / 組人数）

### 6.5 [4] クチコミ画面

**目的**: クチコミの量と質を月次トラッキング + 最新クチコミを店長が読める。

**レイアウト**:
- 上段: 月次クチコミ件数推移グラフ（媒体別積み上げ棒）
- 中段: 月次平均評価推移グラフ（媒体別折れ線）
- 下段: 当月クチコミ一覧テーブル（媒体 / 来店月 / 評価 / 投稿者 / 本文先頭100文字）

ソート: 来店月降順。フィルタ: 媒体・評価範囲。

### 6.6 [5] 競合・SEO 画面

**目的**: 競合内ポジションと検索流入の改善材料を提供。

**レイアウト**:
- 上段: 食べログ エリア順位 12ヶ月推移グラフ（折れ線、Y軸反転で「1位が上」表示）
- 中段: エリア内 TOP10 店舗テーブル（当月スナップショット、自店舗ハイライト）
  - 列: 順位 / 店舗名 / ジャンル / 駅エリア / アクセス数 / 前月比
- 下段: HPG 流入ワード TOP20（当月）
  - 列: 順位 / フリーワード / 流入割合% / 同エリア競合数 / 前月変動

### 6.7 すべての画面で共通の UI 要素

| 要素 | 仕様 |
|---|---|
| 期間ピッカー | ヘッダ常駐。`YYYY-MM` プルダウン。前月デフォルト。 |
| 店舗切替 | 多店舗運用時のヘッダプルダウン（外注ロールではID/PW非表示） |

## 7. 既知の落とし穴・運用注意

| # | 問題 | 対処 |
|---|---|---|
| W1 | レストランボード未契約店舗では媒体予約の統合取得不可 | 食べログ・HPG 個別の予約 CSV / xlsx を取得して合算する fallback パスを fetcher 側に実装 |
| W2 | 食べログノートのウォークイン比率は信用しない | §2 の POS 突合方式で算出（本書の核） |
| W3 | 食べログ ページ別 PV は 13ヶ月集計のスナップショットしか取れない | UI 上「参考値・期間集計」と明示。月次 KPI には組み込まない |
| W4 | 食べログ 日別アクセスは直近1ヶ月のみ | 月初に必ず取得（取り逃すと欠損）。fetcher の失敗通知必須 |
| W5 | クチコミの「来店月」と「投稿月」がズレる | UI では「来店月ベース」で表示。投稿月ベースが必要になったらマート列を追加 |
| W6 | POS 総組数 < 媒体予約合計 で逆転する場合 | データ整合性エラーとして UI で警告 + Slack 通知。原因: 同一予約が複数媒体に重複登録、POS 入力漏れ、媒体側のキャンセル反映漏れ |
| W7 | 媒体管理画面の DOM 変更 | パース失敗を月初に検知（fetcher 戻り値 / parser stderr）→ Slack 通知 → 手動修正 |
| W8 | 食べログ・HPG・レストランボードのアカウントロック | リトライバックオフ + ロック検知時の自動停止（連続失敗3回でアカウントロック想定） |
| W9 | 媒体経路ラベルの揺れ | `media_mapping.yaml` で正規化。未登録ラベルが現れたら `direct_other` に倒し、警告を出す |
| W10 | 自社直予約（電話/LINE）の取りこぼし | レストランボードに登録された分のみ計上。運用ルールで店舗側に登録徹底を依頼（責務分離） |
