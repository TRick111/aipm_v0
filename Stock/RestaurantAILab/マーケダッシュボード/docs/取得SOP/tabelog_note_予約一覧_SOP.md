# 食べログノート — 予約一覧 ダウンロード SOP

最終更新日: 2026-06-29
店舗: 韓国屋台ポックンパ ネネチキン 東岡崎店
対応 Dashboard reportType: `tabelog_note_reservations`（`lib/marketing/reportTypes.ts`）
パーサ実装の正: `lib/marketing/parsers/tabelogNote.ts` の `parseTabelogNoteReservations`

## 1. ログイン URL と必要認証情報

| 項目 | 内容 |
|---|---|
| ログイン URL | https://owner.tabelog.com/owner_account/login/ |
| 認証方式 | 食べログ店舗会員 ID / パスワード |
| 2 要素認証 | **公式ヘルプ上では必須化記載なし**（IP 制限・ロックの可能性あり、本番運用前に要確認） |
| ID / PW の保管 | **要 user 確認**（1Password 等のチーム共有 vault 想定。現時点で TaskRunner 側に未登録） |
| アカウント共有 | 食べログ本体（owner.tabelog.com）・食べログノート共通の店舗会員アカウント |
| セッション保持期間 | Cookie ベース。具体的な TTL は公式記載なし（経験則: 1〜2 週間程度で再認証要求が発生する想定） |
| アカウントロック | 連続失敗で発生する想定。バックオフ必須 |

## 2. ダウンロード手順（手動）

1. ブラウザで https://owner.tabelog.com/owner_account/login/ にアクセスしログイン
2. グローバルナビ／サイドメニューから「食べログノート」を選択
3. 左サイドナビ（PC ブラウザ推奨）の「分析」をクリック
4. URL を `https://owner.tabelog.com/tn/analysis?yearMonth=YYYY-MM` 形式に直接書き換える（対象月を指定）
   - 例: 2025-07 分なら `?yearMonth=2025-07`
5. 画面に「予約一覧」セクションが表示されたら、画面内の **「CSV ダウンロード」ボタン** をクリック
6. ファイル名（既定）: `予約 (N).csv` ※ ブラウザの保存ダイアログで連番が付く
7. 保存後、後述§6 の命名規則に従ってリネーム

> 補足: 公式ヘルプ上は「予約一覧 CSV」の明示記載はないが、`sample_data/README.md` および調査メモ（食べログノート_CSV出力項目.md §2.10）で **2026-04 / 2026-06 で実機ダウンロード確認済**。`yearMonth` クエリで任意月取得可能。

## 3. 期間指定の限界

| 項目 | 仕様 |
|---|---|
| 過去の遡及範囲 | `yearMonth=YYYY-MM` で任意月を指定可能。**公式に過去保持期間の明示記載なし**。実機サンプルは 2026-04, 2026-06 で取得確認済 |
| 1 リクエストあたり最大期間 | **月単位固定**（複数月を 1 リクエストで取得することは不可） |
| 月またぎ境界 | 食べログノート独自の **06:00〜翌06:00（30:00）境界**で日次集計。CSV の `来店日` は来店時刻ベース（要再確認） |
| 月確定タイミング | 月初に前月分が確定。確定後の予約編集は過去 CSV に反映されない（**確定済み月の値は安定**） |
| 当月途中ダウンロード | 値が変動するため、当月の取得値は速報扱い |

## 4. 出力ファイルの仕様

| 項目 | 仕様 |
|---|---|
| 文字コード | **Shift_JIS / CP932 想定**（食べログ系の旧来 CSV 流儀。実機サンプルで要確認） |
| 改行 | CRLF（推定） |
| 区切り | カンマ |
| 拡張子 | `.csv` |
| 行粒度 | 1 行 = 1 予約 |

### 必須カラム（`parseTabelogNoteReservations` が参照）

パーサが必須トークンとして検出: `予約No`, `来店日`

パーサが読むカラム（22 列のうち抜粋）:

```
予約No, 顧客No, 来店日, 来店時間, 人数, お子様人数,
ステータス, 受付窓口, 卓, コース, メニュー/コース
```

> **注**: 実機サンプル（`食べログノート_予約一覧_2026-04.csv`）のヘッダは「受付方法」「お客様人数」表記だが、パーサ側は「受付窓口」「お子様人数」を参照している（命名ゆれ）。**Phase 1.0 取り込み時に名称ずれが発生する可能性あり**。実取り込み時にエラーが出たら `parseTabelogNoteReservations` のキーをサンプル準拠に修正するか、`media_mapping.yaml` 相当のヘッダ正規化を入れる必要あり。

## 5. 12 ヶ月遡及取得の作業見積もり

| 項目 | 見積もり |
|---|---|
| 1 ヶ月あたりの操作 | URL の `yearMonth` を書き換え → CSV ダウンロードボタン → リネーム → 配置 |
| 1 ヶ月あたりの所要時間 | **約 1.5 分**（URL 書き換え 10 秒 + ダウンロード待機 10 秒 + リネーム 30 秒 + 配置 30 秒） |
| 12 ヶ月分の総工数 | **約 18 分**（ログイン込みで 20〜25 分） |
| 一括取得の可否 | **不可**（月単位 1 ファイル DL のみ。月を跨いだ 1 リクエスト不可） |
| 取得方針 | **月別 1 ファイル × 12 回**。`yearMonth` パラメータを 12 回切り替えて連続取得 |

> **過去 12 ヶ月遡及の留意点**: 食べログノートは公式に過去データ保持期間を明示していない。**サンプルでは 2026-04 と 2026-06 のみ動作確認** のため、12 ヶ月遡って 2025-07 〜 2026-06 の全月で取得可能かは **実機で要検証**。最古月から DL 試行し、404 / 空ファイルになる月があれば取得可能範囲を絞る。

## 7. 既知の落とし穴

| # | 内容 | 対処 |
|---|---|---|
| 1 | ヘッダ名のゆれ（「受付方法」vs パーサの「受付窓口」、「お客様人数」vs「お子様人数」） | 取り込み時に NG が出たらパーサ側 or 正規化レイヤを修正 |
| 2 | Shift_JIS のため Excel で開いて再保存すると UTF-8 化されてしまう | ダウンロード後は **そのまま** アップロード。Excel で開かない |
| 3 | 食べログノートの日次境界が 06:00〜30:00 と独特。`来店日` カラムを 0:00 基準で解釈するとズレる可能性 | データ取得・UI 設計書 §1.3 に従い「v1 では 00:00〜24:00 暦月」を採用。深夜来店は要検証 |
| 4 | 他社台帳連携店舗（ebica/Resty/RESZAIKO/レスラク）では分析・CSV 機能が無効化される | ポックンパは食べログノート直利用前提（要 user 確認） |
| 5 | 当月途中の DL は値が変動 | 月確定後（翌月 1〜3 日）に取得する |
| 6 | キャンセル予約も含まれる | パーサ／集計側で `ステータス` を見て除外（`status != cancelled` で集計） |

## 8. 自動化の見通し

| 項目 | 見通し |
|---|---|
| Playwright 自動化の所要工数 | **半日**（ログイン + storageState 保存 + URL ループ + download イベント捕捉）。既存 `scripts/scrapers/tabelog/` の 3 層（認証 / フェッチ / パース）パターンを踏襲可能 |
| Selector の安定性 | CSV ダウンロードボタンは食べログノート分析画面に常設。**HTML 構造が変わらない限り安定** |
| ログイン状態保持 | `state/storageState.json`（Cookie ベース）。食べログ本体（owner.tabelog.com）と同一アカウントのため、**既存の食べログ用 storageState を流用可能性あり**（ドメインが同じ） |
| 食べログ本体スクレイパとの共有 | 既存 `scripts/scrapers/tabelog/login_and_save_state.py` で取得した storageState の流用で初手は動くと推定 |
| 既存スクレイパの位置付け | スキル `tabelog-marketing-scrape` は食べログ本体 5 ページ（access_report_total 等）のみ対象。**食べログノート分析 CSV は別 fetcher が必要** |

### 既存スクレイパでの過去 12 ヶ月遡及可否（脚注）

- 既存スキル `tabelog-marketing-scrape` は `yearMonth` パラメータ未対応（access_report_page を除き月固定スクレイプ）。
- 食べログ本体側 `access_report_total?display_type=monthly` は **13 ヶ月分の月別データを一発取得** 可能（過去 12 ヶ月遡及はカバー）。
- 食べログノート予約一覧の自動化（Phase 1.1）では、`yearMonth` ループで月別 fetch する実装を新規追加する必要あり。
