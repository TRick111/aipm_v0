## 1. はじめに

### 1.1 文書の目的

Chef's Room がクライアント飲食店に対して提供する CMO／CFO プランの外注化を可能にするため、既存ダッシュボード（売上分析・PL 管理）に **マーケ媒体の月次データを集約するビューと、外注コンサル向けの権限制御** を追加する。本書はその要件を定義する。

### 1.2 対象読者

- 田中（実装担当）
- 竹矢（PO・運用責任者）
- 吉田（事業責任者）
- 将来の外注コンサル担当者
- 将来の保守担当者

### 1.3 用語

| 用語 | 意味 |
|---|---|
| マーケ媒体 | 食べログ、食べログノート、ホットペッパー、一休、オズモール、Retty、ぐるなび 等の集客／予約媒体 |
| 月次レポート | 各媒体が提供する月単位の集計レポート（CSV／PDF／Excel／管理画面） |
| 外注コンサル | Chef's Room の CMO／CFO プランを代行する外部の事業者 |
| RESZAIKO | 一休／オズモール等で利用される予約管理 SaaS |

## 2. 背景と目的

### 2.1 現状の課題

- Chef's Room が抱えるクライアント店舗に対する CMO／CFO プランを **外注化** する計画があり、外注コンサルに各媒体の月次データを共有する必要がある
- 各媒体のログイン ID／PW を外注に直接渡すと、**支払情報など機密情報まで見えてしまう** ためセキュリティ上不可
- 現状は社内担当が各媒体に手動ログインして数字を集計しており、**人手依存・スケール不可** の状態
- 媒体ごとにレポート形式（CSV／PDF／管理画面のみ）が違い、横断分析ができない

### 2.2 本機能で実現すること

1. **媒体側ログイン情報を社内に集中保管**し、外注コンサルには数値だけをダッシュボード経由で提供
2. **各種マーケ媒体の月次 KPI をシステムで一元管理** できる状態を作る（第一目標）
3. その上で、外注コンサルが月次レポート（スライド形式）をクライアントに納品できる

### 2.3 ステークホルダーごとの提供価値

| ステークホルダー | 提供価値 |
|---|---|
| 外注コンサル | 担当店舗の月次マーケKPIを一画面で確認。ID/PW なしでクライアント MTG を進行できる |
| 店舗（クライアント） | 自店の集客状況を媒体横断で把握できる |
| Chef's Room | 外注コンサルへの情報統制を担保しつつ、月次データ収集の自動化で人手負荷を削減 |

## 4. 全体アーキテクチャ概要

```
[マーケ媒体 7]                      [ダッシュボード]                     [ユーザー]
食べログ ────┐
食べログノート ┤                   ┌──────────────────────┐         外注コンサル
ホットペッパー ┼──月次CSV/PDF──→ │ Marketing View       │ ──→  店舗
一休 ─────────┤  または            │ ・媒体横断KPI         │       Chef's Room
オズモール ───┤  Playwright自動取得│ ・媒体ごと詳細        │
Retty ────────┤                   │ ・スライド月次レポート │
ぐるなび ─────┘                   └──────────────────────┘
                                            ↑
                                  媒体マスタ（拡張可能）
```

- **データ取得**: 手動 CSV アップロード + Playwright 自動取得のハイブリッド（§5.5）
- **保管**: PostgreSQL（KPI 値）＋ Vercel Blob（CSV 原本）
- **権限**: 管理者 / 店舗ユーザー / **外注コンサル**（新規ロール）/ 店舗グループ
- **出力**: ダッシュボード画面 + HTML スライド（月次レポート）

### 5.5 CSV ダウンロード自動取得機能（M-14 新規・Phase 2 → Phase 1 繰上）

| ID | 機能 | Phase | 受入基準 |
|---|---|:---:|---|
| **M-14** | CSV／レポート自動取得（Playwright） | 1（段階導入） | 媒体ログイン → レポートダウンロード → アップロード API への送信、を月次バッチで実行できる。媒体ごとに段階的に対応。失敗時は人手フォールバックに切り替え可能 |
| M-14a | Playwright 実行基盤 | 1 | スケジューラ（cron or Vercel Cron）＋ 媒体ごとのスクリプト構造（`scripts/marketing/fetch/{channel}.ts`）が整備されている |
| M-14b | 媒体ログイン情報の暗号化保管 | 1 | 媒体 ID／PW を環境変数または暗号化テーブルで保管。コンサルからは見えない（S-03 の運用ルール化と整合） |
| M-14c | 2 要素認証への対応 | 1 | 媒体側で 2 要素認証が要求される場合の運用手順（人手介入のフォールバック）が定義されている |
| M-14d | 媒体規約レビュー | 1 | 食べログ・一休・オズモールほか各媒体の利用規約を確認し、自動取得が許容される媒体を明示する |

> **注**: Playwright 自動取得は **媒体ごとに段階導入** する前提（全 7 媒体一気の実装は工数大）。リリース時点で対応する媒体は §10 のスケジュール案ごとに変動。詳細は §11 リスク参照。
