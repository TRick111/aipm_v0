## 1. はじめに

### 1.1 文書の目的

Chef's Room がクライアント飲食店に対して提供する CMO／CFO プランの外注化を可能にするため、既存ダッシュボード（売上分析・PL 管理）に **マーケ媒体の月次データを集約するビューと、外注コンサル向けの権限制御** を追加する。本書はその要件を定義する。

### 1.2 対象読者

- 田中（実装担当）
- 竹矢（PO・運用責任者）　
- 将来の外注コンサル担当者
- 将来の保守担当者

### 1.3 用語

| 用語 | 意味 |
|---|---|
| マーケ媒体 | 食べログ、食べログノート、ホットペッパー、一休、オズモール、Retty、ぐるなび 等の集客／予約媒体 |
| 月次レポート | 各媒体が提供する月単位の集計レポート（CSV／PDF／Excel／管理画面） |
| 外注コンサル | Chef's Room の CMO／CFO プランを代行する外部の事業者 |
| RESZAIKO | 一休／オズモール等で利用される予約管理 SaaS |

## 2. 背景と目的

### 2.1 現状の課題

- Chef's Room が抱えるクライアント店舗に対する CMO／CFO プランを **外注化** する計画があり、外注コンサルに各媒体の月次データを共有する必要がある
- 各媒体のログイン ID／PW を外注に直接渡すと、**支払情報など機密情報まで見えてしまう** ためセキュリティ上不可
- 現状は社内担当が各媒体に手動ログインして数字を集計しており、**人手依存・スケール不可** の状態
- 媒体ごとにレポート形式（CSV／PDF／管理画面のみ）が違い、横断分析ができない

### 2.2 本機能で実現すること

1. **媒体側ログイン情報を社内に集中保管**し、外注コンサルには数値だけをダッシュボード経由で提供
2. **各種マーケ媒体の月次 KPI をシステムで一元管理** できる状態を作る（第一目標）
3. その上で、外注コンサルが月次レポート（スライド形式）をクライアントに納品できる

### 2.3 ステークホルダーごとの提供価値

| ステークホルダー | 提供価値 |
|---|---|
| 外注コンサル | 担当店舗の月次マーケKPIを一画面で確認。ID/PW なしでクライアント MTG を進行できる |
| 店舗（クライアント） | 自店の集客状況を媒体横断で把握できる |
| Chef's Room | 外注コンサルへの情報統制を担保しつつ、月次データ収集の自動化で人手負荷を削減 |

## 4. 全体アーキテクチャ概要

```
[マーケ媒体 7]        [ローカル管理端末]              [Vercel アプリ]                [ユーザー]
食べログ ────┐
食べログノート ┤      ┌────────────────────┐                                       外注コンサル
ホットペッパー ┼──→ │ Claude Code +       │── HTTP/JSON →┌──────────────────────┐ ──→  店舗
一休 ─────────┤      │ Playwright スキル   │  アップロードAPI│ Marketing View       │  Chef's Room
オズモール ───┤      │ ・媒体ログイン      │              │ ・媒体横断KPI         │
Retty ────────┤      │ ・CSV ダウンロード  │              │ ・媒体ごと詳細        │
ぐるなび ─────┘      │ ・正規化/送信       │              │ ・スライド月次レポート │
                      └────────────────────┘              └──────────────────────┘
   ※ 手動アップロード経路（既存）も併存可
```

- **データ取得**: 手動 CSV アップロード + **ローカル Claude Code 経由の自動取得スキル**（§5.5）
  - 自動取得部分は Vercel 上のアプリには載せず、**運用者のローカル端末上で Claude Code が Playwright スキルを呼び出して実行**する
  - 取得後、Vercel アプリ側のアップロード API（手動 CSV アップロードと同じ口）を叩いてダッシュボードに反映
- **保管**: PostgreSQL（KPI 値）＋ Vercel Blob（CSV 原本）
- **権限**: 管理者 / 店舗ユーザー / **外注コンサル**（新規ロール）/ 店舗グループ
- **出力**: ダッシュボード画面 + HTML スライド（月次レポート）

### 5.4 CSV ダウンロード自動取得スキル（M-14 新規・ローカル Claude Code 実装）

> **重要な前提（変更）**: 本機能は **Vercel 上のアプリ側機能としては実装しない**。
> 運用者のローカル管理端末上で動作する **Claude Code のスキル** として実装する。
> ローカル Claude Code が Playwright スキル（Play Lite 前提）を呼び出して各媒体からデータをダウンロードし、Vercel アプリの **アップロード API**（M-08 の手動アップロードと同じ口）を叩いてダッシュボードに反映する。
> これにより Vercel 上に媒体ログイン情報や常駐スクレイピングプロセスを置かずに済む。

| ID | 機能 | Phase | 実装場所 | 受入基準 |
|---|---|:---:|---|---|
| **M-14** | CSV／レポート自動取得スキル | 1（段階導入） | ローカル端末（Claude Code） | 運用者が Claude Code 上でスキルを呼ぶと、媒体ログイン → レポートダウンロード → Vercel アプリのアップロード API への送信、までを実行できる。媒体ごとに段階的に対応。失敗時は人手フォールバックに切り替え可能 |
| M-14a | スキル定義（SKILL.md）と Playwright 実行 | 1 | ローカル端末（Claude Code） | 媒体ごとに SKILL.md を整備し、ローカル Claude Code が Playwright（Play Lite）スキルを介して該当媒体にログイン・CSV ダウンロードできる。スケジューラは Vercel Cron ではなく **ローカル端末側の運用ループ（運用者の手元起動 or ローカル cron）** とする |
| M-14b | 媒体ログイン情報のローカル保管 | 1 | ローカル端末 | 媒体 ID／PW はローカル端末側（1Password CLI／ローカル暗号化ファイル等）に保管する。**Vercel 側 DB／環境変数には保管しない**（外注コンサルからの完全分離） |
| M-14c | 2 要素認証への対応 | 1 | ローカル端末 | 媒体側で 2 要素認証が要求される場合、ローカル運用者が手元で介入できるフローを定義する |
| M-14d | 媒体規約レビュー | 1 | 共通 | 食べログ・一休・オズモールほか各媒体の利用規約を確認し、自動取得が許容される媒体を明示する |
| M-14e | Vercel アプリ側アップロード API | 1 | Vercel アプリ | M-14 スキルから叩かれる前提の HTTP エンドポイント（CSV / 正規化済み JSON 受け口）を提供。認証はサービストークン |

> **注**: 自動取得スキルは **媒体ごとに段階導入** する前提（全 7 媒体一気の実装は工数大）。リリース時点で対応する媒体は §10 のスケジュールに従う。詳細は §11 リスク参照。

| 国際化 | 日本語のみ対応（既存ダッシュボードと同じ） |

| 環境 | Development / Staging / Production の3環境（既存と同じ） |

## 8. データモデル概要

## 11. リスクと前提

| # | リスク／前提 | 影響 | 対策 |
|---|---|---|---|
| R-1 | 食べログ等の利用規約でスクレイピング／自動取得が禁止されている可能性 | M-14 が一部媒体で実施不可 | M-14d 規約レビューを Phase 1 着手時に実施。不可媒体は手動アップ継続 |
| R-2 | 2 要素認証（特に一休の Yahoo!ID 連携／ホットペッパー）への対応 | 自動取得が安定しない | M-14c のローカル運用者手元介入フローを整備 |
| R-3 | Retty／ぐるなびのレポート仕様が未知 | パーサ実装の見積もり困難 | 6/25 までにサンプル CSV／スクショ提供を依頼（§13 確認事項） |
| R-4 | 媒体側 UI 変更によるローカル Claude Code スキル（Playwright）破壊 | 自動取得が断続的に失敗 | 失敗時はローカル運用者が即時手動切替（SKILL.md にフォールバック手順を明記）。X-02 に含む |
| R-5 | 食べログの 050 番号 + PR／集客契約が未締結の店舗で来店指標が取れない | 該当店舗でコンサル分析の深さが落ちる | 契約状況を §13 で確認。未締結店舗は KPI 一部欠落を許容するか別途協議 |
| R-6 | スコープ拡張による開発期間延伸 | 7 月初旬リリースが守れない可能性 | §10 の B 案（段階リリース）で対処（採用済） |
| R-7 | ローカル管理端末が運用者個人に依存し、属人化／可用性低下のリスク | 月次の自動取得が回らなくなる可能性 | 運用端末・1Password Vault・SKILL.md をチームで共有可能な状態に整備。手動アップロード経路は常時維持 |

## 12. 受入基準（Acceptance Criteria）
