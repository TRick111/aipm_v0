# BFA（BAR FIVE Arrows）固有ルール

- **作成日**: 2026-05-18
- **適用範囲**: BFA（BAR FIVE Arrows / 店舗コード `bfa-001` / 運営会社 `BFA`）の月報・週報・日次分析すべて
- **目的**: BAR FIVE Arrows の業態（バー／深夜営業／ボトル運用／CLP メンバーシップ／インバウンド対応）に固有の集計ルール・抽出キー・データソース仕様を一元管理し、月報共通ルール `Stock/RestaurantAILab/月報/共通ルール.md` ／週報共通ルール `Stock/RestaurantAILab/週報/共通ルール.md` の双方から参照する

> 飲食店一般に通用する集計ルール（業務日基準、3点セット、ランキングTOP15、メッセージ帯、分類条件帯、絵文字運用、カラーパレット 等）は **各週月の共通ルール側** に置き、本ファイルでは **BFA でしか成立しない / 解釈の前提が変わる** ものだけを扱う。

---

## §B-1 ボトル購入の取扱い

| 集計対象 | ボトル購入の扱い |
|---|---|
| 月次／週次の総売上 | **含む** |
| 客単価分子（経営判断用） | **除外**（=（売上−ボトル）÷客数 で算出） |
| 週次・日次・曜日別・時間帯別 | **除外** |
| 会計レベル分析（中央値・分布等） | **除外** |

抽出キー: POSカテゴリ `ボトル購入` 全件（SAKAYA 用／仕入れボトル／CLP 利用分などの内訳を区別せず合算）。

> 他店舗（BBC / 麻布しき / キーポイント）には「ボトル購入」カテゴリが存在しない、または意味が異なるため、本ルールは **BFA のみ** に適用する。

---

## §B-2 客単価2系統（BFAのみ）

| 指標 | 定義 | 用途 |
|---|---|---|
| 客単価（ボトル除く・経営判断用） | （売上 − ボトル購入）÷ 客数 | 傾向分析の基準値 |
| 客単価（全部込み・エアメイト比較用） | 売上 ÷ 客数 | エアメイト／エアレジとの数字一致確認用 |

KPIカード・前月比較・前年比較・前週比較・客単価推移グラフ すべてで **2系統を併記** する。

## §B-4 POS CSV（エアレジ会計明細）カラム参照

BFA は AirRegi POS を使用しているため、CSV カラム名は以下に従う（Dinii / USEN とはスキーマが異なる点に注意）。

| カラム | 用途 |
|---|---|
| 取引No | 会計のキー（ヘッダー行と明細行で共有） |
| 会計日 / 会計時間 | datetime 化に使用 |
| 合計 | ヘッダー行の会計合計金額 |
| 人数 | ヘッダー行の客数 |
| 滞在時間 | 入店時刻算出（= 会計時間 − 滞在時間）に使用 |
| カテゴリー名 | 明細行のカテゴリ |
| メニュー名 | 明細行の商品名 |
| 価格 / 注文数量 | 明細行の単価／数量 |
| メモ | ヘッダー行の自由記入（インバウンド「海外」等の抽出キー） |

**注意**: 1行で「ヘッダー＋明細」を兼ねる行が稀に存在（単発イベント費用等）。
- `is_header = 合計あり`
- `is_detail = カテゴリ＋メニュー名あり`
で別判定する。

> **キーポイント（かめや駅前店）も AirRegi のためカラム仕様は共通だが、業態が異なるためメモ欄運用・カテゴリ運用は別**。本ファイルの仕様はカラム参照部分のみ AirRegi 共通として流用可能。

## §B-5 データソース正本（BFA）

- BFA の組数・売上・客数は、**ダッシュボードDB（prod）の生データに準拠する**
- データソースは `sales_data` ＋ `sales_items`（prod DB ep-rough-bird）
- エアレジ提供CSV（会計明細）または `Dashboard/scripts/_bfa_export_prod_to_airregi.mjs` で抽出した CSV を入力とする
- **エアメイト表示値との完全一致は保証しない**（§B-6 参照）

---

## §B-6 エアメイトとの組数差異について

BFA の組数は **「業務日基準（朝6時境界）＋ 合計>0 フィルタ」** で集計する（業務日基準は各共通ルール §2 参照）。
**エアメイトの表示組数とは一致しないことがある**。これは仕様であり、BFA 側を無理に合わせない。

### B-6.1 差異の主な要因

- **取引No `0001` プレフィックスの会計**（CLP／試飲／イベント／別端末入力等の別系統会計）
  - メインAirRegi（取引No `RB01...`）とは別経路で sales_data に登録される
  - 月により 0〜20件超の幅で発生する
- **エアメイトは月により `0001` 系を採用／除外する基準が一定でない**
  - 例: 2026-03 は 22件中 約18件 を airmate 側でカウントしていない
  - 月報側（業務日基準＋合計>0）では全件カウントするため +18 程度の差が出る

### B-6.2 rawdata.csv の利用上限

- rawdata.csv は **スナップショット時点依存** のため、当月末の追記分が欠落するリスクがある
- 入店時刻と会計日跨ぎのレコード（深夜帯）で **業務日基準と会計日基準が混在** している場合がある
- **当月分は必ず prod DB から直接抽出する**（`_bfa_export_prod_to_airregi.mjs` 推奨）
- rawdata.csv は **前月以前の比較データ用** に限定して使う

### B-6.3 実例（2026-01〜04 BFA／業務日基準・合計>0 適用後）

| 月 | 月報 | エアメイト | 差 | 主因 |
|---|---:|---:|---:|---|
| 2026-01 | 170 | 167 | +3 | 0001系 4件のうち airmate 集計外 |
| 2026-02 | 162 | 163 | -1 | 赤伝1件の扱い違い（許容範囲） |
| 2026-03 | 188 | 170 | +18 | 0001系 22件のうち 18件が airmate 集計外（CLP/試飲一括入力） |
| 2026-04 | 165 | 165 | 0 | 完全一致 ✅ |

詳細は `Flow/.../BFA/_org_count_analysis/march_diff_analysis.md` を参照。

## §B-7 インバウンド比率（BFA-1）の運用ルール

BFA の「客層分析①：インバウンド比率」は、エアレジ会計明細の **メモ欄から「海外」キーワードを部分一致で抽出** することでインバウンド客を識別する（§B-3 BFA-1 参照）。

### B-7.1 運用開始月

- 「海外」キーワードをメモ欄に必ず記入する運用ルールは、**2026年4月以降** で徹底開始
- それ以前（2026年3月以前）はメモ欄記入の徹底がなく、抽出データが不正確（ほぼ全てが「該当0件」として記録されている）

### B-7.2 過去月のインバウンドスライド方針

- **2026年3月以前のレポートでは BFA-1 スライドを非表示** とする
  - データ自体が不正確（=ほぼゼロ）であり、表示するとミスリードになる
  - 表示しないことで「データが取れていない月／週」と明示する
- **2026年4月以降のレポートでは BFA-1 スライドを表示** する（通常通り）

### B-7.3 自動判定ロジック（推奨）

`build_monthly_html.py`（および将来の週報 build スクリプト）で以下のいずれかを満たす場合、BFA-1 スライドを自動で出力しない:

- 対象期間の終了日が 2026-04 より前（YYYY-MM <= "2026-03"）
- または、インバウンド該当件数が極端に少ない（運用未開始の状態）と判断された期間

### B-7.4 過去分の遡及修正（2026-05-18 実施済）

- 2026-01 / 2026-02 / 2026-03 の月報HTMLから BFA-1（旧 P1-1）スライドを削除し、prod DB を再アップロード済
- スライド枚数: 26枚 → 25枚（各月）
- 旧版HTMLは各月の `_db_backup/bfa-001_YYYYMM_prod_blob_*.html` と `_html_backup/*.bak_20260518_inbound` に退避
