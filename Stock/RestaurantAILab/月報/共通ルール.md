## 2. 集計軸の使い分け（最重要）

**すべての集計（月次・週次・日次・曜日別・時間帯別）を「業務日基準（朝6時境界）」で統一する。**
エアレジ売上分析画面も業務日基準で表示しているため、これと一致させるための統一仕様。

| 集計対象 | 使用する集計軸 | 理由 |
|---|---|---|
| **月次総売上／月次客数／月次客単価** | **業務日基準**（朝6時境界） | エアレジ売上分析画面と一致させるため。深夜営業（〜翌5:59）を当日扱い |
| **週次／日次／曜日別／時間帯別** | **業務日基準**（朝6時境界 ／ 0:00〜5:59を前日扱い） | 深夜帯営業（〜翌2時）を1営業日として扱うため |

### 2.1 業務日基準の擬似コード

```python
def to_business_date(checkout_dt_jst):
    """6AM境界：0:00〜5:59 を前日の営業日として扱う"""
    if checkout_dt_jst.hour < 6:
        return (checkout_dt_jst - timedelta(days=1)).date()
    return checkout_dt_jst.date()
```

### 2.2 月帰属（業務日基準）

```python
def to_business_month(checkout_dt_jst):
    """業務日が属するカレンダー月を返す。
    例：5/1 0:21 JST の取引 → business_date = 2026-04-30 → business_month = '2026-04'
    """
    return to_business_date(checkout_dt_jst).strftime('%Y-%m')
```

### 2.3 SQL での業務日基準月次フィルタ（参考）

```sql
-- 2026-04 の業務月（4/1 06:00 JST 〜 5/1 06:00 JST）
WHERE entry_at >= '2026-04-01 06:00:00+09'::timestamptz
  AND entry_at <  '2026-05-01 06:00:00+09'::timestamptz
```

### 2.4 暦日基準の使用禁止（重要）

**月次集計を「カレンダー日 0:00〜23:59」「売上日時ベース」で行うのは禁止。**
過去（〜2026-05-18）にそのように記述していたドキュメント／スクリプト／HTMLは順次業務日基準に置換した（背景: 4月分で 5/1 0:00〜0:30 JST の dawn entries ¥54,300 を取りこぼした事案）。

## 6. 3点セット原則

**全ての切り口分析に「入客数 / 売上 / 構成比」の3点セットを必ず付ける。**

- 「入客数」=「そのメニュー／カテゴリを含む会計の人数合算」（同一会計に同一メニュー複数明細でも人数は1回のみ）
- 「売上」=該当範囲の subtotal 合算
- 「構成比」=該当売上 ÷ 月次総売上（％・小数1桁）

---

## 7. ランキング（P1-7／P1-8）

- フード／ドリンク 各 **TOP15** （TOP20はスライドに収まらないため15に制約）
- 列構成: 出数 / 入客数 / 売上 / 平均販売価格 / 構成比
- 平均販売価格 = 売上 ÷ 出数
- **★マーク** = 平均販売価格と登録価格の乖離が **±5%以上** のメニュー（ダブル販売・特別価格等の存在を示唆 — P0-4 ゆずマティーニ問題の恒久対応）

## 8. グラフ・数字表現の必須ルール

### 8.1 凡例（Legend）— 必須

- 全グラフに凡例必須。系列が1本のグラフでも、右上に「単位: 円 / 出典: エアレジ会計明細」の補助凡例
- 線種違い（実線/破線）の系列は、凡例に `___` `- - -` も反映

### 8.2 キャプション

- グラフ直下に「このグラフが何を語っているか」を1行で記載

### 8.3 メッセージ帯 — 必須

- タイトル直下にオレンジ縦線×薄黄背景の「メッセージ帯」を1本必ず配置
- 文章ルール：**動詞のある完結した1〜2文**。体言止め禁止
- 冒頭にアイコン1つ（✅強み／⚠️課題／💡インサイト）

### 8.4 分類条件帯 — 必須（v2.0 厳格化）

- タイトル直下のメッセージ帯のすぐ下に、緑系の薄色帯で配置
- **必ず箇条書き**（`<ul><li>...</li></ul>`）。スラッシュ区切りの一文化は禁止
- **書く内容**:
  - 集計対象期間
  - そのページで使う抽出キー・データの絞り込み条件
  - 営業日基準の話は、**実際に営業日基準を使うページ（週別概況・曜日別日平均・時間帯別月間売上）のみ**に記載
  - ボトル除外の話は、実際にボトル除外しているページ（週次・日次・曜日・時間帯）のみに記載
- **書かないこと**:
  - 表示されている数値の構造をオウム返しに説明する自明な定義（「売上総額=月次合計」「3点セット…」など）
  - 他ページの集計ロジック（KPIページに週次・時間帯の話を書かない、等）
  - 引用元未確定の仮置き（「対予算: 引用元確認中」等は削除。確定後に追加）

### 8.5 算出根拠フッター

- 最小限の2項目で固定：
  ```
  出典: エアレジ会計明細 ／ 集計対象: YYYY年MM月の業務日（朝6時境界、深夜営業を当日扱い）
  ```
- 「ボトル除く」等の詳細補足は分類条件帯に集約。フッターには書かない

### 8.6 月売上の集計範囲注釈

月売上額を表示するスライド（月間サマリーKPI、売上推移、客数推移、客単価推移、前年同月比、客層分析①②③、商品構造①②③、カテゴリ別、強み、確認事項）の分類条件帯に、必ず以下相当の1行を入れる：

> 集計対象: YYYY年MM月の業務日（MM/1 6:00 〜 翌月1日 5:59 JST、朝6時境界 ／ エアレジ売上分析画面と一致）

## 11. 日本語表記の原則

- 一般化したもの（KPI、SNS等）は許容

## 18. PL分析の取り扱い（v2.1 新設）

### 18.1 データソース

| 月 | 出典 |
|---|---|
| 当月（対象月） | **ダッシュボードDB**（`pl_daily_sales` / `pl_daily_expenses` / `pl_monthly_targets` から JSON ダンプ） |
| 過去月（前月以前・2025年含む） | **【BFA】本部サポート / PLシート**（gws CLI 経由 Google Drive） |
| 2026-01 / 2026-02 | **【BFA】本部サポート / 新PL管理（移管前）シート**（旧シートが空のため移行期データを参照。2026-05-18 反映） |
| 2026-03 | **【BFA】運営管理 / 新PL管理シート**（家賃補完・初期費用除外。2026-05-19 反映） |

どちらの出典かをスライド分類条件帯に必ず明示すること。

#### 18.1.1 家賃の補完ルール（2026-03 以降の運用）

- 【BFA】運営管理 / 新PL管理シート上で家賃が ¥0 で転記されている場合、シート転記漏れとみなし、**直近月（前月）と同額の固定費として補完表示**する。
- 補完した旨は PL関連スライドの分類条件帯（青帯）に必ず明示する（例: 「家賃 ¥1,112,859 は新PL管理シート上では¥0だが、運用上の固定費として補完表示」）。
- `sheet_pl_parsed.json` の `notes_per_month.<YYYY-MM>` に `rent_supplemented: true` / `rent_value` / `rent_reason` を記録する。

#### 18.1.2 初期費用の取り扱い（一過性費用の除外）

- 撮影費／試作費／開業準備費等、**一過性で発生した費用は PL の月次総費用から除外**する（運営定常コストではないため）。
- 除外した金額・項目はスライド注釈に明示し、`sheet_pl_parsed.json` の `notes_per_month.<YYYY-MM>` に `initial_cost_excluded: true` / `initial_cost_value` / `initial_cost_reason` を記録する。
- 2026-03 実例: 初期費用 ¥950,000（撮影/試作費等）を PL から除外。

### 18.2 必須KPI（KPIスライド2段目に配置）

| KPI | 計算式 |
|---|---|
| 総費用 | 当月9費目合計 |
| 営業利益 | 売上 − 総費用 |
| 営業利益率 | 営業利益 ÷ 売上 |
| FL率 | **（食材費 + ドリンク仕入れ）÷ 月次総売上（ボトル含む）** |
| 人件費率 | 人件費 ÷ 月次総売上（ボトル含む） |

### 18.3 PL用計算関数（`monthly_report_pipeline.py` 実装済）

- `calc_fl_rate(food_cost, drink_cost, total_sales)` — FL率（％）
- `calc_labor_rate(labor_cost, total_sales)` — 人件費率（％）
- `calc_op_profit(total_sales, total_expense)` — 営業利益
- `calc_op_profit_rate(total_sales, total_expense)` — 営業利益率（％）

これらの関数は単独でテスト可能。新規KPI追加時も関数化を継続する。

### 18.6 表記ルール

- 業界基準値は「**目安**XX%」と表記（「業界平均」「業界水準」「平均値」等は使わない — 出典が曖昧なため）
- 用途の解釈・推測はしない（事実数値のみ）

## 19.5 時刻列の取り扱い（重要・全集計スクリプト共通）

`rawdata.csv` の `entry_at` / `exit_at` / `ordered_at` 列は、Dashboard API が `formatUTCtoJST` で生成した **JSTナイーブ文字列**（タイムゾーンサフィックスなし、例: `"2026-06-01 19:30:00"`）。

Python 集計側では:
- ✅ 正: `pd.to_datetime(s).dt.tz_localize('Asia/Tokyo')`
- ❌ 誤: `pd.to_datetime(s, utc=True).dt.tz_convert('Asia/Tokyo')` — UTC誤認識で+9hシフトする

`tz_convert` を使うと `business_date` が翌日にズレ、月次・週次・曜日別・時間帯別の全KPIが破綻する。新規スクリプトを書く際も同じパターンを徹底すること。

参照: `Stock/RestaurantAILab/週報/_investigations/2026-06-10_timezone_bug.md`
