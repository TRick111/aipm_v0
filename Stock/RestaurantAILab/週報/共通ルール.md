## 2. 集計軸の使い分け（最重要）

**すべての集計（週次・日次・曜日別・時間帯別）を「業務日基準（朝6時境界）」で統一する。**
深夜営業（〜翌5:59）を当日扱いとする。

| 集計対象 | 使用する集計軸 | 理由 |
|---|---|---|
| **週次総売上／週次客数／週次客単価** | **業務日基準**（朝6時境界） | 深夜営業（〜翌2時）を1営業日として扱うため。POS分析画面と一致させる |
| **日次／曜日別／時間帯別** | **業務日基準**（朝6時境界 ／ 0:00〜5:59を前日扱い） | 同上 |

### 2.1 業務日基準の擬似コード

```python
def to_business_date(checkout_dt_jst):
    """6AM境界：0:00〜5:59 を前日の営業日として扱う"""
    if checkout_dt_jst.hour < 6:
        return (checkout_dt_jst - timedelta(days=1)).date()
    return checkout_dt_jst.date()
```

### 2.2 週帰属（業務日基準＋ISO週）

```python
def to_business_week(checkout_dt_jst):
    """業務日が属するISO週（月曜始まり）を返す。
    例：5/4 0:21 JST の取引 → business_date = 2026-05-03 → 2026-W18
    """
    return to_business_date(checkout_dt_jst).strftime('%G-W%V')
```

### 2.3 SQL での業務日基準週次フィルタ（参考）

```sql
-- 2026-W06（2026-02-02 〜 2026-02-08）
WHERE entry_at >= '2026-02-02 06:00:00+09'::timestamptz
  AND entry_at <  '2026-02-09 06:00:00+09'::timestamptz
```

### 2.4 暦日基準の使用禁止（重要）

**週次集計を「カレンダー日 0:00〜23:59」「売上日時ベース」で行うのは禁止。**
過去にそのように記述していたドキュメント／スクリプト／HTMLは順次業務日基準に置換した（背景: 月報 4月分で 5/1 0:00〜0:30 JST の dawn entries ¥54,300 を取りこぼした事案）。

週報の `analysis_script.py` / `deep_dive_weekday_analysis.py` は既に business_date 基準で実装済（2025-12-31 反映）。

## 3. 比較対象の使い分け

- **前週比**: 直近の変化、短期的な異常値の検出
- **過去4週平均比**: 曜日別比較・基本数字比較で使用（前週との単週比較ではなく、4週移動平均で平滑化）
- **前月平均比**: 対象週開始日から1ヶ月前の月に該当する週の平均値と比較
- **前年同週比**: 第1部（全体トレンド）で使用

### 3.1 KPIカードの比較値（2系統表示・マスター判断 K-6 反映）

KPIカード（売上 / 客数 / 客単価 / 組数）の比較値は **2系統併記** とする：

- **前週比**（直近の変化）
- **過去4週平均比**（平滑化トレンド）

**前年同週比は KPIカードに表示せず**、第1部のトレンド推移スライド（売上トレンド・客数トレンド・客単価トレンド・前年同週比サマリー）でのみ表示する。
理由: KPIカードに3系統を併記すると情報量過多で読み取りコストが上がる。前年比は推移グラフ＋専用ページで扱うのが適切。

「達成」「未達」「達成率」は **明確な目標値（予算）に対してのみ** 使用（§9 参照）。前週比・前年比・4週平均比に「達成率」は使わない。

---

## 4. 3点セット原則

**全ての切り口分析に「入客数 / 売上 / 構成比」の3点セットを必ず付ける。**

- 「入客数」=「そのメニュー／カテゴリを含む会計の人数合算」（同一会計に同一メニュー複数明細でも人数は1回のみ）
- 「売上」= 該当範囲の subtotal 合算
- 「構成比」= 該当売上 ÷ 週次総売上（％・小数1桁）

## 5. ランキング

- フード／ドリンク 各 **TOP15**（スライドに収まる上限。TOP10 表記は週報03の旧仕様 — 段階的にTOP15へ移行）
- 列構成: 出数 / 入客数 / 売上 / 平均販売価格 / 構成比
- 平均販売価格 = 売上 ÷ 出数
- **★マーク** = 平均販売価格と登録価格の乖離が **±5%以上** のメニュー（ダブル販売・特別価格等の存在を示唆）

> BFA のカテゴリ分類は `Stock/RestaurantAILab/月報/カテゴリ分類マスタ.md` を参照（`food` / `drink` / `course` / `bottle` / `event` / `other`）。

## 6. グラフ・数字表現の必須ルール

### 6.1 凡例（Legend）— 必須

- 全グラフに凡例必須。系列が1本のグラフでも、右上に「単位: 円 / 出典: 〇〇」の補助凡例
- 線種違い（実線/破線）の系列は、凡例に `___` `- - -` も反映

### 6.2 キャプション

- グラフ直下に「このグラフが何を語っているか」を1行で記載

### 6.3 主張バー（メッセージ帯）— 全スライド必須

- タイトル直下にオレンジ縦線×薄黄背景の「主張バー」を1本必ず配置（タイトルスライド以外）
- 文章ルール：**動詞のある完結した1〜2文**。体言止め禁止
- 冒頭にアイコン1つ（✅強み／⚠️課題／💡インサイト）
- フォントサイズ **28px以上**、太字、ヘッダー直下に配置（slide-body の外）
- スライド下部に別途 `message-box` 等を追加してはならない（主張バーが唯一のメッセージ表示エリア）

```css
.msg-bar {
  background: #FFF7ED;
  border-left: 5px solid #F97316;
  padding: 12px 40px;
  font-size: 28px;
  font-weight: 800;
  color: #1E293B;
  line-height: 1.4;
}
```

### 6.4 分類条件帯 — 必須

- タイトル直下の主張バーのすぐ下に、緑系の薄色帯で配置
- **必ず箇条書き**（`<ul><li>...</li></ul>`）。スラッシュ区切りの一文化は禁止
- **書く内容**:
  - 集計対象期間（週次の業務日範囲）
  - そのページで使う抽出キー・データの絞り込み条件
  - 営業日基準は、**実際に営業日基準が結果を変えるページ（曜日別・時間帯別）** に明記
- **書かないこと**:
  - 表示されている数値の構造をオウム返しに説明する自明な定義（「売上総額=週次合計」など）
  - 他ページの集計ロジック（KPIページに時間帯の話を書かない、等）
  - 引用元未確定の仮置き（「対予算: 引用元確認中」等は削除。確定後に追加）

### 6.5 算出根拠フッター

最小限の2項目で固定：

```
出典: {店舗POS名} ／ 集計対象: YYYY-Www の業務日（YYYY/MM/DD 6:00 〜 翌週月曜 5:59 JST、朝6時境界）
```

「ボトル除く」等の詳細補足は分類条件帯に集約。フッターには書かない。

### 6.6 週売上の集計範囲注釈

KPI／売上推移／客数推移／客単価推移／前年同週比／曜日別概況／時間帯別／カテゴリ別／商品ランキング／強み・課題スライドの分類条件帯に、必ず以下相当の1行を入れる：

> 集計対象: YYYY-Www の業務日（{開始日} 6:00 〜 {終了日+1} 5:59 JST、朝6時境界 ／ POS売上分析画面と一致）

## 9. 「達成・未達成」の使い分け

- 「達成」「未達」「達成率」は **目標値（予算）に対してのみ** 使用
- 前週比・前年同週比・4週平均比・前月平均比に「達成率」は使わない
- 表現例:
  - NG: 「前年同週比の達成率 78%」
  - OK: 「前年比 -22%」
  - NG: 「4週平均に対する達成率 130%」
  - OK: 「4週平均比 +30%」「4週平均の1.3倍」

## 10. 日本語表記の原則

- カタカナ英語・英語表記は極力避ける
  - NG「サイズ」→ OK「規模／人数」
  - NG「セールス」→ OK「売上」
  - NG「トップ15」→ OK「上位15商品」
  - NG「ケース」→ OK「件／事例」
  - NG「カウント」→ OK「件数／数」
  - NG「Win / Loss」→ OK「強み / 課題」
- 固有名詞（商品名・店舗名・サービス名）は英語のまま
- 一般化したもの（KPI、SNS等）は許容

## 12. 絵文字使用ガイドライン

控えめに。1スライドに3個まで。

| 配置 | 個数 | 用途 |
|---|---|---|
| スライドタイトル先頭 | 1個（必要時のみ） | 第◯部 タグの近辺 等 |
| KPIカードラベル先頭 | 1個 | 💰売上、👥客数、💴客単価、🧾組数、📅営業日数、📈日平均 |
| 主張バー冒頭 | 1個 | ✅強み／⚠️課題／💡インサイト |

**禁止**: ランキング表の順位への絵文字／1スライド内に同じ絵文字を3回以上／装飾目的のスパム。

**カード・箇条書きには内容に合った絵文字を行頭に添える**（success-item / challenge-item カード、bullet-list の .icon 要素 等）。

- 成功・好調: 🎯 🎉 📈 🌱 ★ 🏆
- 課題・不調: ⚠️ 📉 💰 🚨 🔴
- カテゴリ表現: 🍸（カクテル）🍽️（フード）📋（管理）⏰（時間）🏢（法人）📅（スケジュール）🥂（イベント）🎨（ブランド）🥤（ドリンク）
- 比較・分析: 📊 📈 📉 🟢 🔴

## 13. 比率ページの円グラフ必須化

比率を扱うスライド（カテゴリ別売上構成比 等）には **D3.js ドーナツチャートを併設**：

仕様:
- サイズ 180〜200px
- 中央に該当カテゴリの%を大きく表示（24px以上）
- 上位5カテゴリ＋「その他」の6セグメント以内に統合
- 最大セグメントを中心から18px外側に引き出す（explode）＋リーダーライン

## 14. データソース原則（BFA以外も含む）

- 当週の組数・売上・客数は、**ダッシュボードDB（prod）の生データに準拠する**
- rawdata.csv はスナップショット時点依存のため、最新分の追記欠落リスクがある。
  - **当週分は incremental-export で直前まで更新してから集計を開始する**
  - rawdata.csv は前週以前の比較データ用として活用
- POSとダッシュボードDBで組数・売上に差異が出た場合の正本はダッシュボードDB（理由: 入店時刻ベースで業務日基準を適用しているため）
