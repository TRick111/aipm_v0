# マンション内LAN

## 🎯 次のアクション
**Cloudflare WARP は DoH（1.1.1.1）モードのみで運用継続中**。フルトンネル（warp+doh）は MASQUE / WireGuard 双方で `CF_DNS_LOOKUP_FAILURE` が発生したため一旦保留。DoH のみでも curl 計測上の `CONN ぴったり 1 秒（=TCP SYN 再送）`現象が消失したため、まずはこの状態で 1〜2 日体感を評価。

ぶり返した場合: プラン A（WARP の utun インターフェースだけ MTU 1024 を手動指定 → 他ソフト無影響）に進む。

## 🚧 現在のブロッカー
なし（フルトンネル不可は仮置き対処で迂回中）

## 📋 概要
集合住宅で発生する自宅 Wi-Fi の「初回アクセス遅延」問題の真因解明と恒久対策プロジェクト。MTU 1024 で改善することは 2026-04-22 に確認済だが、他ソフトへの影響があるため MTU を変えずに直す方法を探る。

2026-05-04 の再診断で、(1) ダブル NAT 構成、(2) ICMP の重度レート制限／フィルタ、(3) TCP SYN がランダムにロスして 1 秒の再送ペナルティが発生、という新しい所見を得た。「初回が遅く 2 回目以降が速い」体感は HTTP keep-alive・DNS / TLS セッションキャッシュで新規接続フェーズが省略されるため。

同日 Cloudflare WARP を導入。フルトンネルは確立できなかったが、**DoH モード（DNS のみ）だけで TCP SYN 再送現象が消失**するという想定外の効果を得た。マンション上位の DNS 経路に介入／レート制限があり、DoH（TLS）化で覆い隠せたと推察。

関係者: 田中利空（オーナー / 実行者）、マンション管理会社（上位 NAT 機器の運用主体）。

## 🧠 決定事項（Why ログ）
- MTU 1024 への恒久変更は採用しない: Tailscale 等の他ソフトへの影響と、根本原因解決にならないため
- IPv6 は「使用しない」のまま維持: 中途半端な IPv6 開通は Happy Eyeballs の挙動でむしろ症状を悪化させる
- 個別回線契約は当面検討対象外: 賃貸かつ管理会社経由のため初手にしない
- WARP フルトンネル化を一旦保留: DoH のみで体感改善が出たため、追加リスク（接続不能化）を取らず現状で運用観察
- WARP のトンネルプロトコルは WireGuard 設定のままで保持: DoH モード時はトンネル不使用のため実質影響なし。フルトンネル再挑戦時の選択肢を残す
