# マンション内LAN 診断レポート

**日時**: 2026-05-04
**前回診断**: 2026-04-22（`Stock/作業効率化/環境整備/ネットワーク最適化診断_2026-04-22.md`）

---

## 1. 症状（ユーザーヒアリング）

- 集合住宅に引っ越してから自宅 Wi-Fi が遅くなった
- **「あるページにその日最初にアクセスする時は遅く、2 回目以降は軽くなる」**（再現性あり）
- パソコン・スマートフォン双方で発生
- スマホはセルラー通信なら速い → 端末側ではなく自宅回線側が原因
- 過去に **MTU を 1024 に下げたら改善** したが、他ソフトへの影響があるため恒常的には適用したくない
- 時間帯によらず発生（→ 単純な PPPoE 夜間混雑型ではない）
- fast.com 実測 86 Mbps（帯域は出ている）

---

## 2. 構成の確定: ダブル NAT

ルーター管理画面 (`DDNS` 画面) の `Internet 側 IP アドレス` が `192.168.6.114` であることから**プライベート IP の払い出し = ダブル NAT 構成**が確定。

```
[インターネット]
       │
  グローバル網（オプテージ系: 219.59.90.x）
       │
[マンション大元ルーター] 192.168.6.241  ← ★ ここが NAT
       │ プライベートIPを各戸に配布 (192.168.6.x)
       │
[Buffalo WSR-A253DHP3]
   WAN: 192.168.6.114 (DHCPで取得・プライベート)
   LAN: 192.168.11.0/24
   DNS: 1.1.1.1 / 1.0.0.1 (手動)
   IPv6: 「使用しない」
   MTU: 1500
       │
[Mac / iPhone / 他]
```

集合住宅でよくある「マンション一括契約・光集合住宅タイプ」の典型構成。

### 3.3 ICMP（ping）

```
通常ping 8.8.8.8     : 3/3 ロス
通常ping 1.1.1.1     : 2/3 ロス（1 回だけ 5.054ms で応答）
DF付き 1472B → 1.1.1.1: 3/3 ロス
```

→ **ICMP に重度のレート制限 / フィルタがかかっている**。サイズに依らず落ちるため MTU 探索の指標としては使えないが、この事実自体が PMTU Discovery の機能不全を意味する（ICMP "Fragmentation Needed" が返らない）。

### 3.4 HTTP（curl で内訳計測）

| サイト | DNS | CONN | TLS | TTFB | TOTAL | 解釈 |
|--------|-----|------|-----|------|-------|------|
| nikkei  1回目 | 0.014 | 0.020 | 0.029 | 0.068 | 0.129 | 正常 |
| nikkei  2回目 | 0.003 | 0.008 | 0.018 | 0.028 | 0.089 | DNS / TCP キャッシュ効果 |
| asahi   1回目 | 0.021 | 0.026 | 0.047 | 0.305 | 0.342 | 正常（TTFBはサーバ側） |
| **asahi 2回目** | 0.002 | **1.008** | 1.031 | 1.213 | **1.250** | **★ TCP SYN 再送** |
| **yahoo 1回目** | 0.012 | **1.019** | 1.032 | 1.065 | **1.078** | **★ TCP SYN 再送** |
| **yahoo 2回目** | 0.004 | **1.010** | 1.022 | 1.047 | **1.056** | **★ TCP SYN 再送** |

「**CONN がぴったり 1.000 秒**」は OS のデフォルト initial RTO ＝ 1 秒の証拠。SYN パケットがロス → 1 秒待って再送 → 接続成立、という挙動。**ランダムに発生**している点が重要（同じサイトでも 1 回目は速く 2 回目で詰まる、など）。

## 4. 真因の仮説（更新版）

### 主犯: ダブル NAT + 上位機器のセッション／レート制限

- マンション大元ルーターが多数戸の NAT を捌くため、**新規 TCP セッション確立時に確率的にパケロス**が発生
- 既存セッション（HTTP keep-alive）は通るため、**ブラウザでの 2 回目アクセスは速く感じる**
- 1 回目は新規 DNS / TCP / TLS を全部やる必要があり、SYN ロスが 1 件でも発生すると体感が大きく劣化
- 2 回目は DNS キャッシュ + TCP keep-alive + TLS セッション再開で**新規接続フェーズ自体がスキップされる**

### 共犯: ICMP フィルタによる PMTU 黒穴

- ICMP がほぼ通らないため Path MTU Discovery が機能しない
- 経路上のどこかで MTU が 1500 未満（過去診断より、**マンション GW 192.168.6.241 が MTU 1024**）
- DF 付き大きいパケットが「黙って落ちる」 → TCP の中盤でストール
- 過去に MTU 1024 で改善したのはこの黒穴を回避していたため

### 無実

- DNS（5〜8ms）
- 帯域（86 Mbps 出ている）
- IPv6（そもそも OFF。Happy Eyeballs は無関係）
- 時間帯混雑（終日発生）

## 5. 「初回遅い・2 回目速い」の正体

| フェーズ | 1 回目 | 2 回目以降 |
|----------|--------|------------|
| DNS 解決 | 上位リゾルバへ問合せ（数 ms〜） | OS / ブラウザキャッシュ |
| TCP 三方ハンドシェイク | **新規 SYN：ここで確率的に詰まる** | **HTTP keep-alive で再利用** |
| TLS ハンドシェイク | フルハンドシェイク（〜100ms） | TLS セッション再開（数 ms） |
| HTTP リクエスト | 上記の合計が体感に直撃 | ほぼ TTFB のみ |

→ 「初回だけ遅い」は **新規 TCP セッション確立だけが詰まる** 構造のサインそのもの。

## 6. 対策（影響度の小さい順）

### ★ 最優先候補: Cloudflare WARP（無料）

- 全トラフィックを QUIC（UDP）で Cloudflare に集約
- TCP SYN 詰まりを **TCP を使わずに回避**
- MSS 自動調整で MTU 黒穴も回避
- **MTU を 1500 のまま改善する可能性が高い**
- Mac / iPhone とも公式アプリで 5 分導入、合わなければ即アンインストール

### 候補 2: ブラウザの HTTP/3 (QUIC) 優先利用

- Chrome / Edge / Safari (Tahoe 以降) は標準で HTTP/3 対応
- HTTP/3 は UDP ベースなので TCP SYN ロスを回避できる
- ただし HTTP/3 非対応サイト（多い）には効かない → WARP の補助として位置づけ

### 候補 3: 設定の現状維持を確認

- IPv6 は「使用しない」のまま（中途半端な開通は症状を悪化させる）
- DNS は 1.1.1.1 / 1.0.0.1 のまま（実測で速い）
- MTU は 1500 のまま（変えると他ソフトに影響）

### 候補 4（根治）: マンション管理会社へ問い合わせ

- ダブル NAT を解消できないか（戸別ブリッジ or 戸別 PPPoE 払い出し）
- 上位ルーターのセッション数制限・ICMP フィルタの緩和
- 上位ルーターの再起動（NAT テーブルクリア）

### 候補 5（最終手段）: 個別回線契約

- 光コンセントが部屋にあれば、NTT 東西フレッツ + IPoE 業者（v6 プラス等）で根治
- 賃貸かつ管理会社経由のため当面は対象外

---

## 7. 採用しない案

- **MTU を 1024 に下げる**: 過去診断で効果は確認済だが、Tailscale 他ソフトへの影響があるため恒久採用しない（黒穴を踏まないだけで真因解消にはならない）
- **IPv6 を有効化**: マンション側が IPv6 ネイティブ提供しているか不明。半開通だと Happy Eyeballs で逆に遅くなる

## 9. 同日（2026-05-04）WARP 検証結果

### 経緯
- Mac に Cloudflare WARP を `brew install --cask cloudflare-warp` で導入
- フルトンネル化（`warp+doh`）を MASQUE / WireGuard の両プロトコルで試行 → **両方とも `CF_DNS_LOOKUP_FAILURE` で失敗**
  - WireGuard は `162.159.192.3:2408` 宛で 13 秒「connectivity checks」を繰り返したのち Unable
  - 但し curl 単発では `warp=on` で着弾できる「片肺」状態を観測
- DoH のみモード（`Mode: DnsOverHttps`、トンネル無効）に切り戻し

### 観測結果（curl 再ベンチ）

| サイト | 計測タイミング | Before（WARP 無し）CONN | After（WARP DoH）CONN | 評価 |
|---|---|---|---|---|
| yahoo.co.jp 1回目 | 〃 | **1.019s** | 0.031s | SYN 再送消失 ✅ |
| yahoo.co.jp 2回目 | 〃 | **1.010s** | 0.010s | 〃 ✅ |
| asahi.com 2回目 | 〃 | **1.008s** | 0.011s | 〃 ✅ |
| nikkei.com 1回目 | 〃 | 0.020s | 0.019s | 元から正常 |

### 解釈
- **DoH（DNS over HTTPS）化だけで TCP SYN 再送が消失した**ことは想定外の強い効果
- 仮説: マンション上位ネットワークに **DNS 経路への介入／レート制限**が存在し、DNS パケットそのものというより「DNS をトリガにした上位機器の挙動」が SYN ロスを誘発していた可能性
- DoH は TLS 上で DNS を流すため、上位機器からは普通の HTTPS と見分けがつかず、介入を回避できる

### フルトンネル失敗の追加考察
- WARP のトンネル MTU 既定 1280 が、マンション GW の MTU 1024 制限と PMTU Discovery 不能（ICMP フィルタ）の合わせ技で「黒穴」に落ちている可能性が高い
- WARP のトンネル確立用ハンドシェイク（小さい）は通るが、DoH ブートストラップ（TLS ハンドシェイクで大きい応答）が落ちる、という挙動と整合的
- 対処: `sudo ifconfig <utun_X> mtu 1024` で WARP の仮想 IF だけ MTU を縮める（プラン A）。ただし DoH のみで体感改善しているため、現時点では未実施

### 当面の運用方針
- WARP は **DoH モードで常駐** し続ける
- 1〜2 日体感を確認、ぶり返したらプラン A（utun MTU 1024）に進む
- 完全に元へ戻したい場合は `warp-cli disconnect`（緊急時の切戻しは `warp-cli mode doh ; warp-cli tunnel protocol reset`）
