# 定期実行（Cronタスク管理）

## 背景
- 1日のレポート作成など、毎日/毎週/毎月の定型処理を手作業で行うと漏れや遅延が発生しやすい
- AI（主にCursor CLI）を活用して、定期タスクを自動実行・監視できる運用基盤を整える必要がある

## 目的
- Cronを使った定期実行タスクの作成・管理を、AIOSプロジェクトとして一元管理する
- タスク定義、実行ログ、失敗時リカバリ手順を標準化し、運用コストを下げる

## ゴール（完了条件）
- Cronで動く定期実行ジョブの雛形（実行スクリプト + スケジュール + ログ）が運用可能
- 1日レポート作成などの実ジョブを再利用可能な形で追加できる
- 失敗検知と通知（例: LINE/メール/ログ監視）の基本導線が整っている

## 定期実行のルール
- まず手動実行で成功させてから、Cronなどの定期実行に登録する
- 1ジョブ1責務で小さく作る
- 実行ログを必ず残す
- 入力・出力・失敗時の対応をジョブ定義に明記する
- APIキーやトークンなどの秘密情報はコミットしない
- PCがスリープ/電源OFF中の場合、Cronは実行されない可能性がある前提で運用する

## 【最重要】2段階運用フロー（ジョブ案作成 → Cron登録）

> このプロジェクトでは、新規ジョブの追加を**必ず2段階に分ける**。
> 担当AIエージェントは、ユーザー依頼を受けた際にこのフローに従うこと。

### Step 1：ジョブ案を作成する（Cron登録はしない）

ユーザーから「○○するジョブを作って」「定期実行のジョブを追加して」と依頼された場合、AIは以下のみを行う：

1. `ジョブ一覧/` にジョブ説明Markdownを作成（テンプレートをコピーして埋める）
2. `scripts/` に実行スクリプトを作成
3. `ProjectIndex.yaml` と `log.md` を更新
4. **手動で1回実行**し、出力とログを確認
5. 手動実行の結果（成功 / 出力ファイル / ログ）をユーザーに報告
6. **この時点ではCronに登録しない**

### Step 2：Cronに登録する（明示依頼時のみ）

ユーザーから「このジョブをCronに登録して」「毎朝○時に動くようにして」と**明示的に**依頼された場合のみ、AIは以下を行う：

1. `crontab` を編集してジョブを登録
2. `crontab -l` の結果をユーザーに見せて確認させる
3. 該当ジョブMarkdownに「Cron登録済み」と実行時刻を追記
4. `log.md` にCron登録を記録

### 例外と確認

- 例外：ユーザーが明示的に「Step 1とStep 2を一気にやってください」と指示した場合のみ、両方をまとめて実施してOK
- 依頼が曖昧な場合、AIは「Step 1（案作成）のみで進めますか？それとも Step 2（Cron登録）まで実施しますか？」と確認すること

## 定期実行ジョブを作る際のルール
- `scripts/` に実行スクリプトや実行ラッパーを置く
- `ジョブ一覧/` にジョブ説明書をMarkdownで置く
- `logs/` にCronや手動実行のログを保存する
- `テンプレート/` にジョブ定義テンプレートを置き、新規ジョブ作成時はコピーして使う
- 新しいジョブを追加したら `ProjectIndex.yaml` と `log.md` を更新する
- **新規ジョブの追加は上記の「2段階運用フロー」に従うこと**

## フォルダの役割
| フォルダ | 役割 |
|---|---|
| `scripts/` | 実際に動くスクリプトや実行ラッパー |
| `ジョブ一覧/` | ジョブごとの目的・入力・出力・実行タイミングを説明するMarkdown |
| `logs/` | Cronや手動実行のログ保存先 |
| `テンプレート/` | 新しいジョブ説明書を作るための雛形 |
